# Server Monitoring and Alerting System

## Capstone Project — Foundations Path

Congratulations on reaching the capstone project for the Foundations path. This is
where everything comes together. Over the previous modules you learned how to
navigate Linux, write shell scripts, manage code with Git, understand networking,
and automate tasks with Python. Now you will combine all of those skills into a
single, cohesive project: a **server monitoring and alerting system** that you
could genuinely use in a production environment.

---

## What You Will Build

You will build a monitoring system that collects health metrics, evaluates them
against configurable thresholds, sends alerts, and presents data in a dashboard.
The system has five major components:

1. **A Bash monitoring script** that collects raw system metrics (CPU, memory,
   disk, network) and outputs structured JSON data.
2. **A Python automation layer** that reads the metric data, manages alert logic,
   stores historical metrics, and generates reports.
3. **Network health checks** that verify HTTP endpoints, DNS resolution, port
   availability, and response times.
4. **A terminal-based dashboard** that displays current and historical metrics
   in a human-readable format.
5. **A Git repository** with clean history, proper branching, documentation,
   and conventional commit messages.

### Phase 2: System Monitoring Scripts

**Skills Used:** Shell Scripting, Linux

In this phase you write the core data-collection layer: a Bash script that
gathers system metrics and outputs structured JSON. This is the heartbeat of
your monitoring system.

#### Tasks

1. **Implement `check_cpu()`** — Collect CPU usage percentage. Use tools such
   as `top`, `mpstat`, or read from `/proc/stat`. Return a JSON object:
   ```json
   {
     "metric": "cpu",
     "value": 23.5,
     "unit": "percent",
     "timestamp": "2026-03-01T14:30:00Z"
   }
   ```

2. **Implement `check_memory()`** — Collect total, used, and available memory.
   Use `free` or read from `/proc/meminfo`. Return a JSON object:
   ```json
   {
     "metric": "memory",
     "total_mb": 8192,
     "used_mb": 5120,
     "available_mb": 3072,
     "percent": 62.5,
     "timestamp": "2026-03-01T14:30:00Z"
   }
   ```

3. **Implement `check_disk()`** — Collect disk usage for mounted filesystems.
   Use `df`. Return a JSON object per mount point.

4. **Implement `check_network()`** — Collect bytes sent/received and active
   connections. Use `/proc/net/dev` or `ss`. Return a JSON object.

5. **Combine all checks** into a single JSON document written to
   `metrics/metric-YYYY-MM-DD-HHMMSS.json`.

6. **Add error handling** so the script does not crash if a tool is missing.
   Log warnings to stderr and continue with the remaining checks.

7. **Set up a cron job** that runs the script every minute:
   ```
   * * * * * /workspace/monitor.sh
   ```

#### Acceptance Criteria

- Running `./monitor.sh` produces valid JSON to stdout (verify with `jq`).
- Each metric includes a UTC timestamp.
- The script handles missing commands gracefully (warning, not crash).
- Metric files accumulate in `metrics/` with unique filenames.
- ShellCheck reports no errors or warnings for the script.

### Phase 3: Python Automation

**Skills Used:** Python for DevOps

In this phase you build the Python layer that consumes the metric data, applies
alerting logic, stores history, and generates reports.

#### Tasks

1. **Implement the `MetricCollector` class:**
   - Load configuration from `config.yaml`.
   - Read the latest metric file from `metrics/`.
   - Optionally invoke `monitor.sh` directly via `subprocess`.
   - Parse JSON data into Python objects.

2. **Implement the `AlertManager` class:**
   - Compare metric values against thresholds from `config.yaml`.
   - Support multiple alert channels: `console` (print to terminal) and
     `webhook` (HTTP POST to a Slack-compatible URL).
   - Format alert messages with severity, metric name, current value,
     and threshold.
   - Example alert format:
     ```
     [CRITICAL] CPU usage at 92.3% (threshold: 80%)
     [WARNING]  Memory usage at 84.1% (threshold: 85%)
     ```
   - Implement alert cooldown so the same alert is not sent repeatedly
     within a configurable window (e.g., 5 minutes).

3. **Implement the `ReportGenerator` class:**
   - Read all metric files from `metrics/`.
   - Compute summary statistics: min, max, average for each metric.
   - Generate a terminal-formatted report.
   - Generate an HTML report saved to `reports/report-YYYY-MM-DD.html`.
   - Include a trend indicator: is each metric increasing, decreasing,
     or stable compared to the previous collection period?

4. **Implement the main loop:**
   ```python
   def main():
       config = load_config("config.yaml")
       collector = MetricCollector(config)
       alerter = AlertManager(config)

       metrics = collector.collect()
       alerts = alerter.evaluate(metrics)
       alerter.send(alerts)
   ```

5. **Add logging** using Python's `logging` module. Log to both console
   and a file in `logs/monitor.log`. Use appropriate log levels.

6. **Add error handling** with try/except blocks. The script should never
   crash unexpectedly. All exceptions should be logged.

#### Acceptance Criteria

- `python3 monitor.py` runs without errors.
- Alerts are printed to the console when thresholds are exceeded.
- Webhook alerts are sent as HTTP POST with a JSON payload.
- Reports are generated in both terminal and HTML format.
- All functions have docstrings.
- Logging captures key events (startup, metric collection, alerts, errors).

### Phase 4: Network Health Checks

**Skills Used:** Networking

In this phase you add network-level monitoring. Your system will check whether
HTTP endpoints are reachable, DNS names resolve correctly, specific ports are
open, and response times are within acceptable limits.

#### Tasks

1. **Implement HTTP endpoint monitoring:**
   - Read endpoint list from `config.yaml`.
   - Send an HTTP GET request to each endpoint.
   - Record: status code, response time, whether it matches the expected
     status (200 by default).
   - Handle timeouts and connection errors gracefully.
   ```python
   def check_endpoint(url, timeout=5):
       """Check if an HTTP endpoint is healthy."""
       # Returns: {"url": ..., "status": ..., "response_time_ms": ..., "healthy": True/False}
   ```

2. **Implement DNS resolution monitoring:**
   - Resolve a list of hostnames.
   - Record resolution time and resolved IP addresses.
   - Detect DNS failures and alert.
   ```python
   def check_dns(hostname):
       """Check if a hostname resolves correctly."""
       # Returns: {"hostname": ..., "resolved_ips": [...], "resolution_time_ms": ..., "success": True/False}
   ```

3. **Implement port availability checks:**
   - Attempt a TCP connection to specified host:port combinations.
   - Record whether the port is open and how long the connection took.
   ```python
   def check_port(host, port, timeout=3):
       """Check if a TCP port is open."""
       # Returns: {"host": ..., "port": ..., "open": True/False, "response_time_ms": ...}
   ```

4. **Implement response time tracking:**
   - Store response times over time.
   - Calculate average, p95, and p99 response times.
   - Alert when response times exceed a configurable threshold.

5. **Integrate network checks into the main monitoring loop.** Network check
   results should appear in the metric output and in the dashboard.

#### Acceptance Criteria

- HTTP checks correctly identify up/down endpoints.
- DNS resolution failures are detected and reported.
- Port checks work for both open and closed ports.
- Response times are tracked and stored historically.
- Network check results are included in generated reports.

### Phase 5: Dashboard and Reporting

**Skills Used:** Integration of all modules

In this phase you bring everything together into a terminal-based dashboard
and automated reporting system.

#### Tasks

1. **Build a terminal dashboard** that displays:
   - Current system metrics (CPU, memory, disk, network) with color coding:
     - Green: below 60% of threshold
     - Yellow: between 60% and 90% of threshold
     - Red: above 90% of threshold or exceeding threshold
   - Network health check status for each endpoint.
   - Recent alerts (last 10).
   - Uptime and time since last check.
   - Use the `rich` library for formatting, or implement plain-text
     formatting if you prefer.

2. **Implement auto-refresh** so the dashboard updates at a configurable
   interval (default: every 60 seconds).

3. **Generate daily summary reports** that include:
   - Metric averages, peaks, and lows for the day.
   - Number of alerts triggered and their severities.
   - Endpoint availability percentage.
   - Trend analysis: which metrics are trending upward?

4. **Generate weekly summary reports** that include:
   - Day-over-day comparison.
   - Worst-performing metrics.
   - Recommendations (e.g., "Disk usage has increased 5% this week;
     consider cleanup").

5. **Add command-line arguments** using `argparse`:
   ```
   python3 monitor.py                    # Run single collection + alert check
   python3 monitor.py --dashboard        # Launch live dashboard
   python3 monitor.py --report daily     # Generate daily report
   python3 monitor.py --report weekly    # Generate weekly report
   python3 monitor.py --check-network    # Run network checks only
   ```

#### Acceptance Criteria

- Dashboard renders in the terminal with color-coded metrics.
- Dashboard refreshes automatically.
- Daily and weekly reports are generated on demand.
- CLI arguments work as documented.
- All components integrate without errors.

## Hints and Tips

### Getting Started

Do not try to build everything at once. Follow the phases in order. Get each
phase working and committed before moving on.

### Bash Script Tips

- Use `date -u +"%Y-%m-%dT%H:%M:%SZ"` for UTC timestamps.
- Use `jq` for JSON formatting if available; fall back to manual formatting.
- Test with `./monitor.sh | jq .` to verify JSON validity.
- Use `set -euo pipefail` at the top of your script for safety.

### Python Tips

- Use `pathlib.Path` instead of string concatenation for file paths.
- Use `json.dumps(data, indent=2)` for readable JSON output.
- Use `datetime.utcnow().isoformat()` for timestamps.
- The `rich` library makes terminal dashboards easy with `Console`, `Table`,
  and `Panel`.

### Networking Tips

- Use `requests.get(url, timeout=5)` with a timeout to prevent hangs.
- Use `socket.getaddrinfo()` for DNS resolution.
- Use `socket.create_connection((host, port), timeout=3)` for port checks.

### Common Mistakes to Avoid

1. **Committing the virtual environment.** Add `venv/` to `.gitignore`.
2. **Hardcoding thresholds.** Always read from `config.yaml`.
3. **Not handling errors.** Wrap every external call in error handling.
4. **Giant commits.** Make small, focused commits per feature.
5. **No testing.** Write at least a JSON output test and a threshold test.
