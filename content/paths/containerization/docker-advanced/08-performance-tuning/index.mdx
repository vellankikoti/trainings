<Callout type="story" title="The Story">
Your e-commerce platform handles 500 requests per second on a bare-metal server. After containerizing the application with default settings, throughput drops to 180 requests per second. The CEO is furious. The team wants to abandon containers and go back to bare metal. But the problem is not containers — it is configuration. The JVM inside the container thinks it has 64 GB of RAM (the host's total) and allocates a 16 GB heap, triggering constant garbage collection inside a 512 MB container limit. The Node.js worker processes are fighting over two CPU cores with no affinity. The database container has no I/O priority and starves during peak traffic. Today you learn to tune containers until they match — and often exceed — bare-metal performance.
</Callout>

## Benchmarking Containers

Before tuning, you need a baseline. Never optimize based on guesses — measure, change one variable, and measure again.

### CPU Benchmarking with sysbench

```bash
# Run sysbench CPU benchmark inside a container
docker run --rm severalnines/sysbench sysbench cpu \
  --cpu-max-prime=20000 \
  --threads=4 \
  run

# Example output:
# CPU speed:
#     events per second:  3842.15
#
# Throughput:
#     events/s (eps):     3842.1537
#     time elapsed:       10.0003s
#     total number of events: 38423
#
# Latency (ms):
#          min:           0.94
#          avg:           1.04
#          max:           12.31
#          95th percentile: 1.16

# Compare with resource limits
docker run --rm --cpus 1.0 severalnines/sysbench sysbench cpu \
  --cpu-max-prime=20000 \
  --threads=4 \
  run
# events per second: ~960 (approximately 1/4 of unlimited)

docker run --rm --cpus 0.5 severalnines/sysbench sysbench cpu \
  --cpu-max-prime=20000 \
  --threads=4 \
  run
# events per second: ~480 (approximately 1/8 of unlimited)
```

## CPU Scheduling

Docker provides multiple mechanisms to control how containers use CPU resources. Understanding the difference between shares, quotas, and pinning is critical for performance tuning.

```
┌─────────────────────────────────────────────────────────────────┐
│                    CPU Scheduling Mechanisms                     │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │  CPU Shares (--cpu-shares)                               │    │
│  │  ─────────────────────────                               │    │
│  │  Relative weight. Only matters when CPU is contended.    │    │
│  │  Default: 1024. Higher = more CPU time proportionally.   │    │
│  │                                                          │    │
│  │  Container A: shares=1024  ─┐                            │    │
│  │  Container B: shares=512   ─┤ A gets 2x B's CPU time    │    │
│  │  Container C: shares=256   ─┘ A gets 4x C's CPU time    │    │
│  │                                                          │    │
│  │  When CPU is NOT contended, all containers can use 100%  │    │
│  └──────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │  CPU Quota (--cpus or --cpu-quota/--cpu-period)          │    │
│  │  ──────────────────────────────────────                  │    │
│  │  Hard limit. Container cannot exceed this regardless     │    │
│  │  of available CPU.                                       │    │
│  │                                                          │    │
│  │  --cpus 0.5  = max 50% of one core                      │    │
│  │  --cpus 2.0  = max 200% (two full cores)                │    │
│  │  --cpus 0.25 = max 25% of one core                      │    │
│  │                                                          │    │
│  │  Equivalent: --cpu-quota=50000 --cpu-period=100000       │    │
│  │  (50000 microseconds per 100000 microsecond period)      │    │
│  └──────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │  CPU Pinning (--cpuset-cpus)                             │    │
│  │  ───────────────────────────                             │    │
│  │  Bind container to specific CPU cores.                   │    │
│  │  Eliminates context switching between cores.             │    │
│  │  Best for latency-sensitive workloads.                   │    │
│  │                                                          │    │
│  │  --cpuset-cpus="0,1"    = cores 0 and 1 only            │    │
│  │  --cpuset-cpus="0-3"    = cores 0, 1, 2, 3              │    │
│  │  --cpuset-cpus="0,2,4"  = cores 0, 2, 4 (skip 1,3)     │    │
│  └──────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

### CPU Shares — Relative Weight

```bash
# Default shares (1024) — equal priority
docker run -d --name app-default alpine sleep 3600
docker inspect app-default --format '{{.HostConfig.CpuShares}}'
# 0 (means default: 1024)

# High priority container (2x CPU time when contended)
docker run -d --name app-high \
  --cpu-shares 2048 \
  myapp:latest

# Low priority container (half CPU time when contended)
docker run -d --name app-low \
  --cpu-shares 512 \
  worker:latest

# Demonstrate: when CPU is contended, shares determine allocation
# Container with shares=2048 gets 2048/(2048+512) = 80% of CPU
# Container with shares=512 gets 512/(2048+512) = 20% of CPU

# When only one container needs CPU, it gets 100% regardless of shares
```

<Callout type="info" title="Shares Only Matter Under Contention">
CPU shares are a proportional weight, not an absolute limit. A container with `--cpu-shares 256` can use all available CPU if no other container wants it. Shares only determine allocation when multiple containers compete for the same CPU time. Use `--cpus` for hard limits.
</Callout>

### CPU Quota — Hard Limits

```bash
# Limit to 0.5 CPU (50% of one core)
docker run -d --name app-half-cpu \
  --cpus 0.5 \
  myapp:latest

# Limit to 2 CPUs (can use up to 2 full cores)
docker run -d --name app-two-cpu \
  --cpus 2.0 \
  myapp:latest

# Equivalent using quota/period (advanced)
# --cpus 0.5 is the same as:
docker run -d --name app-quota \
  --cpu-quota 50000 \
  --cpu-period 100000 \
  myapp:latest

# Why use quota/period directly?
# For burst-friendly workloads, increase the period:
docker run -d --name app-burst \
  --cpu-quota 100000 \
  --cpu-period 200000 \
  myapp:latest
# This allows the container to use 100ms of CPU time per 200ms period
# Same average (0.5 CPU) but can burst to 100% for 100ms at a time

# Update limits on a running container
docker update --cpus 1.5 myapp
```

### CPU Pinning — Core Affinity

```bash
# Check available CPU cores
nproc
# 8

# Pin to cores 0 and 1
docker run -d --name app-pinned \
  --cpuset-cpus="0,1" \
  --cpus 2.0 \
  myapp:latest

# Pin database to cores 2-3 (no sharing with app)
docker run -d --name db-pinned \
  --cpuset-cpus="2,3" \
  --cpus 2.0 \
  postgres:16-alpine

# This prevents CPU cache thrashing between app and database
# Each has dedicated cores with warm L1/L2 caches

# Verify the CPU affinity
docker inspect app-pinned --format '{{.HostConfig.CpusetCpus}}'
# 0,1

# Pin memory to specific NUMA nodes (for NUMA-aware systems)
docker run -d --name numa-aware \
  --cpuset-cpus="0-3" \
  --cpuset-mems="0" \
  myapp:latest
```

<Callout type="tip" title="CPU Pinning for Latency-Sensitive Applications">
CPU pinning eliminates the overhead of context switching between cores. When a process moves from core 0 to core 3, it loses its L1 and L2 cache (a "cold cache miss"), which can add 10-100 microseconds of latency. For applications where every millisecond matters (trading systems, real-time data processing, game servers), pin the container to specific cores and keep those cores free from other workloads.
</Callout>

## Memory Management

### Memory Limits and Reservations

```bash
# Hard memory limit (container is killed if it exceeds this)
docker run -d --name app-mem \
  --memory 512m \
  myapp:latest

# Memory reservation (soft limit — scheduling hint)
docker run -d --name app-mem-res \
  --memory 512m \
  --memory-reservation 256m \
  myapp:latest
# The container is guaranteed 256MB and can use up to 512MB
# Docker will try to reclaim memory back to 256MB under pressure

# Disable swap (recommended for predictable performance)
docker run -d --name app-no-swap \
  --memory 512m \
  --memory-swap 512m \
  myapp:latest
# memory-swap = memory means NO swap available
# memory-swap > memory means swap = (memory-swap - memory)

# Unlimited swap (dangerous)
docker run -d --name app-unlimited-swap \
  --memory 512m \
  --memory-swap -1 \
  myapp:latest
# Container can swap without limit — very slow but will not be OOM killed

# Check current memory usage
docker stats --no-stream --format \
  "table {{.Name}}\t{{.MemUsage}}\t{{.MemPerc}}" app-mem
# NAME      MEM USAGE / LIMIT     MEM %
# app-mem   45.12MiB / 512MiB     8.81%
```

### OOM Score Adjustment

```bash
# Set OOM kill priority (lower = less likely to be killed)
# Range: -1000 (never kill) to 1000 (kill first)

# Protect critical containers from OOM
docker run -d --name critical-db \
  --memory 1g \
  --oom-score-adj -500 \
  postgres:16-alpine

# Make batch workers the first to be killed
docker run -d --name batch-worker \
  --memory 256m \
  --oom-score-adj 500 \
  worker:latest

# Disable OOM killer entirely (container pauses instead of dying)
docker run -d --name no-oom-kill \
  --memory 512m \
  --oom-kill-disable \
  myapp:latest
# WARNING: If this container exceeds memory, it will hang,
# and may cause the host to become unresponsive
```

### Kernel Memory Limits

```bash
# Limit kernel memory (for preventing fork bombs and excessive file handles)
docker run -d --name app-kmem \
  --memory 512m \
  --kernel-memory 64m \
  myapp:latest

# View memory breakdown
docker stats --no-stream --format \
  "{{.Name}}: {{.MemUsage}} ({{.MemPerc}})"
```

## I/O Throttling

```bash
# Limit write speed to a specific device
docker run -d --name app-io \
  --device-write-bps /dev/sda:50mb \
  --device-read-bps /dev/sda:100mb \
  myapp:latest

# Limit I/O operations per second (IOPS)
docker run -d --name app-iops \
  --device-write-iops /dev/sda:1000 \
  --device-read-iops /dev/sda:2000 \
  myapp:latest

# Relative I/O weight (like CPU shares, 10-1000, default 500)
docker run -d --name app-io-weight \
  --blkio-weight 300 \
  myapp:latest

docker run -d --name db-io-weight \
  --blkio-weight 700 \
  postgres:16-alpine
# Database gets ~70% of I/O bandwidth when contended

# Verify I/O limits
docker inspect app-io --format '{{json .HostConfig.BlkioDeviceWriteBps}}' | jq .
# [
#   {
#     "Path": "/dev/sda",
#     "Rate": 52428800
#   }
# ]
```

<Callout type="warning" title="I/O Throttling Requires Direct Device Access">
The `--device-read-bps` and `--device-write-bps` flags only work with direct device access, not with the overlay filesystem. If your container's workload goes through the overlay storage driver, these limits may not be enforced. For reliable I/O throttling, use volumes backed by specific block devices and set the limits on those devices.
</Callout>

## Network Performance

### Host Networking

```bash
# Default bridge networking adds ~2-5% latency overhead from NAT
docker run -d --name app-bridge \
  -p 8080:3000 \
  myapp:latest

# Host networking eliminates the overhead (no NAT, no bridge)
docker run -d --name app-host \
  --network host \
  myapp:latest
# The container shares the host's network stack directly
# Port 3000 is available on the host without mapping

# Benchmark comparison
# Bridge:
docker run --rm --network bridge williamyeh/wrk \
  -t4 -c100 -d10s http://app-bridge:3000/
# Requests/sec: 42000

# Host:
docker run --rm --network host williamyeh/wrk \
  -t4 -c100 -d10s http://localhost:3000/
# Requests/sec: 48000  (~14% improvement)
```

### Network Tuning for High Throughput

```bash
# Increase container's network buffer sizes
docker run -d --name app-tuned \
  --sysctl net.core.rmem_max=16777216 \
  --sysctl net.core.wmem_max=16777216 \
  --sysctl net.core.somaxconn=65535 \
  --sysctl net.ipv4.tcp_max_syn_backlog=65535 \
  myapp:latest

# Enable TCP_NODELAY for latency-sensitive applications
# (configure in your application, not Docker)

# Use macvlan for near-bare-metal network performance
docker network create -d macvlan \
  --subnet=192.168.1.0/24 \
  --gateway=192.168.1.1 \
  -o parent=eth0 \
  macnet

docker run -d --name app-macvlan \
  --network macnet \
  --ip 192.168.1.100 \
  myapp:latest
# Container gets its own MAC address and IP on the physical network
# No NAT, no bridge — lowest possible latency
```

## JVM in Containers

The JVM has a historically difficult relationship with containers. Older JVM versions do not respect container memory limits and will allocate heap based on the host's total memory, causing OOM kills.

```bash
# BAD: JVM ignoring container limits (Java 8 before update 191)
docker run --rm --memory 256m openjdk:8-jdk java -XX:+PrintFlagsFinal -version \
  2>&1 | grep -i maxheap
# MaxHeapSize = 16873684992  (16 GB — the HOST's memory!)
# Container has 256MB limit — JVM will be OOM killed

# GOOD: Modern JVM (Java 11+) respects container limits by default
docker run --rm --memory 256m eclipse-temurin:21-jdk \
  java -XX:+PrintFlagsFinal -version 2>&1 | grep -i maxheap
# MaxHeapSize = 130023424  (~124 MB — ~50% of container limit)
```

### JVM Tuning for Containers

```bash
# Optimal JVM settings for containers
docker run -d --name java-app \
  --memory 512m \
  --cpus 2.0 \
  -e JAVA_OPTS="\
    -XX:MaxRAMPercentage=75.0 \
    -XX:InitialRAMPercentage=50.0 \
    -XX:+UseG1GC \
    -XX:MaxGCPauseMillis=200 \
    -XX:+UseStringDeduplication \
    -XX:+UseContainerSupport \
    -XX:ActiveProcessorCount=2" \
  eclipse-temurin:21-jdk \
  java $JAVA_OPTS -jar /app/myapp.jar
```

| JVM Flag | Purpose | Recommended Value |
|----------|---------|-------------------|
| `-XX:MaxRAMPercentage` | Max heap as % of container memory | 75% (leave room for non-heap) |
| `-XX:InitialRAMPercentage` | Initial heap as % of container memory | 50% |
| `-XX:+UseContainerSupport` | Respect cgroup memory limits | Always enable (default in JDK 11+) |
| `-XX:ActiveProcessorCount` | Override detected CPU count | Set to `--cpus` value |
| `-XX:+UseG1GC` | Garbage collector for containers | Best for most workloads |
| `-XX:MaxGCPauseMillis` | Target GC pause time | 200ms for APIs, 500ms for batch |

<Callout type="warning" title="Always Set ActiveProcessorCount for JVM">
Even modern JVMs may detect the host's total CPU count instead of the container's CPU limit. This causes the JVM to create too many GC threads and compiler threads, all competing for limited CPU quota. Set `-XX:ActiveProcessorCount` to match your `--cpus` value. For `--cpus 2.0`, set `-XX:ActiveProcessorCount=2`.
</Callout>

## Node.js in Containers

### Thread Pool and Memory Tuning

```bash
# Node.js uses libuv with a default thread pool of 4 threads
# For I/O-heavy applications, increase it
docker run -d --name node-app \
  --memory 512m \
  --cpus 2.0 \
  -e UV_THREADPOOL_SIZE=8 \
  -e NODE_OPTIONS="--max-old-space-size=384" \
  node:20-alpine node /app/server.js

# UV_THREADPOOL_SIZE: default 4, max 1024
# Used for: DNS lookups, filesystem I/O, crypto, zlib
# Rule of thumb: set to 2x the number of CPU cores

# --max-old-space-size: V8 heap limit in MB
# Should be ~75% of container memory limit
# For 512MB container: 384MB
# For 1GB container: 768MB
# For 256MB container: 192MB
```

### Node.js Cluster Mode

```bash
# Single-process Node.js only uses one CPU core
# Use cluster module for multi-core utilization

cat > /tmp/cluster-server.js << 'EOF'
const cluster = require('cluster');
const http = require('http');
const os = require('os');

// Use the number of available CPUs (respects --cpus in modern Node)
const WORKERS = parseInt(process.env.WEB_CONCURRENCY) || os.availableParallelism?.() || os.cpus().length;

if (cluster.isPrimary) {
  console.log(`Primary ${process.pid} starting ${WORKERS} workers`);

  for (let i = 0; i < WORKERS; i++) {
    cluster.fork();
  }

  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died (${signal || code}). Restarting...`);
    cluster.fork();
  });
} else {
  http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Worker ${process.pid}\n`);
  }).listen(3000);

  console.log(`Worker ${process.pid} started`);
}
EOF

# Run with cluster mode
docker run -d --name node-cluster \
  --memory 1g \
  --cpus 4.0 \
  -e WEB_CONCURRENCY=4 \
  -e UV_THREADPOOL_SIZE=16 \
  -e NODE_OPTIONS="--max-old-space-size=192" \
  -p 3000:3000 \
  -v /tmp/cluster-server.js:/app/server.js:ro \
  node:20-alpine node /app/server.js

# Note: --max-old-space-size is PER PROCESS
# With 4 workers: 4 * 192MB = 768MB max heap total
# Plus overhead: ~200MB for V8 internals, libuv, etc.
# Total: ~968MB fits within 1GB container limit
```

| Setting | Default | Recommended | Notes |
|---------|---------|-------------|-------|
| `UV_THREADPOOL_SIZE` | 4 | 2x CPU cores | For I/O-heavy apps |
| `--max-old-space-size` | 1.5GB | 75% container memory / workers | Per process |
| `WEB_CONCURRENCY` | 1 | Match `--cpus` | Number of cluster workers |
| `NODE_ENV` | undefined | `production` | Enables optimizations |

## Python in Containers

### Gunicorn Worker Tuning

```bash
# Gunicorn with sync workers (CPU-bound)
docker run -d --name python-app \
  --memory 512m \
  --cpus 2.0 \
  -p 8000:8000 \
  python-app:latest \
  gunicorn app:application \
    --bind 0.0.0.0:8000 \
    --workers 5 \
    --threads 2 \
    --worker-class gthread \
    --max-requests 1000 \
    --max-requests-jitter 50 \
    --timeout 30 \
    --graceful-timeout 10

# Worker formula: (2 * CPU cores) + 1
# For --cpus 2.0: (2 * 2) + 1 = 5 workers

# For async I/O-bound apps, use uvicorn with uvloop
docker run -d --name python-async \
  --memory 512m \
  --cpus 2.0 \
  -p 8000:8000 \
  python-app:latest \
  uvicorn app:application \
    --host 0.0.0.0 \
    --port 8000 \
    --workers 5 \
    --loop uvloop \
    --http httptools \
    --no-access-log
```

| Setting | Formula | Example (2 CPUs, 512MB) |
|---------|---------|------------------------|
| `--workers` | (2 * CPUs) + 1 | 5 |
| `--threads` | 2-4 per worker | 2 |
| Memory per worker | Container memory / workers | ~100MB |
| `--max-requests` | Recycle workers | 1000 (prevents memory leaks) |
| `--worker-class` | `gthread` for mixed I/O | `gthread` or `uvicorn.workers.UvicornWorker` |

<Callout type="tip" title="Use --max-requests to Prevent Memory Leaks">
Python applications are prone to memory leaks due to reference cycles, C extensions that do not free memory, and growing caches. Set `--max-requests 1000 --max-requests-jitter 50` in gunicorn to automatically restart workers after a set number of requests. The jitter prevents all workers from restarting simultaneously. This is a safety net, not a fix — investigate and fix the actual memory leak.
</Callout>

## Key Takeaways

- **Always benchmark before optimizing** — use sysbench for CPU, wrk/hey for HTTP, and fio for I/O to establish baselines and measure the impact of each change
- **CPU shares** (`--cpu-shares`) are relative weights that only matter under contention; **CPU quotas** (`--cpus`) are hard limits; **CPU pinning** (`--cpuset-cpus`) binds containers to specific cores for cache efficiency
- **Memory limits** (`--memory`) are enforced by the kernel OOM killer; set `--memory-swap` equal to `--memory` to disable swap; use `--memory-reservation` as a soft limit for scheduling
- **I/O throttling** with `--device-read-bps` and `--device-write-bps` controls storage bandwidth per container, preventing I/O-heavy containers from starving others
- **Host networking** (`--network host`) eliminates NAT overhead and can improve throughput by 10-15% for network-intensive workloads, at the cost of port isolation
- **JVM containers** require explicit tuning: set `-XX:MaxRAMPercentage=75.0` for heap sizing, `-XX:ActiveProcessorCount` to match `--cpus`, and use G1GC for containerized workloads
- **Node.js containers** need `UV_THREADPOOL_SIZE` (2x CPU cores), `--max-old-space-size` (75% of container memory divided by worker count), and cluster mode for multi-core utilization
- **Python containers** with gunicorn should use `(2 * CPUs) + 1` workers, `--max-requests` to recycle workers and prevent memory leaks, and uvloop for async applications
- **docker stats** provides real-time monitoring; **cAdvisor** provides historical metrics and Prometheus integration; **perf** and **flame graphs** identify CPU hotspots at the function level
- Performance tuning is iterative: change one variable at a time, measure the impact, and keep the changes that produce measurable improvement
