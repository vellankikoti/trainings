## Creating Services

A swarm service is the definition of a desired state: which image to run, how many replicas, which networks to attach to, and how to update. The swarm manager continuously reconciles the actual state with the desired state.

### Basic Service Creation

```bash
# Create a simple service with 3 replicas
docker service create \
  --name webapp \
  --replicas 3 \
  --publish 80:80 \
  nginx:alpine

# Verify the service
docker service ls
# ID             NAME     MODE         REPLICAS   IMAGE          PORTS
# abc123def456   webapp   replicated   3/3        nginx:alpine   *:80->80/tcp

# See where tasks are running
docker service ps webapp
# ID        NAME       IMAGE          NODE       DESIRED STATE   CURRENT STATE
# abc123    webapp.1   nginx:alpine   manager1   Running         Running 10 seconds ago
# def456    webapp.2   nginx:alpine   worker1    Running         Running 10 seconds ago
# ghi789    webapp.3   nginx:alpine   worker2    Running         Running 10 seconds ago

# Inspect the full service definition
docker service inspect webapp --pretty
# ID:             abc123def456
# Name:           webapp
# Service Mode:   Replicated
#  Replicas:      3
# Placement:
# UpdateConfig:
#  Parallelism:   1
#  On failure:    pause
#  ...
```

### Service Creation with Full Configuration

```bash
# Production-grade service creation
docker service create \
  --name api \
  --replicas 5 \
  --publish published=8080,target=3000,protocol=tcp,mode=ingress \
  --network app-network \
  --env NODE_ENV=production \
  --env PORT=3000 \
  --limit-cpu 1.0 \
  --limit-memory 512M \
  --reserve-cpu 0.25 \
  --reserve-memory 128M \
  --restart-condition on-failure \
  --restart-delay 5s \
  --restart-max-attempts 3 \
  --restart-window 120s \
  --health-cmd "wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1" \
  --health-interval 15s \
  --health-timeout 5s \
  --health-retries 3 \
  --health-start-period 30s \
  --update-delay 10s \
  --update-parallelism 2 \
  --update-failure-action rollback \
  --update-monitor 15s \
  --update-order start-first \
  --rollback-delay 5s \
  --rollback-parallelism 1 \
  --rollback-failure-action pause \
  --rollback-monitor 10s \
  --rollback-order stop-first \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  myregistry.com/api:v1.2.3
```

## Replicated vs Global Services

```
┌────────────────────────────────────────────────────────────┐
│              Replicated vs Global Services                  │
│                                                            │
│  Replicated (--mode replicated)                            │
│  ─────────────────────────────                             │
│  You specify the number of replicas.                       │
│  Swarm distributes tasks across available nodes.           │
│                                                            │
│  Node 1        Node 2        Node 3        Node 4          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ webapp.1 │  │ webapp.2 │  │ webapp.3 │  │          │  │
│  │          │  │          │  │          │  │ (empty)  │  │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘  │
│  3 replicas spread across 4 nodes (1 node idle)            │
│                                                            │
│  Global (--mode global)                                    │
│  ─────────────────────                                     │
│  Exactly one task per node. You cannot set replica count.  │
│  New nodes automatically get a task.                       │
│                                                            │
│  Node 1        Node 2        Node 3        Node 4          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ monitor  │  │ monitor  │  │ monitor  │  │ monitor  │  │
│  │ .abc123  │  │ .def456  │  │ .ghi789  │  │ .jkl012  │  │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘  │
│  One per node — ideal for monitoring, log agents, etc.     │
└────────────────────────────────────────────────────────────┘
```

```bash
# Create a replicated service (default)
docker service create \
  --name webapp \
  --mode replicated \
  --replicas 5 \
  nginx:alpine

# Create a global service (one per node)
docker service create \
  --name node-exporter \
  --mode global \
  --publish 9100:9100 \
  prom/node-exporter:latest

# Global services are ideal for:
# - Monitoring agents (node-exporter, cAdvisor)
# - Log collectors (Fluentd, Filebeat)
# - Security agents
# - Network proxies

# Scale a replicated service
docker service scale webapp=10
# webapp scaled to 10
# (Does not apply to global services — they auto-scale with node count)

# Scale multiple services at once
docker service scale webapp=5 api=3 worker=8
```

## Rolling Updates and Rollback

Rolling updates replace service tasks incrementally, maintaining availability throughout the deployment.

### Update Configuration

```bash
# Update a service to a new image version
docker service update \
  --image myregistry.com/api:v1.3.0 \
  api

# Watch the update in real-time
docker service ps api
# ID       NAME       IMAGE              NODE      DESIRED STATE   CURRENT STATE
# new123   api.1      api:v1.3.0         worker1   Running         Running 5s
# old123   \_api.1    api:v1.2.3         worker1   Shutdown        Shutdown 10s
# new456   api.2      api:v1.3.0         worker2   Running         Running 15s
# old456   \_api.2    api:v1.2.3         worker2   Shutdown        Shutdown 20s
# old789   api.3      api:v1.2.3         manager1  Running         Running 2m
# ^^ api.3 has not been updated yet (rolling update in progress)
```

### Update Parameters Explained

```bash
# Configure how updates are performed
docker service update \
  --update-parallelism 2 \
  --update-delay 10s \
  --update-failure-action rollback \
  --update-monitor 15s \
  --update-order start-first \
  --update-max-failure-ratio 0.25 \
  --image api:v2.0.0 \
  api
```

| Parameter | Default | Description |
|-----------|---------|-------------|
| `--update-parallelism` | 1 | Number of tasks updated simultaneously |
| `--update-delay` | 0s | Wait time between updating each batch |
| `--update-failure-action` | pause | What to do on failure: `pause`, `continue`, or `rollback` |
| `--update-monitor` | 5s | Time to monitor a new task before considering it healthy |
| `--update-order` | stop-first | `stop-first` (stop old, start new) or `start-first` (start new, then stop old) |
| `--update-max-failure-ratio` | 0 | Maximum fraction of tasks that can fail before triggering failure action |

```
┌──────────────────────────────────────────────────────────────┐
│              Rolling Update: stop-first vs start-first       │
│                                                              │
│  stop-first (default):                                       │
│  ┌──────┐                    ┌──────┐                        │
│  │ v1.0 │ ──→ stop ──→      │ v2.0 │ ──→ start ──→ healthy  │
│  └──────┘                    └──────┘                        │
│  Temporarily reduces capacity. Saves resources.              │
│                                                              │
│  start-first:                                                │
│  ┌──────┐     ┌──────┐                                       │
│  │ v1.0 │     │ v2.0 │ ──→ start ──→ healthy ──→            │
│  └──────┘     └──────┘                           │           │
│      │                                           │           │
│      └───────── stop after new is healthy ◄──────┘           │
│  Maintains full capacity. Uses more resources temporarily.   │
│  Preferred for zero-downtime deployments.                    │
└──────────────────────────────────────────────────────────────┘
```

<Callout type="tip" title="Use start-first for Zero-Downtime Deployments">
The `--update-order start-first` option starts the new task before stopping the old one. This means the service always has the desired number of healthy replicas during the update. Combined with health checks and `--update-monitor`, this ensures that bad deployments are caught before the old task is removed. Use `stop-first` only when resources are constrained and you cannot afford to run extra tasks temporarily.
</Callout>

### Rollback

```bash
# Automatic rollback on failure
# (configured with --update-failure-action rollback)

# Manual rollback to the previous version
docker service rollback api
# api
# rollback: manually requested rollback

# Watch the rollback
docker service ps api
# Shows tasks being rolled back to the previous image

# Check rollback configuration
docker service inspect api --format '{{json .Spec.RollbackConfig}}' | jq .
# {
#   "Parallelism": 1,
#   "Delay": 5000000000,
#   "FailureAction": "pause",
#   "Monitor": 10000000000,
#   "MaxFailureRatio": 0
# }

# View the update and rollback history
docker service inspect api --format '{{json .PreviousSpec.TaskTemplate.ContainerSpec.Image}}'
# myregistry.com/api:v1.2.3@sha256:abc123...   (previous version)

docker service inspect api --format '{{json .Spec.TaskTemplate.ContainerSpec.Image}}'
# myregistry.com/api:v1.3.0@sha256:def456...   (current version)

# Update multiple service parameters at once
docker service update \
  --image api:v1.4.0 \
  --env-add NEW_FEATURE=enabled \
  --env-rm OLD_FEATURE \
  --limit-memory 768M \
  --replicas 8 \
  api
```

<Callout type="warning" title="Rollback Only Goes Back One Version">
`docker service rollback` reverts to the immediately previous service specification. It does not maintain a full version history. If you need to roll back multiple versions, you must explicitly specify the target image tag with `docker service update --image api:v1.0.0 api`. For production deployments, always tag your images with specific version numbers so you can target any previous version.
</Callout>

## Placement Constraints and Preferences

### Constraints (Hard Rules)

```bash
# Run only on worker nodes
docker service create \
  --name api \
  --constraint 'node.role == worker' \
  api:latest

# Run only on nodes with specific labels
docker service create \
  --name db \
  --constraint 'node.labels.disk == ssd' \
  --constraint 'node.labels.env == production' \
  postgres:16-alpine

# Run only on a specific hostname
docker service create \
  --name singleton \
  --constraint 'node.hostname == db-server-01' \
  postgres:16-alpine

# Exclude specific nodes
docker service create \
  --name api \
  --constraint 'node.labels.env != staging' \
  api:latest

# Multiple constraints are AND-ed together:
# Both must be true for a node to be eligible
docker service create \
  --name api \
  --constraint 'node.role == worker' \
  --constraint 'node.labels.region == us-east' \
  --constraint 'node.labels.tier == frontend' \
  api:latest
```

### Preferences (Soft Rules)

```bash
# Spread tasks across datacenters (soft preference)
docker service create \
  --name api \
  --replicas 6 \
  --placement-pref 'spread=node.labels.datacenter' \
  api:latest

# With 3 datacenters, each gets 2 replicas:
# us-east: api.1, api.4
# us-west: api.2, api.5
# eu-west: api.3, api.6

# Spread across availability zones
docker service create \
  --name webapp \
  --replicas 9 \
  --placement-pref 'spread=node.labels.zone' \
  webapp:latest

# Combine constraints and preferences
docker service create \
  --name api \
  --replicas 6 \
  --constraint 'node.labels.env == production' \
  --placement-pref 'spread=node.labels.zone' \
  api:latest
# Only runs on production nodes, spread across zones
```

### Resource Reservations and Limits

```bash
# Reserve resources (scheduling guarantee) and set limits (hard cap)
docker service create \
  --name api \
  --replicas 3 \
  --reserve-cpu 0.5 \
  --reserve-memory 256M \
  --limit-cpu 2.0 \
  --limit-memory 512M \
  api:latest

# The scheduler will only place a task on a node that has
# at least 0.5 CPU and 256MB of memory available.
# The task cannot use more than 2.0 CPU or 512MB memory.
```

```
┌──────────────────────────────────────────────────────────┐
│           Resource Reservations vs Limits                 │
│                                                          │
│  Reservation (--reserve-*)                               │
│  ┌────────────────────────────────────────────────────┐  │
│  │  "Promise me at least this much"                   │  │
│  │  - Used by the scheduler for placement decisions   │  │
│  │  - Node must have this much free to accept task    │  │
│  │  - Prevents overcommitting (if set correctly)      │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  Limit (--limit-*)                                       │
│  ┌────────────────────────────────────────────────────┐  │
│  │  "Never let me use more than this"                 │  │
│  │  - Hard enforcement by the kernel (cgroups)        │  │
│  │  - Exceeding memory limit → OOM kill               │  │
│  │  - Exceeding CPU limit → throttling                │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  Example: --reserve-cpu 0.5 --limit-cpu 2.0              │
│  The task is guaranteed 0.5 CPU but can burst to 2.0     │
│  when other containers are not using their allocation.   │
└──────────────────────────────────────────────────────────┘
```

## Docker Stack: Deploying Compose Files to Swarm

Docker Stack lets you deploy a complete application from a Compose file to a swarm. It is the bridge between development (Docker Compose) and production (Swarm).

### Stack Deploy

```yaml
# docker-compose.yml (stack-compatible)
version: "3.9"

services:
  api:
    image: myregistry.com/api:v1.2.3
    ports:
      - "8080:3000"
    networks:
      - frontend
      - backend
    environment:
      NODE_ENV: production
      DB_HOST: db
      REDIS_HOST: redis
    deploy:
      replicas: 5
      update_config:
        parallelism: 2
        delay: 10s
        failure_action: rollback
        monitor: 15s
        order: start-first
      rollback_config:
        parallelism: 1
        delay: 5s
        failure_action: pause
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
        window: 120s
      resources:
        limits:
          cpus: "1.0"
          memory: 512M
        reservations:
          cpus: "0.25"
          memory: 128M
      placement:
        constraints:
          - node.role == worker
          - node.labels.env == production
        preferences:
          - spread: node.labels.zone
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:3000/health"]
      interval: 15s
      timeout: 5s
      retries: 3
      start_period: 30s
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  db:
    image: postgres:16-alpine
    volumes:
      - db-data:/var/lib/postgresql/data
    networks:
      - backend
    environment:
      POSTGRES_USER_FILE: /run/secrets/db_user
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
      POSTGRES_DB: appdb
    secrets:
      - db_user
      - db_password
    configs:
      - source: postgres_conf
        target: /etc/postgresql/postgresql.conf
    deploy:
      replicas: 1
      placement:
        constraints:
          - node.labels.disk == ssd
      resources:
        limits:
          cpus: "2.0"
          memory: 1G
        reservations:
          cpus: "0.5"
          memory: 256M

  redis:
    image: redis:7-alpine
    networks:
      - backend
    deploy:
      replicas: 1
      resources:
        limits:
          cpus: "0.5"
          memory: 256M

  worker:
    image: myregistry.com/worker:v1.2.3
    networks:
      - backend
    environment:
      REDIS_HOST: redis
      DB_HOST: db
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: "0.5"
          memory: 256M

networks:
  frontend:
    driver: overlay
  backend:
    driver: overlay
    internal: true

volumes:
  db-data:
    driver: local

secrets:
  db_user:
    external: true
  db_password:
    external: true

configs:
  postgres_conf:
    file: ./postgres/postgresql.conf
```

```bash
# Deploy the stack
docker stack deploy --compose-file docker-compose.yml myapp
# Creating network myapp_frontend
# Creating network myapp_backend
# Creating secret myapp_db_user
# Creating secret myapp_db_password
# Creating config myapp_postgres_conf
# Creating service myapp_api
# Creating service myapp_db
# Creating service myapp_redis
# Creating service myapp_worker

# List stacks
docker stack ls
# NAME     SERVICES   ORCHESTRATOR
# myapp    4          Swarm

# List services in a stack
docker stack services myapp
# ID        NAME           MODE         REPLICAS   IMAGE                        PORTS
# abc123    myapp_api      replicated   5/5        myregistry.com/api:v1.2.3    *:8080->3000/tcp
# def456    myapp_db       replicated   1/1        postgres:16-alpine
# ghi789    myapp_redis    replicated   1/1        redis:7-alpine
# jkl012    myapp_worker   replicated   3/3        myregistry.com/worker:v1.2.3

# List all tasks in a stack
docker stack ps myapp
# Shows every task across all services

# Update a stack (modify the compose file, then redeploy)
# Edit docker-compose.yml (change image version, replicas, etc.)
docker stack deploy --compose-file docker-compose.yml myapp
# Swarm performs a rolling update for changed services

# Remove a stack
docker stack rm myapp
# Removing service myapp_api
# Removing service myapp_db
# Removing service myapp_redis
# Removing service myapp_worker
# Removing network myapp_frontend
# Removing network myapp_backend
```

<Callout type="info" title="Stack Deploy vs Compose Up">
`docker stack deploy` and `docker compose up` both use Compose files but behave differently:
- `docker compose up` runs containers on a single host with the Compose engine
- `docker stack deploy` distributes services across a swarm with the orchestrator
- Stack deploy ignores `build`, `depends_on`, `container_name`, and `links` directives
- Stack deploy requires pre-built images (it cannot build during deploy)
- The `deploy` section in the Compose file is only used by `docker stack deploy`, not by `docker compose up`
</Callout>

## Secrets Management

Docker secrets provide a secure way to pass sensitive data (passwords, API keys, TLS certificates) to services. Secrets are encrypted at rest in the Raft store and only mounted into containers that need them.

```bash
# Create a secret from a value
echo "mydbpassword123" | docker secret create db_password -
# abc123def456

# Create a secret from a file
echo "dbadmin" > /tmp/db_user.txt
docker secret create db_user /tmp/db_user.txt
rm /tmp/db_user.txt  # Remove the plaintext file

# Create a secret from a certificate file
docker secret create tls_cert ./server.crt
docker secret create tls_key ./server.key

# List secrets
docker secret ls
# ID             NAME           CREATED          UPDATED
# abc123         db_password    5 seconds ago    5 seconds ago
# def456         db_user        3 seconds ago    3 seconds ago
# ghi789         tls_cert       1 second ago     1 second ago
# jkl012         tls_key        1 second ago     1 second ago

# Inspect a secret (does NOT reveal the value)
docker secret inspect db_password
# [
#   {
#     "ID": "abc123",
#     "Version": { "Index": 15 },
#     "CreatedAt": "2025-01-15T10:30:00Z",
#     "UpdatedAt": "2025-01-15T10:30:00Z",
#     "Spec": {
#       "Name": "db_password",
#       "Labels": {}
#     }
#   }
# ]
# Note: The secret value is NEVER shown by docker secret inspect
```

### Using Secrets in Services

```bash
# Grant a service access to secrets
docker service create \
  --name api \
  --secret db_password \
  --secret db_user \
  --secret source=tls_cert,target=server.crt,uid=1001,gid=1001,mode=0400 \
  api:latest

# Inside the container, secrets are mounted at /run/secrets/
# /run/secrets/db_password    → contains "mydbpassword123"
# /run/secrets/db_user        → contains "dbadmin"
# /run/secrets/server.crt     → contains the TLS certificate

# Read secrets in your application:
# Node.js: fs.readFileSync('/run/secrets/db_password', 'utf8').trim()
# Python:  open('/run/secrets/db_password').read().strip()
# Shell:   cat /run/secrets/db_password
```

```

│                    Secrets Architecture                       │
│                                                              │
│  docker secret create         Raft Store (Encrypted)         │
│  ─────────────────────        ──────────────────────         │
│  echo "pass" | docker    ──→  ┌──────────────────────┐      │
│  secret create db_pass -      │  AES-256 encrypted   │      │
│                               │  Replicated to all   │      │
│                               │  manager nodes       │      │
│                               └─────────┬────────────┘      │
│                                         │                    │
│  Service requests secret                │                    │
│  ─────────────────────────              │                    │
│  --secret db_pass              Manager decrypts and          │
│                                sends to task node            │
│                                         │                    │
│                                         ▼                    │
│                               ┌──────────────────────┐      │
│                               │  Container tmpfs     │      │
│                               │  /run/secrets/       │      │
│                               │  └── db_pass         │      │
│                               │                      │      │
│                               │  In-memory only      │      │
│                               │  Never written to    │      │
│                               │  disk on worker      │      │
│                               └──────────────────────┘      │
│                                                              │

│  Workers never have access to secrets they do not need.      │
└──────────────────────────────────────────────────────────────┘
```

### Rotating Secrets

```bash
# Secrets are immutable — you cannot update a secret's value.
# To rotate, create a new secret and update the service.

# Step 1: Create the new secret
echo "newpassword456" | docker secret create db_password_v2 -

# Step 2: Update the service to use the new secret
docker service update \
  --secret-rm db_password \
  --secret-add source=db_password_v2,target=db_password \
  api
# The service sees the new value at /run/secrets/db_password
# Rolling update replaces containers one by one

# Step 3: Remove the old secret
docker secret rm db_password
```

<Callout type="warning" title="Secrets Are for Swarm Services Only">
Docker secrets only work with swarm services (`docker service create`), not with standalone containers (`docker run`). If you need to pass secrets to standalone containers, use environment variables (less secure) or mount a file from the host (manage permissions carefully). In Docker Compose with `docker compose up` (not stack deploy), secrets are simulated by mounting files from the host.
</Callout>

## Configs Management

Docker configs store non-sensitive configuration files in the swarm. Unlike secrets, configs are not encrypted, but they provide the same versioned, immutable, service-attached workflow.

```bash
# Create a config from a file
cat > /tmp/nginx.conf << 'EOF'
worker_processes auto;
events { worker_connections 1024; }
http {
    server {
        listen 80;
        location / {
            proxy_pass http://api:3000;
        }
        location /health {
            return 200 'ok';
        }
    }
}
EOF

docker config create nginx_conf_v1 /tmp/nginx.conf

# List configs
docker config ls
# ID             NAME              CREATED
# abc123         nginx_conf_v1     5 seconds ago

# Use a config in a service
docker service create \
  --name proxy \
```

## Key Takeaways

- **Swarm services** define a desired state (image, replicas, resources, networks) and the swarm manager continuously reconciles actual state to match, automatically replacing failed tasks
- **Replicated services** run a specified number of tasks spread across nodes; **global services** run exactly one task per node — use global for monitoring agents and log collectors
- **Rolling updates** replace tasks incrementally; configure `--update-parallelism` (how many at once), `--update-delay` (wait between batches), `--update-order start-first` (maintain capacity), and `--update-failure-action rollback` (auto-revert on failure)
- **Rollback** reverts to the previous service specification; it only goes back one version — always use versioned image tags so you can target any previous version explicitly
- **Placement constraints** are hard rules (`node.labels.env == production`) that determine which nodes can run a service; **preferences** (`spread=node.labels.zone`) are soft rules for distributing tasks evenly
- **Resource reservations** guarantee minimum resources for scheduling; **resource limits** set hard caps enforced by cgroups — always set both for predictable behavior
- **Docker Stack** deploys Compose files to Swarm with `docker stack deploy`; the `deploy` section in the Compose file controls swarm-specific behavior (replicas, update config, placement)
- **Secrets** are encrypted at rest, transmitted only to nodes running tasks that need them, mounted as in-memory files at `/run/secrets/`, and never exposed via `docker inspect` — rotate by creating a new version and updating the service
- **Configs** store non-sensitive configuration files with the same versioned, immutable workflow as secrets — useful for nginx configs, application settings, and other configuration files
- **Service logs** aggregate output from all tasks with `docker service logs`; use `-f` to follow, `--tail` to limit, and `--since` to filter by time
