## What Is Docker Swarm?

Docker Swarm is Docker's native clustering and orchestration system. It is built directly into the Docker Engine — no additional software to install. When you enable Swarm mode, your Docker hosts form a cluster that can deploy and manage services across multiple machines.


### Swarm vs Kubernetes

| Feature | Docker Swarm | Kubernetes |
|---------|-------------|------------|
| **Setup complexity** | Single command: `docker swarm init` | Complex: etcd, API server, scheduler, etc. |
| **Learning curve** | Low — uses Docker CLI | High — new concepts, kubectl, YAML manifests |
| **Scale** | Hundreds of nodes | Thousands of nodes |
| **Ecosystem** | Docker native | Massive (Helm, operators, CRDs, etc.) |
| **Auto-scaling** | Manual or scripted | Built-in HPA/VPA |
| **Service mesh** | Basic routing mesh | Istio, Linkerd, etc. |
| **Storage** | Docker volumes | CSI drivers, PVs, PVCs |
| **Health checks** | Container health checks | Liveness, readiness, startup probes |
| **Configuration** | Docker Compose files | YAML manifests, Helm charts |
| **Community** | Smaller | Very large |

<Callout type="info" title="When to Choose Swarm">
Choose Docker Swarm when you need simple orchestration for a small-to-medium deployment (3-20 nodes), when your team already knows Docker, when you want to reuse Docker Compose files, or when setup time matters more than ecosystem breadth. Choose Kubernetes when you need auto-scaling, service mesh, custom resources, or you are running at very large scale. Many teams start with Swarm and migrate to Kubernetes only when they outgrow it.
</Callout>

## Initializing a Swarm

### Creating the First Manager

```bash
# Initialize the swarm on the first node
docker swarm init --advertise-addr 192.168.1.10
# Swarm initialized: current node (abc123def456) is now a manager.
#
# To add a worker to this swarm, run the following command:
#
#     docker swarm join --token SWMTKN-1-abc123...xyz789 192.168.1.10:2377
#
# To add a manager to this swarm, run 'docker swarm join-token manager'
# and follow the instructions.

# The swarm is now active
docker info | grep -A 5 "Swarm"
# Swarm: active
#  NodeID: abc123def456
#  Is Manager: true
#  ClusterID: cluster-id-here
#  Managers: 1
#  Nodes: 1

# Check node status
docker node ls
# ID                            HOSTNAME   STATUS   AVAILABILITY   MANAGER STATUS   ENGINE VERSION
# abc123def456 *                manager1   Ready    Active         Leader           27.0.3
```

### Join Tokens

Swarm uses two types of join tokens — one for workers and one for managers. These tokens authenticate nodes joining the cluster.

```bash
# View the worker join token
docker swarm join-token worker
# To add a worker to this swarm, run the following command:
#     docker swarm join --token SWMTKN-1-worker-token 192.168.1.10:2377

# View the manager join token
docker swarm join-token manager
# To add a manager to this swarm, run the following command:
#     docker swarm join --token SWMTKN-1-manager-token 192.168.1.10:2377

# Rotate tokens (if compromised)
docker swarm join-token --rotate worker
# Successfully rotated worker join token.
# (Old tokens are immediately invalidated)

docker swarm join-token --rotate manager
# Successfully rotated manager join token.

# The join token format:
# SWMTKN-1-<cluster-hash>-<role-hash>
# ├──────┘ └─────────────┘ └─────────┘
# │        │               └── Role-specific hash (worker or manager)
# │        └── Cluster identity hash
# └── Swarm token prefix and version
```

### Adding Nodes

```bash
# On Worker Node 1 (192.168.1.11):
docker swarm join --token SWMTKN-1-worker-token 192.168.1.10:2377
# This node joined a swarm as a worker.

# On Worker Node 2 (192.168.1.12):
docker swarm join --token SWMTKN-1-worker-token 192.168.1.10:2377
# This node joined a swarm as a worker.

# On Manager Node 2 (192.168.1.13):
docker swarm join --token SWMTKN-1-manager-token 192.168.1.10:2377
# This node joined a swarm as a manager.

# On Manager Node 3 (192.168.1.14):
docker swarm join --token SWMTKN-1-manager-token 192.168.1.10:2377
# This node joined a swarm as a manager.

# Verify the cluster (run on any manager)
docker node ls
# ID                   HOSTNAME    STATUS   AVAILABILITY   MANAGER STATUS   ENGINE
# abc123 *             manager1    Ready    Active         Leader           27.0.3
# def456               manager2    Ready    Active         Reachable        27.0.3
# ghi789               manager3    Ready    Active         Reachable        27.0.3
# jkl012               worker1     Ready    Active                          27.0.3
# mno345               worker2     Ready    Active                          27.0.3
```

<Callout type="warning" title="Ports Required for Swarm Communication">
Swarm requires these ports to be open between all nodes:
- **TCP 2377** — Cluster management communication (managers only)
- **TCP/UDP 7946** — Node discovery and gossip protocol (all nodes)
- **UDP 4789** — VXLAN overlay network traffic (all nodes)

If these ports are blocked by a firewall, nodes cannot join the swarm or communicate. Open them before initializing the swarm.
</Callout>

## Manager and Worker Nodes

### Roles and Responsibilities

```
┌────────────────────────────────────────────────────────┐
│                    Manager Nodes                        │
│                                                        │
│  Responsibilities:                                     │
│  ├── Maintain cluster state (Raft consensus)           │
│  ├── Schedule tasks to nodes                           │
│  ├── Serve the Swarm API                               │
│  ├── Assign IP addresses to tasks                      │
│  ├── Manage overlay networks                           │
│  ├── Issue and rotate TLS certificates                 │
│  └── Can also run tasks (unless drained)               │
│                                                        │
│  Manager Status:                                       │
│  ├── Leader    — actively manages the swarm            │
│  ├── Reachable — can become leader if current fails    │
│  └── Unreachable — lost connectivity                   │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│                    Worker Nodes                         │
│                                                        │
│  Responsibilities:                                     │
│  ├── Execute tasks (containers) assigned by managers   │
│  ├── Report task status back to managers               │
│  └── Participate in overlay networking                 │
│                                                        │
│  Workers:                                              │
│  ├── Cannot manage the swarm                           │
│  ├── Cannot view or modify cluster state               │
│  ├── Cannot run docker node/service commands           │
│  └── Can be promoted to managers                       │
└────────────────────────────────────────────────────────┘
```

### Raft Consensus

Swarm uses the Raft consensus algorithm to maintain a consistent cluster state across all manager nodes. Raft requires a majority (quorum) of managers to agree before any change is committed.

```
┌──────────────────────────────────────────────────────────┐
│                  Raft Consensus Quorum                    │
│                                                          │
│  Managers    Quorum    Tolerated Failures                │
│  ─────────  ────────  ──────────────────                 │
│     1          1          0  (no HA)                     │
│     2          2          0  (no benefit over 1)         │
│     3          2          1  ← minimum for HA            │
│     5          3          2  ← recommended               │
│     7          4          3  ← maximum recommended       │
│                                                          │
│  Formula: Quorum = (N / 2) + 1                           │
│           Failures = N - Quorum = (N - 1) / 2            │
│                                                          │
│  IMPORTANT: Always use an ODD number of managers!        │
│  Even numbers provide no additional fault tolerance       │
│  over odd-1. (4 managers tolerate 1 failure, same as 3)  │
└──────────────────────────────────────────────────────────┘
```

```bash
# Example: What happens when a manager fails?

# 3-manager cluster: quorum = 2
# If 1 manager dies: 2 remaining > quorum. Cluster works.
# If 2 managers die: 1 remaining < quorum. Cluster is READ-ONLY.
#   Existing tasks keep running but no new scheduling occurs.

# 5-manager cluster: quorum = 3
# If 1 manager dies: 4 remaining > quorum. Cluster works.
# If 2 managers die: 3 remaining = quorum. Cluster works.
# If 3 managers die: 2 remaining < quorum. Cluster is READ-ONLY.

# Check Raft status
docker node ls --format "table {{.Hostname}}\t{{.ManagerStatus}}\t{{.Status}}"
# HOSTNAME    MANAGER STATUS   STATUS
# manager1    Leader           Ready
# manager2    Reachable        Ready
# manager3    Reachable        Ready
# worker1                      Ready
# worker2                      Ready
```

<Callout type="tip" title="Spread Managers Across Failure Domains">
Place manager nodes in different availability zones or racks. If all three managers are on the same physical host or in the same data center, a single hardware failure or network outage will take down the entire control plane. With 3 managers across 3 availability zones, you can lose an entire zone and the swarm continues operating.
</Callout>

## Node Management

### Promoting and Demoting Nodes

```bash
# Promote a worker to manager
docker node promote worker1
# Node worker1 promoted to a manager in the swarm.

docker node ls
# worker1 now shows "Reachable" in MANAGER STATUS

# Demote a manager to worker
docker node demote manager3
# Manager manager3 demoted in the swarm.

docker node ls
# manager3 no longer shows a MANAGER STATUS

# You can also use docker node update:
docker node update --role manager worker2
docker node update --role worker manager2
```

### Draining Nodes for Maintenance

```bash
# Drain a node — all tasks are moved to other nodes
docker node update --availability drain worker1
# worker1 availability set to "drain"

# Tasks on worker1 are rescheduled to other available nodes
docker node ps worker1
# NAME                IMAGE          NODE      DESIRED STATE   CURRENT STATE
# webapp.3            nginx:alpine   worker1   Shutdown        Shutdown 5 seconds ago

# The task is restarted on another node automatically
docker service ps webapp
# NAME              IMAGE          NODE      DESIRED STATE   CURRENT STATE
# webapp.3          nginx:alpine   worker2   Running         Running 3 seconds ago
# \_webapp.3        nginx:alpine   worker1   Shutdown        Shutdown 8 seconds ago

# Pause a node — no new tasks, existing tasks keep running
docker node update --availability pause worker2

# Bring a node back to active
docker node update --availability active worker1
# New tasks can now be scheduled on worker1 again

# Note: drained tasks are NOT automatically moved back.
# Rebalancing requires: docker service update --force <service>
```

### Removing Nodes

```bash
# Remove a worker from the swarm (run on the worker node)
docker swarm leave
# Node left the swarm.

# Remove a manager from the swarm (requires --force)
docker swarm leave --force
# Node left the swarm.

# Remove a node record from the swarm (run on a manager)
docker node rm worker1
# worker1

# If the node is still running, force remove it
docker node rm --force worker1
```

### Node Labels for Placement

```bash
# Add labels to nodes for placement constraints
docker node update --label-add env=production manager1
docker node update --label-add env=production manager2
docker node update --label-add env=staging worker1
docker node update --label-add datacenter=us-east worker1
docker node update --label-add datacenter=us-west worker2
docker node update --label-add disk=ssd manager1
docker node update --label-add disk=hdd worker2

# View node labels
docker node inspect worker1 --format '{{json .Spec.Labels}}' | jq .
# {
#   "datacenter": "us-east",
#   "disk": "ssd",
#   "env": "staging"
# }

# Use labels with service placement (covered in the next lesson)
docker service create \
  --name production-api \
  --constraint 'node.labels.env == production' \
  --constraint 'node.labels.disk == ssd' \
  nginx:alpine
```

## Swarm Networking

### Overlay Networks

Overlay networks span across all nodes in the swarm, allowing containers on different physical hosts to communicate as if they were on the same network.

```bash
# Create an overlay network
docker network create \
  --driver overlay \
  --subnet 10.10.0.0/24 \
  --attachable \
  app-network

# The --attachable flag allows standalone containers to join
# (not just swarm services)

# List overlay networks
docker network ls --filter driver=overlay
# NETWORK ID     NAME            DRIVER    SCOPE
# abc123         ingress         overlay   swarm
# def456         app-network     overlay   swarm

# Inspect the overlay network
docker network inspect app-network --format '{{json .IPAM}}' | jq .
# {
#   "Config": [
#     {
#       "Subnet": "10.10.0.0/24",
#       "Gateway": "10.10.0.1"
#     }
#   ]
# }
```


### The Ingress Routing Mesh

The routing mesh is a built-in load balancer that makes every node in the swarm listen on published service ports. Any node can receive traffic for any service, even if that node is not running a task for that service.

```bash
# Create a service with a published port
docker service create \
  --name webapp \
  --publish 80:80 \
  --replicas 3 \
  nginx:alpine

# The routing mesh makes port 80 available on ALL nodes:
# curl http://192.168.1.10:80  → routed to any webapp replica
# curl http://192.168.1.11:80  → routed to any webapp replica
# curl http://192.168.1.12:80  → routed to any webapp replica
# Even nodes that are NOT running a webapp container!
```

### Service Discovery

Swarm provides built-in DNS-based service discovery. Every service gets a DNS entry that resolves to the service's virtual IP (VIP).

```bash
# Create two services on the same overlay network
docker network create --driver overlay backend
docker service create --name api --network backend --replicas 3 myapi:latest
docker service create --name redis --network backend redis:alpine

# Inside any container in the "api" service:
# "redis" resolves to the VIP of the redis service
# nslookup redis
# Name:   redis
# Address: 10.10.1.5  ← Virtual IP (load-balanced)

# DNS round-robin mode (resolve to individual task IPs)
# nslookup tasks.redis
# Name:   tasks.redis
# Address: 10.10.1.6
# Address: 10.10.1.7
# (One entry per running task)

# The VIP approach (default) is preferred because it uses
# IPVS for connection-level load balancing. DNS round-robin
# suffers from client-side caching of DNS responses.

# You can force DNS round-robin mode:
docker service create \
  --name redis-rr \
  --network backend \
  --endpoint-mode dnsrr \
  redis:alpine
```

## Swarm Security

### Mutual TLS

Docker Swarm automatically encrypts all communication between nodes using mutual TLS. Each node gets a certificate, and both sides verify each other's identity before communicating.

```bash
# View the current TLS configuration
docker info --format '{{json .Swarm.Cluster}}' | jq '{
  ID: .ID,
  RootRotationInProgress: .RootRotationInProgress,
  TLSInfo: .TLSInfo
}'

# View a node's TLS certificate
docker node inspect self --format '{{json .Description.TLSInfo}}' | jq .
# {
#   "TrustRoot": "-----BEGIN CERTIFICATE-----\n...",
#   "CertIssuerSubject": "...",
#   "CertIssuerPublicKey": "..."
# }

# The certificate includes:
# - Node ID
# - Role (manager or worker)
# - Organization (swarm cluster ID)
# - Expiry time (default: 90 days)
```

### Certificate Rotation

```bash
# Check the current certificate rotation interval
docker swarm inspect --format '{{.Spec.CAConfig.NodeCertExpiry}}'
# 2160h0m0s  (90 days)

# Change the rotation interval (e.g., every 24 hours for high security)
docker swarm update --cert-expiry 24h
# Swarm updated.

# Rotate the root CA certificate (regenerate all node certificates)
docker swarm ca --rotate
# desired root digest: sha256:abc123...
# rotated TLS certificates:
#   manager1: 10.10.0.2:2377 [OK]
#   manager2: 10.10.0.3:2377 [OK]
#   worker1:  10.10.0.4:2377 [OK]
#   worker2:  10.10.0.5:2377 [OK]

# Use an external CA
docker swarm init --external-ca \
  protocol=cfssl,url=https://ca.example.com
```

<Callout type="info" title="Swarm Encryption Is Automatic">
Unlike Kubernetes, which requires you to configure TLS certificates manually or use tools like cert-manager, Docker Swarm handles all TLS setup automatically. When a node joins the swarm, it receives a certificate from the manager's CA. Certificates are automatically rotated before they expire. All control plane traffic (Raft consensus, task assignments, status updates) is encrypted by default. You do not need to configure anything.
</Callout>

### Encrypting Data Plane Traffic

```bash
# By default, overlay network data traffic is NOT encrypted
# (control plane traffic IS encrypted)

# Create an encrypted overlay network
docker network create \
  --driver overlay \
  --opt encrypted \
  secure-network

# All traffic between containers on this network is now encrypted
# using IPsec (ESP) in transport mode.

# Note: encryption adds ~10-15% CPU overhead for packet encryption
# Use only for sensitive data that crosses physical network boundaries
```

### Locking the Swarm (Autolock)

```bash
# Enable autolock — requires an unlock key to restart managers
docker swarm update --autolock=true
# Swarm updated.
# To unlock a swarm manager after it restarts, run the
# `docker swarm unlock` command and provide the following key:
#
#     SWMKEY-1-abc123def456ghi789jkl012mno345pqr678stu901vwx234

# rejoin the cluster and their Raft data is inaccessible.

# View the current unlock key
```

<Callout type="warning" title="Store the Unlock Key Securely">
If autolock is enabled and you lose the unlock key, restarted manager nodes cannot decrypt their Raft data and rejoin the cluster. Store the key in a secrets manager (HashiCorp Vault, AWS Secrets Manager, etc.), not in a text file on the same server. Rotate the key periodically with `docker swarm unlock-key --rotate`.

Practice advanced node management operations including labeling, availability changes, and service placement constraints.

**Hints:**

**Validation Criteria:**

- A service with constraints is created and tasks are placed correctly
- Changing availability to drain reschedules tasks
- Labels can be removed and the node reverts to unlabeled state

<CollapsibleSolution>

```bash
# ─── Setup: Ensure swarm is active ───
docker swarm init 2>/dev/null || true
NODE_ID=$(docker node ls --format '{{.ID}}' -q | head -1)
HOSTNAME=$(docker node inspect $NODE_ID --format '{{.Description.Hostname}}')

echo "=== Working with node: $HOSTNAME ($NODE_ID) ==="
echo ""

# ─── Step 1: Label Management ───
echo "=== Step 1: Add Labels ==="
docker node update --label-add tier=frontend $NODE_ID
docker node update --label-add region=us-east $NODE_ID
docker node update --label-add ssd=true $NODE_ID

echo "Current labels:"
docker node inspect $NODE_ID --format '{{range $k, $v := .Spec.Labels}}  {{$k}} = {{$v}}{{"\n"}}{{end}}'

echo ""
echo "=== Step 2: Create Service with Constraints ==="
docker service create \
  --name frontend-app \
  --constraint 'node.labels.tier == frontend' \
  --replicas 2 \
  --publish 8080:80 \
  nginx:alpine

sleep 3

echo "Service tasks:"
docker service ps frontend-app --format "table {{.Name}}\t{{.Node}}\t{{.CurrentState}}"

echo ""
echo "=== Step 3: Create Service with Non-matching Constraint ==="
docker service create \
  --name backend-app \
  --constraint 'node.labels.tier == backend' \
  --replicas 2 \
  nginx:alpine

sleep 5

echo "Backend tasks (should be pending — no matching nodes):"
docker service ps backend-app --format "table {{.Name}}\t{{.Node}}\t{{.CurrentState}}\t{{.Error}}" 2>/dev/null || \
  echo "  No tasks running (expected — no nodes match constraint)"

echo ""
echo "=== Step 4: Add Label to Fix Constraint ==="
docker node update --label-add tier=backend $NODE_ID
sleep 3
echo "Backend tasks (should now be running):"
docker service ps backend-app --format "table {{.Name}}\t{{.Node}}\t{{.CurrentState}}"

echo ""
echo "=== Step 5: Remove Labels ==="
docker node update --label-rm tier $NODE_ID
docker node update --label-rm region $NODE_ID
docker node update --label-rm ssd $NODE_ID

echo "Labels after removal:"
docker node inspect $NODE_ID --format '{{json .Spec.Labels}}'

echo ""
echo "=== Cleanup ==="
docker service rm frontend-app backend-app 2>/dev/null
echo "Services removed."
```

This exercise demonstrates how node labels serve as metadata for service placement. When no nodes match a service's constraints, tasks remain pending indefinitely. Adding the matching label causes tasks to be immediately scheduled. Removing labels does not automatically reschedule running tasks — they continue running until the next scheduling event (scale, update, or node failure).

## Key Takeaways

- **Docker Swarm** is Docker's built-in orchestration system — it requires no additional software and can be initialized with a single `docker swarm init` command
- **Choose Swarm** for small-to-medium deployments (3-20 nodes) where simplicity matters; choose Kubernetes for large-scale deployments requiring auto-scaling, service mesh, and a rich ecosystem
- **Manager nodes** maintain cluster state via Raft consensus, schedule tasks, and manage certificates; **worker nodes** execute tasks and report status
- **Raft consensus** requires a quorum (majority) of managers to be available; use **odd numbers** of managers (3 or 5); 3 managers tolerate 1 failure, 5 managers tolerate 2
- **Join tokens** authenticate new nodes — rotate them with `docker swarm join-token --rotate` if compromised; there are separate tokens for workers and managers
- **Node management** includes promoting workers to managers, draining nodes for maintenance (tasks are rescheduled), and labeling nodes for placement constraints
- **Overlay networks** span all swarm nodes using VXLAN encapsulation; add `--opt encrypted` for data plane encryption (IPsec)
- **The routing mesh** (ingress network) makes published service ports available on every node, automatically routing traffic to healthy task containers via IPVS
- **Service discovery** is built-in: services are accessible by name via Docker's internal DNS; VIP-based load balancing is the default and preferred mode
- **Mutual TLS** is automatic: all control plane traffic between nodes is encrypted; certificates are automatically issued and rotated; use autolock (`--autolock`) to protect Raft data at rest
