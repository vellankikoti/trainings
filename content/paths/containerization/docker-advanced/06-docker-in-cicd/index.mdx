## Docker-in-Docker vs Socket Mounting

When running Docker commands inside a CI pipeline, you face a fundamental question: how does the CI runner access a Docker daemon? There are two primary approaches, and choosing the wrong one can create security vulnerabilities or break your builds.

```
┌──────────────────────────────────────────────────────────────────────┐
│                        CI Runner Host                                │
│                                                                      │
│  ┌──────────────────────────────────────┐                            │
│  │     Option A: Docker-in-Docker       │                            │
│  │                                      │                            │
│  │  ┌────────────────────────────────┐  │                            │
│  │  │   CI Job Container             │  │                            │
│  │  │                                │  │                            │
│  │  │  ┌──────────────────────────┐  │  │                            │
│  │  │  │  Nested Docker Daemon    │  │  │                            │
│  │  │  │  (dockerd inside)        │  │  │                            │
│  │  │  │                          │  │  │                            │
│  │  │  │  docker build ...        │  │  │                            │
│  │  │  │  docker push ...         │  │  │                            │
│  │  │  └──────────────────────────┘  │  │                            │
│  │  │  --privileged required         │  │                            │
│  │  │  Isolated layer cache          │  │                            │
│  │  └────────────────────────────────┘  │                            │
│  └──────────────────────────────────────┘                            │
│                                                                      │
│  ┌──────────────────────────────────┐  ┌──────────────────────────┐  │
│  │   Option B: Socket Mounting      │  │   Host Docker Daemon     │  │
│  │                                  │  │                          │  │
│  │  ┌────────────────────────────┐  │  │  ┌────────────────────┐  │  │
│  │  │   CI Job Container         │  │  │  │  dockerd           │  │  │
│  │  │                            │──┼──┼─>│                    │  │  │
│  │  │  -v /var/run/docker.sock   │  │  │  │  docker build ...  │  │  │
│  │  │     :/var/run/docker.sock  │  │  │  │  docker push ...   │  │  │
│  │  │                            │  │  │  │                    │  │  │
│  │  │  Shares host daemon        │  │  │  │  Shared cache      │  │  │
│  │  └────────────────────────────┘  │  │  └────────────────────┘  │  │
│  └──────────────────────────────────┘  └──────────────────────────┘  │
└──────────────────────────────────────────────────────────────────────┘
```

### Docker-in-Docker (DinD)

DinD runs a full Docker daemon inside a container. Each CI job gets its own daemon, layer cache, and network namespace.

```bash
# Running DinD manually for testing
docker run --privileged --name dind-test -d \
  docker:27-dind

# Connect a client container to it
docker run --rm --link dind-test:docker \
  -e DOCKER_HOST=tcp://docker:2376 \
  -e DOCKER_TLS_CERTDIR=/certs \
  docker:27 \
  docker info

# In GitLab CI, DinD is a service
# services:
#   - docker:27-dind
```

**Advantages of DinD:**
- Complete isolation between jobs — no shared state
- Each job gets a clean Docker environment
- No risk of one job affecting another's builds
- Works well in Kubernetes-based CI runners

**Disadvantages of DinD:**
- Requires `--privileged` mode (security concern)
- No shared layer cache between jobs — slower builds
- Storage driver limitations (overlayfs inside overlayfs)
- Higher resource consumption

### Docker Socket Mounting

Socket mounting shares the host's Docker daemon with the CI job container by mounting `/var/run/docker.sock`.

```bash
# Socket mounting example
docker run --rm \
  -v /var/run/docker.sock:/var/run/docker.sock \
  docker:27 \
  docker ps

# The container can see ALL containers on the host
# Including other CI jobs' containers
```

**Advantages of Socket Mounting:**
- Shared layer cache — faster builds
- Lower resource consumption
- No `--privileged` required
- Simpler setup

**Disadvantages of Socket Mounting:**
- CI jobs can see and manipulate other containers on the host
- A compromised job can access the Docker daemon and escape to the host
- Build artifacts may collide between concurrent jobs
- Not truly isolated

<Callout type="warning" title="Socket Mounting Grants Root Access">
Mounting the Docker socket into a container is equivalent to giving that container root access on the host. Any process inside the container can run `docker run -v /:/host --privileged alpine` and access the entire host filesystem. In multi-tenant CI environments, always use DinD or a rootless alternative like Kaniko.
</Callout>

### Choosing the Right Approach

| Factor | DinD | Socket Mounting |
|--------|------|-----------------|
| **Isolation** | Full isolation | Shared daemon |
| **Layer cache** | Per-job (cold) | Shared (warm) |
| **Security** | Needs `--privileged` | Grants host access |
| **Performance** | Slower first builds | Faster with cache |
| **Multi-tenant** | Safe | Unsafe |
| **Complexity** | More setup | Simpler |
| **Kubernetes** | Works well | Requires DaemonSet |

### Rootless Alternatives

For environments where neither DinD nor socket mounting is acceptable, consider rootless build tools:

```bash
# Kaniko — builds images without a Docker daemon
# Used in Kubernetes-based CI (no --privileged needed)
docker run --rm \
  -v $(pwd):/workspace \
  gcr.io/kaniko-project/executor:latest \
  --dockerfile /workspace/Dockerfile \
  --destination registry.example.com/myapp:latest \
  --context /workspace

# Buildah — OCI image builder without a daemon
buildah bud -t myapp:latest .
buildah push myapp:latest docker://registry.example.com/myapp:latest
```

## Building Images in GitHub Actions

GitHub Actions is the most popular CI platform for open-source projects. Docker provides official actions that handle authentication, buildx setup, and caching.

### Basic Docker Build Workflow

```yaml
# .github/workflows/docker-build.yml
name: Build and Push Docker Image

on:
  push:
    branches: [main]
    tags: ['v*']
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      # Step 1: Check out the repository
      - name: Checkout repository
        uses: actions/checkout@v4

      # Step 2: Set up Docker Buildx
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # Step 3: Log in to the container registry
      - name: Log in to GitHub Container Registry
        if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # Step 4: Extract metadata for tags and labels
      - name: Extract Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix=

      # Step 5: Build and push
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

### Understanding Layer Caching in CI

Layer caching is the single biggest performance optimization for Docker builds in CI. Without caching, every build downloads base images and reinstalls dependencies from scratch.

```
Without Cache                          With Cache
─────────────                          ──────────
Step 1: FROM node:20-alpine            Step 1: FROM node:20-alpine
  ↓ Pull 80MB base image                 ↓ CACHED (already in cache)
  Time: 15s                              Time: 0s

Step 2: COPY package*.json ./          Step 2: COPY package*.json ./
  ↓ Copy files                           ↓ CACHED (files unchanged)
  Time: 1s                              Time: 0s

Step 3: RUN npm ci                     Step 3: RUN npm ci
  ↓ Install all dependencies             ↓ CACHED (package.json unchanged)
  Time: 45s                              Time: 0s

Step 4: COPY . .                       Step 4: COPY . .
  ↓ Copy source code                     ↓ Source changed, rebuild from here
  Time: 2s                              Time: 2s

Step 5: RUN npm run build             Step 5: RUN npm run build
  ↓ Build application                   ↓ Must rebuild (cache invalidated)
  Time: 30s                              Time: 30s

Total: ~93s                            Total: ~32s
       ^^^^^^                                 ^^^^^
       Every single build                     Only changed layers
```

GitHub Actions supports three cache backends:

```yaml
# Option 1: GitHub Actions cache (gha) — simplest, recommended
- name: Build with GHA cache
  uses: docker/build-push-action@v5
  with:
    context: .
    cache-from: type=gha
    cache-to: type=gha,mode=max

# Option 2: Registry cache — stores cache in your container registry
- name: Build with registry cache
  uses: docker/build-push-action@v5
  with:
    context: .
    cache-from: type=registry,ref=ghcr.io/myorg/myapp:buildcache
    cache-to: type=registry,ref=ghcr.io/myorg/myapp:buildcache,mode=max

# Option 3: Local cache — useful for self-hosted runners
- name: Build with local cache
  uses: docker/build-push-action@v5
  with:
    context: .
    cache-from: type=local,src=/tmp/.buildx-cache
    cache-to: type=local,dest=/tmp/.buildx-cache-new,mode=max
```

<Callout type="tip" title="Use mode=max for Complete Caching">
The `mode=max` option caches all layers from all build stages, including intermediate layers from multi-stage builds. Without it, only the final stage layers are cached. This can make a dramatic difference for multi-stage builds where the build stage has expensive dependency installation steps.
</Callout>

## GitLab CI with Docker Executor

GitLab CI has first-class Docker support through its Docker executor and the DinD service container pattern.

### GitLab CI Docker Build Pipeline

```yaml
# .gitlab-ci.yml
stages:
  - lint
  - build
  - scan
  - test
  - push

variables:
  DOCKER_HOST: tcp://docker:2376
  DOCKER_TLS_CERTDIR: "/certs"
  DOCKER_TLS_VERIFY: 1
  DOCKER_CERT_PATH: "$DOCKER_TLS_CERTDIR/client"
  IMAGE_TAG: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

# Use DinD as a service for all jobs that need Docker
.docker_base:
  image: docker:27
  services:
    - docker:27-dind
  before_script:
    - docker info
    - echo "$CI_REGISTRY_PASSWORD" | docker login -u "$CI_REGISTRY_USER" --password-stdin $CI_REGISTRY

# Lint the Dockerfile
lint:
  stage: lint
  image: hadolint/hadolint:latest-alpine
  script:
    - hadolint Dockerfile
  rules:
    - changes:
        - Dockerfile

# Build the Docker image
build:
  extends: .docker_base
  stage: build
  script:
    - docker build
        --build-arg BUILD_DATE=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
        --build-arg VCS_REF=$CI_COMMIT_SHORT_SHA
        --build-arg VERSION=$CI_COMMIT_TAG
        -t $IMAGE_TAG
        -t $CI_REGISTRY_IMAGE:build-$CI_PIPELINE_ID
        .
    - docker save $IMAGE_TAG | gzip > image.tar.gz
  artifacts:
    paths:
      - image.tar.gz
    expire_in: 1 hour

# Scan the image for vulnerabilities
scan:
  extends: .docker_base
  stage: scan
  script:
    - docker load < image.tar.gz
    - docker run --rm
        -v /var/run/docker.sock:/var/run/docker.sock
        aquasec/trivy:latest image
        --exit-code 1
        --severity HIGH,CRITICAL
        --no-progress
        $IMAGE_TAG
  allow_failure: false

# Run integration tests using the built image
test:
  extends: .docker_base
  stage: test
  script:
    - docker load < image.tar.gz
    - docker run -d --name test-app -p 8080:8080 $IMAGE_TAG
    - sleep 5
    - docker run --rm --network host curlimages/curl:latest
        curl -f http://localhost:8080/health
    - docker logs test-app
    - docker rm -f test-app

# Push to registry on main branch only
push:
  extends: .docker_base
  stage: push
  script:
    - docker load < image.tar.gz
    - docker tag $IMAGE_TAG $CI_REGISTRY_IMAGE:latest
    - docker push $IMAGE_TAG
    - docker push $CI_REGISTRY_IMAGE:latest
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

<Callout type="info" title="GitLab Container Registry">
GitLab provides a built-in container registry at `registry.gitlab.com/your-group/your-project`. Authentication uses `CI_REGISTRY_USER` and `CI_REGISTRY_PASSWORD` variables that are automatically available in CI jobs. You do not need to configure any external registry for basic workflows.
</Callout>

## Jenkins with Docker Agents

Jenkins can use Docker containers as build agents, running each pipeline stage in an isolated container.

### Jenkinsfile with Docker Agents

```groovy
// Jenkinsfile
pipeline {
    agent none

    environment {
        REGISTRY = 'registry.example.com'
        IMAGE_NAME = 'myorg/webapp'
        REGISTRY_CREDENTIALS = 'registry-creds'
    }

    stages {
        stage('Lint Dockerfile') {
            agent {
                docker {
                    image 'hadolint/hadolint:latest-alpine'
                    args '--entrypoint=""'
                }
            }
            steps {
                sh 'hadolint Dockerfile'
            }
        }

        stage('Build Image') {
            agent any
            steps {
                script {
                    def commitHash = sh(
                        script: 'git rev-parse --short HEAD',
                        returnStdout: true
                    ).trim()

                    docker.withRegistry(
                        "https://${REGISTRY}",
                        REGISTRY_CREDENTIALS
                    ) {
                        def image = docker.build(
                            "${REGISTRY}/${IMAGE_NAME}:${commitHash}",
                            "--build-arg BUILD_DATE=\$(date -u +%Y-%m-%dT%H:%M:%SZ) " +
                            "--build-arg VCS_REF=${commitHash} " +
                            "--no-cache ."
                        )

                        // Store for later stages
                        env.BUILT_IMAGE = "${REGISTRY}/${IMAGE_NAME}:${commitHash}"
                        env.COMMIT_HASH = commitHash
                    }
                }
            }
        }

        stage('Security Scan') {
            agent any
            steps {
                sh """
                    docker run --rm \
                      -v /var/run/docker.sock:/var/run/docker.sock \
                      aquasec/trivy:latest image \
                      --exit-code 1 \
                      --severity CRITICAL \
                      --no-progress \
                      ${env.BUILT_IMAGE}
                """
            }
        }

        stage('Integration Test') {
            agent any
            steps {
                sh """
                    docker run -d --name test-\${BUILD_NUMBER} \
                      -p 0:8080 ${env.BUILT_IMAGE}
                    sleep 5

                    PORT=\$(docker port test-\${BUILD_NUMBER} 8080 | cut -d: -f2)
                    curl -f http://localhost:\${PORT}/health

                    docker rm -f test-\${BUILD_NUMBER}
                """
            }
        }

        stage('Push') {
            agent any
            when {
                branch 'main'
            }
            steps {
                script {
                    docker.withRegistry(
                        "https://${REGISTRY}",
                        REGISTRY_CREDENTIALS
                    ) {
                        def image = docker.image(env.BUILT_IMAGE)
                        image.push()
                        image.push('latest')
                    }
                }
            }
        }
    }

    post {
        always {
            sh 'docker system prune -f --filter "until=24h" || true'
        }
        failure {
            echo "Pipeline failed. Check the logs above."
        }
    }
}
```

Vulnerability scanning catches known CVEs before images reach production. Multiple tools exist, each with different strengths.

```bash
docker run --rm \
  -v /var/run/docker.sock:/var/run/docker.sock \

  --severity HIGH,CRITICAL \
```

### Trivy in GitHub Actions

```yaml
# .github/workflows/scan.yml
- name: Run Trivy vulnerability scanner
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
    format: 'sarif'
    output: 'trivy-results.sarif'
    severity: 'CRITICAL,HIGH'
    exit-code: '1'

- name: Upload Trivy scan results to GitHub Security
  uses: github/codeql-action/upload-sarif@v3
  if: always()
  with:
    sarif_file: 'trivy-results.sarif'
```

### Snyk Container Scanning

```yaml
# In GitHub Actions
- name: Run Snyk container scan
  uses: snyk/actions/docker@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
  with:
    image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
    args: --severity-threshold=high --file=Dockerfile

# Snyk CLI locally
snyk container test myapp:latest --severity-threshold=high
snyk container monitor myapp:latest  # Continuous monitoring
```

### Docker Scout

```bash
# Docker Scout is built into Docker Desktop and Docker Hub
docker scout cves myapp:latest

# Quickview — summary of vulnerabilities
docker scout quickview myapp:latest

# Compare two images
docker scout compare --to myapp:v1.0 myapp:v1.1

# In GitHub Actions
- name: Docker Scout scan
  uses: docker/scout-action@v1
  with:
    command: cves
    image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
    sarif-file: scout-results.sarif
    exit-code: true
    only-severities: critical,high
```

### Scanner Comparison

| Feature | Trivy | Snyk | Docker Scout |
|---------|-------|------|--------------|
| **Cost** | Free / OSS | Free tier + paid | Free tier + paid |
| **OS packages** | Yes | Yes | Yes |
| **Language deps** | Yes | Yes | Yes |
| **IaC scanning** | Yes | Yes | No |
| **License check** | Yes | Yes | Yes |
| **SBOM generation** | Yes | Yes | Yes |
| **Fix suggestions** | Basic | Detailed | Detailed |
| **CI integration** | All platforms | All platforms | GitHub, GitLab |
| **Speed** | Fast | Medium | Fast |

<Callout type="warning" title="Fail the Build on Critical CVEs">
Always configure your scanner to fail the CI pipeline when critical vulnerabilities are found. Without an enforced gate, scan results become informational noise that developers ignore. Set `--exit-code 1` in Trivy or `--severity-threshold=critical` in Snyk to make the pipeline fail and block the deployment.
</Callout>

## Multi-Platform Builds

Modern applications need to run on multiple CPU architectures. Apple Silicon Macs use arm64, while most cloud servers run amd64. Docker buildx makes it possible to build for both from a single machine.

### Setting Up Buildx for Multi-Platform

```bash
# Check current buildx builders
docker buildx ls
# NAME/NODE    DRIVER/ENDPOINT    STATUS    BUILDKIT    PLATFORMS
# default      docker
#   default    default            running   v0.12.4     linux/amd64

# Create a new builder with multi-platform support
docker buildx create --name multiplatform \
  --driver docker-container \
  --bootstrap \
  --use

# Verify — you should see multiple platforms
docker buildx inspect multiplatform
# Platforms: linux/amd64, linux/arm64, linux/arm/v7, linux/arm/v6,
#            linux/386, linux/ppc64le, linux/s390x

# Build for multiple platforms and push to registry
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag registry.example.com/myapp:v1.0.0 \
  --push \
  .

# Build for multiple platforms and load locally (single platform only)
docker buildx build \
  --platform linux/amd64 \
  --tag myapp:v1.0.0 \
  --load \
  .
```

<Callout type="info" title="How Multi-Platform Builds Work">
Under the hood, buildx uses QEMU emulation to run build steps for foreign architectures. When you build for `linux/arm64` on an `amd64` host, the `RUN` instructions in your Dockerfile are executed inside a QEMU-emulated arm64 environment. This is slower than native builds but produces genuine arm64 binaries. For performance-critical builds, you can use native arm64 build nodes instead of QEMU emulation.
</Callout>

### Multi-Platform in GitHub Actions

```yaml
# .github/workflows/multi-platform.yml
name: Multi-Platform Build

on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to DockerHub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build and push multi-platform image
        uses: docker/build-push-action@v5
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: true
          tags: |
            ${{ secrets.DOCKERHUB_USERNAME }}/myapp:${{ github.ref_name }}
            ${{ secrets.DOCKERHUB_USERNAME }}/myapp:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Inspect manifest
        run: |
          docker buildx imagetools inspect \
            ${{ secrets.DOCKERHUB_USERNAME }}/myapp:${{ github.ref_name }}
```

### Platform-Specific Dockerfile Instructions

Sometimes you need different build steps for different architectures:

```dockerfile
FROM --platform=$BUILDPLATFORM node:20-alpine AS builder

# These args are automatically set by buildx
ARG TARGETPLATFORM
ARG BUILDPLATFORM
ARG TARGETOS
ARG TARGETARCH

# Platform-aware dependency installation
RUN echo "Building on $BUILDPLATFORM for $TARGETPLATFORM"

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

# Production stage — uses target platform
FROM node:20-alpine

WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./

RUN addgroup -g 1001 -S appgroup && \
    adduser -u 1001 -S appuser -G appgroup

USER appuser

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

## CI Best Practices

### Reproducible Builds

```dockerfile
# BAD: Non-reproducible
FROM node:latest                    # Tag changes over time
RUN apt-get update && apt-get install -y curl  # Versions vary
COPY . .
RUN npm install                     # Uses ^ ranges from package.json

# GOOD: Reproducible
FROM node:20.11.0-alpine3.19@sha256:abc123...  # Pinned digest
RUN apk add --no-cache curl=8.5.0-r0           # Pinned version
```

```yaml
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix=

        with:

          tags: ${{ env.IMAGE_NAME }}:scan
          cache-from: type=gha

            BUILD_DATE=${{ github.event.head_commit.timestamp }}
            VCS_REF=${{ github.sha }}

        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.IMAGE_NAME }}:scan
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'

      - name: Upload scan results
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: 'trivy-results.sarif'

      - name: Build and push multi-platform

        with:
          context: .

            VCS_REF=${{ github.sha }}
```

<Callout type="tip" title="Separate Build and Push Steps for Scanning">
Build the image with `--load` first (single platform), run your security scan on it, and only then build and push the multi-platform version with `--push`. This avoids pushing vulnerable images to your registry. The multi-platform build reuses the cached layers from the first build, so the overhead is minimal.

## Key Takeaways

- **Docker-in-Docker (DinD)** provides full isolation between CI jobs but requires `--privileged` mode and has no shared layer cache; **socket mounting** is simpler and faster but shares the host Docker daemon and grants equivalent root access
- **Rootless alternatives** like Kaniko and Buildah build images without a Docker daemon and without `--privileged`, making them ideal for Kubernetes-based CI runners
- **GitHub Actions** provides official Docker actions (`docker/build-push-action`, `docker/metadata-action`) that handle buildx setup, caching, tagging, and registry authentication
- **Layer caching** (`cache-from: type=gha`, `cache-to: type=gha,mode=max`) is the single biggest performance optimization for CI Docker builds — it can reduce build times by 60-80%
- **Vulnerability scanning** with Trivy, Snyk, or Docker Scout should run on every build, and the pipeline must fail on critical CVEs to prevent vulnerable images from reaching production
- **Multi-platform builds** with buildx and QEMU let you build for `linux/amd64` and `linux/arm64` from a single CI runner, producing a manifest list that Docker automatically resolves
- **Reproducible builds** require pinning base image versions (ideally by digest), using lockfiles (`npm ci` instead of `npm install`), and embedding build metadata as OCI labels
- **Never tag images only as `:latest`** — use semantic versioning, commit SHAs, and branch names so you can always trace a running container back to its source code
- **Separate the scan step from the push step** — build with `--load` for scanning, then build and push multi-platform only after all quality gates pass
