<Callout type="story" title="The Story">
Your team just got approval to containerize the company's main application. The project kicks off Monday, and your manager asks you to prepare the development machines. There are three developers on Ubuntu, two on macOS, and one on a CentOS staging server. You need Docker running correctly on all of them by Monday morning. One developer tries to install Docker using `apt install docker` and gets a completely wrong package — a system tray docking application, not Docker Engine. Another tries `snap install docker` and runs into permission issues. ## Prerequisites and System Requirements

Before installing Docker, make sure your system meets these requirements:

### Linux Requirements

| Requirement | Details |
|---|---|
| **Architecture** | x86_64 (amd64), ARM64 (aarch64), ARM (armhf), s390x |
| **Kernel version** | 3.10 or higher (4.x+ recommended) |
| **Storage driver** | overlay2 (requires kernel 4.0+, d_type support in filesystem) |
| **iptables** | Version 1.4 or higher |
| **Filesystem** | ext4 or xfs recommended for overlay2 |
| **RAM** | Minimum 512 MB (2 GB+ recommended) |

```bash
# Check your kernel version:
$ uname -r
5.15.0-91-generic

# Check your architecture:
$ uname -m
x86_64

# Check if your filesystem supports d_type (needed for overlay2):
$ xfs_info / 2>/dev/null | grep ftype || stat -f -c %T /
ext2/ext3
```

### macOS Requirements

| Requirement | Details |
|---|---|
| **macOS version** | 12 (Monterey) or newer |
| **Hardware** | Apple Silicon (M1/M2/M3) or Intel |
| **RAM** | 4 GB minimum |
| **Disk** | ~2 GB for Docker Desktop |

### Windows Requirements

| Requirement | Details |
|---|---|
| **Windows version** | Windows 10 (Build 19041+) or Windows 11 |
| **Backend** | WSL 2 (recommended) or Hyper-V |
| **RAM** | 4 GB minimum |
| **BIOS** | Hardware virtualization enabled |

<Callout type="warning" title="Do NOT Install the Wrong Package">
On Linux, there are several packages named "docker" that are NOT Docker Engine:

```bash
# WRONG — this is a system tray docking application:
sudo apt install docker

# WRONG — this is an outdated version:
sudo apt install docker.io

# CORRECT — Docker Engine from Docker's official repository:
# (installation steps below)
```

Always install from Docker's official repositories to get the latest, supported version.
</Callout>

## Installing Docker Engine on Ubuntu/Debian

This is the most common installation path. Follow these steps exactly.

### Step 1: Remove Old Versions

Old versions of Docker may be installed under different package names. Remove them first:

```bash
# Remove any old Docker packages:
sudo apt-get remove -y docker docker-engine docker.io containerd runc 2>/dev/null

# This is safe to run even if none of these packages are installed.
# Existing images, containers, volumes, and networks in /var/lib/docker/
# are preserved.
```

### Step 2: Install Prerequisites

Install packages needed to use Docker's repository over HTTPS:

```bash
# Update the package index:
sudo apt-get update

# Install prerequisite packages:
sudo apt-get install -y \
    ca-certificates \
    curl \
    gnupg \
    lsb-release
```

### Step 3: Add Docker's Official GPG Key

```bash
# Create the keyrings directory:
sudo install -m 0755 -d /etc/apt/keyrings

# Download Docker's official GPG key:
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
    -o /etc/apt/keyrings/docker.asc

# Set proper permissions:
sudo chmod a+r /etc/apt/keyrings/docker.asc
```

<Callout type="info" title="For Debian Instead of Ubuntu">
If you are on Debian, replace `ubuntu` with `debian` in the URLs:

```bash
# Debian GPG key:
sudo curl -fsSL https://download.docker.com/linux/debian/gpg \
    -o /etc/apt/keyrings/docker.asc

# Debian repository (Step 4):
echo "deb [arch=$(dpkg --print-architecture) \
  signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/debian \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```
</Callout>

### Step 4: Set Up the Docker Repository

```bash
# Add Docker's official apt repository:
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Verify the repository was added:
cat /etc/apt/sources.list.d/docker.list
# Should show something like:
# deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.asc]
#   https://download.docker.com/linux/ubuntu jammy stable
```

### Step 5: Install Docker Engine

```bash
# Update package index with the new repository:
sudo apt-get update

# Install Docker Engine, CLI, containerd, and plugins:
sudo apt-get install -y \
    docker-ce \
    docker-ce-cli \
    containerd.io \
    docker-buildx-plugin \
    docker-compose-plugin
```

Here is what each package provides:

| Package | What It Provides |
|---|---|
| **docker-ce** | Docker Engine (daemon) |
| **docker-ce-cli** | Docker CLI (`docker` command) |
| **containerd.io** | containerd runtime |
| **docker-buildx-plugin** | BuildKit-based build system |
| **docker-compose-plugin** | Docker Compose V2 (`docker compose`) |

## Installing Docker Engine on CentOS/RHEL

### Step 1: Remove Old Versions

```bash
# Remove any old Docker packages:
sudo yum remove -y docker \
    docker-client \
    docker-client-latest \
    docker-common \
    docker-latest \
    docker-latest-logrotate \
    docker-logrotate \
    docker-engine 2>/dev/null
```

### Step 2: Install Prerequisites and Add Repository

```bash
# Install yum-utils for yum-config-manager:
sudo yum install -y yum-utils

# Add Docker's official repository:
sudo yum-config-manager --add-repo \
    https://download.docker.com/linux/centos/docker-ce.repo
```

<Callout type="info" title="For RHEL Instead of CentOS">
For Red Hat Enterprise Linux, the repository URL is the same (CentOS repo works on RHEL). However, you may need to enable the `extras` repo for some dependencies:

```bash
# RHEL 8/9:
sudo yum-config-manager --add-repo \
    https://download.docker.com/linux/rhel/docker-ce.repo
```
</Callout>

### Step 3: Install Docker Engine

```bash
# Install Docker Engine, CLI, containerd, and plugins:
sudo yum install -y \
    docker-ce \
    docker-ce-cli \
    containerd.io \
    docker-buildx-plugin \
    docker-compose-plugin

# If prompted to accept the GPG key, verify the fingerprint:
# 060A 61C5 1B55 8A7F 742B 77AA C52F EB6B 621E 9F35
```

### Step 4: Start and Enable Docker

On CentOS/RHEL, Docker does not start automatically after installation:

```bash
# Start the Docker daemon:
sudo systemctl start docker

# Enable Docker to start on boot:
sudo systemctl enable docker

# Verify it is running:
$ sudo systemctl status docker
● docker.service - Docker Application Container Engine
     Loaded: loaded (/usr/lib/systemd/system/docker.service; enabled)
     Active: active (running) since Mon 2025-01-15 08:30:00 UTC
       Docs: https://docs.docker.com
   Main PID: 1234 (dockerd)
      Tasks: 12
     Memory: 98.4M
     CGroup: /system.slice/docker.service
             ├─1234 /usr/bin/dockerd -H fd:// --containerd=/run/containerd/containerd.sock
             └─1235 /usr/bin/containerd

# Run the hello-world test:
$ sudo docker run hello-world
Hello from Docker!
...
```

<Callout type="warning" title="Firewalld Compatibility">
On CentOS/RHEL with firewalld, Docker may have issues with iptables rules. If containers cannot reach the internet, try:

```bash
# Option 1: Restart Docker after firewalld:
sudo systemctl restart docker

# Option 2: Add Docker's interface to the trusted zone:
sudo firewall-cmd --permanent --zone=trusted --add-interface=docker0
sudo firewall-cmd --reload
```
</Callout>

**Installation Steps:**

1. **Enable WSL 2** (recommended backend):

```powershell

4. Select "Use WSL 2 instead of Hyper-V" during installation
5. Follow the setup wizard

...
```

<Callout type="tip" title="WSL 2 vs Hyper-V Backend">
Docker Desktop on Windows offers two backends:

| Feature | WSL 2 Backend | Hyper-V Backend |
|---|---|---|
| **Performance** | Better (native Linux kernel) | Good |
| **Memory usage** | Dynamic (grows/shrinks) | Fixed allocation |
| **File sharing** | Fast (native Linux FS) | Slower (SMB mount) |
| **Linux integration** | Full WSL distro access | Limited |
| **Windows version** | Home, Pro, Enterprise | Pro, Enterprise only |

**Recommendation**: Use WSL 2 — it is faster, uses less memory, and works on all Windows editions.
</Callout>

## Post-Installation Steps (Linux)

After installing Docker on Linux, there are important configuration steps to complete.

### Run Docker Without sudo

By default, the Docker daemon runs as root and requires `sudo` for every command. Add your user to the `docker` group to avoid this:

```bash
# Create the docker group (may already exist):
sudo groupadd docker 2>/dev/null

# Add your user to the docker group:
sudo usermod -aG docker $USER

# IMPORTANT: Log out and back in for the group change to take effect.
# Or run this to activate the group in the current session:
newgrp docker

# Verify it works without sudo:
$ docker run hello-world
Hello from Docker!
...

# Verify your groups:
$ groups
username sudo docker
```

<Callout type="warning" title="Security Implication of the Docker Group">
Adding a user to the `docker` group grants them the equivalent of root access to the host. Anyone in the docker group can:

```bash
# Mount the host filesystem into a container:
docker run -v /:/host -it ubuntu bash
# Now they can read/write ANY file on the host as root!
```

Only add trusted users to the docker group. For shared servers, consider using rootless Docker or Podman instead.
</Callout>

### Configure Docker to Start on Boot

```bash
# Enable Docker daemon to start on boot:
sudo systemctl enable docker.service
sudo systemctl enable containerd.service

# Verify:
$ systemctl is-enabled docker
enabled
$ systemctl is-enabled containerd
enabled
```

### Configure Default Logging

Docker's default logging driver (`json-file`) can fill up disk space because it stores logs indefinitely. Configure log rotation:

```bash
# Create or edit the daemon configuration file:
sudo mkdir -p /etc/docker

sudo tee /etc/docker/daemon.json <<'EOF'
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
EOF

# Restart Docker to apply:
sudo systemctl restart docker
```

This limits each container's log file to 10 MB and keeps at most 3 rotated files (30 MB max per container).

## Docker Daemon Configuration

The Docker daemon is configured through `/etc/docker/daemon.json`. This file controls everything from storage drivers to networking defaults.

### Common daemon.json Configuration

```json
{
  "data-root": "/var/lib/docker",
  "storage-driver": "overlay2",
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "default-address-pools": [
    {
      "base": "172.17.0.0/16",
      "size": 24
    }
  ],
  "dns": ["8.8.8.8", "8.8.4.4"],
  "live-restore": true,
  "userland-proxy": false,
  "experimental": false,
  "metrics-addr": "0.0.0.0:9323",
  "max-concurrent-downloads": 10,
  "max-concurrent-uploads": 5
}
```

### Configuration Options Explained

| Option | Description | Default |
|---|---|---|
| `data-root` | Directory where Docker stores all data | `/var/lib/docker` |
| `storage-driver` | Filesystem driver for layers | `overlay2` |
| `log-driver` | Default logging driver for containers | `json-file` |
| `log-opts` | Options for the logging driver | No limits |
| `dns` | DNS servers for containers | Host DNS |
| `live-restore` | Keep containers running during daemon downtime | `false` |
| `userland-proxy` | Use userland proxy for port mapping | `true` |
| `metrics-addr` | Expose Prometheus metrics | Disabled |
| `max-concurrent-downloads` | Parallel layer downloads per pull | `3` |
| `max-concurrent-uploads` | Parallel layer uploads per push | `5` |
| `insecure-registries` | Registries to allow HTTP connections | `[]` |
| `registry-mirrors` | Mirror registries for Docker Hub | `[]` |

<Callout type="info" title="live-restore Is Critical for Production">
The `live-restore` option allows containers to keep running when the Docker daemon is stopped or restarted. Without this, restarting Docker kills all containers.

```json
{
  "live-restore": true
}
```

This is essential for production servers where you need to upgrade Docker without downtime. Enable it on every production host.
</Callout>

### Moving Docker's Data Directory

If `/var/lib/docker` is on a small root partition, move Docker's data to a larger disk:

```bash
# Stop Docker:
sudo systemctl stop docker
sudo systemctl stop containerd

# Create the new directory:
sudo mkdir -p /data/docker

# Move existing data:
sudo rsync -aP /var/lib/docker/ /data/docker/

# Configure Docker to use the new location:
sudo tee /etc/docker/daemon.json <<'EOF'
{
  "data-root": "/data/docker",
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "live-restore": true
}
EOF

# Start Docker:
sudo systemctl start docker

# Verify:
$ docker info --format '{{.DockerRootDir}}'
/data/docker

# Once verified, remove old data:
sudo rm -rf /var/lib/docker
```

### Configuring an Insecure Registry

For development environments with a private registry that does not have TLS:

```json
{
  "insecure-registries": ["registry.dev.internal:5000"]
}
```

<Callout type="warning" title="Never Use Insecure Registries in Production">
The `insecure-registries` option disables TLS verification, making connections vulnerable to man-in-the-middle attacks. Use it only for local development. In production, always configure proper TLS certificates for your registries.
</Callout>

### Configuring Registry Mirrors

If Docker Hub pulls are slow, configure a mirror:

```json
{
  "registry-mirrors": [
    "https://mirror.gcr.io",
    "https://registry-mirror.company.com"
  ]
}
```

### Applying Configuration Changes

After editing `daemon.json`, you must restart Docker:

```bash
# Validate the JSON syntax first (avoid breaking Docker):
python3 -c "import json; json.load(open('/etc/docker/daemon.json'))" && echo "Valid JSON" || echo "Invalid JSON!"

# Restart Docker daemon:
sudo systemctl restart docker

# Check that Docker started successfully:
sudo systemctl status docker

# If Docker fails to start, check the logs:
sudo journalctl -u docker.service --no-pager -n 50
```

## Troubleshooting Common Installation Issues

### Issue 1: "Cannot connect to the Docker daemon"

```bash
$ docker ps
Cannot connect to the Docker daemon at unix:///var/run/docker.sock.
Is the docker daemon running?

# Fix: Start the Docker daemon
sudo systemctl start docker

# Check if it started:
sudo systemctl status docker

# If it failed, check the logs:
sudo journalctl -u docker.service --no-pager -n 30
```

### Issue 2: "Permission denied" when running Docker commands

```bash
$ docker ps
Got permission denied while trying to connect to the Docker daemon socket
at unix:///var/run/docker.sock: ...dial unix /var/run/docker.sock:
connect: permission denied

# Fix: Add your user to the docker group
sudo usermod -aG docker $USER

# IMPORTANT: Log out and log back in, or run:
newgrp docker

# Verify:
docker ps
```

### Issue 3: Docker daemon fails to start with storage driver errors

```bash
# Check the daemon logs:
$ sudo journalctl -u docker.service --no-pager | grep -i error
level=error msg="failed to mount overlay: invalid argument"

# Fix: Ensure filesystem supports overlay2
# Check the filesystem type:
$ df -T /var/lib/docker
Filesystem     Type  1K-blocks     Used Available Use% Mounted on
/dev/sda1      ext4  102400000  5000000  92179024   6% /

# If using XFS, ensure ftype=1:
$ xfs_info /var/lib/docker | grep ftype
# ftype must be 1 for overlay2
```

### Issue 4: Containers cannot reach the internet

```bash
# Test DNS from inside a container:
$ docker run --rm alpine ping -c 3 google.com
ping: bad address 'google.com'

# Fix 1: Configure DNS in daemon.json
sudo tee /etc/docker/daemon.json <<'EOF'
{
  "dns": ["8.8.8.8", "8.8.4.4"]
}
EOF
sudo systemctl restart docker

# Fix 2: Check iptables/firewall:
sudo iptables -L -n | grep DOCKER
# Docker should have its own chain

# Fix 3: On CentOS/RHEL with firewalld:
sudo firewall-cmd --permanent --zone=trusted --add-interface=docker0
sudo firewall-cmd --reload
sudo systemctl restart docker
```

### Issue 5: "No space left on device"

## Key Takeaways

- Always install Docker from Docker's official repository — the `docker` or `docker.io` packages from default OS repositories are outdated or incorrect
- Docker Engine on Linux requires five packages: docker-ce, docker-ce-cli, containerd.io, docker-buildx-plugin, and docker-compose-plugin
- On Ubuntu/Debian, installation uses `apt` with Docker's GPG key and repository; on CentOS/RHEL, it uses `yum` with Docker's repo
- Docker Desktop provides Docker Engine, CLI, Compose, and Kubernetes in one package for macOS and Windows, running a lightweight Linux VM behind the scenes
- Post-installation steps are critical: add users to the docker group, enable Docker on boot, and configure log rotation in daemon.json
- The `/etc/docker/daemon.json` file controls daemon behavior including storage driver, logging, DNS, live-restore, and performance tuning
- Enable `live-restore` on all production hosts so containers survive daemon restarts
- Always verify installation with `docker version`, `docker info`, `docker run hello-world`, and `docker compose version`
- Common issues include wrong packages, permission errors, firewall conflicts, and disk space — all have straightforward fixes documented in this lesson
