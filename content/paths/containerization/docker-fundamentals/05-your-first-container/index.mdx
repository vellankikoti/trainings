## docker run hello-world: What Actually Happens

The `docker run hello-world` command is the simplest Docker command, but understanding what happens behind the scenes teaches you the fundamentals of every container operation.

```bash
$ docker run hello-world
```

Here is exactly what happens, step by step:

```
What happens when you run "docker run hello-world":
═══════════════════════════════════════════════════════════════════

Step 1: Docker CLI sends "run hello-world" to the Docker daemon
        ┌──────────┐  POST /containers/create  ┌──────────┐
        │ docker   │ ─────────────────────────→ │ dockerd  │
        │ CLI      │                             │          │
        └──────────┘                             └────┬─────┘
                                                      │
Step 2: Daemon checks for "hello-world:latest" image locally
        ┌──────────────────────────────────────────────────┐
        │  Local Image Cache: hello-world NOT FOUND         │
        └──────────────────────────────────────────────────┘
                                                      │
Step 3: Daemon pulls the image from Docker Hub
        ┌──────────┐  GET /v2/library/hello-world  ┌──────────┐
        │ dockerd  │ ────────────────────────────→ │ Docker   │
        │          │ ←────────────────────────────  │ Hub      │
        │          │  (downloads image layers)      │          │
        └────┬─────┘                                └──────────┘
             │
Step 4: Daemon creates a container from the image
        • Creates a writable layer on top of image layers
        • Assigns a container ID (e.g., a1b2c3d4e5f6)
        • Configures networking (bridge network)
             │
Step 5: Daemon starts the container
        • containerd creates the OCI bundle
        • shim starts runc
        • runc creates namespaces and cgroups
        • The hello-world binary executes
             │
Step 6: Container output is streamed to your terminal
        "Hello from Docker!"
        (plus explanation text)
             │
Step 7: The hello-world process exits (exit code 0)
        • Container enters "Exited" state
        • Container is NOT removed (still exists as stopped)
```

Let us run it and observe each part:

```bash
# Run hello-world:
$ docker run hello-world
Unable to find image 'hello-world:latest' locally      ← Step 2: not found
latest: Pulling from library/hello-world                ← Step 3: pulling
c1ec31eb5944: Pull complete                             ← Step 3: layer downloaded
Digest: sha256:4bd78111b6914a99dbc560e6a20eab57ff6655aea4a80c50b0c5036dd5dc7e5b
Status: Downloaded newer image for hello-world:latest   ← Step 3: done

Hello from Docker!                                      ← Step 6: output
This message shows that your installation appears to be working correctly.

To generate this message, Docker took the following steps:
 1. The Docker client contacted the Docker daemon.
 2. The Docker daemon pulled the "hello-world" image from the Docker Hub.
    (amd64)
 3. The Docker daemon created a new container from that image which runs the
    executable that produces the output you are currently reading.
 4. The Docker daemon streamed that output to the Docker client, which sent it
    to your terminal.

To try something more ambitious, you can run an Ubuntu container with:
 $ docker run -it ubuntu bash
...

# The container has exited, but still exists:
$ docker ps -a
CONTAINER ID   IMAGE         COMMAND    CREATED          STATUS                      NAMES
a1b2c3d4e5f6   hello-world   "/hello"   30 seconds ago   Exited (0) 29 seconds ago   kind_tesla

# Run it again — this time the image is cached (no download):
$ docker run hello-world
Hello from Docker!
...
# Notice: no "Pulling from library/hello-world" — the image was already local!
```

<Callout type="info" title="Container Names">
Docker automatically generates a random name for each container (like `kind_tesla` above). These names are two words: an adjective and a famous scientist or hacker. They are used when you do not specify a name with `--name`.
</Callout>

## Running Interactive Containers

The `hello-world` container runs a single command and exits. But Docker can also run **interactive** containers where you get a shell inside the container.

### docker run -it: Interactive Mode

```bash
$ docker run -it ubuntu:22.04 bash
root@c3d4e5f6a7b8:/#
```

Let us break down every flag:

| Flag | Meaning |
|---|---|
| `-i` | **Interactive** — Keep STDIN open (so you can type) |
| `-t` | **TTY** — Allocate a pseudo-terminal (so you see a prompt) |
| `ubuntu:22.04` | The image to run |
| `bash` | The command to run inside the container (overrides the image default) |

```bash
# You are now INSIDE the container:
root@c3d4e5f6a7b8:/# whoami
root

root@c3d4e5f6a7b8:/# hostname
c3d4e5f6a7b8          # The container ID IS the hostname

root@c3d4e5f6a7b8:/# cat /etc/os-release
PRETTY_NAME="Ubuntu 22.04.3 LTS"
NAME="Ubuntu"
VERSION_ID="22.04"

root@c3d4e5f6a7b8:/# ls /
bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var

# This is a complete Ubuntu filesystem, but the container is tiny:
root@c3d4e5f6a7b8:/# apt list --installed 2>/dev/null | wc -l
89    # Only 89 packages — a minimal Ubuntu

# The container is isolated — it only sees its own processes:
root@c3d4e5f6a7b8:/# ps aux
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.0   4624  3712 pts/0    Ss   10:30   0:00 bash
root        12  0.0  0.0   7060  1536 pts/0    R+   10:31   0:00 ps aux

# Only 2 processes! bash (PID 1) and the ps command we just ran.

# Exit the container:
root@c3d4e5f6a7b8:/# exit
exit

# Back on your host. The container has stopped:
$ docker ps -a
CONTAINER ID   IMAGE          COMMAND   STATUS                     NAMES
c3d4e5f6a7b8   ubuntu:22.04   "bash"    Exited (0) 5 seconds ago   musing_curie
```

<Callout type="warning" title="Exit Means Stop">
When you exit an interactive container (by typing `exit` or pressing Ctrl+D), the container stops. This is because the main process (bash) has exited. A container runs only as long as its main process (PID 1) is running.

To detach from an interactive container without stopping it, press **Ctrl+P, Ctrl+Q** (press Ctrl+P first, then Ctrl+Q while still holding Ctrl).
</Callout>

### Experimenting Inside a Container

Containers are disposable — you can install things, break things, and throw the container away:

```bash
# Start a fresh Ubuntu container:
$ docker run -it --name playground ubuntu:22.04 bash

# Install software inside the container:
root@abc123:/# apt-get update && apt-get install -y curl vim
...
Setting up curl (7.81.0-1ubuntu1.14) ...
Setting up vim (2:8.2.3995-1ubuntu2.13) ...

# Use curl:
root@abc123:/# curl -s https://httpbin.org/ip
{
  "origin": "203.0.113.42"
}

# Create files:
root@abc123:/# echo "Hello from my container!" > /tmp/message.txt
root@abc123:/# cat /tmp/message.txt
Hello from my container!

# Exit and stop the container:
root@abc123:/# exit

# The container and all changes still exist (but stopped):
$ docker ps -a --filter name=playground
CONTAINER ID   IMAGE          COMMAND   STATUS                     NAMES
abc123def456   ubuntu:22.04   "bash"    Exited (0) 5 seconds ago   playground

# Restart and reattach:
$ docker start -ai playground
root@abc123:/# cat /tmp/message.txt
Hello from my container!    # Changes are preserved when restarting!

# Exit and clean up:
root@abc123:/# exit
$ docker rm playground
```

<Callout type="tip" title="--rm Flag: Auto-Remove on Exit">
If you want a container to be automatically deleted when it stops, use the `--rm` flag:

```bash
# Container is removed automatically when you exit:
docker run -it --rm ubuntu:22.04 bash

# After exit, docker ps -a will NOT show this container.
# Great for quick experiments you do not need to keep.
```
</Callout>

## Running Detached Containers

Most real-world containers run in the **background** (detached mode). Web servers, databases, and APIs do not need an interactive terminal — they just need to run.

### docker run -d: Detached Mode

```bash
# Run nginx in the background:
$ docker run -d nginx
7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b

# Docker returns the container ID and gives you your terminal back.

# The container is running in the background:
$ docker ps
CONTAINER ID   IMAGE   COMMAND                  CREATED          STATUS          PORTS     NAMES
7a8b9c0d1e2f   nginx   "/docker-entrypoint.…"   10 seconds ago   Up 9 seconds    80/tcp    cranky_euler
```

But wait — nginx is running on port 80 inside the container. Can you access it?

```bash
# Try to access nginx on the host:
$ curl http://localhost:80
curl: (7) Failed to connect to localhost port 80: Connection refused

# It does not work! The container's port 80 is NOT mapped to the host.
# The container has its own network namespace — port 80 inside the
# container is NOT the same as port 80 on your host machine.
```

To access the containerized service from outside, you need **port mapping**.

## Port Mapping: -p host:container

Port mapping connects a port on your **host machine** to a port inside the **container**. This is how you make containerized services accessible.

```
Port Mapping Diagram:
═══════════════════════════════════════════════════════

Your Browser                     Host Machine
http://localhost:8080             ┌─────────────────┐
        │                        │                  │
        │                        │  Port 8080       │
        └───────────────────────→│    │             │
                                 │    │  -p 8080:80 │
                                 │    ▼             │
                                 │  ┌───────────┐  │
                                 │  │ Container  │  │
                                 │  │            │  │
                                 │  │ nginx      │  │
                                 │  │ Port 80    │  │
                                 │  └───────────┘  │
                                 └─────────────────┘

Traffic flow:
  Browser → localhost:8080 → Docker proxy → container:80 → nginx
```

```bash
# Run nginx with port mapping:
$ docker run -d --name web -p 8080:80 nginx
a1b2c3d4e5f6...

# Now port 8080 on the host forwards to port 80 in the container:
$ docker ps
CONTAINER ID   IMAGE   COMMAND                  STATUS         PORTS                  NAMES
a1b2c3d4e5f6   nginx   "/docker-entrypoint.…"   Up 5 seconds   0.0.0.0:8080->80/tcp   web

# Access nginx through the host port:
$ curl http://localhost:8080
<!DOCTYPE html>
<html>
<head>
<title>Welcome to nginx!</title>
<style>
html { color-scheme: light dark; }
body { width: 35em; margin: 0 auto;
font-family: Tahoma, Verdana, Arial, sans-serif; }
</style>
</head>
<body>
<h1>Welcome to nginx!</h1>
<p>If you see this page, the nginx web server is successfully installed and
working. Further configuration is required.</p>
...
</body>
</html>
```

### Port Mapping Variations

```bash
# Map host port 8080 to container port 80:
docker run -d -p 8080:80 nginx

# Map multiple ports:
docker run -d -p 8080:80 -p 8443:443 nginx

# Map to a specific host interface (bind to localhost only):
docker run -d -p 127.0.0.1:8080:80 nginx

# Let Docker choose a random host port:
docker run -d -p 80 nginx
# Check which port was assigned:
$ docker port <container_id>
80/tcp -> 0.0.0.0:49153

# Map a UDP port:
docker run -d -p 5353:53/udp my-dns-server

# Map all exposed ports to random host ports:
docker run -d -P nginx
```

<Callout type="warning" title="Port Conflicts">
You cannot map the same host port to multiple containers:

```bash
$ docker run -d --name web1 -p 8080:80 nginx
# Success

$ docker run -d --name web2 -p 8080:80 nginx
# Error! Port 8080 is already in use:
# docker: Error response from daemon: driver failed programming
# external connectivity on endpoint web2: Bind for 0.0.0.0:8080
# failed: port is already allocated.

# Solution: use different host ports:
$ docker run -d --name web2 -p 8081:80 nginx
# Now web1 is on :8080 and web2 is on :8081
```
</Callout>

<Callout type="info" title="Why -p 8080:80 and Not -p 80:8080?">
The format is always **host:container** (left is host, right is container). Think of it as "from:to" from the outside perspective:

```
-p 8080:80    → Host port 8080 forwards TO container port 80
-p 3000:3000  → Host port 3000 forwards TO container port 3000
-p 5432:5432  → Host port 5432 forwards TO container port 5432
```

A common pattern is to map the same port number when there is no conflict.
</Callout>

## Naming Containers: --name

By default, Docker gives containers random names. Use `--name` to give them meaningful names:

```bash
# Without --name (random name):
$ docker run -d nginx
$ docker ps --format "{{.Names}}"
hungry_darwin

# With --name (explicit name):
$ docker run -d --name my-website -p 8080:80 nginx
$ docker ps --format "{{.Names}}"
my-website

# Now you can reference the container by name:
docker stop my-website
docker start my-website
docker logs my-website
docker rm my-website
```

Container naming rules:
- Must match the regex `[a-zA-Z0-9][a-zA-Z0-9_.-]+`
- Must be unique (no two running containers with the same name)
- Tip: Use descriptive names related to the service, like `web`, `api`, `db`, `redis`

```bash
# Practical example — running a full stack:
docker run -d --name web      -p 8080:80  nginx
docker run -d --name api      -p 3000:3000 node-api:latest
docker run -d --name database -p 5432:5432 postgres:15
docker run -d --name cache    -p 6379:6379 redis:7
```

## Container Lifecycle Commands

These are the commands you will use every day to manage containers.

### docker ps: List Containers

```bash
# List running containers:
$ docker ps
CONTAINER ID   IMAGE     COMMAND                  CREATED         STATUS         PORTS                  NAMES
a1b2c3d4e5f6   nginx     "/docker-entrypoint.…"   2 minutes ago   Up 2 minutes   0.0.0.0:8080->80/tcp   web
b2c3d4e5f6a7   redis:7   "docker-entrypoint.s…"   5 minutes ago   Up 5 minutes   6379/tcp               cache

# List ALL containers (including stopped):
$ docker ps -a
CONTAINER ID   IMAGE         COMMAND     CREATED          STATUS                     NAMES
a1b2c3d4e5f6   nginx         "nginx…"    2 minutes ago    Up 2 minutes               web
b2c3d4e5f6a7   redis:7       "redis…"    5 minutes ago    Up 5 minutes               cache
c3d4e5f6a7b8   ubuntu:22.04  "bash"      10 minutes ago   Exited (0) 8 minutes ago   playground
d4e5f6a7b8c9   hello-world   "/hello"    15 minutes ago   Exited (0) 15 minutes ago  kind_tesla

# Show only container IDs (useful for scripting):
$ docker ps -q
a1b2c3d4e5f6
b2c3d4e5f6a7

# Show all container IDs:
$ docker ps -aq
a1b2c3d4e5f6
b2c3d4e5f6a7
c3d4e5f6a7b8
d4e5f6a7b8c9

# Format output:
$ docker ps --format "table {{.ID}}\t{{.Image}}\t{{.Status}}\t{{.Names}}"
CONTAINER ID   IMAGE     STATUS         NAMES
a1b2c3d4e5f6   nginx     Up 2 minutes   web
b2c3d4e5f6a7   redis:7   Up 5 minutes   cache

# Filter containers:
$ docker ps --filter "status=exited"
$ docker ps --filter "name=web"
$ docker ps --filter "ancestor=nginx"
```

### docker stop: Stop a Running Container

```bash
# Stop a container gracefully (SIGTERM, then SIGKILL after 10s):
$ docker stop web
web

# Stop with a custom timeout (wait 30 seconds before SIGKILL):
$ docker stop -t 30 web
web

# Stop multiple containers:
$ docker stop web cache database
web
cache
database

# Stop ALL running containers:
$ docker stop $(docker ps -q)
```

```
What docker stop does:
═══════════════════════════════

1. Sends SIGTERM to the main process (PID 1)
   → Application should handle this and shut down cleanly
   → Close database connections, flush buffers, etc.

2. Waits for timeout (default: 10 seconds)
   → Gives the process time to clean up

3. If still running, sends SIGKILL
   → Forcefully terminates the process
   → No cleanup possible
```

### docker start: Start a Stopped Container

```bash
# Start a stopped container:
$ docker start web
web

# Start and attach to it (see output):
$ docker start -a web
172.17.0.1 - - [15/Jan/2025:10:30:00 +0000] "GET / HTTP/1.1" 200 615

# Start and attach interactively:
$ docker start -ai playground
root@abc123:/#
```

### docker restart: Restart a Container

```bash
# Restart a container (stop + start):
$ docker restart web
web

# Restart with a custom stop timeout:
$ docker restart -t 5 web
web
```

### docker rm: Remove a Container

```bash
# Remove a stopped container:
$ docker rm playground
playground

# Try to remove a running container (fails!):
$ docker rm web
Error response from daemon: You cannot remove a running container a1b2c3d4e5f6.
Stop the container before attempting removal or force remove.

# Force-remove a running container (stop + remove):
$ docker rm -f web
web

# Remove all stopped containers:
$ docker container prune
WARNING! This will remove all stopped containers.
Are you sure you want to continue? [y/N] y
Deleted Containers:
c3d4e5f6a7b8
d4e5f6a7b8c9
Total reclaimed space: 0B

# Remove all stopped containers (without confirmation):
$ docker container prune -f
```

```

═══════════════════════════════════════

```

## docker exec: Run Commands in Running Containers

`docker exec` lets you run additional commands inside a **running** container. This is essential for debugging and administration.

```bash
# First, make sure a container is running:
$ docker run -d --name web -p 8080:80 nginx

# Run a command inside the running container:
$ docker exec web hostname
a1b2c3d4e5f6

$ docker exec web cat /etc/nginx/nginx.conf
user  nginx;
worker_processes  auto;
error_log  /var/log/nginx/error.log notice;
pid        /var/run/nginx.pid;
...

$ docker exec web ls /usr/share/nginx/html/
50x.html
index.html

# Get an interactive shell inside the running container:
$ docker exec -it web bash
root@a1b2c3d4e5f6:/# whoami
root
root@a1b2c3d4e5f6:/# nginx -v
nginx version: nginx/1.25.3
root@a1b2c3d4e5f6:/# exit
# The container keeps running! exec does NOT affect the main process.

# Run a command as a specific user:
$ docker exec -u nginx web whoami
nginx

# Set environment variables for the exec session:
$ docker exec -e MY_VAR=hello web env | grep MY_VAR
MY_VAR=hello

# Run a command in the background (detached exec):
$ docker exec -d web touch /tmp/background-task-done
```

```
docker exec vs docker run:
═══════════════════════════════════════

docker run:  Creates a NEW container from an image
docker exec: Runs a command in an EXISTING running container

docker run -it ubuntu bash    → New container, new process
docker exec -it web bash      → Same container, additional process
```

<Callout type="warning" title="exec Only Works on Running Containers">
You cannot `docker exec` into a stopped container:

```bash
$ docker stop web
$ docker exec web hostname
Error response from daemon: Container a1b2c3d4e5f6 is not running

# To inspect a stopped container, start it first:
$ docker start web
$ docker exec web hostname
a1b2c3d4e5f6
```

For debugging stopped containers, use `docker logs` (for output) or `docker cp` (to copy files out).
</Callout>

## docker inspect: Container Metadata

`docker inspect` shows the complete configuration and state of a container in JSON format. It is your deep-dive debugging tool.


### Extracting Specific Information

Use Go template formatting to extract specific fields:

```bash
# Get the container's IP address:
$ docker inspect web --format '{{.NetworkSettings.IPAddress}}'
172.17.0.2

# Get the container's MAC address:
$ docker inspect web --format '{{.NetworkSettings.MacAddress}}'
02:42:ac:11:00:02

# Get the container's PID on the host:
$ docker inspect web --format '{{.State.Pid}}'
12345

# Get the container's start time:
$ docker inspect web --format '{{.State.StartedAt}}'
2025-01-15T10:30:00.456789012Z

# Get the container's port mappings:
$ docker inspect web --format '{{json .NetworkSettings.Ports}}' | python3 -m json.tool
{
    "80/tcp": [
        {
            "HostIp": "0.0.0.0",
            "HostPort": "8080"
        }
    ]
}

# Get the container's environment variables:
$ docker inspect web --format '{{json .Config.Env}}' | python3 -m json.tool
[
    "PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin",
    "NGINX_VERSION=1.25.3",
    "NJS_VERSION=0.8.2",
    "PKG_RELEASE=1~bookworm"
]

# Get the container's mounts/volumes:
$ docker inspect web --format '{{json .Mounts}}' | python3 -m json.tool

# Get the container's log file path:
$ docker inspect web --format '{{.LogPath}}'
/var/lib/docker/containers/a1b2c3d4.../a1b2c3d4...-json.log

# Get the image the container was created from:
$ docker inspect web --format '{{.Config.Image}}'
nginx

# Get the container's restart policy:
$ docker inspect web --format '{{.HostConfig.RestartPolicy.Name}}'
no
```

<Callout type="info" title="docker inspect Is Your Best Friend for Debugging">
When something is not working, `docker inspect` is usually the first place to look:

- Container not accessible? Check `.NetworkSettings.IPAddress` and `.NetworkSettings.Ports`
- Container crashing? Check `.State.ExitCode` and `.State.Error`
- Wrong environment? Check `.Config.Env`
- Storage issues? Check `.GraphDriver.Data` and `.Mounts`
</Callout>

## Key Takeaways

- `docker run hello-world` involves seven steps: CLI sends request, daemon checks local images, pulls from Hub if needed, creates container, starts it, streams output, container exits
- Interactive mode (`-it`) gives you a shell inside a container; detached mode (`-d`) runs it in the background — most production containers run detached
- Port mapping (`-p host:container`) is required to access containerized services from outside; without it, container ports are isolated
- Container lifecycle follows: `docker run` (create + start) → `docker stop` → `docker start` → `docker rm`; use `docker ps -a` to see all containers including stopped ones
- `docker exec -it <name> bash` gives you a shell in a running container for debugging; `docker exec` does not affect the main container process
- `docker logs` shows container output, with `-f` for real-time following and `--tail` for the last N lines — this is your primary debugging tool for containerized applications
- `docker inspect` returns the complete JSON metadata for a container; use `--format` to extract specific fields like IP address, PID, ports, and environment variables
- Containers are ephemeral by default — use `--rm` for throwaway containers and `--name` for containers you want to manage by name
- `docker stats` provides real-time resource monitoring, `docker diff` shows filesystem changes, and `docker cp` copies files to/from containers
