## What Is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. Instead of running multiple `docker run` commands with long lists of flags, you describe your entire application stack in a single YAML file called `docker-compose.yml`.

```
Without Compose (what you've been doing):
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  $ docker network create myapp                                   │
│  $ docker volume create db-data                                  │
│  $ docker run -d --name db --network myapp -v db-data:/var/...  │
│  $ docker run -d --name redis --network myapp redis:7            │
│  $ docker run -d --name api --network myapp -p 3000:3000 ...    │
│  $ docker run -d --name web --network myapp -p 80:80 ...        │
│                                                                  │
│  4 commands, dozens of flags, easy to make mistakes              │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

With Compose:
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  $ docker compose up -d                                          │
│                                                                  │
│  1 command. Everything starts. Networks and volumes created      │
│  automatically.                                                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

<Callout type="info" title="Compose V2 vs V1">
Docker Compose V2 is integrated directly into the Docker CLI as `docker compose` (with a space). The older V1 was a separate binary called `docker-compose` (with a hyphen). V1 is deprecated. This lesson uses the modern V2 syntax: `docker compose`.
</Callout>

### Why Use Docker Compose?

| Problem | Compose Solution |
|---------|-----------------|
| Running multiple `docker run` commands | Single `docker compose up` |
| Remembering complex flags | Declared in YAML file |
| Creating networks manually | Automatic default network |
| Managing volume lifecycle | Volumes declared and created automatically |
| Ensuring services start in order | `depends_on` directive |
| Sharing setup across team | Commit `docker-compose.yml` to version control |
| Environment consistency | Everyone runs the same stack |

## The docker-compose.yml File

The Compose file is a YAML file that defines your entire application stack. Let us start with the structure.

### Basic Structure

```yaml
# docker-compose.yml

services:
  # Define each container as a service
  web:
    image: nginx:alpine
    ports:
      - "80:80"

  api:
    build: ./api
    ports:
      - "3000:3000"

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data

# Named volumes
volumes:
  db-data:

# Custom networks (optional — Compose creates a default)
networks:
  frontend:
  backend:
```

<Callout type="info" title="No More 'version' Key">
In Compose V2, the top-level `version` key is no longer required. Older Compose files used `version: '3.8'` or similar, but this is now obsolete. You can safely omit it. If present, it is ignored.
</Callout>

### Top-Level Keys

```
docker-compose.yml
├── services:      # Required. Define your containers here
│   ├── web:       # Service name (becomes the hostname on the network)
│   ├── api:       # Another service
│   └── db:        # Another service
├── volumes:       # Optional. Named volumes for persistent data
│   └── db-data:
├── networks:      # Optional. Custom networks
│   ├── frontend:
│   └── backend:
├── secrets:       # Optional. Sensitive data
│   └── db-pass:
└── configs:       # Optional. Configuration files
    └── nginx-conf:
```

## Service Configuration Deep Dive

Each service in the `services` block defines a container. Here are the most important configuration options.

### Image vs Build

You either pull an existing image or build one from a Dockerfile:

```yaml
services:
  # Use an existing image from a registry
  redis:
    image: redis:7-alpine

  # Build from a Dockerfile
  api:
    build: ./api                    # Path to directory containing Dockerfile

  # Build with more control
  api-advanced:
    build:
      context: ./api               # Build context directory
      dockerfile: Dockerfile.prod  # Custom Dockerfile name
      args:                        # Build arguments
        NODE_ENV: production
        APP_VERSION: "2.1.0"
      target: production           # Multi-stage build target
    image: mycompany/api:latest    # Tag the built image
```

### Ports

Map container ports to host ports:

```yaml
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"           # HOST:CONTAINER
      - "443:443"         # HTTPS
      - "8080:80"         # Map container 80 to host 8080
      - "127.0.0.1:3000:3000"  # Bind only to localhost
```

<Callout type="warning" title="Quote Your Ports">
Always quote port mappings in YAML. Without quotes, YAML may interpret `80:80` as a base-60 number. Use `"80:80"` to be safe.
</Callout>

### Environment Variables

Set environment variables for a service:

```yaml
services:
  api:
    image: node:20-alpine
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgres://user:pass@db:5432/myapp
      - REDIS_URL=redis://redis:6379

    # Or use the map syntax:
    environment:
      NODE_ENV: production
      DATABASE_URL: postgres://user:pass@db:5432/myapp
      REDIS_URL: redis://redis:6379

    # Or load from a file:
    env_file:
      - .env
      - .env.local
```

### Volumes

Mount volumes for persistent data or development file sharing:

```yaml
services:
  db:
    image: postgres:16
    volumes:
      # Named volume (persistent data)
      - db-data:/var/lib/postgresql/data

      # Bind mount (share host files with container)
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql

      # Read-only bind mount
      - ./config/nginx.conf:/etc/nginx/nginx.conf:ro

  api:
    build: ./api
    volumes:
      # Bind mount source code for development
      - ./api/src:/app/src

      # Anonymous volume to preserve node_modules
      - /app/node_modules

volumes:
  db-data:              # Declare the named volume
    driver: local       # Default driver
```

### Networking

Services can communicate by name on shared networks:

```yaml
services:
  web:
    image: nginx:alpine
    networks:
      - frontend

  api:
    build: ./api
    networks:
      - frontend      # Web can reach API
      - backend       # API can reach DB

  db:
    image: postgres:16
    networks:
      - backend       # Only API can reach DB

networks:
  frontend:
  backend:
```

```
┌─────────────────────────────────────────────────┐
│           frontend network                       │
│                                                  │
│  ┌─────────┐         ┌─────────┐                │
│  │   web   │ ──────> │   api   │                │
│  │ (nginx) │         │ (node)  │                │
│  └─────────┘         └────┬────┘                │
│                           │                      │
└───────────────────────────┼──────────────────────┘
                            │
┌───────────────────────────┼──────────────────────┐
│           backend network │                       │
│                           │                       │
│                      ┌────┴────┐                 │
│                      │   api   │                 │
│                      │ (node)  │                 │
│                      └────┬────┘                 │
│                           │                      │
│                      ┌────┴────┐                 │
│                      │   db    │                 │
│                      │(postgres)│                │
│                      └─────────┘                 │
│                                                  │
└──────────────────────────────────────────────────┘
```

### depends_on

Control startup order:

```yaml
services:
  web:
    build: ./web
    depends_on:
      - api

  api:
    build: ./api
    depends_on:
      db:
        condition: service_healthy     # Wait for health check to pass
      redis:
        condition: service_started     # Just wait for container to start

  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
```

<Callout type="warning" title="depends_on Does Not Wait for 'Ready'">
By default, `depends_on` only waits for the container to start, not for the application inside to be ready. A PostgreSQL container might start in 1 second, but the database takes 5 seconds to accept connections. Use `condition: service_healthy` with a `healthcheck` to wait for actual readiness.
</Callout>

### Restart Policies

```yaml
services:
  api:
    image: myapp/api:latest
    restart: unless-stopped   # Restart unless manually stopped

  # Options:
  # restart: "no"            # Never restart (default)
  # restart: always          # Always restart
  # restart: on-failure      # Only restart on non-zero exit
  # restart: unless-stopped  # Restart unless explicitly stopped
```

## Essential Compose Commands

Here is every command you need to manage your Compose application.

### Starting and Stopping

```bash
# Start all services (foreground — see logs)
docker compose up

# Start all services (detached — background)
docker compose up -d

# Start specific services only
docker compose up -d api db

# Stop all services (containers stop but are preserved)
docker compose stop

# Stop and REMOVE containers, networks
docker compose down

# Stop, remove containers, AND delete volumes (destroys data!)
docker compose down -v

# Stop, remove containers, AND delete images
docker compose down --rmi all
```

### Viewing Status

```bash
# List running services
docker compose ps

# Example output:
# NAME                  IMAGE          COMMAND                  STATUS          PORTS
# myapp-api-1           myapp/api      "node server.js"         Up 2 minutes    0.0.0.0:3000->3000/tcp
# myapp-db-1            postgres:16    "docker-entrypoint..."   Up 2 minutes    5432/tcp
# myapp-redis-1         redis:7        "docker-entrypoint..."   Up 2 minutes    6379/tcp

# View logs from all services
docker compose logs

# Follow logs from a specific service
docker compose logs -f api

# View last 50 lines from a service
docker compose logs --tail 50 api
```

### Running Commands in Services

```bash
# Execute a command in a running service container
docker compose exec api sh

# Run a one-off command (creates a new container)
docker compose run api npm test

# Run database migrations
docker compose exec api npx prisma migrate deploy
```

### Building and Updating

```bash
# Build or rebuild service images
docker compose build

# Build without cache
docker compose build --no-cache

# Build a specific service
docker compose build api

# Pull latest images for services using image:
docker compose pull

# Rebuild and restart
docker compose up -d --build
```

<Callout type="tip" title="Use --build When Code Changes">
If you change your application code but the Compose file has not changed, `docker compose up -d` will NOT rebuild the image. Always use `docker compose up -d --build` during development to ensure your latest code changes are included.
</Callout>

## Building with Compose

### Using Custom Dockerfiles

```yaml
# Directory structure:
# project/
# ├── docker-compose.yml
# ├── api/
# │   ├── Dockerfile
# │   ├── Dockerfile.dev
# │   └── src/
# └── web/
#     ├── Dockerfile
#     └── src/

services:
  api:
    build:
      context: ./api
      dockerfile: Dockerfile.dev     # Use development Dockerfile
    volumes:
      - ./api/src:/app/src           # Mount source for hot-reload
    environment:
      - NODE_ENV=development

  web:
    build:
      context: ./web
      target: development            # Multi-stage build target
    volumes:
      - ./web/src:/app/src
    ports:
      - "3000:3000"
```

### Development vs Production Compose Files

Use multiple Compose files to override settings:

```yaml
# docker-compose.yml (base)
services:
  api:
    build: ./api
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/myapp

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```

```yaml
# docker-compose.override.yml (auto-loaded for development)
services:
  api:
    build:
      context: ./api
      dockerfile: Dockerfile.dev
    volumes:
      - ./api/src:/app/src       # Hot-reload source code
    ports:
      - "3000:3000"
      - "9229:9229"             # Debug port
    environment:
      - DEBUG=true

  db:
    ports:
      - "5432:5432"             # Expose DB port for local tools
```

```yaml
# docker-compose.prod.yml (explicitly loaded for production)
services:
  api:
    build:
      context: ./api
      dockerfile: Dockerfile.prod
    restart: unless-stopped
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: "0.5"
```

```bash
# Development (auto-loads docker-compose.yml + docker-compose.override.yml)
docker compose up -d

# Production (explicitly specify files)
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## Scaling Services

Docker Compose lets you run multiple instances of a service:

```bash
# Scale the API to 3 instances
docker compose up -d --scale api=3

# Check running instances
docker compose ps
# NAME                  IMAGE          STATUS          PORTS
# myapp-api-1           myapp/api      Up 2 minutes    3000/tcp
# myapp-api-2           myapp/api      Up 2 minutes    3000/tcp
# myapp-api-3           myapp/api      Up 2 minutes    3000/tcp
# myapp-db-1            postgres:16    Up 2 minutes    5432/tcp
```

<Callout type="warning" title="Scaling and Port Conflicts">
If a service has a fixed host port mapping like `"3000:3000"`, you cannot scale it because the host port can only be used by one container. Remove the host port or use a range:

```yaml
services:
  api:
    build: ./api
    # Instead of "3000:3000", use:
    ports:
      - "3000-3010:3000"    # Each instance gets a unique host port
    # Or expose without host binding:
    expose:
      - "3000"               # Accessible only to other services
```

In practice, you would put a load balancer (like Nginx or Traefik) in front of scaled services.
</Callout>

### Load Balancing Scaled Services

```yaml
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - api

  api:
    build: ./api
    expose:
      - "3000"
    # No host port — only accessible through nginx

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```

```nginx
# nginx.conf
upstream api_servers {
    server myapp-api-1:3000;
    server myapp-api-2:3000;
    server myapp-api-3:3000;
}

server {
    listen 80;

    location /api/ {
        proxy_pass http://api_servers;
    }
}
```

## Compose Profiles

Profiles let you selectively start services based on the environment or use case:

```yaml
services:
  api:
    build: ./api
    ports:
      - "3000:3000"

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data

  # Only start in development
  adminer:
    image: adminer
    ports:
      - "8080:8080"
    profiles:
      - debug

  # Only start when running tests
  test-runner:
    build:
      context: ./api
      dockerfile: Dockerfile.test
    profiles:
      - testing

  # Monitoring tools — only in staging/prod
  prometheus:
    image: prom/prometheus
    profiles:
      - monitoring

  grafana:
    image: grafana/grafana
    profiles:
      - monitoring

volumes:
  db-data:
```

```bash
# Start only services WITHOUT a profile (api, db)
docker compose up -d

# Start default services + debug profile
docker compose --profile debug up -d

# Start default services + monitoring
docker compose --profile monitoring up -d

# Start multiple profiles
docker compose --profile debug --profile monitoring up -d
```

<Callout type="tip" title="Profiles for Optional Tools">
Profiles are perfect for optional services like database admin UIs, monitoring dashboards, test runners, or debug tools. The core application starts clean, and developers enable additional tools as needed.
</Callout>

## Real Example: Full-Stack Application

Let us build a complete application with React frontend, Node.js API, PostgreSQL database, and Redis cache.

### Project Structure

```
fullstack-app/
├── docker-compose.yml
├── .env
├── api/
│   ├── Dockerfile
│   ├── package.json
│   └── server.js
├── web/
│   ├── Dockerfile
│   ├── package.json
│   └── src/
│       └── index.html
├── nginx/
│   └── nginx.conf
└── db/
    └── init.sql
```

### The Compose File

```yaml
# docker-compose.yml
services:
  # Nginx reverse proxy
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - web
      - api
    restart: unless-stopped

  # React frontend
  web:
    build:
      context: ./web
      target: development
    volumes:
      - ./web/src:/app/src
    expose:
      - "5173"
    environment:
      - VITE_API_URL=http://localhost/api

  # Node.js API
  api:
    build: ./api
    expose:
      - "3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgres://${DB_USER}:${DB_PASSWORD}@db:5432/${DB_NAME}
      - REDIS_URL=redis://redis:6379
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    volumes:
      - ./api/src:/app/src
    restart: unless-stopped

  # PostgreSQL database
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - db-data:/var/lib/postgresql/data
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 5s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  # Redis cache
  redis:
    image: redis:7-alpine
    command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
    volumes:
      - redis-data:/data
    restart: unless-stopped

  # Database admin (debug profile only)
  adminer:
    image: adminer
    ports:
      - "8080:8080"
    profiles:
      - debug

volumes:
  db-data:
    driver: local
  redis-data:
    driver: local
```

### The .env File

### The Nginx Config

```nginx
# nginx/nginx.conf
events {
    worker_connections 1024;
}

http {
    upstream web_app {
        server web:5173;
    }

    upstream api_app {
        server api:3000;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://web_app;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
        }

        location /api/ {
            rewrite ^/api/(.*) /$1 break;
            proxy_pass http://api_app;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}
```

const http = require('http');

Build a Docker Compose setup for a blog application with three services:

2. **api** - A simple Python Flask API on port 5000
3. **db** - A PostgreSQL 16 database with persistent data

Requirements:
- The API should depend on the database being healthy
- The database should use a named volume for data persistence
- Environment variables for database credentials should be defined in the Compose file
- Nginx should be accessible on host port 8080

Create the project structure with all necessary files.

app.use(express.json());

  connectionString: process.env.DATABASE_URL

  try {
    await pool.query('SELECT 1');
    res.json({ status: 'healthy', database: 'connected' });

    res.status(500).json({ status: 'unhealthy', error: err.message });
  }

  try {
    const cached = await redis.get('tasks:all');
    if (cached) {
      return res.json({ source: 'cache', tasks: JSON.parse(cached) });
    }

    const result = await pool.query('SELECT * FROM tasks ORDER BY created_at DESC');
    await redis.setEx('tasks:all', 60, JSON.stringify(result.rows));
    res.json({ source: 'database', tasks: result.rows });

    res.status(500).json({ error: err.message });

});

## Key Takeaways

- Docker Compose defines multi-container applications in a single YAML file, replacing many `docker run` commands with one `docker compose up`
- The Compose file uses top-level `services`, `volumes`, and `networks` keys to organize your application stack
- Services communicate by name on the default Compose network (the service name IS the hostname)
- Use `depends_on` with `condition: service_healthy` and `healthcheck` to ensure services start in the right order
- `docker compose up -d --build` is your go-to command during development to rebuild and restart
- Profiles let you define optional services (debug tools, monitoring) that only start when explicitly requested
- Multiple Compose files (`docker-compose.yml` + `docker-compose.override.yml`) support different environments without duplication
