## The Problem: Bloated Images

When you build an application inside a Docker image, the image ends up containing everything used during the build — compilers, build tools, package managers, intermediate files, source code, and development dependencies. None of these are needed to run the application.

### A Typical Single-Stage Go Build

```dockerfile
FROM golang:1.22
WORKDIR /app
COPY . .
RUN go build -o server .
CMD ["./server"]
```

What's in this image:

```
┌─────────────────────────────────────────────┐
│  Final Image: 843 MB                         │
├─────────────────────────────────────────────┤
│  ✗ Go compiler and toolchain       (600 MB) │ ← NOT needed at runtime
│  ✗ gcc, make, build tools          (150 MB) │ ← NOT needed at runtime
│  ✗ Source code                      (5 MB)  │ ← NOT needed at runtime
│  ✗ Go module cache                  (75 MB) │ ← NOT needed at runtime
│  ✓ Compiled binary                  (12 MB) │ ← The only thing we need
│  ✓ Base OS (Debian)                 (1 MB)  │ ← Actually need some libs
└─────────────────────────────────────────────┘

  Useful: 13 MB  |  Waste: 830 MB  |  Efficiency: 1.5%
```

<Callout type="warning" title="Why Bloat Matters">
Bloated images cause real problems:
- **Slower deployments**: More data to transfer across networks
- **Higher costs**: Storage, bandwidth, and egress fees in cloud registries
- **Larger attack surface**: Every extra package is a potential vulnerability
- **Slower scaling**: New nodes in Kubernetes must pull the full image before starting pods
- **Security audit failures**: Scanners flag tools that shouldn't be in production
</Callout>

## What Are Multi-Stage Builds?

Multi-stage builds let you use **multiple `FROM` statements** in a single Dockerfile. Each `FROM` starts a new **stage** with a fresh filesystem. You can selectively copy artifacts from one stage to another, leaving behind everything you don't need.

```dockerfile
# Stage 1: Build
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN go build -o server .

# Stage 2: Production
FROM alpine:3.19
COPY --from=builder /app/server /server
CMD ["/server"]
```

### How It Works

```
Stage 1: "builder"                    Stage 2: Production
┌────────────────────────┐           ┌────────────────────────┐
│ golang:1.22 (843 MB)   │           │ alpine:3.19 (7 MB)     │
│                        │           │                        │
│ Go compiler            │           │                        │
│ gcc, make              │    COPY   │                        │
│ Source code             │──────────▶│ server binary (12 MB)  │
│ Module cache            │  --from=  │                        │
│ server binary ──────────│──builder──│                        │
│                        │           │                        │
└────────────────────────┘           └────────────────────────┘
   DISCARDED                           FINAL IMAGE: 19 MB
   (never shipped)                     (shipped to production)
```

The build stage is used to compile the code, and then **only the compiled binary** is copied to the production stage. Everything else — the compiler, build tools, source code — is discarded.

<Callout type="info" title="Named Stages">
The `AS builder` syntax gives a stage a name. You reference it with `COPY --from=builder`. Without a name, stages are numbered starting from 0: `COPY --from=0`.
</Callout>

## The Builder Pattern in Detail

The builder pattern is the most common multi-stage pattern. It has two stages:

1. **Build stage**: Contains all build tools, compiles the application
2. **Production stage**: Contains only the runtime and compiled artifacts

### Go Application: 843 MB to 12.5 MB

```dockerfile
# ============================================
# Stage 1: Build the Go binary
# ============================================
FROM golang:1.22-alpine AS builder

WORKDIR /app

# Download dependencies first (cache optimization)
COPY go.mod go.sum ./
RUN go mod download

# Copy source and build
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-w -s" -o /app/server .

# ============================================
# Stage 2: Create minimal production image
# ============================================
FROM scratch

# Copy the binary from the build stage
COPY --from=builder /app/server /server

# Copy CA certificates for HTTPS (if needed)
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
CMD ["/server"]
```

```bash
# Build and check size
docker build -t go-app:single -f Dockerfile.single .
docker build -t go-app:multi -f Dockerfile.multi .

docker images go-app
REPOSITORY   TAG      IMAGE ID       SIZE
go-app       single   abc123def456   843MB
go-app       multi    789ghi012jkl   12.5MB
```

<Callout type="tip" title="Go Build Flags Explained">
- `CGO_ENABLED=0`: Disables cgo, producing a statically linked binary (no external C library dependencies)
- `GOOS=linux`: Cross-compile for Linux (useful when building on macOS)
- `-ldflags="-w -s"`: Strip debug information and symbol tables, reducing binary size by ~30%
</Callout>

### Node.js Application: 1.1 GB to 150 MB

Node.js apps need a runtime (you can't compile them to a static binary like Go), but you can still eliminate build-time dependencies:

```dockerfile
# ============================================
# Stage 1: Install dependencies and build
# ============================================
FROM node:20 AS builder

WORKDIR /app

COPY package.json package-lock.json ./
RUN npm ci

COPY . .
RUN npm run build    # TypeScript compilation, bundling, etc.

# Remove dev dependencies
RUN npm prune --production

# ============================================
# Stage 2: Production image
# ============================================
FROM node:20-alpine

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy only production dependencies and built code
COPY --from=builder --chown=appuser:appgroup /app/node_modules ./node_modules
COPY --from=builder --chown=appuser:appgroup /app/dist ./dist
COPY --from=builder --chown=appuser:appgroup /app/package.json ./

USER appuser

EXPOSE 3000
CMD ["node", "dist/server.js"]
```

What changed:

```
Single-stage:                         Multi-stage:
┌─────────────────────────┐          ┌─────────────────────────┐
│ node:20 (1.1 GB)        │          │ node:20-alpine (135 MB) │
│ + devDependencies       │          │ + production deps only  │
│ + TypeScript compiler   │          │ + compiled JS only      │
│ + Source .ts files      │          │ + package.json          │
│ + test files            │          │                         │
│ + build tools           │          │                         │
├─────────────────────────┤          ├─────────────────────────┤
│ Total: ~1.4 GB          │          │ Total: ~150 MB          │
└─────────────────────────┘          └─────────────────────────┘
```

### Python Application: 1.2 GB to 95 MB

```dockerfile
# ============================================
# Stage 1: Build wheels for all dependencies
# ============================================
FROM python:3.12 AS builder

WORKDIR /app

# Install build dependencies for packages with C extensions
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      gcc \
      libffi-dev \
      libpq-dev \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip wheel --no-cache-dir --wheel-dir /app/wheels -r requirements.txt

# ============================================
# Stage 2: Production image
# ============================================
FROM python:3.12-slim

# Install runtime libraries only (no compiler!)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      libpq5 \
    && rm -rf /var/lib/apt/lists/*

RUN groupadd -r appuser && useradd -r -g appuser appuser

WORKDIR /app

# Install pre-built wheels (no compilation needed!)
COPY --from=builder /app/wheels /tmp/wheels
COPY --from=builder /app/requirements.txt .
RUN pip install --no-cache-dir /tmp/wheels/* && \
    rm -rf /tmp/wheels

COPY --chown=appuser:appuser . .

USER appuser

EXPOSE 8000
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "4", "app:app"]
```

<Callout type="info" title="Python Wheels Strategy">
The builder stage compiles Python packages with C extensions into **wheel** files (pre-built binary packages). The production stage installs these wheels without needing a C compiler. This means the production image only needs runtime libraries (like `libpq5` for PostgreSQL) — not build-time tools (like `gcc` and `libpq-dev`).
</Callout>

### Java Application: 800 MB to 180 MB

```dockerfile
# ============================================
# Stage 1: Build with Maven
# ============================================
FROM maven:3.9-eclipse-temurin-21 AS builder

WORKDIR /app

# Cache dependencies
COPY pom.xml .
RUN mvn dependency:go-offline -B

# Build the application
COPY src ./src
RUN mvn package -DskipTests -B

# ============================================
# Stage 2: Production with JRE only
# ============================================
FROM eclipse-temurin:21-jre-alpine

RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy only the JAR file
COPY --from=builder /app/target/*.jar app.jar

RUN chown appuser:appgroup app.jar

USER appuser

EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget --quiet --tries=1 --spider http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["java", "-jar", "app.jar"]
```

Key difference: the builder uses `maven` (JDK + Maven, ~800 MB) while production uses `eclipse-temurin:21-jre-alpine` (JRE only, ~85 MB).

## When to Use Multi-Stage Builds

### Use Multi-Stage When:

- **Compiled languages** (Go, Rust, C/C++, Java): Build tools are large and unnecessary at runtime
- **TypeScript/transpiled languages**: TypeScript compiler and source files aren't needed at runtime
- **Applications with build steps**: Bundling, minification, asset compilation
- **C extension dependencies**: Python or Ruby packages that need gcc to compile
- **Security requirements**: You need to minimize the attack surface

### Single Stage Is Fine When:

- **Simple scripting languages** (Python, Ruby) with no C extensions and no build step
- **Quick prototypes** where image size doesn't matter
- **Development images** where you want build tools available for debugging
- **Very simple applications** with no build step and minimal dependencies

<Callout type="tip" title="Rule of Thumb">
If your Dockerfile has `RUN npm run build`, `RUN go build`, `RUN mvn package`, `RUN cargo build`, or installs a compiler/build tool — you should probably use a multi-stage build.
</Callout>

## Advanced Multi-Stage Patterns

### Pattern 1: Named Stages for Clarity

```dockerfile
# Stage: dependencies
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# Stage: builder
FROM deps AS builder
COPY . .
RUN npm run build
RUN npm run test

# Stage: production
FROM node:20-alpine AS production
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
COPY package.json ./
CMD ["node", "dist/server.js"]

# Stage: development (optional, for dev containers)
FROM deps AS development
COPY . .
CMD ["npm", "run", "dev"]
```

You can target a specific stage during build:

```bash
# Build only the production stage
docker build --target production -t myapp:prod .

# Build the development stage
docker build --target development -t myapp:dev .
```

### Pattern 2: External Images as Stages

You can copy files from any existing image, not just stages in your Dockerfile:

```dockerfile
FROM alpine:3.19

# Copy nginx binary from the official nginx image
COPY --from=nginx:1.25-alpine /usr/sbin/nginx /usr/sbin/nginx

# Copy the configuration tool from another image
COPY --from=hashicorp/consul:1.17 /bin/consul /usr/local/bin/consul

# Copy a static binary you built in a separate repo
COPY --from=myregistry.com/tools:latest /usr/local/bin/my-tool /usr/local/bin/
```

### Pattern 3: Parallel Stages

Docker BuildKit can build independent stages in parallel:

```dockerfile
# These two stages run in PARALLEL
FROM node:20-alpine AS frontend-builder
WORKDIR /frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

FROM golang:1.22-alpine AS backend-builder
WORKDIR /backend
COPY backend/go.* ./
RUN go mod download
COPY backend/ .
RUN CGO_ENABLED=0 go build -o /server .

# Final stage combines both
FROM alpine:3.19
COPY --from=backend-builder /server /server
COPY --from=frontend-builder /frontend/dist /static
CMD ["/server"]
```

```
Build timeline:

  Without parallel:
  ──[frontend build]────────[backend build]────────[combine]──
  0s                  30s                   60s              65s

  With BuildKit parallel:
  ──[frontend build]────────┐
  ──[backend build]─────────┤──[combine]──
  0s                  30s   31s         36s

  Time saved: ~29 seconds (48%)
```

### Pattern 4: Test Stage

Include a test stage that runs during build but doesn't contribute to the final image:

```dockerfile
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN go build -o /server .

# Test stage — runs during build, results not in final image
FROM builder AS tester
RUN go test -v ./...
RUN go vet ./...

# Production stage
FROM alpine:3.19
COPY --from=builder /server /server
CMD ["/server"]
```

```bash
# Build including tests (will fail if tests fail)
docker build --target tester -t myapp:test .

# Build production image (skips test stage unless targeted)
docker build -t myapp:prod .
```

### Pattern 5: Distroless Images

Google's distroless images contain only the runtime — no shell, no package manager, no utilities. Maximum security:

```dockerfile
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o /server .

# Distroless: no shell, no package manager, nothing extra
FROM gcr.io/distroless/static-debian12
COPY --from=builder /server /server
CMD ["/server"]
```

<Callout type="warning" title="Distroless Limitations">
Distroless images have no shell. You cannot `docker exec` into them for debugging. This is a security feature — but it makes troubleshooting harder. Consider keeping a debug variant:

```bash
# Production: distroless (no shell)
docker build --target production -t myapp:prod .

# Debug: includes busybox shell
docker build --target debug -t myapp:debug .
```

```dockerfile
FROM gcr.io/distroless/static-debian12 AS production
COPY --from=builder /server /server
CMD ["/server"]

FROM gcr.io/distroless/static-debian12:debug AS debug
COPY --from=builder /server /server
CMD ["/server"]
```
</Callout>

## Debugging Multi-Stage Builds

### Build a Specific Stage

```bash
# Stop at the builder stage to inspect contents
docker build --target builder -t myapp:builder-debug .

# Run the builder stage interactively
docker run -it --rm myapp:builder-debug /bin/sh

# Check what files exist
docker run --rm myapp:builder-debug ls -la /app/
```

### Show Build Progress

```bash
# Detailed build output (not collapsed)
docker build --progress=plain -t myapp .

# See the size of each stage
docker build --target builder -t myapp:builder .
docker build --target production -t myapp:production .
docker images myapp
```

### Inspect Layer Differences

```bash
# Compare what's in each stage
docker history myapp:builder
docker history myapp:production
```

Take a bloated single-stage Dockerfile and convert it to an optimized multi-stage build.

**Step 1: Create a TypeScript Node.js project**

```bash
mkdir -p ~/multi-stage-exercise && cd ~/multi-stage-exercise

cat > package.json << 'EOF'
{
  "name": "multi-stage-demo",
  "version": "1.0.0",
  "scripts": {
    "build": "tsc",
    "start": "node dist/server.js"
  },
  "dependencies": {
    "express": "^4.18.2"
  },
  "devDependencies": {
    "typescript": "^5.3.3",
    "@types/express": "^4.17.21",
    "@types/node": "^20.10.0"
  }
}
EOF

cat > tsconfig.json << 'EOF'
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "commonjs",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true
  },
  "include": ["src/**/*"]
}
EOF

mkdir -p src
cat > src/server.ts << 'EOF'
import express, { Request, Response } from 'express';

const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req: Request, res: Response) => {
  res.json({
    message: 'Hello from multi-stage build!',
    environment: process.env.NODE_ENV || 'development',
    timestamp: new Date().toISOString()
  });
});

app.listen(PORT, () => {
  console.log(`Server listening on port ${PORT}`);
});
EOF

cat > .dockerignore << 'EOF'
node_modules
dist
.git
*.md
Dockerfile*
.dockerignore
EOF
```

**Step 2: Create the single-stage Dockerfile**

```bash
cat > Dockerfile.single << 'EOF'
FROM node:20
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build
ENV NODE_ENV=production
EXPOSE 3000
CMD ["node", "dist/server.js"]
EOF
```

**Step 3: Build and measure the single-stage image**

```bash
docker build -f Dockerfile.single -t demo:single .
docker images demo:single
```

**Step 4: Now write the multi-stage Dockerfile**

```bash
cat > Dockerfile.multi << 'EOF'
# Stage 1: Build
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --production

# Stage 2: Production
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
ENV NODE_ENV=production
EXPOSE 3000
CMD ["node", "dist/server.js"]
EOF
```

**Step 5: Build and compare**

```bash
docker build -f Dockerfile.multi -t demo:multi .

# Compare sizes
docker images demo
# demo:single should be ~1.1 GB
# demo:multi should be ~150 MB

# Test the multi-stage image
docker run -d -p 3000:3000 --name demo-test demo:multi
curl http://localhost:3000
docker stop demo-test && docker rm demo-test
```

**Step 1: Create the Go application**

```bash
mkdir -p ~/go-scratch-exercise && cd ~/go-scratch-exercise

cat > go.mod << 'EOF'
module go-scratch-demo

go 1.22
EOF

cat > main.go << 'EOF'
package main

import (
	"encoding/json"
	"fmt"
	"log"
	"net/http"
	"os"
	"runtime"
	"time"
)

type Response struct {
	Message   string `json:"message"`
	GoVersion string `json:"go_version"`
	OS        string `json:"os"`
	Arch      string `json:"arch"`
	Hostname  string `json:"hostname"`
	Timestamp string `json:"timestamp"`
}

func main() {
	hostname, _ := os.Hostname()

	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
		resp := Response{
			Message:   "Hello from scratch image!",
			GoVersion: runtime.Version(),
			OS:        runtime.GOOS,
			Arch:      runtime.GOARCH,
			Hostname:  hostname,
			Timestamp: time.Now().Format(time.RFC3339),
		}
		w.Header().Set("Content-Type", "application/json")
		json.NewEncoder(w).Encode(resp)
	})

	http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
		fmt.Fprintf(w, "OK")
	})

	port := os.Getenv("PORT")
	if port == "" {
		port = "8080"
	}
	log.Printf("Server starting on :%s", port)
	log.Fatal(http.ListenAndServe(":"+port, nil))
}
EOF
```

**Step 2: Create the single-stage Dockerfile (for comparison)**

```bash
cat > Dockerfile.single << 'EOF'
```

**Step 3: Create the multi-stage Dockerfile with scratch**

```bash
cat > Dockerfile.multi << 'EOF'
# Build stage
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod ./
COPY main.go ./
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build \
    -ldflags="-w -s" \

COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

# Copy the binary
COPY --from=builder /server /server

EXPOSE 8080
CMD ["/server"]
EOF
```

```bash
package main

import (
	"encoding/json"
	"log"
	"net/http"
	"os"
	"runtime"
	"time"
)

func main() {
	hostname, _ := os.Hostname()

	// Serve static files
	fs := http.FileServer(http.Dir("/static"))
	http.Handle("/", fs)

	// API endpoint
	http.HandleFunc("/api/status", func(w http.ResponseWriter, r *http.Request) {
		w.Header().Set("Content-Type", "application/json")
		json.NewEncoder(w).Encode(map[string]interface{}{
			"service":    "multi-stage-demo",
			"version":    os.Getenv("APP_VERSION"),
			"hostname":   hostname,
			"go_version": runtime.Version(),
			"timestamp":  time.Now().Format(time.RFC3339),
			"status":     "healthy",
		})
	})

	// Health check
	http.HandleFunc("/api/health", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
		w.Write([]byte(`{"healthy":true}`))
	})

	port := os.Getenv("PORT")
	if port == "" {
		port = "8080"
	}
	log.Printf("Server starting on :%s", port)
	log.Fatal(http.ListenAndServe(":"+port, nil))
}
```

**Step 2: Create the multi-stage Dockerfile**

```bash
cat > Dockerfile << 'DOCKERFILE'
# =============================================
# Stage: frontend-builder
# Minify and prepare static assets
# (runs in PARALLEL with backend-builder)
# =============================================
FROM node:20-alpine AS frontend-builder
WORKDIR /frontend
COPY frontend/ .
# In a real project, you'd run npm build here
# For this demo, we just copy the files
RUN mkdir -p /output && cp -r * /output/

# =============================================
# Stage: backend-builder
# Compile the Go binary
# (runs in PARALLEL with frontend-builder)
# =============================================
FROM golang:1.22-alpine AS backend-builder
WORKDIR /app
COPY backend/go.mod ./
RUN go mod download
COPY backend/ .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-w -s" -o /server .

# =============================================
# Stage: tester
# Run tests — build fails if tests fail
# =============================================
FROM backend-builder AS tester
RUN go test -v ./...
RUN go vet ./...

# =============================================
# Stage: development
# Full tooling for local development
# =============================================
FROM golang:1.22-alpine AS development
RUN apk add --no-cache curl wget jq bash
WORKDIR /app
COPY backend/ .
COPY --from=frontend-builder /output /static
ENV APP_VERSION=dev
ENV PORT=8080
EXPOSE 8080
CMD ["go", "run", "."]

# =============================================
# Stage: production
# Minimal image for deployment
# =============================================
FROM scratch AS production
COPY --from=backend-builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=backend-builder /server /server
COPY --from=frontend-builder /output /static
ENV APP_VERSION=1.0.0
ENV PORT=8080
EXPOSE 8080
CMD ["/server"]
DOCKERFILE
```

**Step 3: Build all targets**

```bash
cd ~/multi-stage-project

# Build and run tests (will fail the build if tests fail)
docker build --target tester -t demo:tester .

# Build development image
docker build --target development -t demo:dev .

# Build production image
docker build --target production -t demo:prod .

# Compare sizes
echo "=== Image Size Comparison ==="
docker images demo --format "table {{.Tag}}\t{{.Size}}"
```

## Key Takeaways

- Multi-stage builds use **multiple FROM statements** to separate build-time tools from runtime artifacts
- The `COPY --from=stagename` instruction copies files between stages — everything else is discarded
- Go applications can be built from **scratch** (empty image), achieving 98%+ size reduction
- Node.js and Python apps benefit from multi-stage by excluding devDependencies and build tools
- Use **named stages** (`AS builder`) for clarity and to support `--target` builds
- **BuildKit** builds independent stages in parallel, speeding up builds significantly
- Use separate **development** and **production** targets for different environments
- **Distroless** images provide maximum security but no shell for debugging
- Test stages can run during the build and **fail the build** if tests don't pass
- Multi-stage builds are one of the most impactful optimizations you can make to your Docker workflow
