<Callout type="story" title="The Story">
It is 2 AM and your pager goes off. The production API is returning 500 errors. You SSH into the server and check the logs: "Connection refused: database at localhost:5432." Someone deployed the development configuration to production. The app is trying to connect to a database that does not exist on the production server. ## The 12-Factor App: Configuration Principle

The [12-Factor App](https://12factor.net) methodology defines best practices for building modern applications. Factor III states:

> **Store config in the environment.**

Configuration is everything that varies between deployments — database URLs, API keys, feature flags, log levels. Code does not change between environments, but configuration does.

```
┌──────────────────────────────────────────────────────────┐
│                    Same Docker Image                      │
│                     myapp:v2.1.0                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Development          Staging           Production       │
│  ┌────────────┐    ┌────────────┐    ┌────────────┐     │
│  │DB=localhost │    │DB=staging- │    │DB=prod-db. │     │
│  │PORT=3000   │    │  db.intern │    │  aws.com   │     │
│  │DEBUG=true  │    │PORT=3000   │    │PORT=3000   │     │
│  │LOG=debug   │    │DEBUG=false │    │DEBUG=false │     │
│  └────────────┘    │LOG=info    │    │LOG=warn    │     │
│                    └────────────┘    └────────────┘     │
│                                                          │
│  One image. Three environments. Only config changes.     │
└──────────────────────────────────────────────────────────┘
```

<Callout type="info" title="Why Not Config Files?">
Hardcoding configuration in files inside the image means you need a different image for each environment. Environment variables let you build once, deploy everywhere. The same image binary runs in dev, staging, and production with different settings injected at runtime.
</Callout>

## ENV Instruction in Dockerfiles

The `ENV` instruction in a Dockerfile sets environment variables that persist in the built image and are available at runtime.

### Setting Default Values

```dockerfile
FROM node:20-alpine

# Set environment variables with defaults
ENV NODE_ENV=production
ENV PORT=3000
ENV LOG_LEVEL=info
ENV APP_NAME="My Application"

WORKDIR /app
COPY . .
RUN npm ci --only=production

EXPOSE $PORT

CMD ["node", "server.js"]
```

When you run this image, `NODE_ENV`, `PORT`, `LOG_LEVEL`, and `APP_NAME` are available inside the container:

```bash
# Run with default values
docker run myapp
# NODE_ENV=production, PORT=3000

# Override at runtime
docker run -e NODE_ENV=development -e PORT=8080 myapp
# NODE_ENV=development, PORT=8080
```

### ENV vs ARG

A common source of confusion is `ENV` vs `ARG`:

```dockerfile
# ARG — only available during build
ARG NODE_VERSION=20

# ENV — available during build AND at runtime
ENV NODE_ENV=production

FROM node:${NODE_VERSION}-alpine

# ARG values are NOT available here unless re-declared
# ENV values ARE available here and in the running container
```

| Feature | `ARG` | `ENV` |
|---------|-------|-------|
| Available during build | Yes | Yes |
| Available at runtime | No | Yes |
| Set from command line | `--build-arg` | `-e` / `--env` |
| Persists in image layers | No | Yes |
| Use case | Build-time configuration | Runtime configuration |
| Visible in image history | Yes (security risk!) | Yes |

<Callout type="warning" title="Never Put Secrets in ARG or ENV">
Both `ARG` and `ENV` values are visible in the image layer history via `docker history`. Never use them for passwords, API keys, or other secrets during the build process. Use multi-stage builds or Docker BuildKit secrets instead.

```bash
# Anyone can see your ARG/ENV values:
docker history myapp
# STEP 3: ENV DATABASE_PASSWORD=supersecret   <-- exposed!
```
</Callout>

### Best Practices for ENV in Dockerfiles

```dockerfile
# GOOD: Set sensible defaults that work for most environments
ENV NODE_ENV=production
ENV PORT=3000
ENV LOG_LEVEL=info

# GOOD: Use ENV for non-sensitive configuration
ENV APP_VERSION=2.1.0
ENV TZ=UTC

# BAD: Never put secrets in ENV
# ENV DATABASE_PASSWORD=mysecret     # Don't do this!
# ENV API_KEY=sk-abc123              # Don't do this!

# BAD: Don't set environment-specific values
# ENV DATABASE_URL=postgres://prod-server:5432/mydb  # Don't do this!
```

## Runtime Environment Variables

### With docker run

Pass environment variables when starting a container:

```bash
# Using -e flag (one variable at a time)
docker run -d \
  -e NODE_ENV=production \
  -e DATABASE_URL="postgres://user:pass@db:5432/myapp" \
  -e REDIS_URL="redis://redis:6379" \
  -e LOG_LEVEL=warn \
  myapp:latest

# Using --env (same as -e)
docker run -d \
  --env NODE_ENV=production \
  --env DATABASE_URL="postgres://user:pass@db:5432/myapp" \
  myapp:latest

# Pass a variable from your host shell
export API_KEY="sk-abc123"
docker run -d -e API_KEY myapp:latest
# The container gets API_KEY=sk-abc123 from the host environment
```

### With --env-file

Load variables from a file:

```bash
# Create an env file
cat > app.env << 'EOF'
NODE_ENV=production
DATABASE_URL=postgres://user:pass@db:5432/myapp
REDIS_URL=redis://redis:6379
LOG_LEVEL=warn
API_KEY=sk-abc123
EOF

# Pass the file to docker run
docker run -d --env-file app.env myapp:latest

# Multiple env files
docker run -d \
  --env-file common.env \
  --env-file production.env \
  myapp:latest
```

### Inspecting Environment Variables

```bash
# See environment variables of a running container
docker exec mycontainer env

# Output:
# PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
# NODE_ENV=production
# DATABASE_URL=postgres://user:pass@db:5432/myapp
# REDIS_URL=redis://redis:6379
# LOG_LEVEL=warn

# Or inspect the container configuration
docker inspect mycontainer --format '{{range .Config.Env}}{{println .}}{{end}}'

# Check a specific variable
docker exec mycontainer printenv DATABASE_URL
```

<Callout type="warning" title="Environment Variables Are Visible">
Anyone with access to `docker inspect` or `docker exec` can see all environment variables. This includes secrets passed via `-e`. For highly sensitive data in production, consider Docker secrets or a secrets manager like HashiCorp Vault.
</Callout>

## Docker Compose Environment Configuration

Compose provides several ways to configure environment variables for services.

### The environment Key

```yaml
services:
  api:
    image: myapp:latest
    environment:
      # Map syntax (recommended — cleaner)
      NODE_ENV: production
      PORT: "3000"
      LOG_LEVEL: info
      DATABASE_URL: postgres://user:pass@db:5432/myapp

  api-alt:
    image: myapp:latest
    environment:
      # List syntax (also valid)
      - NODE_ENV=production
      - PORT=3000
      - LOG_LEVEL=info
      - DATABASE_URL=postgres://user:pass@db:5432/myapp
```

### The env_file Key

Load variables from external files:

```yaml
services:
  api:
    image: myapp:latest
    env_file:
      - .env              # Shared defaults
      - .env.production   # Environment-specific overrides

  worker:
    image: myapp:latest
    env_file:
      - .env
      - .env.worker       # Worker-specific settings
```

```bash
# .env
NODE_ENV=production
LOG_LEVEL=info
TZ=UTC

# .env.production
DATABASE_URL=postgres://produser:prodpass@prod-db:5432/myapp
REDIS_URL=redis://prod-redis:6379
API_KEY=sk-production-key

# .env.worker
WORKER_CONCURRENCY=4
WORKER_QUEUE=high,medium,low
```

### Variable Substitution

Compose can interpolate variables from your shell or `.env` file:

```yaml
services:
  db:
    image: postgres:${POSTGRES_VERSION:-16}   # Default to 16 if not set
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    ports:
      - "${DB_PORT:-5432}:5432"

  api:
    image: myapp:${APP_VERSION}
    environment:
      DATABASE_URL: postgres://${DB_USER}:${DB_PASSWORD}@db:5432/${DB_NAME}
```

```bash
# .env (loaded automatically by Compose)
POSTGRES_VERSION=16
DB_USER=appuser
DB_PASSWORD=secret123
DB_NAME=myapp
DB_PORT=5432
APP_VERSION=2.1.0
```

<Callout type="info" title="Compose .env File">
Docker Compose automatically loads a file named `.env` from the project directory (where `docker-compose.yml` is). These variables are available for substitution in the Compose file itself. This is different from `env_file`, which loads variables into the container.

```
.env file    -->  Variable substitution in docker-compose.yml
env_file     -->  Environment variables inside the container
```
</Callout>

### Variable Substitution Syntax

```yaml
services:
  app:
    image: myapp:${VERSION}              # Required — fails if not set
    environment:
      PORT: ${PORT:-3000}                # Default: use 3000 if PORT not set
      HOST: ${HOST:-0.0.0.0}            # Default: use 0.0.0.0 if HOST not set
      DB: ${DATABASE_URL:?Database URL required}  # Error if not set
```

| Syntax | Meaning |
|--------|---------|
| `${VAR}` | Value of VAR, empty string if not set |
| `${VAR:-default}` | Value of VAR, or "default" if not set or empty |
| `${VAR-default}` | Value of VAR, or "default" if not set (allows empty) |
| `${VAR:?error message}` | Value of VAR, or exit with error if not set or empty |
| `${VAR?error message}` | Value of VAR, or exit with error if not set |

## .env File Best Practices

### Syntax Rules

```bash
# .env file syntax

# Simple key=value
DATABASE_URL=postgres://user:pass@localhost:5432/mydb

# Quoted values (for spaces or special characters)
APP_NAME="My Application"
GREETING='Hello World'

# No spaces around =
GOOD=value
# BAD = value      <-- This breaks

# Comments start with #
# This is a comment
PORT=3000   # Inline comments are NOT supported in all parsers

# Empty values
EMPTY_VAR=
# This sets EMPTY_VAR to an empty string

# Multi-line values (with quotes)
PRIVATE_KEY="-----BEGIN RSA PRIVATE KEY-----
MIIEowIBAAKCAQEA...
-----END RSA PRIVATE KEY-----"
```

### Managing Multiple Environments

```
project/
├── docker-compose.yml
├── .env                    # Default/shared values (committed to git)
├── .env.example            # Template with placeholder values (committed)
├── .env.local              # Local overrides (NOT committed — in .gitignore)
├── .env.development        # Development settings
├── .env.staging            # Staging settings
└── .env.production         # Production settings (NOT committed)
```

```bash
# .env.example (committed to version control)
DATABASE_URL=postgres://user:password@localhost:5432/dbname
REDIS_URL=redis://localhost:6379
API_KEY=your-api-key-here
SECRET_KEY=generate-a-random-key

# .gitignore
.env.local
.env.production
.env*.local
```

<Callout type="tip" title="Always Commit .env.example">
Create a `.env.example` file with placeholder values and commit it to version control. This documents what environment variables your application needs without exposing real credentials. New developers copy it: `cp .env.example .env.local` and fill in their values.
</Callout>

### Priority Order

When the same variable is defined in multiple places, Compose uses this priority (highest to lowest):

```
1. Environment variables from the shell (export VAR=value)
2. Variables in the .env file
3. Variables in env_file
4. Variables in environment key
5. ENV instruction in the Dockerfile
```

```bash
# Example: PORT is defined everywhere
# Dockerfile:           ENV PORT=3000
# docker-compose.yml:   environment: PORT=4000
# .env file:            PORT=5000
# Shell:                export PORT=6000

# Result: PORT=6000 (shell wins)
```

## Docker Secrets

For sensitive data in production, Docker provides a secrets mechanism that is more secure than environment variables.

### Why Secrets Instead of Environment Variables?

| Feature | Environment Variables | Docker Secrets |
|---------|----------------------|---------------|
| Visible in `docker inspect` | Yes | No |
| Visible in process list | Sometimes | No |
| Stored on disk | In container config | Encrypted (Swarm) or tmpfs |
| Access method | `$VAR` | Read file from `/run/secrets/` |
| Rotation | Restart required | Can update without restart |

### Using Secrets with Docker Compose

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
      POSTGRES_DB: myapp
    secrets:
      - db_password

  api:
    build: ./api
    environment:
      DATABASE_PASSWORD_FILE: /run/secrets/db_password
      API_KEY_FILE: /run/secrets/api_key
    secrets:
      - db_password
      - api_key

secrets:
  db_password:
    file: ./secrets/db_password.txt    # Read from a local file
  api_key:
    file: ./secrets/api_key.txt
```

```bash
# Create secrets directory
mkdir -p secrets

# Create secret files
echo -n "supersecretpassword" > secrets/db_password.txt
echo -n "sk-api-key-12345" > secrets/api_key.txt

# Add to .gitignore
echo "secrets/" >> .gitignore
```

<Callout type="info" title="The _FILE Convention">
Many official Docker images (PostgreSQL, MySQL, MariaDB) support the `_FILE` suffix convention. Instead of `POSTGRES_PASSWORD=mysecret`, you use `POSTGRES_PASSWORD_FILE=/run/secrets/db_password`. The entrypoint script reads the file and sets the variable. This way, the secret never appears in environment variables.
</Callout>

### Reading Secrets in Your Application

Inside the container, secrets are mounted as files at `/run/secrets/<secret_name>`:

```bash
# Inside the container
cat /run/secrets/db_password
# Output: supersecretpassword

ls -la /run/secrets/
# -r--r--r-- 1 root root 19 Jan 15 10:00 db_password
# -r--r--r-- 1 root root 16 Jan 15 10:00 api_key
```

Your application needs to read from files instead of environment variables:

```javascript
// Node.js — reading secrets
const fs = require('fs');

function getSecret(name) {
  const secretPath = `/run/secrets/${name}`;
  try {
    return fs.readFileSync(secretPath, 'utf8').trim();
  } catch (err) {
    // Fall back to environment variable
    return process.env[name.toUpperCase()];
  }
}

const dbPassword = getSecret('db_password');
const apiKey = getSecret('api_key');
```

```python
# Python — reading secrets
import os

def get_secret(name):
    secret_path = f"/run/secrets/{name}"
    try:
        with open(secret_path, 'r') as f:
            return f.read().strip()
    except FileNotFoundError:
        return os.environ.get(name.upper())

db_password = get_secret('db_password')
api_key = get_secret('api_key')
```

## Configuration Patterns

### Pattern 1: Environment Variables for Simple Config

Best for: non-sensitive settings, feature flags, log levels.

```yaml
services:
  api:
    image: myapp:latest
    environment:
      NODE_ENV: production
      PORT: "3000"
      LOG_LEVEL: warn
      FEATURE_NEW_UI: "true"
      MAX_UPLOAD_SIZE: "10MB"
```

### Pattern 2: Config Files for Complex Configuration

Best for: Nginx configs, application config files, complex structured data.

```yaml
services:
  nginx:
    image: nginx:alpine
    volumes:
      - ./config/nginx.conf:/etc/nginx/nginx.conf:ro

  prometheus:
    image: prom/prometheus
    volumes:
      - ./config/prometheus.yml:/etc/prometheus/prometheus.yml:ro
```

### Pattern 3: Secrets for Sensitive Data

Best for: passwords, API keys, TLS certificates, private keys.

```yaml
services:
  api:
    image: myapp:latest
    secrets:
      - db_password
      - api_key
      - tls_cert
      - tls_key

secrets:
  db_password:
    file: ./secrets/db_password.txt
  api_key:
    file: ./secrets/api_key.txt
  tls_cert:
    file: ./certs/server.crt
  tls_key:
    file: ./certs/server.key
```

### Pattern 4: Combining All Three

```yaml
services:
  api:
    build: ./api
    # Non-sensitive config as environment variables
    environment:
      NODE_ENV: production
      PORT: "3000"
      LOG_LEVEL: warn
      DB_HOST: db
      DB_PORT: "5432"
      DB_NAME: myapp
    # Sensitive data as secrets
    secrets:
      - db_password
      - jwt_secret
    # Complex config as mounted files
    volumes:
      - ./config/app.yml:/app/config/app.yml:ro

secrets:
  db_password:
    file: ./secrets/db_password.txt
  jwt_secret:
    file: ./secrets/jwt_secret.txt
```

## Sensitive Data Handling

### What Counts as Sensitive?

| Sensitive (Use Secrets) | Not Sensitive (Use ENV) |
|------------------------|------------------------|
| Database passwords | Database host/port |
| API keys / tokens | Log level |
| TLS private keys | Application port |
| JWT signing secrets | Feature flags |
| OAuth client secrets | Timezone |
| Encryption keys | Environment name |
| Cloud credentials | Max upload size |

### Common Mistakes to Avoid

```dockerfile
# MISTAKE 1: Hardcoding secrets in Dockerfile
ENV DATABASE_PASSWORD=mysecretpass
# Fix: Pass at runtime with -e or use secrets

# MISTAKE 2: Copying .env into the image
COPY .env /app/.env
# Fix: Mount at runtime or use env_file in Compose

# MISTAKE 3: Using ARG for secrets (visible in history)
ARG GITHUB_TOKEN
RUN git clone https://${GITHUB_TOKEN}@github.com/private/repo.git
# Fix: Use BuildKit secrets

# MISTAKE 4: Logging secrets
RUN echo "Password is $DB_PASSWORD"
# Fix: Never echo secrets, even in build steps
```

### Using BuildKit Secrets for Build-Time Secrets

When you need a secret during the Docker build (like accessing a private registry or repository):

```dockerfile
# syntax=docker/dockerfile:1

FROM node:20-alpine
WORKDIR /app

# Mount the secret only during this RUN step — never stored in a layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci --only=production

COPY . .
CMD ["node", "server.js"]
```

```bash
# Build with the secret
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .

# The .npmrc file is available during build but NOT in the final image
docker history myapp   # No trace of the secret
```

<Callout type="tip" title="Always Scan for Leaked Secrets">
Use tools like `trufflehog`, `git-secrets`, or `docker scan` to check your images and repositories for accidentally committed secrets. Add a pre-commit hook to catch secrets before they reach version control.
</Callout>

<CollapsibleSolution>

```bash
# Create project structure
mkdir -p ~/secrets-demo/{api,secrets}

# Create secret files
echo -n "db-super-secret-pass" > ~/secrets-demo/secrets/db_password.txt
echo -n "sk-api-key-98765" > ~/secrets-demo/secrets/api_key.txt
echo -n "jwt-signing-secret-xyz" > ~/secrets-demo/secrets/jwt_secret.txt

# Create .gitignore (always do this!)
cat > ~/secrets-demo/.gitignore << 'EOF'
secrets/
.env.local
.env.production
EOF

# Create the API server
cat > ~/secrets-demo/api/server.js << 'SERVEREOF'
const http = require('http');
const fs = require('fs');

// Helper function to read secrets
function getSecret(name) {
  const secretPath = `/run/secrets/${name}`;
  try {
    const value = fs.readFileSync(secretPath, 'utf8').trim();
    console.log(`Secret '${name}' loaded from ${secretPath}`);
    return { value, source: 'docker-secret' };
  } catch (err) {
    const envValue = process.env[name.toUpperCase()];
    if (envValue) {
      console.log(`Secret '${name}' loaded from environment variable`);
      return { value: envValue, source: 'environment-variable' };
    }
```

## Key Takeaways

- Follow the 12-Factor App principle: store configuration in the environment, never in the code or image
- Use `ENV` in Dockerfiles for sensible defaults, override them at runtime with `-e` or Compose `environment`
- Docker Compose loads `.env` automatically for variable substitution in the Compose file; `env_file` loads variables into containers
- Environment variable priority: shell > `.env` file > `env_file` > Compose `environment` > Dockerfile `ENV`
- Never bake secrets into images — use Docker secrets, the `_FILE` convention, or a secrets manager
- Always commit a `.env.example` with placeholder values; add real `.env` files to `.gitignore`
- Use Compose override files (`docker-compose.override.yml`, `docker-compose.prod.yml`) to manage per-environment differences from a single base configuration
