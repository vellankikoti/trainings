<Callout type="story" title="The Story">
Friday afternoon. A chart upgrade rolls out to staging and immediately crashes -- a typo in a template produces invalid YAML, a required value is missing, and the health check endpoint was renamed two versions ago. None of this was caught because nobody tested the chart. You decide that Helm charts deserve the same testing rigor as application code: linting for syntax, template validation for correctness, schema checks for required values, and integration tests to verify the deployed application actually works.

## The Testing Pyramid for Helm Charts

Testing Helm charts happens at multiple levels, from fast static checks to full cluster integration tests:

```
        /  Integration Tests  \      ← helm test, ct install
       /   Template Validation  \    ← helm template + kubeconform
      /    Schema Validation     \   ← values.schema.json
     /      Chart Linting         \  ← helm lint
    /________________________________\
```

Each layer catches different problems. Use all of them together for comprehensive coverage.

## Linting with helm lint

The fastest check -- catches structural issues, missing required fields, and template syntax errors:

```bash
# Basic lint
helm lint ./my-app

# Lint with specific values
helm lint ./my-app -f values-production.yaml

# Lint with --strict to treat warnings as errors
helm lint ./my-app --strict

# Lint with debug for verbose output
helm lint ./my-app --debug
```

Common issues `helm lint` catches:

- Missing `Chart.yaml` required fields
- Invalid template syntax (unclosed brackets, undefined variables)
- Deprecated Kubernetes API versions
- Files in wrong directories

## Template Rendering with helm template

Render templates locally to verify the output is valid:

```bash
# Render all templates
helm template my-release ./my-app

# Render a specific template
helm template my-release ./my-app -s templates/deployment.yaml

# Render with custom values
helm template my-release ./my-app -f values-production.yaml

# Include hooks in the output
helm template my-release ./my-app --show-only templates/pre-install-job.yaml
```

### Validating Against Kubernetes Schemas

Use tools like `kubeconform` or `kubeval` to validate rendered output against Kubernetes API schemas:

```bash
# Install kubeconform
go install github.com/yannh/kubeconform/cmd/kubeconform@latest

# Render and validate
helm template my-release ./my-app | kubeconform -strict -summary

# Validate against a specific Kubernetes version
helm template my-release ./my-app | \
  kubeconform -kubernetes-version 1.28.0 -strict

# Validate with custom resource definitions
helm template my-release ./my-app | \
  kubeconform -schema-location default \
  -schema-location 'https://raw.githubusercontent.com/datreeio/CRDs-catalog/main/{{.Group}}/{{.ResourceKind}}_{{.ResourceAPIVersion}}.json'
```

## Dry-Run Against the Cluster

Validate templates with the actual Kubernetes API server:

```bash
# Client-side dry-run (renders templates, no cluster needed)
helm install my-release ./my-app --dry-run

# Server-side dry-run (validates against the API server)
helm install my-release ./my-app --dry-run=server

# With debug output
helm install my-release ./my-app --dry-run --debug
```

Server-side dry-run catches issues that static validation misses:
- Invalid resource references (missing Secrets, ConfigMaps)
- RBAC permission issues
- Namespace conflicts
- Custom resource validation

## Schema Validation with values.schema.json

Define a JSON schema to validate user-provided values:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["image"],
  "properties": {
    "replicaCount": {
      "type": "integer",
      "minimum": 1,
      "maximum": 100,
      "description": "Number of pod replicas"
    },
    "image": {
      "type": "object",
      "required": ["repository"],
      "properties": {
        "repository": {
          "type": "string",
          "pattern": "^[a-z0-9.-/]+$"
        },
        "tag": {
          "type": "string",
          "minLength": 1
        }
      }
    },
    "service": {
      "type": "object",
      "properties": {
        "type": {
          "enum": ["ClusterIP", "NodePort", "LoadBalancer"]
        },
        "port": {
          "type": "integer",
          "minimum": 1,
          "maximum": 65535
        }
      }
    }
  }
}
```

The schema is validated automatically during `helm install`, `helm upgrade`, `helm lint`, and `helm template`:

```bash
# This fails with a clear error message
helm install my-app ./my-app --set replicaCount=-1
# Error: values don't meet the specifications of the schema
```

## Writing Helm Tests

Helm tests are templates with the `helm.sh/hook: test` annotation. They run as Pods when you execute `helm test`:

```yaml
# templates/tests/test-connection.yaml
apiVersion: v1
kind: Pod
metadata:
  name: {{ include "my-app.fullname" . }}-test-connection
  labels:
    {{- include "my-app.labels" . | nindent 4 }}
  annotations:
    "helm.sh/hook": test
spec:
  restartPolicy: Never
  containers:
    - name: wget
      image: busybox:1.36
      command: ['wget']
      args: ['{{ include "my-app.fullname" . }}:{{ .Values.service.port }}']
```

```yaml
# templates/tests/test-health.yaml
apiVersion: v1
kind: Pod
metadata:
  name: {{ include "my-app.fullname" . }}-test-health
  annotations:
    "helm.sh/hook": test
    "helm.sh/hook-weight": "1"
spec:
  restartPolicy: Never
  containers:
    - name: health-check
      image: curlimages/curl:latest
      command:
        - sh
        - -c
        - |
          echo "Testing health endpoint..."
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
            http://{{ include "my-app.fullname" . }}:{{ .Values.service.port }}/healthz)
          if [ "$STATUS" = "200" ]; then
            echo "Health check passed"
            exit 0
          else
            echo "Health check failed with status $STATUS"
            exit 1
          fi
```

```yaml
# templates/tests/test-api.yaml
apiVersion: v1
kind: Pod
metadata:
  name: {{ include "my-app.fullname" . }}-test-api
  annotations:
    "helm.sh/hook": test
    "helm.sh/hook-weight": "2"
spec:
  restartPolicy: Never
  containers:
    - name: api-test
      image: curlimages/curl:latest
      command:
        - sh
        - -c
        - |
          echo "Testing API endpoint..."
          RESPONSE=$(curl -s http://{{ include "my-app.fullname" . }}:{{ .Values.service.port }}/api/v1/status)
          echo "Response: $RESPONSE"
          echo "$RESPONSE" | grep -q '"status":"healthy"' || exit 1
          echo "API test passed"
```

### Running Tests

```bash
# Run all tests for a release
helm test my-release

# Run with a timeout
helm test my-release --timeout 5m

# View test logs
helm test my-release --logs
```

## Chart Testing Tool (ct)

The `chart-testing` tool (`ct`) from the Helm project automates chart testing in CI:

```bash
# Install ct
pip install chart-testing

# Or use the Docker image
docker run --rm -it quay.io/helmpack/chart-testing ct version
```

### Linting with ct

```bash
# Lint all changed charts in a Git repository
ct lint --target-branch main

# Lint with custom configuration
ct lint --config ct.yaml
```

```yaml
# ct.yaml
target-branch: main
chart-dirs:
  - charts
helm-extra-args: --timeout 600s
validate-maintainers: false
check-version-increment: true
```

### Integration Testing with ct

```bash
# Install changed charts and run tests
ct install --target-branch main

# Install with upgrade testing
ct install --target-branch main --upgrade
```

This workflow:
1. Detects which charts changed compared to the target branch
2. Lints them
3. Installs them in a fresh namespace
4. Runs `helm test`
5. Optionally tests upgrades from the previous version
6. Cleans up

## Unit Testing with helm-unittest

For testing template logic without a cluster, use `helm-unittest`:

```bash
# Install the plugin
helm plugin install https://github.com/helm-unittest/helm-unittest
```

Write test files in `tests/`:

```yaml
# tests/deployment_test.yaml
suite: deployment tests
templates:
  - deployment.yaml
tests:
  - it: should set the correct replica count
    set:
      replicaCount: 5
    asserts:
      - equal:
          path: spec.replicas
          value: 5

  - it: should use the correct image
    set:
      image:
        repository: myapp
        tag: "v2.0"
    asserts:
      - equal:
          path: spec.template.spec.containers[0].image
          value: "myapp:v2.0"

  - it: should not create HPA when autoscaling is disabled
    templates:
      - hpa.yaml
    set:
      autoscaling:
        enabled: false
    asserts:
      - hasDocuments:
          count: 0

  - it: should set resource limits
    set:
      resources:
        limits:
          cpu: "1"
          memory: 512Mi
    asserts:
      - equal:
          path: spec.template.spec.containers[0].resources.limits.cpu
          value: "1"
      - equal:
          path: spec.template.spec.containers[0].resources.limits.memory
          value: 512Mi
```

```bash
# Run unit tests
helm unittest ./my-app

# Run with verbose output
helm unittest -v ./my-app
```

## CI/CD Pipeline Example

A complete testing pipeline for Helm charts:

```yaml
# .github/workflows/helm-test.yaml
name: Helm Chart CI
on:
  pull_request:
    paths:
      - 'charts/**'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: azure/setup-helm@v4
      - name: Lint charts
        run: |
          helm lint charts/my-app --strict
          helm lint charts/my-app -f charts/my-app/values-production.yaml --strict

  template-validation:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: azure/setup-helm@v4
      - name: Render and validate templates
        run: |
          helm template test charts/my-app | kubeconform -strict -summary
          helm template test charts/my-app -f charts/my-app/values-production.yaml | kubeconform -strict

  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: azure/setup-helm@v4
      - name: Install unittest plugin
        run: helm plugin install https://github.com/helm-unittest/helm-unittest
      - name: Run unit tests
        run: helm unittest charts/my-app

  integration:
    runs-on: ubuntu-latest
    needs: [lint, template-validation, unit-tests]
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: azure/setup-helm@v4
      - name: Create Kind cluster
        uses: helm/kind-action@v1
      - name: Install and test chart
        run: |
          helm install test-release charts/my-app --wait --timeout 5m
          helm test test-release --logs
```

## Summary

- `helm lint` catches structural and syntax errors -- run it first and always
- `helm template` renders manifests locally; pipe to `kubeconform` for Kubernetes schema validation
- `values.schema.json` enforces value types, ranges, and required fields at install time
- `helm test` runs integration tests as Pods in the cluster after deployment
- `helm-unittest` lets you unit test template logic without a cluster
- `ct` (chart-testing) automates linting and integration testing in CI pipelines
- Combine all layers in CI/CD for comprehensive chart validation before production
