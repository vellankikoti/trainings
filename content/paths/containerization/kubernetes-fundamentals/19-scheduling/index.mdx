<Callout type="story" title="The Story">
Your cluster has a mix of nodes: some with GPUs for ML workloads, some with fast SSDs for databases, and general-purpose nodes for everything else. Without scheduling constraints, Kubernetes might place a GPU-intensive training job on a general-purpose node (wasting money on the unused GPU node) or pack all your API replicas on the same node (losing availability if that node fails). Scheduling controls let you direct pods to the right nodes and spread them for resilience.
</Callout>

## Node Selectors

The simplest scheduling constraint — place pods on nodes with matching labels:

```bash
# Label a node
kubectl label node worker-3 disk=ssd
```

```yaml
spec:
  nodeSelector:
    disk: ssd
```

Simple but limited: only equality-based matching, no soft preferences.

## Taints and Tolerations

Taints **repel** pods from nodes. Tolerations let specific pods **ignore** the taint.

```bash
# Taint a node — only GPU workloads should run here
kubectl taint nodes gpu-node-1 gpu=true:NoSchedule
```

Only pods with a matching toleration will be scheduled:

```yaml
spec:
  tolerations:
    - key: "gpu"
      operator: "Equal"
      value: "true"
      effect: "NoSchedule"
```

### Taint Effects

| Effect | Behavior |
|--------|----------|
| `NoSchedule` | New pods won't be scheduled (existing pods stay) |
| `PreferNoSchedule` | Try to avoid, but schedule if necessary |
| `NoExecute` | Evict existing pods that don't tolerate |

## Node Affinity

More expressive than nodeSelector — supports "preferred" (soft) and "required" (hard) rules:

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                  - us-east-1a
                  - us-east-1b
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 80
          preference:
            matchExpressions:
              - key: disk
                operator: In
                values:
                  - ssd
```

## Pod Affinity and Anti-Affinity

### Pod Affinity — Co-locate Related Pods

Run the cache on the same node as the API for low-latency access:

```yaml
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values:
                  - backend-api
          topologyKey: kubernetes.io/hostname
```

### Pod Anti-Affinity — Spread Pods Apart

Ensure API replicas run on different nodes for high availability:

```yaml
spec:
  affinity:
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: backend-api
            topologyKey: kubernetes.io/hostname
```

## Topology Spread Constraints

Evenly distribute pods across zones or nodes:

```yaml
spec:
  topologySpreadConstraints:
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule
      labelSelector:
        matchLabels:
          app: backend-api
```

## Summary

- **Node selectors**: Simple label-based node targeting
- **Taints/tolerations**: Repel pods from nodes (or allow specific ones)
- **Node affinity**: Flexible required/preferred node placement
- **Pod affinity**: Co-locate related pods for performance
- **Pod anti-affinity**: Spread pods apart for availability
- **Topology spread**: Even distribution across failure domains
