<Callout type="story" title="The Story">
A script that started as 50 lines of bash grew to 500 lines. It needed JSON parsing from an API, conditional logic based on nested response fields, error handling with retries, and Slack notifications on failure. Every change broke something else. One afternoon, we rewrote it in Python — half the code, twice the readability, proper error handling, and unit tests. Python isn't replacing bash — it's for when bash isn't enough.
</Callout>

## The DevOps Language Problem

Every DevOps engineer faces the same crossroads: your bash script works, but it's getting complicated. You need to parse JSON, call REST APIs, handle errors gracefully, or process complex data. This is where Python steps in.

Python isn't the only option — Go, Ruby, and Perl all have their place. But Python has won the DevOps ecosystem for very specific reasons.

### When Bash vs Python

The decision isn't bash OR Python. It's knowing when each is the right tool.

| Criteria | Use Bash | Use Python |
|---|---|---|
| Combining CLI commands | Yes | Overkill |
| Quick file manipulation | Yes | Overkill |
| JSON/YAML parsing | Painful (jq helps) | Native and easy |
| REST API interactions | Hacky (curl + jq) | Clean (requests) |
| Error handling | Limited (set -e) | Full try/except |
| Complex data structures | Arrays only | Lists, dicts, sets, classes |
| Unit testing | Difficult | Built-in (pytest) |
| Cross-platform | Linux/Mac only | Everywhere |
| Lines of code threshold | Under 100 lines | Over 100 lines |

<Callout type="tip" title="The 100-Line Rule">
If your bash script passes 100 lines, or if you find yourself fighting with string parsing and nested conditionals, it's time to rewrite in Python. The investment pays off immediately in maintainability.
</Callout>

## Python's DevOps Strengths

### 1. Readability Is Maintainability

In DevOps, scripts get handed between team members, revisited at 3 AM during incidents, and maintained for years. Readability matters more than cleverness.

```python
# Python: clear intent, easy to understand at 3 AM
def check_server_health(servers):
    """Check health of all servers and return unhealthy ones."""
    unhealthy = []
    for server in servers:
        try:
            response = requests.get(f"http://{server}/health", timeout=5)
            if response.status_code != 200:
                unhealthy.append(server)
        except requests.ConnectionError:
            unhealthy.append(server)
    return unhealthy
```

Compare the equivalent bash:

```bash
# Bash: works but harder to maintain and extend
check_servers() {
    local unhealthy=()
    for server in "$@"; do
        status=$(curl -s -o /dev/null -w "%{http_code}" \
            --connect-timeout 5 "http://${server}/health" 2>/dev/null)
        if [ "$status" != "200" ]; then
            unhealthy+=("$server")
        fi
    done
    echo "${unhealthy[@]}"
}
```

### 2. Error Handling That Actually Works

# Python: granular error handling
try:

### A Real Example: Deployment Notifier

```python
#!/usr/bin/env python3
"""Notify Slack when a deployment starts, succeeds, or fails."""

import json
import os
import sys
import requests

SLACK_WEBHOOK = os.environ.get("SLACK_WEBHOOK_URL")

def notify(message, color="good"):
    """Send a formatted message to Slack."""
    payload = {
        "attachments": [{
            "color": color,
            "text": message,
            "footer": f"Deploy Bot | {os.environ.get('CI_PIPELINE_ID', 'manual')}",
        }]
    }
    requests.post(SLACK_WEBHOOK, json=payload, timeout=10)

if __name__ == "__main__":
    action = sys.argv[1] if len(sys.argv) > 1 else "unknown"
    service = os.environ.get("SERVICE_NAME", "unknown-service")
    version = os.environ.get("VERSION", "unknown")

    messages = {
        "start":   (f"Deploying *{service}* v{version}...", "warning"),
        "success": (f"Deployed *{service}* v{version} successfully", "good"),
        "failure": (f"FAILED to deploy *{service}* v{version}", "danger"),
    }

    msg, color = messages.get(action, (f"Unknown action: {action}", "warning"))
    notify(msg, color)
```

## Python vs Other DevOps Languages

| Language | Strengths | When to Use |
|---|---|---|
| **Bash** | Already installed, great for CLI glue | Quick tasks, system commands |
| **Python** | Readable, huge ecosystem, rapid development | Automation, APIs, data processing |

Python sits in the sweet spot: more powerful than bash, faster to write than Go, with the largest DevOps ecosystem of any language.

<Exercise number={2} title="Your First DevOps Python Script">

Write a Python script that does something useful for a DevOps engineer. This script checks if critical ports are open:

```python
#!/usr/bin/env python3
"""Check if critical services are responding on expected ports."""

import socket
import sys

# Services to check: (host, port, name)
SERVICES = [
    ("localhost", 22,   "SSH"),
    ("localhost", 80,   "HTTP"),
    ("localhost", 443,  "HTTPS"),
    ("localhost", 5432, "PostgreSQL"),
    ("localhost", 6379, "Redis"),
]

def check_port(host, port, timeout=3):
    """Return True if the port is open and accepting connections."""
    try:
        with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as sock:
            sock.settimeout(timeout)
            result = sock.connect_ex((host, port))
            return result == 0
    except socket.error:
        return False

def main():
    print(f"{'Service':<15} {'Host':<15} {'Port':<8} {'Status'}")
    print("-" * 50)

    failures = 0
    for host, port, name in SERVICES:
        is_open = check_port(host, port)
        status = "UP" if is_open else "DOWN"
        symbol = "+" if is_open else "!"
        print(f"  [{symbol}] {name:<13} {host:<15} {port:<8} {status}")
        if not is_open:
            failures += 1

    print(f"\nResult: {len(SERVICES) - failures}/{len(SERVICES)} services healthy")
    sys.exit(1 if failures > 0 else 0)

if __name__ == "__main__":
    main()
```

Run it with `python3 port_check.py`. Notice how the exit code signals success or failure — just like any good DevOps tool.

<MiniProject title="Bash vs Python Comparison">

Implement the same task in both bash and Python to see the difference firsthand. The task: check disk usage on all mounted filesystems and alert if any exceed 80%.

**Bash version:**

```bash
#!/bin/bash
# disk_check.sh — Alert on high disk usage

THRESHOLD=80

echo "=== Disk Usage Check ==="
echo "Threshold: ${THRESHOLD}%"
echo ""

alert=0
while read -r line; do
    usage=$(echo "$line" | awk '{print $5}' | tr -d '%')
    mount=$(echo "$line" | awk '{print $6}')
    filesystem=$(echo "$line" | awk '{print $1}')

    if [ "$usage" -ge "$THRESHOLD" ] 2>/dev/null; then
        echo "[ALERT] $mount is at ${usage}% ($filesystem)"
        alert=1
    else
        echo "[  OK ] $mount is at ${usage}% ($filesystem)"
    fi
done < <(df -h | grep -v "^Filesystem" | grep -v "tmpfs")

echo ""
if [ "$alert" -eq 1 ]; then
    echo "WARNING: One or more filesystems exceeded ${THRESHOLD}%"
    exit 1
else
    echo "All filesystems within threshold"
    exit 0
fi
```

**Python version:**

```python
#!/usr/bin/env python3
"""disk_check.py — Alert on high disk usage with detailed reporting."""

import shutil
import os
import json
from pathlib import Path
from datetime import datetime

THRESHOLD = 80

def get_disk_usage():
    """Get disk usage for all mount points."""
    results = []
    seen_devices = set()

    with open("/proc/mounts") as f:
        for line in f:
            parts = line.split()
            device, mount_point = parts[0], parts[1]

            # Skip virtual filesystems
            if not device.startswith("/dev/"):
                continue
            if device in seen_devices:
                continue
            seen_devices.add(device)

            try:
                usage = shutil.disk_usage(mount_point)
                percent = (usage.used / usage.total) * 100
                results.append({
                    "device": device,
                    "mount": mount_point,
                    "total_gb": round(usage.total / (1024**3), 1),
                    "used_gb": round(usage.used / (1024**3), 1),
                    "free_gb": round(usage.free / (1024**3), 1),
                    "percent": round(percent, 1),
                    "status": "ALERT" if percent >= THRESHOLD else "OK",
                })
            except PermissionError:
                continue

    return sorted(results, key=lambda x: x["percent"], reverse=True)

def main():
    disks = get_disk_usage()
    alerts = [d for d in disks if d["status"] == "ALERT"]

    print(f"Disk Usage Report — {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}")
    print(f"Threshold: {THRESHOLD}%")
    print(f"{'Mount':<20} {'Device':<15} {'Used':<10} {'Free':<10} {'Usage'}")
    print("-" * 70)

    for disk in disks:
        symbol = "!" if disk["status"] == "ALERT" else " "
        bar_filled = int(disk["percent"] / 5)
        bar = "#" * bar_filled + "-" * (20 - bar_filled)
        print(f"[{symbol}] {disk['mount']:<18} {disk['device']:<15} "
              f"{disk['used_gb']}G/{disk['total_gb']}G  [{bar}] {disk['percent']}%")

    print(f"\nSummary: {len(alerts)} alert(s) out of {len(disks)} filesystems")

    # Python advantage: easy to output as JSON for other tools
    if "--json" in __import__("sys").argv:
        print(json.dumps(disks, indent=2))

    return 1 if alerts else 0

if __name__ == "__main__":
    exit(main())
```

Notice the differences: the Python version has structured data, JSON output for pipeline integration, proper error handling, and is easier to extend with email alerts, Slack notifications, or database logging.

## Key Takeaways

- Python is the dominant language in the DevOps ecosystem — Ansible, AWS CLI, and dozens of major tools are built with it
- Use bash for quick command-line glue under 100 lines; use Python when complexity grows
- Python's strengths for DevOps are readability, error handling, testing, and its massive library ecosystem
- Every major cloud provider and infrastructure tool has a Python SDK
- Always use Python 3 — Python 2 is dead
- Python doesn't replace bash; it complements it for more complex automation
