<Callout type="story" title="The Story">
"The dashboard was working fine last month." The product manager's words echo as you stare at 847 commits between then and now. You could check each one — that's 847 builds and tests. Or you could use git bisect, which uses binary search to find the exact commit in just 10 steps (log2(847) = 10). You'll go from "somewhere in 847 commits" to "commit abc123 by Bob on March 3rd changed the query timeout" in minutes.
</Callout>

## The Binary Search Concept

Binary search is one of the most efficient search algorithms. Instead of checking every commit linearly (O(n)), bisect cuts the search space in half at each step (O(log n)):

```
Linear search through 1000 commits:
  Check 1, Check 2, Check 3, ... Check 1000
  Worst case: 1000 checks

Binary search through 1000 commits:
  Step 1:  Check commit 500  → bad  → bug is in 1-500
  Step 2:  Check commit 250  → good → bug is in 251-500
  Step 3:  Check commit 375  → bad  → bug is in 251-375
  Step 4:  Check commit 313  → good → bug is in 314-375
  Step 5:  Check commit 344  → bad  → bug is in 314-344
  Step 6:  Check commit 329  → good → bug is in 330-344
  Step 7:  Check commit 337  → bad  → bug is in 330-337
  Step 8:  Check commit 333  → good → bug is in 334-337
  Step 9:  Check commit 335  → bad  → bug is in 334-335
  Step 10: Check commit 334  → bad  → FOUND IT! Commit 334 introduced the bug.

  10 checks instead of 1000. That's the power of binary search.
```

| Commits | Linear (worst case) | Bisect (worst case) |
|---|---|---|
| 10 | 10 checks | 4 checks |
| 100 | 100 checks | 7 checks |
| 1,000 | 1,000 checks | 10 checks |
| 10,000 | 10,000 checks | 14 checks |
| 100,000 | 100,000 checks | 17 checks |

## Manual Bisect Workflow

### Step 1: Start Bisect

```bash
# Begin the bisect session
git bisect start

# Tell Git about a known bad commit (usually HEAD — current state has the bug)
git bisect bad
# or specify a commit: git bisect bad HEAD

# Tell Git about a known good commit (where things definitely worked)
git bisect good v2.0.0
# or use a hash: git bisect good abc1234
# Bisecting: 423 revisions left to test after this (roughly 9 steps)
# [def5678...] Some commit message
```

Git checks out the middle commit automatically. Now you need to test it.

### Step 2: Test and Mark Each Commit

```bash
# Git has checked out the middle commit. Test it.
npm test
# or: python -m pytest
# or: ./run_tests.sh
# or: manually check if the bug exists

# If this commit has the bug:
git bisect bad
# Bisecting: 211 revisions left to test after this (roughly 8 steps)

# If this commit does NOT have the bug:
git bisect good
# Bisecting: 105 revisions left to test after this (roughly 7 steps)

# Repeat: Git checks out the next middle commit, you test, mark good/bad
```

### Step 3: Found the Culprit

```bash
# After enough steps, Git identifies the exact commit:
# abc1234 is the first bad commit
# commit abc1234
# Author: Bob <bob@company.com>
# Date:   Mon Mar 3 14:30:00 2026
#
#     Change query timeout to 100ms
#
#  src/database/connection.js | 2 +-
#  1 file changed, 1 insertion(+), 1 deletion(-)

# NOW you know exactly which commit introduced the bug
# and who wrote it, and what they changed
```

### Step 4: Clean Up

```bash
# End the bisect session and return to your original branch
git bisect reset
# Previous HEAD position was abc1234... Change query timeout
# Switched to branch 'main'

# You can also reset to a specific ref
git bisect reset HEAD      # stay at current bisect commit
git bisect reset main      # go to main
git bisect reset feature   # go to a specific branch
```

## Bisect Commands Reference

```bash
# Start a session
git bisect start

# Mark commits
git bisect good [ref]     # this commit doesn't have the bug
git bisect bad [ref]      # this commit has the bug

# Alternative terms (Git 2.36+): for things that aren't "bugs"
git bisect old [ref]      # same as good
git bisect new [ref]      # same as bad
# Useful for: "when did the performance regress?" or "when was the feature added?"

# Skip a commit that can't be tested
git bisect skip           # skip the current commit
git bisect skip abc1234   # skip a specific commit
git bisect skip abc..def  # skip a range

# View the bisect log (progress so far)
git bisect log
# git bisect start
# # good: [abc1234] v2.0 release
# git bisect good abc1234
# # bad: [def5678] HEAD
# git bisect bad def5678
# # good: [111aaaa] Add user preferences
# git bisect good 111aaaa
# ...

# Save and replay a bisect session
git bisect log > bisect-session.log
# Later, replay it:
git bisect replay bisect-session.log

# Visualize the remaining commits to test
git bisect visualize
# Opens gitk (or use --stat for terminal output)
git bisect visualize --oneline

# End the session
git bisect reset
```

## Skipping Untestable Commits

Sometimes a commit can't be tested — it might not compile, have broken dependencies, or be a merge commit that's hard to evaluate:

```bash
# Can't test this commit (won't compile)
git bisect skip

# Git picks a nearby commit instead
# Bisecting: 15 revisions left to test after this (roughly 4 steps)

# If too many adjacent commits are skipped, Git may not be able to
# pinpoint the exact first bad commit. It will give a range instead:
# There are only 'skip'ped commits left to test.
# The first bad commit could be any of:
#  abc1234 Commit message 1
#  def5678 Commit message 2
#  ghi9012 Commit message 3
# We cannot bisect more!
```

<Callout type="tip" title="Reducing the Need to Skip">
The number one reason for skipping is "this commit doesn't build." This is preventable:
- **Enforce CI**: every commit on main should build
- **Squash merge**: instead of many small commits, squash to one tested commit
- **Test before push**: run tests locally before pushing

Clean, buildable commits make bisect dramatically more effective.
</Callout>

## Automated Bisect with Scripts

The real power of bisect comes from automation. Instead of manually testing each commit, write a script and let Git run it automatically.

### git bisect run

```bash
# Syntax: git bisect run <script> [args...]
# The script must exit with:
#   0   = good (no bug)
#   1-124, 126-127 = bad (has bug)
#   125 = skip (can't test this commit)

# Start the bisect
git bisect start
git bisect bad HEAD
git bisect good v2.0.0

# Let a script test automatically
git bisect run npm test
# Git runs "npm test" on each commit automatically
# If tests pass (exit 0) → marks as good
# If tests fail (exit non-zero) → marks as bad
# Continues until the first bad commit is found
```

### Writing Test Scripts for Bisect

```bash
# Simple: does a specific test pass?
git bisect run pytest tests/test_dashboard.py -x

# Simple: does the project build?
git bisect run make

# Custom script for more complex checks
cat > /tmp/bisect-test.sh << 'EOF'
#!/bin/bash

# Exit 125 (skip) if the project won't build
npm run build 2>/dev/null || exit 125

# Run the specific test that catches the bug
npm test -- --testPathPattern="dashboard" 2>/dev/null
# Exit code passes through: 0 = good, non-zero = bad
EOF
chmod +x /tmp/bisect-test.sh

git bisect run /tmp/bisect-test.sh
```

### Advanced Test Scripts

```bash
# Test for a performance regression
cat > /tmp/bisect-perf.sh << 'EOF'
#!/bin/bash

# Build first, skip if build fails
make clean && make 2>/dev/null || exit 125

# Run the benchmark
RESULT=$(./benchmark --iterations 100 2>/dev/null)
LATENCY=$(echo "$RESULT" | grep "avg_latency" | awk '{print $2}')

# If latency is above threshold, it's "bad"
if (( $(echo "$LATENCY > 200" | bc -l) )); then
    echo "SLOW: ${LATENCY}ms (threshold: 200ms)"
    exit 1  # bad
else
    echo "OK: ${LATENCY}ms"
    exit 0  # good
fi
EOF
chmod +x /tmp/bisect-perf.sh

git bisect start
git bisect bad HEAD                # currently slow
git bisect good v2.0.0             # was fast in v2.0
git bisect run /tmp/bisect-perf.sh
```

```bash
# Test for a specific string/pattern (e.g., was a config value changed?)
cat > /tmp/bisect-config.sh << 'EOF'
#!/bin/bash

# Check if the config file has the expected value
if grep -q 'timeout: 5000' src/config.js 2>/dev/null; then
    exit 0  # good — timeout is correct
else
    exit 1  # bad — timeout was changed
fi
EOF
chmod +x /tmp/bisect-config.sh

git bisect run /tmp/bisect-config.sh
```

```bash
# Test for file size regression (binary grew too large?)
cat > /tmp/bisect-size.sh << 'EOF'
#!/bin/bash

npm run build 2>/dev/null || exit 125

BUNDLE_SIZE=$(stat -f%z dist/bundle.js 2>/dev/null || stat -c%s dist/bundle.js 2>/dev/null)
MAX_SIZE=512000  # 500KB

if [ "$BUNDLE_SIZE" -gt "$MAX_SIZE" ]; then
    echo "LARGE: ${BUNDLE_SIZE} bytes (max: ${MAX_SIZE})"
    exit 1
else
    echo "OK: ${BUNDLE_SIZE} bytes"
    exit 0
fi
EOF
chmod +x /tmp/bisect-size.sh

git bisect run /tmp/bisect-size.sh
```

<Callout type="warning" title="Script Exit Codes Matter">
Your bisect script MUST use the correct exit codes:
- **0** = this commit is good (no bug)
- **1-124, 126-127** = this commit is bad (has bug)
- **125** = skip (can't test, e.g., won't build)
- **128+** = bisect aborts entirely (something is seriously wrong)

Common mistake: a script that always exits 0 regardless of test results. Bisect will mark everything as "good" and give wrong results. Always test your script manually on a known-good and known-bad commit first.
</Callout>

## Bisect with Merge Commits

Merge commits can complicate bisect because the merge itself might not be where the bug was introduced — it could be in one of the merged branch's commits.

```bash
# If bisect lands on a merge commit:
# Option 1: Test it normally (the merge result should show the bug or not)
git bisect good  # or bad

# Option 2: Skip the merge and test the parents
git bisect skip

# To focus bisect on first-parent commits only (main branch commits):
git bisect start --first-parent
git bisect bad HEAD
git bisect good v2.0.0
# Only tests merge commits and direct commits to main
# Once you find the bad merge, you can bisect within that branch
```

<Callout type="tip" title="First-Parent Bisect Strategy">
For repositories with many merge commits, `git bisect start --first-parent` is very useful. It only walks the main branch's merge commits, quickly identifying which feature branch introduced the problem. Then you can do a second, focused bisect within that specific branch's commits.
</Callout>

## Bisect for Non-Bug Investigations

Bisect isn't just for bugs. You can use it to find when any change happened:

```bash
# When did a test start failing?
git bisect start
git bisect new HEAD          # test fails now (using new/old instead of bad/good)
git bisect old v1.0.0        # test passed in v1.0
git bisect run pytest tests/test_specific.py

# When did performance degrade?
git bisect start
git bisect bad HEAD          # slow now
git bisect good v1.0.0       # was fast
git bisect run /tmp/perf-check.sh

# When was a function added?
git bisect start
git bisect new HEAD          # function exists now
git bisect old v1.0.0        # function didn't exist
git bisect run bash -c 'grep -q "function myNewFunction" src/utils.js'

# When did the binary size exceed the limit?
git bisect start
git bisect bad HEAD
git bisect good v1.0.0
git bisect run /tmp/size-check.sh
```

## Practical Bisect Tips

### 1. Start with Known Good/Bad

```bash
# Use tags for known-good versions
git bisect good v2.0.0

# Or find a good commit by date
git log --oneline --before="2026-01-01" -1
# abc1234 Last commit of December
git bisect good abc1234

# Quick test before bisecting:
git stash
git checkout v2.0.0
npm test  # confirm it actually works
git checkout main
git stash pop
```

### 2. Narrow the Range First

```bash
# Don't bisect across 10,000 commits if you can narrow it down
git log --oneline --since="2 weeks ago" | wc -l
# 47 commits — much more manageable

# Use known deployment dates
git log --oneline --after="2026-02-15" --before="2026-02-22"
# The bug appeared during this week
```

### 3. Make Scripts Idempotent

```bash
# Good: script cleans up after itself
cat > /tmp/bisect-test.sh << 'EOF'
#!/bin/bash
# Clean build
rm -rf node_modules/.cache dist/
npm ci --silent 2>/dev/null || exit 125
npm run build --silent 2>/dev/null || exit 125
npm test -- --testPathPattern="dashboard" 2>/dev/null
EOF

# Bad: script depends on state from previous run
# npm test  ← might use cached builds from wrong commit
```

### 4. Save Your Bisect Session

```bash
# Before a long bisect, save progress periodically
git bisect log > ~/bisect-progress.log

# If something goes wrong, replay from saved state
git bisect reset
git bisect replay ~/bisect-progress.log
# Continues from where you left off
```

# Create a repository with a bug hidden in 20 commits
REPO=$(mktemp -d) && cd "$REPO" && git init -q

# Base commit (known good)
cat > calculator.js << 'EOF'
function add(a, b) { return a + b; }
function subtract(a, b) { return a - b; }
function multiply(a, b) { return a * b; }
function divide(a, b) {
  if (b === 0) throw new Error("Division by zero");
  return a / b;
}
module.exports = { add, subtract, multiply, divide };

cat > test.sh << 'TESTEOF'
#!/bin/bash
# Test: divide(10, 3) should return approximately 3.333
result=$(node -e "const c = require('./calculator.js'); console.log(c.divide(10, 3))")
expected="3.3333333333333335"
if [ "$result" = "$expected" ]; then
    exit 0
else
    echo "FAIL: divide(10, 3) = $result (expected $expected)"
    exit 1
fi
TESTEOF
chmod +x test.sh
git add . && git commit -q -m "v1.0: Basic calculator"
git tag v1.0

# Generate 20 commits, one of which introduces a bug
for i in $(seq 1 20); do
    case $i in
        3)
            echo "// Added logging support" >> calculator.js
            git add . && git commit -q -m "Commit $i: Add logging support"
            ;;
        5)
            echo "function sqrt(a) { return Math.sqrt(a); }" >> calculator.js
            git add . && git commit -q -m "Commit $i: Add square root"
            ;;
        8)
            echo "function power(a, b) { return Math.pow(a, b); }" >> calculator.js
            git add . && git commit -q -m "Commit $i: Add power function"
            ;;
        12)
            # THE BUG: change divide to use integer division
            sed -i 's|return a / b;|return Math.floor(a / b);|' calculator.js
            git add . && git commit -q -m "Commit $i: Optimize division performance"
            BUG_COMMIT=$(git rev-parse HEAD)
            ;;
        15)
            echo "function modulo(a, b) { return a % b; }" >> calculator.js
            git add . && git commit -q -m "Commit $i: Add modulo function"
            ;;
        18)
            echo "function abs(a) { return Math.abs(a); }" >> calculator.js
            git add . && git commit -q -m "Commit $i: Add absolute value"
            ;;
        *)
            echo "// Enhancement $i" >> calculator.js
            git add . && git commit -q -m "Commit $i: Code improvement"
            ;;
    esac
done

echo "=== Repository Setup ==="
echo "Total commits: $(git log --oneline | wc -l)"
echo ""
echo "Current test result:"
./test.sh 2>&1 || true
echo ""
echo "The bug is in commit: $(git log --oneline -1 $BUG_COMMIT)"
echo "(But pretend you don't know this!)"
echo ""
echo "=== Starting Manual Bisect ==="
echo ""

# Start bisect
git bisect start
git bisect bad HEAD
git bisect good v1.0
echo ""

# Simulate manual testing
STEP=1
while true; do
    CURRENT=$(git rev-parse --short HEAD)
    echo "--- Step $STEP: Testing commit $CURRENT ---"
    echo "$(git log --oneline -1)"

    if ./test.sh 2>/dev/null; then
        echo "Result: GOOD (test passes)"
        RESULT=$(git bisect good 2>&1)
    else
        echo "Result: BAD (test fails)"
        RESULT=$(git bisect bad 2>&1)
    fi

    echo "$RESULT"
    echo ""

    # Check if bisect found the answer
    if echo "$RESULT" | grep -q "is the first bad commit"; then
        echo "=== FOUND THE BUG! ==="
        echo ""
        echo "$RESULT"
        break
    fi

    STEP=$((STEP + 1))

    # Safety: prevent infinite loop
    if [ $STEP -gt 20 ]; then
        echo "Something went wrong — too many steps"
        break
    fi
done

echo ""
echo "=== Bisect took $STEP steps to search $(git log --oneline v1.0..HEAD | wc -l) commits ==="
echo "Expected steps: ~$(echo "l(20)/l(2)" | bc -l | cut -d. -f1) (log2 of 20)"
echo ""
echo "=== Bisect Log ==="
git bisect log

# Reset
git bisect reset
echo ""
echo "Back to $(git branch --show-current)"

# Clean up
cd / && rm -rf "$REPO"
```

cat > ~/git-bisect-tool.sh << 'SCRIPT'
#!/bin/bash
set -euo pipefail

# Git Bisect Automation Tool
# Wraps git bisect with nicer output, session management, and reports

RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
CYAN='\033[0;36m'
BOLD='\033[1m'
NC='\033[0m'

# Config
BISECT_DIR="${HOME}/.git-bisect-sessions"
mkdir -p "$BISECT_DIR"

usage() {
    echo -e "${BLUE}${BOLD}Git Bisect Automation Tool${NC}"
    echo ""
    echo "Usage: $0 <command> [options]"
    echo ""
    echo "Commands:"
    echo "  start <good> <bad>              Start a new bisect session"
    echo "  run <good> <bad> <test-cmd>     Automated bisect with a test command"
    echo "  test-grep <good> <bad> <file> <pattern>  Find when a pattern appeared/vanished"
    echo "  test-perf <good> <bad> <cmd> <threshold>  Find performance regression"
    echo "  status                           Show current bisect status"
    echo "  history                          Show past bisect sessions"
    echo "  report [session-id]              Generate a detailed report"
    echo "  save                             Save current bisect session"
    echo "  restore <session-id>             Restore a saved session"
    echo ""
    echo "Examples:"
    echo "  $0 start v2.0 HEAD"
    echo "  $0 run v2.0 HEAD 'npm test'"
    echo "  $0 run v2.0 HEAD './test.sh' --name 'dashboard-bug'"
    echo "  $0 test-grep v1.0 HEAD src/config.js 'timeout: 5000'"
    echo "  $0 test-perf v1.0 HEAD './benchmark.sh' 200"
}

# Generate a unique session ID
session_id() {
    date +"%Y%m%d-%H%M%S"
}

# Pretty print a bisect step
print_step() {
    local step=$1 total_range=$2 commit=$3 message=$4 result=$5
    local remaining
    remaining=$(echo "l($total_range)/l(2)" | bc -l 2>/dev/null | cut -d. -f1)

    echo -e "${BLUE}Step ${step}${NC} | ${CYAN}${commit}${NC}"
    echo -e "  Message: ${message}"

    case "$result" in
        good) echo -e "  Result:  ${GREEN}GOOD${NC}" ;;
        bad)  echo -e "  Result:  ${RED}BAD${NC}" ;;
        skip) echo -e "  Result:  ${YELLOW}SKIP${NC}" ;;
    esac
    echo ""
}

# Start a bisect with nice output
cmd_start() {
    local good="$1"
    local bad="${2:-HEAD}"

    echo -e "${BLUE}${BOLD}Starting Bisect Session${NC}"
    echo -e "Good: ${GREEN}${good}${NC}"
    echo -e "Bad:  ${RED}${bad}${NC}"

    local range_size
    range_size=$(git log --oneline "${good}..${bad}" 2>/dev/null | wc -l | tr -d ' ')
    local max_steps
    max_steps=$(echo "l($range_size)/l(2) + 1" | bc -l 2>/dev/null | cut -d. -f1)

    echo -e "Range: ${CYAN}${range_size} commits${NC}"
    echo -e "Max steps: ${CYAN}~${max_steps}${NC}"
    echo ""

    git bisect start
    git bisect good "$good"
    git bisect bad "$bad"

    echo ""
    echo -e "${YELLOW}Bisect is ready. Current commit:${NC}"
    git log --oneline -1
    echo ""
    echo "Test this commit and run:"
    echo "  git bisect good   (if the bug is NOT present)"
    echo "  git bisect bad    (if the bug IS present)"
    echo "  git bisect skip   (if you can't test this commit)"
}

# Automated bisect with nice output
cmd_run() {
    local good="$1"
    local bad="$2"
    local test_cmd="$3"
    local session_name="${4:-bisect-$(session_id)}"

    echo -e "${BLUE}${BOLD}Automated Bisect Session${NC}"
    echo -e "Session: ${CYAN}${session_name}${NC}"
    echo -e "Good:    ${GREEN}${good}${NC}"
    echo -e "Bad:     ${RED}${bad}${NC}"
    echo -e "Test:    ${CYAN}${test_cmd}${NC}"
    echo ""

    local range_size
    range_size=$(git log --oneline "${good}..${bad}" 2>/dev/null | wc -l | tr -d ' ')
    local max_steps
    max_steps=$(echo "l($range_size)/l(2) + 1" | bc -l 2>/dev/null | cut -d. -f1)

    echo -e "Searching through ${CYAN}${range_size}${NC} commits (max ${CYAN}~${max_steps}${NC} steps)"
    echo -e "${BLUE}$(printf '=%.0s' $(seq 1 60))${NC}"
    echo ""

    # Create wrapper script that logs each step
    local log_file="${BISECT_DIR}/${session_name}.log"
    local step_log="${BISECT_DIR}/${session_name}-steps.log"
    echo "# Bisect Session: $session_name" > "$log_file"
    echo "# Date: $(date)" >> "$log_file"
    echo "# Good: $good" >> "$log_file"
    echo "# Bad: $bad" >> "$log_file"
    echo "# Test: $test_cmd" >> "$log_file"
    echo "# Range: $range_size commits" >> "$log_file"
    echo "" >> "$log_file"
    : > "$step_log"

    local wrapper="/tmp/bisect-wrapper-$$.sh"
    cat > "$wrapper" << WEOF
#!/bin/bash
STEP=\$(cat "${step_log}" | wc -l)
STEP=\$((STEP + 1))
COMMIT=\$(git rev-parse --short HEAD)
MESSAGE=\$(git log --oneline -1 --format="%s")

echo "Step \$STEP: \$COMMIT \$MESSAGE" >> "${step_log}"

# Run the test
$test_cmd > /tmp/bisect-output-$$.txt 2>&1
EXIT_CODE=\$?

if [ \$EXIT_CODE -eq 0 ]; then
    echo "  Result: GOOD" >> "${step_log}"
    echo -e "\033[0;34mStep \${STEP}\033[0m | \033[0;36m\${COMMIT}\033[0m | \033[0;32mGOOD\033[0m | \${MESSAGE}"
elif [ \$EXIT_CODE -eq 125 ]; then
    echo "  Result: SKIP" >> "${step_log}"
    echo -e "\033[0;34mStep \${STEP}\033[0m | \033[0;36m\${COMMIT}\033[0m | \033[1;33mSKIP\033[0m | \${MESSAGE}"
else
    echo "  Result: BAD" >> "${step_log}"
    echo -e "\033[0;34mStep \${STEP}\033[0m | \033[0;36m\${COMMIT}\033[0m | \033[0;31mBAD\033[0m  | \${MESSAGE}"
fi

exit \$EXIT_CODE
WEOF
    chmod +x "$wrapper"

    # Start bisect
    git bisect start
    git bisect good "$good"
    git bisect bad "$bad"

    # Run automated bisect
    local start_time
    start_time=$(date +%s)

    BISECT_OUTPUT=$(git bisect run "$wrapper" 2>&1)

    local end_time
    end_time=$(date +%s)
    local duration=$((end_time - start_time))

    echo ""
    echo -e "${BLUE}$(printf '=%.0s' $(seq 1 60))${NC}"
    echo ""

    # Parse the result
    local culprit_hash culprit_msg
    culprit_hash=$(echo "$BISECT_OUTPUT" | grep "is the first bad commit" | head -1 | awk '{print $1}')

    if [ -n "$culprit_hash" ]; then
        culprit_msg=$(git log --oneline -1 "$culprit_hash" 2>/dev/null || echo "unknown")
        local total_steps
        total_steps=$(cat "$step_log" | grep "^Step" | wc -l | tr -d ' ')

        echo -e "${GREEN}${BOLD}BUG FOUND!${NC}"
        echo ""
        echo -e "  Culprit:  ${RED}${culprit_hash}${NC}"
        echo -e "  Message:  $(git log --format='%s' -1 "$culprit_hash" 2>/dev/null)"
        echo -e "  Author:   $(git log --format='%an <%ae>' -1 "$culprit_hash" 2>/dev/null)"
        echo -e "  Date:     $(git log --format='%ci' -1 "$culprit_hash" 2>/dev/null)"
        echo -e "  Steps:    ${CYAN}${total_steps}${NC} (searched ${range_size} commits)"
        echo -e "  Duration: ${CYAN}${duration}s${NC}"
        echo ""
        echo -e "${YELLOW}Files changed in the culprit commit:${NC}"
        git show --stat --format="" "$culprit_hash" 2>/dev/null | head -20
        echo ""

        # Save result to log
        echo "RESULT: $culprit_hash" >> "$log_file"
        echo "STEPS: $total_steps" >> "$log_file"
        echo "DURATION: ${duration}s" >> "$log_file"
        echo "" >> "$log_file"
        cat "$step_log" >> "$log_file"
    else
        echo -e "${YELLOW}Bisect completed but no clear result was found.${NC}"
        echo "This can happen when too many commits were skipped."
        echo ""
        echo "Output:"
        echo "$BISECT_OUTPUT"
    fi

    # Clean up
    rm -f "$wrapper" "/tmp/bisect-output-$$.txt"
    git bisect reset 2>/dev/null || true

    echo -e "Session log saved: ${CYAN}${log_file}${NC}"
}

# Find when a pattern appeared or disappeared
cmd_test_grep() {
    local good="$1"
    local bad="$2"
    local file="$3"
    local pattern="$4"

    echo -e "${BLUE}Finding when pattern changed in ${file}${NC}"
    echo -e "Pattern: ${CYAN}${pattern}${NC}"
    echo ""

    # Determine if we're looking for when it appeared or disappeared
    if git show "${good}:${file}" 2>/dev/null | grep -q "$pattern"; then
        echo "Pattern EXISTS in good commit, MISSING in bad commit."
        echo "Looking for when it was removed..."
        # Good = pattern exists, bad = pattern missing
        cmd_run "$good" "$bad" "grep -q '$pattern' '$file'"
    else
        echo "Pattern MISSING in good commit, EXISTS in bad commit."
        echo "Looking for when it was added..."
        cmd_run "$good" "$bad" "! grep -q '$pattern' '$file'"
    fi
}

# Find performance regression
cmd_test_perf() {
    local good="$1"
    local bad="$2"
    local bench_cmd="$3"
    local threshold="$4"

    echo -e "${BLUE}Finding performance regression${NC}"
    echo -e "Benchmark: ${CYAN}${bench_cmd}${NC}"
    echo -e "Threshold: ${CYAN}${threshold}${NC}"
    echo ""

    local perf_script="/tmp/bisect-perf-$$.sh"
    cat > "$perf_script" << PEOF
#!/bin/bash
RESULT=\$($bench_cmd 2>/dev/null)
if [ -z "\$RESULT" ]; then
    exit 125  # skip — can't benchmark
fi
if [ "\$RESULT" -le "$threshold" ] 2>/dev/null; then
    exit 0  # good — within threshold
else
    exit 1  # bad — exceeds threshold
fi
PEOF
    chmod +x "$perf_script"

    cmd_run "$good" "$bad" "$perf_script" "perf-regression-$(session_id)"

    rm -f "$perf_script"
}

# Show bisect history
cmd_history() {
    echo -e "${BLUE}${BOLD}Bisect Session History${NC}"
    echo ""

    if [ -z "$(ls -A "$BISECT_DIR"/*.log 2>/dev/null)" ]; then
        echo "No saved sessions."
        return 0
    fi

    for log in "$BISECT_DIR"/*.log; do
        [ -f "$log" ] || continue
        local name
        name=$(basename "$log" .log)
        if echo "$name" | grep -q "steps$"; then
            continue  # skip step logs
        fi
        local date_line result_line
        date_line=$(grep "^# Date:" "$log" | head -1 | sed 's/# Date: //')
        result_line=$(grep "^RESULT:" "$log" | head -1 | sed 's/RESULT: //')

        if [ -n "$result_line" ]; then
            echo -e "  ${GREEN}${name}${NC} | ${date_line} | Culprit: ${RED}${result_line}${NC}"
        else
            echo -e "  ${YELLOW}${name}${NC} | ${date_line} | No result"
        fi
    done
}

# Generate report for a session
cmd_report() {
    local name="${1:-}"

    if [ -z "$name" ]; then
        echo "Available sessions:"
        cmd_history
        echo ""
        read -r -p "Enter session name: " name
    fi

    local log="${BISECT_DIR}/${name}.log"
    local step_log="${BISECT_DIR}/${name}-steps.log"

    if [ ! -f "$log" ]; then
        echo -e "${RED}Session '${name}' not found.${NC}"
        return 1
    fi

    echo -e "${BLUE}${BOLD}Bisect Report: ${name}${NC}"
    echo -e "${BLUE}$(printf '=%.0s' $(seq 1 60))${NC}"
    echo ""
    cat "$log"

    if [ -f "$step_log" ]; then
        echo ""
        echo -e "${YELLOW}Step Details:${NC}"
        cat "$step_log"
    fi
}

# Main dispatch
case "${1:-help}" in
    start)
        shift
        cmd_start "$@"
        ;;
    run)
        shift
        local_good="$1"; local_bad="$2"; local_cmd="$3"
        shift 3
        local_name=""
        while [[ $# -gt 0 ]]; do
            case "$1" in
                --name) local_name="$2"; shift 2 ;;
                *) shift ;;
            esac
        done
        cmd_run "$local_good" "$local_bad" "$local_cmd" "${local_name:-bisect-$(session_id)}"
        ;;
    test-grep)
        shift
        cmd_test_grep "$@"
        ;;
    test-perf)
        shift
        cmd_test_perf "$@"
        ;;
    status)
        if git bisect log &>/dev/null; then
            echo -e "${BLUE}Active bisect session:${NC}"
            git bisect log
        else
            echo "No active bisect session."
        fi
        ;;
    history)
        cmd_history
        ;;
    report)
        shift
        cmd_report "${1:-}"
        ;;
    help|--help|-h)
        usage
        ;;
    *)
        echo -e "${RED}Unknown command: $1${NC}"
        usage
        exit 1
        ;;
esac
SCRIPT

chmod +x ~/git-bisect-tool.sh

echo "=== Git Bisect Tool Created ==="
echo ""
~/git-bisect-tool.sh help
echo ""

# Demo with a real repository
REPO=$(mktemp -d) && cd "$REPO" && git init -q

cat > app.conf << 'EOF'
database_pool: 10
query_timeout: 5000
cache_size: 256
EOF
git add . && git commit -q -m "v1.0: Initial config"
git tag v1.0

for i in $(seq 1 8); do
    if [ $i -eq 4 ]; then
        sed -i 's/query_timeout: 5000/query_timeout: 50/' app.conf
        git add . && git commit -q -m "Commit $i: Optimize query settings"
    else
        echo "# update $i" >> app.conf
        git add . && git commit -q -m "Commit $i: Configuration update"
    fi
done

echo ""
echo "=== Demo: Automated bisect finding config regression ==="
echo ""
~/git-bisect-tool.sh run v1.0 HEAD "bash -c 'val=\$(grep query_timeout app.conf | awk \"{print \\\$2}\"); [ \"\$val\" -ge 1000 ]'" --name "demo-config-regression"
echo ""

echo "=== Demo: Session history ==="
~/git-bisect-tool.sh history

# Clean up
cd / && rm -rf "$REPO"
```

## Key Takeaways

- `git bisect` uses binary search to find the exact commit that introduced a bug in O(log n) steps
- Start with `git bisect start`, then mark `git bisect good <ref>` and `git bisect bad <ref>`
- At each step, test the current commit and mark it as `good`, `bad`, or `skip`
- `git bisect run <script>` automates the process — the script returns 0 for good, non-zero for bad, and 125 for skip
- Always clean up with `git bisect reset` when done
- Use `git bisect log` to save and `git bisect replay` to restore a session
- `git bisect skip` handles untestable commits but may reduce precision if too many are skipped
- `--first-parent` is useful for repositories with many merge commits
- Bisect works for any binary question: bugs, performance regressions, config changes, file size increases
- Clean, buildable commit history makes bisect far more effective — enforce CI on every commit
