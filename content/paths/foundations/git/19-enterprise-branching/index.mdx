<Callout type="story" title="The Story">
Your startup is growing. With 3 developers, "just push to main" worked fine. Now you have 30 developers, 5 teams, staging and production environments, and monthly releases. Developers are stepping on each other's branches. Hotfixes are getting lost. Nobody knows which code is in production. You need a branching strategy — a set of rules everyone follows so that Git serves your process instead of fighting it.
</Callout>

## Why Branching Strategies Matter

A branching strategy is a contract your team agrees on. It answers questions that come up every day:

```
- "Where do I create my feature branch from?"
- "Where do I merge when I'm done?"
- "How do we handle hotfixes?"
- "Which branch represents production?"
- "How do we release?"
- "Can I push directly to main?"
```

Without a strategy, every developer invents their own workflow. Branches pile up. Merges become confusing. Deployments become risky. With a strategy, everyone speaks the same language.

The four major strategies, from most structured to least:

```
Most structured ──────────────────────────── Least structured
   GitFlow     GitLab Flow    GitHub Flow    Trunk-Based
   (complex)   (moderate)     (simple)       (minimal)
```

## GitFlow

GitFlow, created by Vincent Driessen in 2010, is the most structured branching model. It defines specific branch types with strict rules about where code flows.

### Branch Types

```
main (or master)
  │   Always reflects production. Every commit is a release tag.
  │
develop
  │   Integration branch. Features merge here first.
  │
feature/*
  │   One per feature. Branch from develop, merge back to develop.
  │
release/*
  │   Preparation for a release. Branch from develop, merge to main AND develop.
  │
hotfix/*
      Emergency fixes. Branch from main, merge to main AND develop.
```

### GitFlow Diagram

```
main:     ──●──────────────────●──────────●──────●──
             \                / \        /      /
hotfix:       \              /   \      /      /
               \            /     \    /    ●─●
                \          /       \  /    /
release:         \      ●─●────●   \/    /
                  \    /    \   \   /\   /
develop:  ──●──●───●──●──●──●───●──●──●──●──●──
              \   /    \      /        \
feature:    ●──●──●     ●──●──●         ●──●
            (auth)     (payments)      (search)
```

### GitFlow Workflow

```bash
# === Starting a new feature ===
# Branch from develop
git switch develop
git pull origin develop
git switch -c feature/user-authentication

# Work on the feature
# ... commits ...

# Merge back to develop (via PR in most teams)
git switch develop
git merge --no-ff feature/user-authentication
git branch -d feature/user-authentication
git push origin develop

# === Creating a release ===
# When develop has enough features for a release
git switch develop
git switch -c release/2.1.0

# Only bug fixes and version bumps on the release branch
echo "2.1.0" > VERSION
git add . && git commit -m "Bump version to 2.1.0"
# Fix any release bugs...

# Merge release to main
git switch main
git merge --no-ff release/2.1.0
git tag -a v2.1.0 -m "Release 2.1.0"

# Merge release back to develop (to capture bug fixes)
git switch develop
git merge --no-ff release/2.1.0
git branch -d release/2.1.0

# Push everything
git push origin main develop --tags

# === Emergency hotfix ===
# Branch from main (production)
git switch main
git switch -c hotfix/security-patch

# Fix the bug
# ... commits ...

# Merge to main
git switch main
git merge --no-ff hotfix/security-patch
git tag -a v2.1.1 -m "Hotfix: security patch"

# Merge to develop (so the fix is in future releases too)
git switch develop
git merge --no-ff hotfix/security-patch
git branch -d hotfix/security-patch

git push origin main develop --tags
```

### When to Use GitFlow

```
GOOD FIT:                              POOR FIT:
✓ Versioned software (v1.0, v2.0)     ✗ Continuous deployment (web apps)
✓ Scheduled releases (monthly)          ✗ Small teams (< 5 developers)
✓ Multiple versions in production       ✗ Fast iteration required
✓ Formal QA process                     ✗ Teams new to Git
✓ Desktop/mobile apps                   ✗ Microservices
✓ Libraries and frameworks              ✗ High deployment frequency
```

<Callout type="warning" title="GitFlow Complexity">
GitFlow's biggest drawback is complexity. It has 5 branch types, strict merge directions, and requires discipline to follow. For web applications that deploy continuously, GitFlow is usually overkill. It was designed for software that ships versioned releases (like desktop apps or libraries), not for web apps that deploy multiple times per day.
</Callout>

## GitHub Flow

GitHub Flow is the simplest branching strategy. Created by GitHub, it has just two concepts: `main` is always deployable, and everything else is a feature branch.

### Branch Types

```
main
  │   Always deployable. Protected. Deploy on every merge.
  │
feature/*
      Short-lived. Branch from main, PR back to main, deploy on merge.
```

### GitHub Flow Diagram

```
main:     ──●──●──●──●──●──●──●──●──●──●──
              \   /  \     /     \     /
feature:    ●──●──●    ●──●──●    ●──●
            (login)   (dashboard) (search)
                ↑          ↑          ↑
            PR + review  PR + review  PR + review
            then merge   then merge   then merge
            then deploy  then deploy  then deploy
```

### GitHub Flow Workflow

```bash
# === Starting work ===
# Always branch from main
git switch main
git pull origin main
git switch -c feature/add-search

# Work on the feature (commit early and often)
git add . && git commit -m "Add search index"
git add . && git commit -m "Add search UI component"
git add . && git commit -m "Add search API endpoint"

# Push and create a Pull Request
git push -u origin feature/add-search
# Create PR on GitHub (or via CLI: gh pr create)

# === Review process ===
# Teammates review the PR
# CI runs tests automatically
# Address feedback with more commits

# === Merge and deploy ===
# Once approved and CI passes:
# Merge the PR (usually via GitHub UI)
# Deploy automatically (CI/CD triggers on merge to main)

# === Clean up ===
git switch main
git pull origin main
git branch -d feature/add-search
```

### The Rules of GitHub Flow

```
1. main is ALWAYS deployable
2. Branch from main for any change
3. Push to your branch regularly
4. Open a PR when you want feedback (or when you're done)
5. Merge only after PR review AND CI passes
6. Deploy immediately after merging to main
```

### When to Use GitHub Flow

```
GOOD FIT:                              POOR FIT:
✓ Continuous deployment                 ✗ Multiple versions in production
✓ Web applications                      ✗ Scheduled releases
✓ Small to medium teams                 ✗ Formal QA stages
✓ Fast iteration                        ✗ Desktop/mobile apps with versions
✓ Microservices                         ✗ Regulated industries (audit trail)
✓ Teams new to branching strategies     ✗ Need staging environments
```

<Callout type="tip" title="GitHub Flow Is the Default Choice">
If you're unsure which strategy to use, start with GitHub Flow. It's simple, well-understood, and works for most web development teams. You can always add complexity later (environment branches, release branches) if needed. You can't easily simplify a complex strategy that's already in place.
</Callout>

## GitLab Flow

GitLab Flow sits between GitHub Flow's simplicity and GitFlow's structure. It adds environment branches for staged deployments.

### Branch Types

```
main (or production)
  │   Reflects production. Deployments happen from here.
  │
staging (or pre-production)
  │   Code goes here for QA before production.
  │
develop (optional)
  │   Integration branch for features.
  │
feature/*
      Short-lived. Branch from main, merge to main, flows downstream.
```

### GitLab Flow — Environment Branches

```
feature → main → staging → production

main:        ──●──●──●──●──●──●──●──
                \   /        ↓
feature:      ●──●──●      (merge down)
                          ↓
staging:     ────────────●──●──●──
                              ↓
                          (merge down when ready)
                              ↓
production:  ─────────────────●──●──
```

### GitLab Flow — Release Branches

```
feature → main → release/X.Y

main:           ──●──●──●──●──●──●──●──●──
                    \   /              \
feature:          ●──●──●              (cherry-pick)
                                        ↓
release/2.5:     ──────────●──●──●──●──●──
                           ↑
                     (branch from main at release time)
                     (only bug fixes after branching)

release/2.6:     ─────────────────●──●──●──
```

### GitLab Flow Workflow

```bash
# === Environment branch model ===

# Work on a feature (same as GitHub Flow)
git switch main
git pull origin main
git switch -c feature/new-dashboard

# ... work and commits ...

# Merge to main (via PR)
git push -u origin feature/new-dashboard
# Create merge request, get review, CI passes, merge

# Promote to staging
git switch staging
git merge main
git push origin staging
# Staging deployment triggers automatically

# After QA approval, promote to production
git switch production
git merge staging
git push origin production
# Production deployment triggers automatically

# === Release branch model ===

# When it's time to release
git switch main
git switch -c release/2.5
git push origin release/2.5

# Bug fixes go to main first, then cherry-pick to release
git switch main
# ... fix bug, commit, push ...
git switch release/2.5
git cherry-pick <fix-commit>
git push origin release/2.5

# Tag the release
git tag -a v2.5.1 -m "Release 2.5.1"
git push origin release/2.5 --tags
```

### When to Use GitLab Flow

```
GOOD FIT:                              POOR FIT:
✓ Staged environments (dev/staging/prod) ✗ Simple continuous deployment
✓ QA process before production           ✗ Very small teams
✓ Need to control what's in production   ✗ No staging environments
✓ Medium to large teams                  ✗ Teams wanting simplicity
✓ SaaS with release trains               ✗ Pure trunk-based development
✓ Regulated industries                   ✗ Microservices (per-service deploy)
```

## Trunk-Based Development

Trunk-based development is the most minimalist strategy. Everyone commits to a single branch (trunk/main). Feature branches, if used at all, are extremely short-lived (hours, not days).

### Branch Types

```
main (trunk)
  │   The only long-lived branch. Everyone commits here.
  │
short-lived branches (optional)
      Live for hours, not days. Maximum 1-2 days.
```

### Trunk-Based Development Diagram

```
No branches (pure trunk):
main: ──●──●──●──●──●──●──●──●──●──●──●──●──
         A   B   A   C   B   A   C   B   A
      (developers A, B, C all commit to main)

With short-lived branches (common variant):
main:   ──●──●──●──●──●──●──●──●──●──●──●──
            \  / \   /   \  /
feature:  ●──●    ●──●    ●──●
          (< 1 day) (< 1 day) (< 1 day)
```

### Trunk-Based Workflow

```bash
# === Pure trunk-based (no branches) ===

# Pull latest
git switch main
git pull origin main

# Make small, incremental changes
git add . && git commit -m "Add search button to header"
git push origin main
# CI runs, deploys if green

# Another small change
git add . && git commit -m "Wire search button to API"
git push origin main
# CI runs, deploys if green

# === With short-lived branches ===

# Small, focused branch
git switch -c add-search-button
git add . && git commit -m "Add search button to header"
git push -u origin add-search-button
# Create PR immediately
# Review happens within hours (not days)
# Merge same day

# Key: the branch never lives more than 1-2 days
```

### Feature Flags (Essential for Trunk-Based)

Since everyone pushes to main and it deploys immediately, you need feature flags to hide incomplete features:

```bash
# Don't merge half-finished features behind branches
# Instead, merge them behind feature flags

cat > src/features.js << 'EOF'
const features = {
  newSearch: process.env.FEATURE_NEW_SEARCH === 'true',
  darkMode: process.env.FEATURE_DARK_MODE === 'true',
  newCheckout: process.env.FEATURE_NEW_CHECKOUT === 'true',
};

module.exports = features;
EOF

# In your code:
cat > src/header.js << 'EOF'
const features = require('./features');

function renderHeader() {
  return `
    <header>
      <nav>Home | Products | About</nav>
      ${features.newSearch ? '<SearchBar />' : ''}
    </header>
  `;
}
EOF

# Deploy safely: feature is in main but hidden
# Enable for testing: FEATURE_NEW_SEARCH=true (in staging)
# Enable for everyone: FEATURE_NEW_SEARCH=true (in production)
# Rollback without code change: FEATURE_NEW_SEARCH=false
```

### When to Use Trunk-Based Development

```
GOOD FIT:                              POOR FIT:
✓ High CI/CD maturity                   ✗ No CI/CD pipeline
✓ Excellent test coverage               ✗ Poor test coverage
✓ Small, senior teams                   ✗ Junior-heavy teams
✓ Feature flag infrastructure           ✗ No feature flag system
✓ Continuous deployment (multiple/day)  ✗ Scheduled releases
✓ Google, Meta, Netflix scale           ✗ Versioned software
✓ Microservices                         ✗ Teams learning Git
```

<Callout type="warning" title="Trunk-Based Requires Maturity">
Trunk-based development looks simple but requires the most engineering maturity. You need: excellent CI (tests run in minutes, not hours), comprehensive test coverage, feature flags, monitoring, and the ability to roll back deployments quickly. Without these, pushing directly to main becomes dangerous rather than efficient.
</Callout>

## Comparison Table

| Aspect | GitFlow | GitHub Flow | GitLab Flow | Trunk-Based |
|---|---|---|---|---|
| **Complexity** | High (5 branch types) | Low (2 branch types) | Medium (3-4 types) | Minimal (1-2 types) |
| **Long-lived branches** | main, develop | main | main + env branches | main only |
| **Feature branches** | From develop | From main | From main | Optional, very short |
| **Release process** | Release branches | Deploy on merge | Promote through envs | Deploy on push |
| **Hotfix process** | Hotfix branch → main + develop | Branch → main | Fix main → cherry-pick | Fix main → deploy |
| **Deployment frequency** | Weekly/monthly | On every merge | Controlled promotion | Multiple per day |
| **Best team size** | 10-100+ | 2-20 | 5-50 | 3-30 (senior) |
| **CI/CD requirement** | Moderate | High | High | Very high |
| **Learning curve** | Steep | Gentle | Moderate | Gentle (but discipline) |
| **Versioned releases** | Excellent | Poor | Good | Poor |
| **Multiple environments** | Manual | No built-in | Built-in | Via feature flags |

## Choosing the Right Strategy

Use this decision flowchart to choose:

```
Start
│
├── Do you ship versioned software (v1.0, v2.0)?
│   ├── Yes → Do you maintain multiple versions simultaneously?
│   │         ├── Yes → GitFlow
│   │         └── No  → GitLab Flow (release branches)
│   └── No  → Continue ↓
│
├── Do you have staging/pre-production environments?
│   ├── Yes → Do you need controlled promotion between environments?
│   │         ├── Yes → GitLab Flow (environment branches)
│   │         └── No  → GitHub Flow + CI/CD stages
│   └── No  → Continue ↓
│
├── Do you deploy to production multiple times per day?
│   ├── Yes → Do you have feature flags and strong CI/CD?
│   │         ├── Yes → Trunk-Based Development
│   │         └── No  → GitHub Flow (build maturity first)
│   └── No  → GitHub Flow
│
└── Default → GitHub Flow (simplest, most versatile)
```

<Callout type="tip" title="Strategy Evolution">
Most teams evolve their strategy over time:
1. **Startup (2-3 devs):** Push to main (informal trunk-based)
2. **Growing (5-15 devs):** GitHub Flow (add PRs, CI)
3. **Scaling (15-50 devs):** GitLab Flow (add environments) or GitFlow (add releases)
4. **Mature (50+ devs):** Trunk-based with feature flags (requires investment in tooling)

Don't jump to a complex strategy before you need it. Let your process evolve with your team.
</Callout>

## Release Tagging and Versioning

Regardless of which strategy you choose, proper versioning is essential:

### Semantic Versioning (SemVer)

```bash
# MAJOR.MINOR.PATCH
# v2.4.1
# │ │ │
# │ │ └── PATCH: Bug fixes, no API changes
# │ └──── MINOR: New features, backwards compatible
# └────── MAJOR: Breaking changes

# Examples:
git tag -a v1.0.0 -m "Initial release"
git tag -a v1.1.0 -m "Add search feature"
git tag -a v1.1.1 -m "Fix search pagination bug"
git tag -a v2.0.0 -m "New API (breaking changes)"
```

### Annotated Tags for Releases

```bash
# Annotated tags (recommended for releases)
git tag -a v2.1.0 -m "Release 2.1.0

Features:
- User dashboard redesign
- Search improvements
- API rate limiting

Bug fixes:
- Fix memory leak in worker pool
- Fix timezone handling in reports"

# List tags
git tag -l "v2.*"

# Show tag details
git show v2.1.0

# Push tags
git push origin v2.1.0
git push origin --tags  # push all tags
```

### Release Notes from Git Log

```bash
# Generate release notes from commit messages
git log --oneline v2.0.0..v2.1.0 --format="- %s (%an)"
# - Fix timezone handling in reports (Alice)
# - Add API rate limiting (Bob)
# - Search improvements (Charlie)
# - User dashboard redesign (Alice)
# - Fix memory leak in worker pool (Bob)

# Group by type (if using conventional commits)
echo "## Features"
git log v2.0.0..v2.1.0 --oneline --grep="^feat" --format="- %s"
echo ""
echo "## Bug Fixes"
git log v2.0.0..v2.1.0 --oneline --grep="^fix" --format="- %s"
```

## Branch Naming Conventions

Consistent naming makes branches self-documenting:

```bash
# Feature branches
feature/user-authentication
feature/JIRA-1234-add-search
feature/add-payment-gateway

# Bug fix branches
fix/login-timeout
bugfix/JIRA-5678-null-pointer
fix/memory-leak-worker

# Hotfix branches (production emergencies)
hotfix/security-patch
hotfix/JIRA-9999-sql-injection

# Release branches
release/2.1.0
release/2024-Q1

# Experimental branches
experiment/new-search-algorithm
spike/evaluate-redis-caching

# Convention: lowercase, hyphens, optional ticket numbers
# Bad:  Feature_UserAuth, my-branch, test123
# Good: feature/JIRA-1234-user-auth
```

<Callout type="tip" title="Enforce Naming with Git Hooks">
You can enforce branch naming conventions with a pre-push hook:
```bash
#!/bin/bash
branch=$(git branch --show-current)
pattern="^(feature|fix|hotfix|release|experiment)/"
if [[ ! $branch =~ $pattern ]] && [[ $branch != "main" ]] && [[ $branch != "develop" ]]; then
  echo "Branch name '$branch' doesn't match pattern: $pattern"
  exit 1
fi
```
</Callout>

```bash
# Walk through a complete GitFlow cycle
REPO=$(mktemp -d) && cd "$REPO" && git init -q

# === Setup: Initial release ===
echo '{"version": "1.0.0"}' > package.json
cat > src/app.js << 'EOF'
function app() {
  console.log("MyApp v1.0.0");
  return { status: "running" };
}
module.exports = app;
EOF
mkdir -p src
git add . && git commit -q -m "v1.0.0: Initial release"
git tag -a v1.0.0 -m "Release 1.0.0"

# Create develop branch
git switch -c develop
echo ""
echo "=== GitFlow Setup Complete ==="
echo "Branches: $(git branch | tr '\n' ' ')"
echo ""

# === Phase 1: Feature Development ===
echo "=== Phase 1: Feature Development ==="
echo ""

# Feature 1: User Authentication
git switch -c feature/user-auth develop
cat > src/auth.js << 'EOF'
function login(username, password) {
  // Authenticate user
  return { token: "jwt-token", user: username };
}
function logout(token) {
  return { success: true };
}
module.exports = { login, logout };
EOF
git add . && git commit -q -m "Add user authentication module"
echo "auth.js: $(head -1 src/auth.js)"
git add . && git commit -q -m "Add authentication tests"
echo "Feature/user-auth: $(git log --oneline develop..HEAD | wc -l) commits"

# Merge feature to develop
git switch develop
git merge --no-ff feature/user-auth -m "Merge feature/user-auth into develop"
git branch -d feature/user-auth
echo "Merged to develop."
echo ""

# Feature 2: Search
git switch -c feature/search develop
cat > src/search.js << 'EOF'
function search(query) {
  return { results: [], query: query };
}
module.exports = { search };
EOF
git add . && git commit -q -m "Add search functionality"
echo "Feature/search: $(git log --oneline develop..HEAD | wc -l) commit(s)"

git switch develop
git merge --no-ff feature/search -m "Merge feature/search into develop"
git branch -d feature/search
echo "Merged to develop."
echo ""

echo "Develop branch history:"
git log --oneline --graph develop
echo ""

# === Phase 2: Release Preparation ===
echo "=== Phase 2: Release Preparation ==="
echo ""

git switch -c release/1.1.0 develop
echo "Created release/1.1.0 from develop"

# Version bump
sed -i 's/1.0.0/1.1.0/g' package.json src/app.js
git add . && git commit -q -m "Bump version to 1.1.0"

# Bug fix found during release QA
echo "// Fixed edge case in search" >> src/search.js
git add . && git commit -q -m "Fix search edge case found in QA"

echo "Release branch commits:"
git log --oneline develop..release/1.1.0
echo ""

# Merge release to main
git switch main
git merge --no-ff release/1.1.0 -m "Release 1.1.0"
git tag -a v1.1.0 -m "Release 1.1.0 - Auth + Search"
echo "Merged release to main and tagged v1.1.0"

# Merge release back to develop
git switch develop
git merge --no-ff release/1.1.0 -m "Merge release/1.1.0 back to develop"
git branch -d release/1.1.0
echo "Merged release back to develop"
echo ""

echo "Main branch:"
git log --oneline main
echo ""
echo "Tags:"
git tag -l
echo ""

# === Phase 3: Hotfix ===
echo "=== Phase 3: Emergency Hotfix ==="
echo ""

# Critical bug in production!
git switch -c hotfix/auth-crash main
echo "Created hotfix branch from main (production)"

# Fix the bug
cat > src/auth.js << 'EOF'
function login(username, password) {
  if (!username || !password) {
    throw new Error("Username and password required");
  }
  return { token: "jwt-token", user: username };
}
function logout(token) {
  if (!token) return { success: false, error: "No token" };
  return { success: true };
}
module.exports = { login, logout };
EOF
git add . && git commit -q -m "Hotfix: Handle null credentials in auth"
sed -i 's/1.1.0/1.1.1/g' package.json
git add . && git commit -q -m "Bump version to 1.1.1"

echo "Hotfix commits:"
git log --oneline main..hotfix/auth-crash
echo ""

# Merge hotfix to main
git switch main
git merge --no-ff hotfix/auth-crash -m "Hotfix 1.1.1: auth crash fix"
git tag -a v1.1.1 -m "Hotfix 1.1.1"
echo "Merged hotfix to main, tagged v1.1.1"

# Merge hotfix to develop (critical!)
git switch develop
git merge --no-ff hotfix/auth-crash -m "Merge hotfix/auth-crash into develop"
git branch -d hotfix/auth-crash
echo "Merged hotfix to develop"
echo ""

# === Final State ===
echo "=== Final GitFlow State ==="
echo ""
echo "Main (production):"
git log --oneline main
echo ""
echo "Develop (integration):"
git log --oneline develop
echo ""
echo "Tags (releases):"
git tag -l -n1
echo ""
echo "Active branches:"
git branch
echo ""
echo "Full history graph:"
git log --oneline --graph --all

# Clean up
cd / && rm -rf "$REPO"
```

app.get('/products', (req, res) => {

## Key Takeaways

- **GitFlow** uses five branch types (main, develop, feature, release, hotfix) and is best for versioned software with scheduled releases
- **GitHub Flow** is the simplest strategy: main + feature branches, deploy on every merge to main
- **GitLab Flow** adds environment branches (staging, production) for controlled promotion between environments
- **Trunk-based development** has everyone commit to main with feature flags — requires high CI/CD maturity
- Start with GitHub Flow if unsure — it works for most web development teams and you can add complexity later
- All strategies benefit from consistent branch naming conventions (feature/*, fix/*, hotfix/*)
- Use semantic versioning (MAJOR.MINOR.PATCH) and annotated tags for releases
- The best strategy is one your team actually follows — a simple strategy used consistently beats a complex one used inconsistently
- Teams typically evolve: informal pushing to main, then GitHub Flow, then GitLab Flow or trunk-based with feature flags
- Branch protection rules, required reviews, and CI checks are essential regardless of which strategy you choose
