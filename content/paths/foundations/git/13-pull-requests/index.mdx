You have been coding for 3 days on a feature. You push your branch and create a pull request. Within an hour, you get 12 comments. ## What Is a Pull Request?

A pull request (PR) is a request to merge one branch into another. But it is much more than that — it is a conversation about code changes that includes:

- The diff (what changed)
- A description (why it changed)
- Reviews (what others think)
- CI checks (whether it works)
- A merge decision (how it enters the target branch)

```
Feature Branch                      Pull Request                      Main Branch
     │                                  │                                  │
     │  git push                        │                                  │
     ├─────────────►  PR Created        │                                  │
     │                  │               │                                  │
     │                  ├── Description  │                                  │
     │                  ├── Reviewers    │                                  │
     │                  ├── CI Checks    │                                  │
     │                  ├── Comments     │                                  │
```


### Writing Great PR Descriptions

A good PR description answers three questions: What changed? Why did it change? How can reviewers verify it?

```markdown
## Summary
<!-- One or two sentences explaining the purpose of this PR -->
Add rate limiting to the API to prevent abuse and ensure fair usage
across all clients.

## Motivation
<!-- Why is this change needed? Link to issues or discussions -->
We've been seeing intermittent API slowdowns caused by individual clients
making thousands of requests per minute. This affects all users.
See incident report: #89

## Changes
<!-- Specific list of what was modified -->
- Add rate limiting middleware using token bucket algorithm
- Configure default limit: 100 requests per minute per API key
- Add 429 (Too Many Requests) response with Retry-After header
- Add rate limit headers to all API responses (X-RateLimit-*)
- Store rate limit state in Redis for multi-instance support
- Add configuration via environment variables

## How It Works
<!-- Technical details for reviewers -->
The middleware uses a token bucket algorithm stored in Redis. Each API
key gets a bucket with 100 tokens that refills at 100 tokens/minute.
When a request arrives, a token is consumed. If the bucket is empty,
the request gets a 429 response.

## Test Plan
- [ ] Unit tests for token bucket logic
- [ ] Integration tests with Redis
- [ ] Load test: verify 429s are returned after limit exceeded
- [ ] Verify Retry-After header is accurate
- [ ] Test behavior when Redis is unavailable (fail-open)

## Rollback Plan
Set RATE_LIMIT_ENABLED=false to disable without deploying.
```

<Callout type="tip" title="The Smaller the PR, the Better the Review">
Large PRs get superficial reviews. Small PRs get thorough reviews. Aim for PRs under 400 lines of changes. If your feature is larger, break it into a series of smaller PRs that build on each other. A reviewer who sees 50 lines of focused changes will catch subtle bugs that they would miss in a 2000-line PR.
</Callout>

## Code Review: The Reviewer's Guide

### What to Look For

```
1. Correctness
   - Does the code do what the PR description says?
   - Are edge cases handled?
   - Are error conditions handled properly?

2. Security
   - Input validation present?
   - SQL injection, XSS, CSRF risks?
   - Secrets or credentials exposed?
   - Proper authentication/authorization checks?

3. Performance
   - N+1 queries?
   - Unnecessary loops or allocations?
   - Missing indexes for database queries?
   - Large payloads or memory usage?

4. Maintainability
   - Is the code readable?
   - Are names descriptive?
   - Is there unnecessary complexity?
   - Will future developers understand this?

5. Testing
   - Are there tests for the happy path?
   - Are there tests for edge cases?
   - Are error paths tested?
   - Do tests actually assert meaningful things?
```

### Writing Constructive Comments

```markdown
# Bad review comments:
"This is wrong."
"Why did you do it this way?"
"No."

# Good review comments:

# Suggestion with explanation:
"Consider using `Map` instead of a plain object here — it handles
non-string keys better and has O(1) `.has()` checks. This matters
because user IDs could be numeric."

# Question that helps understanding:
"I see this retries 3 times on failure. What happens if all retries
fail? Should we alert the on-call team after exhausting retries?"

# Pointing out a bug with context:
"This will throw a TypeError if `user.address` is null/undefined.
We've seen this in production with users who haven't completed
onboarding yet. Consider: `user.address?.city || 'Unknown'`"

# Acknowledging good work:
"Nice approach! Using the strategy pattern here makes it easy to
add new payment providers without touching existing code."
```

<Callout type="tip" title="Prefix Your Comments">
Many teams use prefixes to clarify the intent of review comments:
- **nit:** Minor style issue, not blocking (e.g., "nit: trailing whitespace")
- **suggestion:** An improvement idea, take it or leave it
- **question:** Seeking clarification, not requesting a change
- **blocking:** Must be addressed before merge
- **praise:** Something done well (do not skip these!)
</Callout>

## Addressing Review Feedback

### Pushing New Commits

```bash
# Make changes based on review feedback
# ... edit files ...

# Commit the fixes
git add src/profiles.js
git commit -m "Address review: add null check for user.address"

git add src/profiles.test.js
git commit -m "Address review: add test for missing address"

# Push the new commits
git push origin feature/user-profiles

# The PR automatically updates with the new commits
# Reviewers can see exactly what changed since their review
```

### Force-Pushing After Rebase

Sometimes reviewers want you to clean up your commit history before merging:

```bash
# Squash your "address review" commits into the original
git rebase -i HEAD~3
# Mark the fixup commits as "fixup" or "squash"

# Force push to update the PR
git push --force-with-lease origin feature/user-profiles
# The PR shows the cleaned-up history
```

<Callout type="warning" title="Force-Push Affects PR History">
When you force-push to a PR branch, GitHub loses the individual commit-by-commit view that reviewers use to see incremental changes. Some teams prefer adding new commits during review and squashing at merge time. Others prefer a clean history at all times. Follow your team's convention.
</Callout>

## Draft Pull Requests

Draft PRs signal that your work is in progress and not ready for review:

```bash
# Create a draft PR
gh pr create --draft \
  --title "WIP: Add user profiles" \
  --body "Work in progress. Opening early for visibility."

# Mark a draft as ready for review
gh pr ready 42

# Convert a regular PR back to draft
gh pr ready 42 --undo
```

Use draft PRs when:
- You want early feedback on your approach
- You want CI to run on your changes
- You want to track progress visually
- You are blocked and want others to see the state

## Merge Strategies

When a PR is approved and CI passes, you choose how to merge it into the target branch.

### Merge Commit (Default)

```bash
# Creates a merge commit that combines the feature branch
gh pr merge 42 --merge

# What happens:
#   main ── C1 ── C2 ── C3 ────── M (merge commit)
#                  │                │
#                  └── F1 ── F2 ───┘ (feature commits preserved)
```

**Pros:** Preserves full branch history. You can see exactly when a feature was merged and all its individual commits.

**Cons:** History can become noisy with many merge commits.

**Use when:** You want to preserve the complete development history, or the feature commits are clean and meaningful.

### Squash Merge

```bash
# Squashes all feature commits into a single commit on main
gh pr merge 42 --squash

# What happens:
#   main ── C1 ── C2 ── C3 ── S (single squashed commit)
#
#   Feature branch had F1, F2, F3 — all squashed into S
#   S's commit message is the PR title + description
```

**Pros:** Clean, linear history. Each feature is one commit on main. Hides messy WIP commits.

**Cons:** Loses individual commit granularity. If the squashed commit is large, `git bisect` is less useful.

**Use when:** The feature branch has messy commits (WIP, fixup, typo), or you want a clean main branch history.

### Rebase Merge

```bash
# Replays feature commits on top of main (no merge commit)
gh pr merge 42 --rebase

# What happens:
#   main ── C1 ── C2 ── C3 ── F1' ── F2' (commits replayed)
#
#   Feature commits are individually applied on top of main
#   Commit hashes change (F1 → F1', F2 → F2')
```

**Pros:** Linear history with individual commits preserved. Best of both worlds if commits are clean.

**Cons:** Commit hashes change, breaking references. Can be confusing if the branch had many commits.

**Use when:** The feature branch has clean, well-structured commits and you want linear history.

### Choosing the Right Strategy

```
Feature has 23 messy commits?      → Squash merge
Feature has 3 clean, logical commits? → Rebase merge
Feature is a long-running release?  → Merge commit
Team prefers linear history?        → Squash or rebase
Team wants full audit trail?        → Merge commit

Many teams default to squash merge because:
  - Most feature branches have messy history
  - One commit per feature makes main easy to read
  - You can always look at the closed PR for full history
```

<Callout type="tip" title="Configure Default Merge Strategy">
Repository admins can configure which merge strategies are allowed in Settings → General → Pull Requests. Many teams disable the plain merge commit option and only allow squash merge to keep main's history clean.
</Callout>

## CI Checks on Pull Requests

GitHub Actions (or other CI tools) run automatically on PRs:

```yaml
# .github/workflows/pr-checks.yml
name: PR Checks

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint

  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
```

## Stacked PRs

For large features, create a chain of PRs where each builds on the previous:

```
PR #1: Add user model          (base: main)
PR #2: Add user API            (base: PR #1's branch)
PR #3: Add user UI             (base: PR #2's branch)

Review and merge order: PR #1 → PR #2 → PR #3
Each PR is small and focused, but together they form the feature.
```

```bash
# Create the stack
git switch main && git pull
git switch -c feature/user-model
# ... add model code ...
git push -u origin feature/user-model

git switch -c feature/user-api  # branches from feature/user-model
# ... add API code ...
git push -u origin feature/user-api

git switch -c feature/user-ui   # branches from feature/user-api
# ... add UI code ...
git push -u origin feature/user-ui

# Create PRs
gh pr create --base main --head feature/user-model --title "Add user model"
gh pr create --base feature/user-model --head feature/user-api --title "Add user API"
gh pr create --base feature/user-api --head feature/user-ui --title "Add user UI"

# After PR #1 merges, update PR #2's base to main:
# GitHub often does this automatically, or use:
gh pr edit 2 --base main
```

<Callout type="warning" title="Stacked PRs Need Careful Management">
When you update an earlier PR in the stack (e.g., address review feedback on PR #1), you need to rebase all downstream PRs. This can be tedious. Use stacked PRs only when the feature is truly too large for a single PR and the pieces are logically separable. Many teams prefer feature flags over stacked PRs.
</Callout>

```bash
class AuthService {
    constructor(db) {
        this.db = db;

    async createUser(username, password) {
        const salt = crypto.randomBytes(16).toString('hex');
        const hash = crypto.pbkdf2Sync(password, salt, 10000, 64, 'sha512').toString('hex');

        this.sessions.set(sessionId, { userId: user.id, createdAt: Date.now() });
        return { sessionId, userId: user.id };
    }

        return this.sessions.get(sessionId) || null;

}

        this.sessions.set(sessionId, { userId: user.id, createdAt: Date.now() });
        return { sessionId, userId: user.id };
    }

    validateSession(sessionId) {
        return this.sessions.get(sessionId) || null;
    }

    // New: API key management
    async createApiKey(userId) {
        var key = "sk_" + crypto.randomBytes(24).toString('hex');
        console.log("Created API key for user " + userId + ": " + key);
        await this.db.insert('api_keys', {
            key: key,
            userId: userId,
            created: new Date()
        });
        return key;
    }

    async validateApiKey(key) {
        const record = await this.db.findOne('api_keys', { key: key });
        if (record) {
            return record;
        }
        return null;
    }

    async deleteApiKey(key) {
        this.db.delete('api_keys', { key: key });
    }

    async listApiKeys(userId) {
        const keys = await this.db.find('api_keys', { userId });
        return keys;
    }

    async rotateApiKey(oldKey) {
        const record = await this.validateApiKey(oldKey);
        if (!record) return null;
        this.deleteApiKey(oldKey);
        const newKey = await this.createApiKey(record.userId);
        return newKey;
    }
}

module.exports = AuthService;
EOF
```

## Key Takeaways

- A pull request is a conversation about code changes — not just a merge mechanism
- Write PR descriptions that answer What, Why, and How — include a test plan and link related issues
- Keep PRs small and focused (under 400 lines) for thorough reviews
- Use PR templates to standardize descriptions across your team
- Review code for correctness, security, performance, and maintainability — not just style
- Write constructive review comments: explain the problem, suggest a fix, and share context
- Use prefixes (nit, suggestion, blocking, question) to clarify the severity of review comments
- Choose merge strategies thoughtfully: squash for messy history, rebase for clean commits, merge commit for full audit trail
- Draft PRs signal work-in-progress and enable early feedback without triggering reviews
- CI checks on PRs catch bugs automatically before code reaches main
- Stacked PRs break large features into reviewable pieces but require careful branch management
