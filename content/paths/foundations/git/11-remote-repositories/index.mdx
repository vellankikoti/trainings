<Callout type="story" title="The Story">
Your code works perfectly on your laptop. Now your team needs it. You need to push your changes to a shared remote, pull your colleague's updates, and handle the inevitable conflicts. Understanding remotes — especially the difference between fetch and pull — is what makes distributed Git work. Without it, you're just working alone with a fancy undo system. Remotes are named URL bookmarks that point to other copies of your repository. Once you understand how they work, the entire world of collaboration opens up.
</Callout>

## What Is a Remote?

A remote is a named reference to another copy of your repository, usually hosted on a server. Think of it as a bookmark — a short name that maps to a URL.

```bash
# When you clone a repository, Git automatically creates a remote called "origin"
git clone https://github.com/your-org/project.git
cd project

# "origin" now points to where you cloned from
git remote
# origin

# See the full URLs
git remote -v
# origin  https://github.com/your-org/project.git (fetch)
# origin  https://github.com/your-org/project.git (push)
```

```
Your Machine                          GitHub Server
┌──────────────┐                    ┌──────────────┐
│  Local Repo  │  ── origin ──►    │  Remote Repo │
│              │                    │              │
│  main        │  ◄── fetch ───    │  main        │
│  feature/x   │  ─── push ──►    │  feature/x   │
│              │                    │              │
│  origin/main │  (tracking branch) │              │
└──────────────┘                    └──────────────┘
```

## Managing Remotes

### Adding a Remote

```bash
# Add a new remote
git remote add origin https://github.com/your-org/project.git

# Add a second remote (common for forks)
git remote add upstream https://github.com/original-org/project.git

# Add a remote for a colleague's fork
git remote add alice https://github.com/alice/project.git
```

### Viewing Remote Details

```bash
# List all remotes with URLs
git remote -v
# origin    https://github.com/your-org/project.git (fetch)
# origin    https://github.com/your-org/project.git (push)
# upstream  https://github.com/original-org/project.git (fetch)
# upstream  https://github.com/original-org/project.git (push)

# Show detailed info about a specific remote
git remote show origin
# * remote origin
#   Fetch URL: https://github.com/your-org/project.git
#   Push  URL: https://github.com/your-org/project.git
#   HEAD branch: main
#   Remote branches:
#     main           tracked
#     feature/auth   tracked
#     feature/deploy tracked
#   Local branches configured for 'git pull':
#     main merges with remote main
#   Local refs configured for 'git push':
#     main pushes to main (up to date)
```

### Renaming and Removing Remotes

```bash
# Rename a remote
git remote rename origin github

# Remove a remote
git remote remove alice

# Change a remote's URL (e.g., switch from HTTPS to SSH)
git remote set-url origin git@github.com:your-org/project.git
```

<Callout type="tip" title="The 'origin' Convention">
There is nothing special about the name `origin` — it is just a convention. When you `git clone`, Git names the source remote `origin` automatically. You could rename it to anything. But every tool and tutorial assumes `origin`, so stick with the convention unless you have a good reason not to.
</Callout>

## Pushing Changes

`git push` uploads your local commits to a remote repository.

### Basic Push

```bash
# Push current branch to origin
git push origin main

# If your branch tracks a remote branch (set up with -u), just:
git push
```

### Setting Up Tracking with -u

The `-u` (or `--set-upstream`) flag creates a tracking relationship between your local branch and the remote branch. After setting it once, you can use `git push` and `git pull` without specifying the remote and branch.

```bash
# First push of a new branch — set up tracking
git push -u origin feature/login
# Branch 'feature/login' set up to track remote branch 'feature/login' from 'origin'.

# From now on, just use:
git push    # pushes to origin/feature/login
git pull    # pulls from origin/feature/login
```

### Pushing a Specific Branch

```bash
# Push a local branch to a differently-named remote branch
git push origin local-branch:remote-branch

# Push all branches
git push origin --all

# Push all tags
git push origin --tags

# Push a specific tag
git push origin v1.0.0
```

### Force Push (Danger Zone)

```bash
# Force push overwrites the remote branch with your local version
# WARNING: This rewrites remote history and can destroy others' work
git push --force origin feature/my-branch

# Safer alternative: force-with-lease
# Only force pushes if the remote hasn't changed since you last fetched
git push --force-with-lease origin feature/my-branch
```

<Callout type="warning" title="Never Force Push to Shared Branches">
Force pushing to `main` or any branch others are working on will overwrite their commits on the remote. Their next pull will create chaos. Only force push to YOUR OWN feature branches that nobody else has checked out. Even then, prefer `--force-with-lease` which fails if someone else has pushed to the branch since your last fetch.
</Callout>

## Fetching Changes

`git fetch` downloads commits, branches, and tags from a remote — but does NOT modify your working directory or local branches. It updates your remote-tracking branches (like `origin/main`).

```bash
# Fetch from origin
git fetch origin

# Fetch from all remotes
git fetch --all

# Fetch a specific branch
git fetch origin main

# Fetch and prune deleted remote branches
git fetch --prune origin
# Removes local tracking branches for branches that no longer exist on the remote
```

### What Fetch Actually Does

```
Before fetch:
  Local:   main (commit C3)    origin/main (commit C3)
  Remote:  main (commit C5)    ← two new commits

After git fetch origin:
  Local:   main (commit C3)    origin/main (commit C5) ← updated!
  Remote:  main (commit C5)

Your local "main" hasn't changed.
origin/main now reflects the remote's state.
You can inspect the new commits before merging.
```

```bash
# After fetching, see what's new
git log main..origin/main --oneline
# abc1234 Add payment processing
# def5678 Fix authentication bug

# See the diff between your branch and the remote
git diff main origin/main

# When you're ready, merge the changes
git merge origin/main
```

<Callout type="tip" title="Fetch First, Then Decide">
`git fetch` is always safe. It never changes your local branches or working directory. Develop a habit of fetching first, reviewing what changed with `git log` or `git diff`, and then merging when you are ready. This is especially important before rebasing or starting new work.
</Callout>

## Pulling Changes

`git pull` is a convenience command that combines `git fetch` and `git merge` (or `git rebase`) in one step.

```bash
# Pull = fetch + merge
git pull origin main
# Equivalent to:
#   git fetch origin main
#   git merge origin/main

# Pull with rebase instead of merge
git pull --rebase origin main
# Equivalent to:
#   git fetch origin main
#   git rebase origin/main
```

### Fetch vs Pull

| Aspect | git fetch | git pull |
|---|---|---|
| Downloads new commits | Yes | Yes |
| Modifies working directory | No | Yes |
| Updates local branches | No | Yes (merges or rebases) |
| Can cause conflicts | No | Yes |
| Safe to run anytime | Yes | Depends on state |
| Lets you review first | Yes | No (auto-merges) |

```bash
# The fetch-first workflow (recommended for teams)
git fetch origin
git log --oneline main..origin/main    # see what's new
git diff main origin/main              # inspect the changes
git merge origin/main                  # merge when ready

# The pull workflow (fine for solo work or simple cases)
git pull origin main                   # fetch and merge in one step
```

### Pull Strategies: Merge vs Rebase

When you pull and your local branch has diverged from the remote, Git needs to reconcile the histories. You have two strategies:

```bash
# Pull with merge (default) — creates a merge commit
git pull origin main
# History: your commits + remote commits + merge commit

# Pull with rebase — replays your commits on top of remote
git pull --rebase origin main
# History: remote commits + your commits (linear, no merge commit)
```

```
Pull with merge:
  C1 ── C2 ── C3 (your commit) ── M (merge commit)
         │                          │
         └── C4 ── C5 (remote) ────┘

Pull with rebase:
  C1 ── C2 ── C4 ── C5 (remote) ── C3' (your commit, replayed)
```

```bash
# Set rebase as the default pull strategy
git config --global pull.rebase true

# Or per-repository
git config pull.rebase true

# Override for a single pull
git pull --no-rebase origin main    # force merge even if config says rebase
git pull --rebase origin main       # force rebase even if config says merge
```

<Callout type="tip" title="Which Pull Strategy?">
Use `pull --rebase` for feature branches where you want clean, linear history. Use `pull` with merge for shared branches where preserving the merge context matters. Many teams set `pull.rebase true` globally because it keeps the history cleaner in the common case where you just need to sync up.
</Callout>

## Tracking Branches

A tracking branch is a local branch linked to a remote branch. Git uses this link for `git push`, `git pull`, and status messages like "Your branch is ahead of 'origin/main' by 2 commits."

### Remote-Tracking Branches

Remote-tracking branches are read-only references like `origin/main` that represent the state of branches on the remote as of your last fetch.

```bash
# List remote-tracking branches
git branch -r
# origin/main
# origin/feature/auth
# origin/feature/deploy

# List all branches (local + remote-tracking)
git branch -a
# * main
#   feature/login
#   remotes/origin/main
#   remotes/origin/feature/auth
#   remotes/origin/feature/deploy

# See tracking relationships
git branch -vv
# * main          abc1234 [origin/main] Latest commit message
#   feature/login def5678 [origin/feature/login: ahead 2] Add login form
#   experiment    ghi9012 No tracking branch set
```

### Setting Up Tracking

```bash
# When you push with -u, tracking is set up automatically
git push -u origin feature/login

# Set tracking on an existing branch
git branch --set-upstream-to=origin/feature/login feature/login

# Shorthand when you're on the branch
git branch -u origin/feature/login

# Create a local branch that tracks a remote branch
git switch --track origin/feature/auth
# Creates local "feature/auth" tracking "origin/feature/auth"

# Or the shorter version
git switch feature/auth
# If origin/feature/auth exists and no local branch matches,
# Git automatically creates a tracking branch
```

### Ahead/Behind Status

```bash
# See how your branch compares to its upstream
git status
# On branch main
# Your branch is ahead of 'origin/main' by 2 commits.

# Detailed ahead/behind count
git rev-list --left-right --count origin/main...main
# 3    2
# (remote is 3 ahead, local is 2 ahead — branches have diverged)

# Fetch first to get accurate counts
git fetch origin
git status
# On branch main
# Your branch and 'origin/main' have diverged,
# and have 2 and 3 different commits each, respectively.
```

## Push Rejected: Non-Fast-Forward

When someone else has pushed to the remote branch since your last fetch, your push will be rejected:

```bash
git push origin main
# ! [rejected]        main -> main (non-fast-forward)
# error: failed to push some refs to 'origin'
# hint: Updates were rejected because the tip of your current branch is behind
# hint: its remote counterpart. Integrate the remote changes
# hint: (e.g., 'git pull ...') before pushing again.
```

### The Fetch-Merge-Push Cycle

```bash
# Step 1: Fetch the latest
git fetch origin

# Step 2: Merge (or rebase) the remote changes
git merge origin/main
# OR
git rebase origin/main

# Step 3: Resolve any conflicts if needed
# ... edit files, git add, git commit ...

# Step 4: Push your merged result
git push origin main
```

```
Timeline:
  You:   C1 ── C2 ── C3 (your work)
  Remote: C1 ── C2 ── C4 ── C5 (others' work)

  Push rejected! (non-fast-forward)

  After fetch + merge:
  You:   C1 ── C2 ── C3 ── M (merge of C5 and C3)
                │             │
                └── C4 ── C5 ─┘

  Push succeeds! (fast-forward from remote's perspective)
```

<Callout type="warning" title="Never Force Push to Fix Rejected Pushes">
When a push is rejected, the solution is fetch-merge-push (or fetch-rebase-push), NOT force push. Force pushing overwrites whatever the remote has, destroying other people's work. The only exception is your own feature branch that nobody else uses.
</Callout>

## Working with Multiple Remotes

Multiple remotes are common when working with forks or deploying to multiple locations.

### Fork Workflow Remotes

```bash
# After forking a repo on GitHub and cloning your fork:
git clone https://github.com/you/project.git
cd project

# origin = your fork
git remote -v
# origin  https://github.com/you/project.git (fetch)
# origin  https://github.com/you/project.git (push)

# Add the original repo as "upstream"
git remote add upstream https://github.com/original-org/project.git

# Now you have two remotes
git remote -v
# origin    https://github.com/you/project.git (fetch)
# origin    https://github.com/you/project.git (push)
# upstream  https://github.com/original-org/project.git (fetch)
# upstream  https://github.com/original-org/project.git (push)

# Fetch from the original repo
git fetch upstream

# Merge upstream changes into your main
git switch main
git merge upstream/main

# Push the updated main to your fork
git push origin main
```

### Deployment Remotes

```bash
# Add a remote for your staging server
git remote add staging git@staging.example.com:project.git

# Add a remote for production
git remote add production git@production.example.com:project.git

# Deploy to staging
git push staging main

# Deploy to production
git push production main:release
```

## Tags and Remotes

By default, `git push` does NOT push tags. You need to do it explicitly.

```bash
# Push a specific tag
git push origin v1.0.0

# Push all tags
git push origin --tags

# Push only annotated tags (recommended)
git push origin --follow-tags

# Delete a remote tag
git push origin --delete v1.0.0-beta

# Fetch tags from remote
git fetch --tags origin
```

<Callout type="tip" title="Use --follow-tags for Releases">
`git push --follow-tags` pushes annotated tags that are reachable from the commits being pushed. This is safer than `--tags` which pushes ALL local tags. Configure it as default with:
```bash
git config --global push.followTags true
```
</Callout>

## Pruning Stale Remote References

When branches are deleted on the remote, your local remote-tracking branches remain. Prune them:

```bash
# See stale remote-tracking branches
git remote prune origin --dry-run
# Pruning origin
# * [would prune] origin/feature/old-feature
# * [would prune] origin/feature/merged-branch

# Actually prune them
git remote prune origin

# Or fetch and prune in one command
git fetch --prune origin

# Configure automatic pruning on fetch
git config --global fetch.prune true
```

```bash
git push origin main -q
echo ""
echo "Developer B pushed successfully after merging!"
echo ""
echo "Final history on main:"
git log --oneline --graph -6

# Clean up
cd / && rm -rf "$WORK_DIR"
```

```bash
# Demonstrate the critical difference between fetch and pull
WORK_DIR=$(mktemp -d)

# Create the shared "remote" repository
git init --bare "$WORK_DIR/shared.git" -q

# Create two clones simulating two developers
git clone "$WORK_DIR/shared.git" "$WORK_DIR/alice" -q
git clone "$WORK_DIR/shared.git" "$WORK_DIR/bob" -q

# Alice sets up the initial project
cd "$WORK_DIR/alice"
cat > server.py << 'EOF'
class Server:
    def __init__(self, port=8080):
        self.port = port
        self.routes = {}

    def add_route(self, path, handler):
        self.routes[path] = handler

    def start(self):
        print(f"Server running on port {self.port}")
        print(f"Routes: {list(self.routes.keys())}")
EOF
git add . && git commit -q -m "Initial server implementation"
git push -u origin main -q

# Bob pulls the initial code
cd "$WORK_DIR/bob"
git pull origin main -q

echo "=== Setup complete: Both Alice and Bob have the same code ==="
echo ""

# === Alice makes changes and pushes ===
cd "$WORK_DIR/alice"
cat > server.py << 'EOF'
import logging

class Server:
    def __init__(self, port=8080):
        self.port = port
        self.routes = {}
        self.logger = logging.getLogger('server')

    def add_route(self, path, handler):
        self.routes[path] = handler
        self.logger.info(f"Route added: {path}")

    def start(self):
        self.logger.info(f"Server starting on port {self.port}")
        print(f"Server running on port {self.port}")
        print(f"Routes: {list(self.routes.keys())}")
EOF
git add . && git commit -q -m "Add logging to server"
git push origin main -q
echo "Alice pushed logging changes."

# === Bob makes local changes (without pulling) ===
cd "$WORK_DIR/bob"
cat > server.py << 'EOF'
class Server:
    def __init__(self, port=8080, host='localhost'):
        self.port = port
        self.host = host
        self.routes = {}

    def add_route(self, path, handler):
        self.routes[path] = handler

    def start(self):
        print(f"Server running on {self.host}:{self.port}")
        print(f"Routes: {list(self.routes.keys())}")
EOF
git add . && git commit -q -m "Add host configuration to server"

echo ""
echo "=== Current state: Alice and Bob have diverged ==="
echo ""
echo "Bob's log:"
git log --oneline -3
echo ""
echo "Bob's origin/main (stale — hasn't fetched):"
git log origin/main --oneline -3

# === Demonstration 1: git fetch (safe inspection) ===
echo ""
echo "========================================="
echo "  DEMONSTRATION 1: git fetch"
echo "========================================="
echo ""

git fetch origin
echo "After fetch:"
echo ""
echo "Bob's local main (unchanged!):"
git log --oneline -3
echo ""
echo "Bob's origin/main (updated with Alice's work):"
git log origin/main --oneline -3

echo ""
echo "Bob can now INSPECT what Alice changed:"
echo "--- Changes Alice made ---"
git diff main origin/main
echo "--- End of diff ---"
echo ""
echo "Bob's working directory is UNTOUCHED."
echo "Bob can decide how to integrate (merge or rebase)."

echo ""
echo "Bob sees the divergence:"
git log --oneline --graph --all

echo ""
echo "Ahead/behind count:"
COUNTS=$(git rev-list --left-right --count origin/main...main)
BEHIND=$(echo "$COUNTS" | cut -f1)
AHEAD=$(echo "$COUNTS" | cut -f2)
echo "  Local is $AHEAD commit(s) ahead, $BEHIND commit(s) behind origin/main"

# === Now Bob merges manually ===
echo ""
echo "Bob decides to merge after reviewing:"
git merge origin/main -m "Merge Alice's logging changes" 2>&1 || true

# Resolve conflict if any
if git diff --name-only --diff-filter=U | grep -q .; then
    echo ""
    echo "Conflict! Resolving by combining both changes..."
    cat > server.py << 'EOF'
import logging

class Server:
    def __init__(self, port=8080, host='localhost'):
        self.port = port
        self.host = host
        self.routes = {}
        self.logger = logging.getLogger('server')

    def add_route(self, path, handler):
        self.routes[path] = handler
        self.logger.info(f"Route added: {path}")

    def start(self):
        self.logger.info(f"Server starting on {self.host}:{self.port}")
        print(f"Server running on {self.host}:{self.port}")
        print(f"Routes: {list(self.routes.keys())}")
EOF
    git add server.py
    git commit -q -m "Merge: combine logging and host configuration"
fi

echo ""
echo "After merge:"
git log --oneline --graph -6

# === Reset Bob to demonstrate pull ===
echo ""
echo "========================================="
echo "  DEMONSTRATION 2: git pull (for comparison)"
echo "========================================="
echo ""

# Reset Bob's state to before the merge
BEFORE_MERGE=$(git reflog --format="%H %gs" | grep "commit: Add host" | head -1 | cut -d' ' -f1)
git reset --hard "$BEFORE_MERGE" -q

echo "Bob reset to before-merge state."
echo "Bob's log:"
git log --oneline -3

echo ""
echo "Now using git pull (fetch + merge in one step):"
```

```bash
cat > ~/remote-sync-status.sh << 'SCRIPT'
#!/bin/bash
set -euo pipefail

# Remote Sync Tool — shows sync status between local branches and all remotes

# Colors
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
CYAN='\033[0;36m'
BOLD='\033[1m'
NC='\033[0m'

# Check we're in a git repo
if ! git rev-parse --git-dir &>/dev/null; then
    echo -e "${RED}Error: Not in a git repository${NC}"
    exit 1
fi

REPO_NAME=$(basename "$(git rev-parse --show-toplevel)")

echo -e "${BOLD}${BLUE}============================================${NC}"
echo -e "${BOLD}${BLUE}  Remote Sync Status: ${REPO_NAME}${NC}"
echo -e "${BOLD}${BLUE}============================================${NC}"
echo ""

# === Step 1: Fetch all remotes ===
if [ "${1:-}" != "--no-fetch" ]; then
    echo -e "${CYAN}Fetching all remotes...${NC}"
    git fetch --all --prune --quiet 2>/dev/null || true
    echo ""
fi

# === Step 2: Show remotes ===
echo -e "${BOLD}Configured Remotes:${NC}"
while IFS= read -r line; do
    REMOTE_NAME=$(echo "$line" | awk '{print $1}')
    REMOTE_URL=$(echo "$line" | awk '{print $2}')
    REMOTE_TYPE=$(echo "$line" | awk '{print $3}')
    if [ "$REMOTE_TYPE" = "(fetch)" ]; then
        echo -e "  ${GREEN}$REMOTE_NAME${NC} → $REMOTE_URL"
    fi
done <<< "$(git remote -v)"
echo ""

# === Step 3: Show branch sync status ===
echo -e "${BOLD}Branch Sync Status:${NC}"
echo ""

# Header
printf "  ${BOLD}%-30s %-20s %-12s %-12s %s${NC}\n" "LOCAL BRANCH" "TRACKS" "AHEAD" "BEHIND" "STATUS"
printf "  %-30s %-20s %-12s %-12s %s\n" "──────────────" "──────" "─────" "──────" "──────"

# Process each local branch
git for-each-ref --format='%(refname:short) %(upstream:short) %(upstream:track)' refs/heads/ | while read -r LOCAL UPSTREAM TRACK; do
    if [ -z "$UPSTREAM" ]; then
        # No tracking branch
        printf "  %-30s %-20s %-12s %-12s " "$LOCAL" "(none)" "-" "-"
        echo -e "${YELLOW}no upstream${NC}"
        continue
    fi

    # Count ahead/behind
    AHEAD=0
    BEHIND=0
    if git rev-parse "$UPSTREAM" &>/dev/null; then
        COUNTS=$(git rev-list --left-right --count "$UPSTREAM...$LOCAL" 2>/dev/null || echo "0 0")
        BEHIND=$(echo "$COUNTS" | awk '{print $1}')
        AHEAD=$(echo "$COUNTS" | awk '{print $2}')
    fi

    # Determine status
    if [ "$AHEAD" -eq 0 ] && [ "$BEHIND" -eq 0 ]; then
        STATUS="${GREEN}up to date${NC}"
    elif [ "$AHEAD" -gt 0 ] && [ "$BEHIND" -eq 0 ]; then
        STATUS="${YELLOW}needs push${NC}"
    elif [ "$AHEAD" -eq 0 ] && [ "$BEHIND" -gt 0 ]; then
        STATUS="${CYAN}needs pull${NC}"
    else
        STATUS="${RED}diverged${NC}"
    fi

    printf "  %-30s %-20s %-12s %-12s " "$LOCAL" "$UPSTREAM" "$AHEAD" "$BEHIND"
    echo -e "$STATUS"
done

echo ""

# === Step 4: Show remote branches not tracked locally ===
echo -e "${BOLD}Remote Branches Not Tracked Locally:${NC}"
UNTRACKED=0
for REMOTE in $(git remote); do
    git for-each-ref --format='%(refname:short)' "refs/remotes/$REMOTE/" | while read -r REMOTE_BRANCH; do
        # Skip HEAD reference
        if [[ "$REMOTE_BRANCH" == *"/HEAD" ]]; then
            continue
        fi

        # Extract branch name without remote prefix
        BRANCH_NAME="${REMOTE_BRANCH#$REMOTE/}"

        # Check if a local branch tracks this remote branch
        LOCAL_EXISTS=$(git for-each-ref --format='%(upstream:short)' refs/heads/ | grep -x "$REMOTE_BRANCH" || true)
        LOCAL_NAME_EXISTS=$(git rev-parse --verify "$BRANCH_NAME" 2>/dev/null || true)

        if [ -z "$LOCAL_EXISTS" ] && [ -z "$LOCAL_NAME_EXISTS" ]; then
            LAST_COMMIT=$(git log -1 --format="%h %s" "$REMOTE_BRANCH" 2>/dev/null || echo "unknown")
            echo -e "  ${CYAN}$REMOTE_BRANCH${NC} → $LAST_COMMIT"
            UNTRACKED=$((UNTRACKED + 1))
        fi
    done
done

if [ "$UNTRACKED" -eq 0 ]; then
    echo "  (none)"
fi

echo ""

# === Step 5: Show stale remote-tracking branches ===
echo -e "${BOLD}Summary:${NC}"
LOCAL_COUNT=$(git branch | wc -l | tr -d ' ')
REMOTE_COUNT=$(git branch -r | grep -v HEAD | wc -l | tr -d ' ')
REMOTE_NAMES=$(git remote | wc -l | tr -d ' ')
echo "  Local branches:  $LOCAL_COUNT"
echo "  Remote branches: $REMOTE_COUNT"
echo "  Remotes:         $REMOTE_NAMES"

# Count branches needing action
NEEDS_PUSH=$(git for-each-ref --format='%(refname:short) %(upstream:short)' refs/heads/ | while read -r L U; do
    if [ -n "$U" ] && git rev-parse "$U" &>/dev/null; then
        COUNTS=$(git rev-list --left-right --count "$U...$L" 2>/dev/null || echo "0 0")
        A=$(echo "$COUNTS" | awk '{print $2}')
        if [ "$A" -gt 0 ]; then echo "x"; fi
    fi
done | wc -l | tr -d ' ')

NEEDS_PULL=$(git for-each-ref --format='%(refname:short) %(upstream:short)' refs/heads/ | while read -r L U; do
    if [ -n "$U" ] && git rev-parse "$U" &>/dev/null; then
        COUNTS=$(git rev-list --left-right --count "$U...$L" 2>/dev/null || echo "0 0")
        B=$(echo "$COUNTS" | awk '{print $1}')
        if [ "$B" -gt 0 ]; then echo "x"; fi
    fi
done | wc -l | tr -d ' ')

echo ""
if [ "$NEEDS_PUSH" -gt 0 ]; then
    echo -e "  ${YELLOW}$NEEDS_PUSH branch(es) need pushing${NC}"
fi
if [ "$NEEDS_PULL" -gt 0 ]; then
    echo -e "  ${CYAN}$NEEDS_PULL branch(es) need pulling${NC}"
fi
if [ "$NEEDS_PUSH" -eq 0 ] && [ "$NEEDS_PULL" -eq 0 ]; then
    echo -e "  ${GREEN}All tracked branches are in sync!${NC}"
fi

echo ""
```

## Key Takeaways

- A remote is a named URL bookmark pointing to another copy of your repository — `origin` is the conventional name for where you cloned from
- `git push -u origin branch` sets up tracking so future push/pull commands work without arguments
- `git fetch` downloads remote changes without modifying your local branches — always safe to run
- `git pull` is `git fetch` + `git merge` (or rebase) in one step — convenient but less controllable
- Prefer the fetch-inspect-merge workflow for important branches so you can review changes before integrating
- Tracking branches (`origin/main`) are read-only local references that reflect the remote's state at last fetch
- Use `git branch -vv` to see which local branches track which remotes and their ahead/behind status
- When a push is rejected (non-fast-forward), fetch and merge first — never force push to shared branches
- `--force-with-lease` is a safer alternative to `--force` that fails if the remote has unexpected changes
- Multiple remotes are common: `origin` for your fork, `upstream` for the original project
- Tags must be pushed explicitly with `git push --tags` or `git push --follow-tags`
