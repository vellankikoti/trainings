<Callout type="story" title="The Story">
You open your terminal and see the dreaded message: "CONFLICT (content): Merge conflict in payment.js". Your heart rate goes up. But here's the truth: a merge conflict is just Git saying "two people changed the same lines and I need a human to decide which version is correct." Once you understand the markers, resolution is straightforward. Every developer hits conflicts. The ones who resolve them in five minutes instead of five hours are the ones who understand the mechanics. Conflicts aren't scary — they're communication.
</Callout>

## What Causes Conflicts?

Git can automatically merge most changes. Conflicts only happen when Git can't decide which version is correct:

### Same Lines Changed

```bash
# On main: line 10 of app.js says
const timeout = 3000;

# On feature: line 10 of app.js says
const timeout = 5000;

# Git: "Both branches changed line 10. Which timeout value is correct?"
```

### File Deleted vs Modified

```bash
# On main: someone deletes old-utils.js
git rm old-utils.js

# On feature: someone adds a function to old-utils.js
echo "function newHelper() {}" >> old-utils.js

# Git: "Main deleted the file but feature modified it. What do you want?"
```

### Both Branches Add at the Same Location

```bash
# Both branches add a new function after line 50
# Main adds: function validateEmail() {}
# Feature adds: function validatePhone() {}

# Git: "Both added content at the same spot. What order should they be in?"
```

### What Doesn't Cause Conflicts

```bash
# Different files — Git handles this automatically
# Main edits: src/auth.js
# Feature edits: src/payment.js
# Result: both changes are included, no conflict

# Same file, different sections — Git handles this too
# Main edits line 10
# Feature edits line 90
# Result: both changes are included, no conflict
```

## Reading Conflict Markers

When a conflict occurs, Git marks the conflicting sections in the file:

```
<<<<<<< HEAD
const timeout = 3000;
const retries = 3;
=======
const timeout = 5000;
const retries = 5;
>>>>>>> feature/increase-timeouts
```

Here's what each marker means:

```
<<<<<<< HEAD
(your current branch's version — what you're merging INTO)
=======
(the incoming branch's version — what you're merging FROM)
>>>>>>> feature/increase-timeouts
```

With `merge.conflictStyle = diff3`, you also see the common ancestor:

```
<<<<<<< HEAD
const timeout = 3000;
||||||| merged common ancestors
const timeout = 1000;
=======
const timeout = 5000;
>>>>>>> feature/increase-timeouts
```

<Callout type="tip" title="Enable diff3 for Better Context">
The `diff3` conflict style shows the original version (common ancestor) alongside both branches. This makes it much easier to understand what each side changed. Enable it globally:
```bash
git config --global merge.conflictStyle diff3
```
</Callout>

## Resolving Conflicts Step by Step

### Step 1: Identify Conflicting Files

```bash
# After a merge conflict, git status shows you exactly what's wrong
git status
# On branch main
# You have unmerged paths.
#   (fix conflicts and run "git add")
#
# Unmerged paths:
#   (use "git add <file>..." to mark resolution)
#       both modified:   src/payment.js
#       both modified:   src/config.js
```

### Step 2: Open the File and Find Markers

```bash
# Search for conflict markers
grep -rn "<<<<<<" src/
# src/payment.js:10:<<<<<<< HEAD
# src/config.js:5:<<<<<<< HEAD
```

### Step 3: Edit the File — Choose the Resolution

You have several options:

```javascript
// Option A: Keep your version (HEAD / current branch)
const timeout = 3000;
const retries = 3;

// Option B: Keep their version (incoming branch)
const timeout = 5000;
const retries = 5;

// Option C: Combine both (often the right answer)
const timeout = 5000;   // Use the longer timeout
const retries = 3;      // Keep the original retry count

// Option D: Write something completely new
const timeout = parseInt(process.env.TIMEOUT) || 5000;
const retries = parseInt(process.env.RETRIES) || 3;
```

**Remove ALL conflict markers** (`<<<<<<<`, `=======`, `>>>>>>>`). The file should look like normal code when you're done.

### Step 4: Mark as Resolved and Commit

```bash
# Stage the resolved file
git add src/payment.js
git add src/config.js

# Commit the merge
git commit
# Git opens your editor with a pre-filled merge commit message
# Merge branch 'feature/increase-timeouts' into main
#
# Conflicts:
#   src/payment.js
#   src/config.js
```

## Using Merge Tools

For complex conflicts, visual merge tools make resolution easier:

### Configure a Merge Tool

```bash
# Use VS Code as your merge tool
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait --merge $REMOTE $LOCAL $BASE $MERGED'

# Use vimdiff (built-in, no setup needed)
git config --global merge.tool vimdiff

# Use meld (popular GUI tool)
git config --global merge.tool meld
```

### Launch the Merge Tool

```bash
# After a conflict occurs, launch the tool
git mergetool
# Opens each conflicting file in the configured tool

# The tool shows:
# LOCAL  — your branch's version
# REMOTE — the incoming branch's version
# BASE   — the common ancestor
# MERGED — the result you're creating
```

<Callout type="tip" title="Clean Up .orig Files">
`git mergetool` creates `.orig` backup files. To prevent them:
```bash
git config --global mergetool.keepBackup false
```
Or clean them up after resolving:
```bash
find . -name "*.orig" -delete
```
</Callout>

### Using git checkout for Specific Sides

Sometimes you know you want one entire side of the conflict:

```bash
# Accept all of their changes for a specific file
git checkout --theirs src/payment.js
git add src/payment.js

# Accept all of your changes for a specific file
git checkout --ours src/config.js
git add src/config.js
```

## Aborting a Merge

If you're overwhelmed or need to rethink, abort the merge entirely:

```bash
# Cancel the merge and go back to the state before it started
git merge --abort

# Everything is back to normal — no changes, no conflicts
git status
# On branch main
# nothing to commit, working tree clean
```

<Callout type="warning" title="Abort Loses Resolution Progress">
`git merge --abort` discards ALL conflict resolution work you've done. If you've resolved 9 out of 10 files, aborting means you'll redo all of them. Consider carefully before aborting a partially-resolved merge.
</Callout>

## Preventing Conflicts

The best conflict is the one that never happens:

### 1. Merge Main into Your Branch Frequently

```bash
# Every day (or more often), pull main into your feature branch
git switch feature/payment
git merge main
# Small conflicts now instead of massive conflicts later
```

### 2. Keep Branches Short-Lived

```bash
# Bad: 3-month feature branch diverging from main
# Good: small branches that merge within days

# Break big features into smaller pieces
git switch -c feature/payment-step-1   # Add payment model
git switch -c feature/payment-step-2   # Add payment API
git switch -c feature/payment-step-3   # Add payment UI
```

### 3. Communicate with Your Team

```bash
# If you know someone is editing the same file:
# - Talk about it
# - Divide the file into sections
# - One person merges first, the other rebases
```

### 4. Pull Before You Push

```bash
# Always pull the latest before pushing
git pull origin main
# If there are conflicts, resolve them locally
# Then push your clean merge
git push origin main
```

## Reuse Recorded Resolution (rerere)

Git can remember how you resolved a conflict and automatically apply the same resolution next time:

```bash
# Enable rerere globally
git config --global rerere.enabled true

# Now when you resolve a conflict, Git records the resolution
# Next time the same conflict appears, Git auto-resolves it

# See recorded resolutions
git rerere status

# Forget a specific resolution
git rerere forget src/payment.js
```

<Callout type="tip" title="rerere Saves Time with Rebase">
If you regularly rebase your feature branch onto main, you'll often see the same conflicts repeatedly. With `rerere` enabled, you resolve the conflict once and Git remembers the resolution for all future occurrences. This is especially valuable for long-running branches.
</Callout>

## Special Conflict Cases

### Binary File Conflicts

Git can't merge binary files (images, PDFs, compiled files):

```bash
# Conflict on a binary file
git merge feature/new-logo
# warning: Cannot merge binary files: logo.png

# You must choose one version or the other
git checkout --theirs logo.png    # use their version
# OR
git checkout --ours logo.png      # use our version

git add logo.png
git commit
```

### Deleted vs Modified Conflicts

```bash
# One branch deletes a file, the other modifies it
git merge feature/refactor
# CONFLICT (modify/delete): old-utils.js deleted in feature/refactor
# and modified in HEAD.

# To keep the file (with your modifications):
git add old-utils.js

# To accept the deletion:
git rm old-utils.js

git commit
```

### Rename Conflicts

```bash
# Both branches rename the same file differently
git merge feature/reorganize
# CONFLICT (rename/rename): src/utils.js renamed to
# src/helpers.js in HEAD and src/lib/utils.js in feature/reorganize

# Resolve by choosing the correct path and content
git add src/helpers.js     # or whichever path you want
git rm src/lib/utils.js    # remove the other
git commit
```

    constructor(host, port, poolSize = 10) {
        this.host = host;
        this.port = port;
        this.poolSize = poolSize;
        this.connections = [];
    }

    connect() {
        for (let i = 0; i < this.poolSize; i++) {

        return this.connections.find(c => c.active);

        return `[conn-${conn.id}] Executing: ${sql}`;
    }
}

module.exports = Database;

## Key Takeaways

- A merge conflict is Git asking a human to decide between two valid changes to the same lines
- Conflict markers: `<<<<<<<` (yours), `=======` (divider), `>>>>>>>` (theirs)
- Resolution process: edit file, remove markers, `git add`, `git commit`
- Use `git merge --abort` to cancel a merge entirely and start over
- `git checkout --ours` and `--theirs` accept one entire side of a conflict
- Enable `diff3` conflict style for better context showing the common ancestor
- `git mergetool` opens a visual tool for complex multi-file conflicts
- Enable `rerere` to have Git remember and replay your conflict resolutions
- Prevent conflicts by merging main into your branch frequently, keeping branches short-lived, and communicating with your team
- Special cases: binary files (choose one side), deleted vs modified (keep or remove), renamed files (choose the path)
