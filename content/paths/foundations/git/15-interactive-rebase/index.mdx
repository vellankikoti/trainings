## Rebase Basics

Before diving into interactive rebase, let's understand what regular rebase does. Rebase replays your commits onto a new base — it moves where your branch diverged from.

```
Before rebase (feature branched from C2, main moved to C4):

  main ──── C1 ── C2 ── C3 ── C4
                    │
                    └── F1 ── F2 ── F3 (feature)

After git rebase main (on feature branch):

  main ──── C1 ── C2 ── C3 ── C4
                                 │
                                 └── F1' ── F2' ── F3' (feature)

  F1', F2', F3' are NEW commits with the same changes but new hashes.
  The feature branch now starts from C4 instead of C2.
```

```bash
# Basic rebase: replay feature commits onto main
git switch feature
git rebase main

# This is equivalent to:
# 1. Find where feature diverged from main (C2)
# 2. Save all feature commits (F1, F2, F3)
# 3. Reset feature to the tip of main (C4)
# 4. Replay F1, F2, F3 one by one on top of C4
```

## Rebase vs Merge

Both combine work from two branches, but they create different histories:

```
                    MERGE                           REBASE

  C1─C2─C3─C4─────M (merge commit)       C1─C2─C3─C4─F1'─F2'─F3'
       │           │
       └─F1─F2─F3─┘                      (linear history, no merge commit)

  (preserves parallel development)        (looks like sequential development)
```

| Aspect | Merge | Rebase |
|--------|-------|--------|
| History | Non-linear (shows branching) | Linear (one straight line) |
| Merge commit | Creates one | No merge commit |
| Original commits | Preserved (same hashes) | Recreated (new hashes) |
| When branches diverged | Visible in graph | Hidden (looks sequential) |
| Conflicts | Resolve once in merge commit | Resolve per replayed commit |
| Safe for shared branches | Yes | No (rewrites history) |

```bash
# Merge approach
git switch main
git merge feature
# Creates: main──C3──C4──M  with feature's commits accessible via M's second parent

# Rebase approach
git switch feature
git rebase main
git switch main
git merge feature    # Fast-forward merge (no merge commit needed)
# Creates: main──C3──C4──F1'──F2'──F3'  (clean linear history)
```

<Callout type="tip" title="When to Rebase vs Merge">
**Rebase** when: cleaning up your own feature branch before creating a PR, syncing with the latest main, making history linear and easy to read.

**Merge** when: combining a feature branch into main (especially with squash merge), preserving the record that parallel development happened, working on shared branches.

The most common workflow: rebase your feature branch onto main (to get the latest changes), then merge the feature into main via a PR (possibly with squash).
</Callout>

## Interactive Rebase

Interactive rebase (`git rebase -i`) opens an editor showing your commits and lets you choose what to do with each one. This is where the real power lies.

```bash
# Rebase the last 5 commits interactively
git rebase -i HEAD~5

# Rebase all commits since branching from main
git rebase -i main
```

When you run `git rebase -i HEAD~5`, Git opens your editor with something like:

```
pick abc1234 Add user model
pick def5678 WIP: working on API
pick 9ab0123 Fix typo in model
pick cde4567 Actually fix the model
pick fgh7890 Add API endpoint for users

# Rebase abc1234..fgh7890 onto xyz9999 (5 commands)
#
# Commands:
# p, pick   = use commit as-is
# r, reword = use commit, but edit the commit message
# e, edit   = use commit, but stop for amending
# s, squash = use commit, but meld into previous commit
# f, fixup  = like "squash", but discard this commit's message
# d, drop   = remove commit entirely
#
# These lines can be re-ordered; they are executed from top to bottom.
# If you remove a line here, THAT COMMIT WILL BE LOST.
```

### The Commands

#### pick — Keep as-is

```
pick abc1234 Add user model
# Uses the commit exactly as it is. No changes.
```

#### reword — Change the message

```
reword abc1234 Add user model
# Keeps the code changes but opens the editor to change the commit message.
# Useful for fixing typos or making messages more descriptive.
```

#### edit — Stop and modify

```
edit abc1234 Add user model
# Stops the rebase at this commit so you can:
#   - Amend the commit (change code + message)
#   - Split it into multiple commits
#   - Add files you forgot
# After editing: git rebase --continue
```

#### squash — Combine with previous

```
pick abc1234 Add user model
squash def5678 WIP: working on API
# Combines def5678 into abc1234.
# Opens editor to combine the commit messages.
```

#### fixup — Combine silently

```
pick abc1234 Add user model
fixup def5678 Fix typo in model
# Combines def5678 into abc1234.
# Discards def5678's commit message (keeps abc1234's).
# Use when the "fix" commit was just correcting a mistake.
```

#### drop — Remove entirely

```
drop def5678 WIP: debug logging left in
# Completely removes this commit and its changes.
# The code changes are LOST.
```

## Practical Workflows

### Squashing WIP Commits

The most common use of interactive rebase. Clean up messy development history:

```bash
# Your branch has these commits:
git log --oneline -7
# fgh7890 Add tests for user API
# cde4567 Please work
# 9ab0123 Actually fix the model
# def5678 Fix typo in model
# abc1234 WIP: working on API
# 111aaa1 Add user model
# 000bbb0 (main) Previous commit

# Clean this up:
git rebase -i HEAD~6
```

In the editor, reorganize and squash:

```
pick 111aaa1 Add user model
fixup def5678 Fix typo in model
fixup 9ab0123 Actually fix the model

pick abc1234 WIP: working on API
fixup cde4567 Please work

pick fgh7890 Add tests for user API
```

After saving:

```bash
git log --oneline -4
# aaa9999 Add tests for user API
# bbb8888 Add user API endpoint
# ccc7777 Add user model
# 000bbb0 (main) Previous commit
```

You went from 6 messy commits to 3 clean ones.

### Reordering Commits

Sometimes commits are in the wrong logical order:

```
# Current order:
pick abc1234 Add API endpoint
pick def5678 Add database migration
pick 9ab0123 Add model definition

# Better order (model → migration → API):
pick 9ab0123 Add model definition
pick def5678 Add database migration
pick abc1234 Add API endpoint
```

<Callout type="warning" title="Reordering Can Cause Conflicts">
When you reorder commits, earlier commits might depend on later ones. If commit 3 modifies a file created in commit 1, moving commit 3 before commit 1 will cause a conflict. Git will stop and ask you to resolve it.
</Callout>

### Splitting a Large Commit

Use `edit` to split one commit into multiple smaller ones:

```bash
git rebase -i HEAD~3
# Mark the large commit as "edit":
# edit abc1234 Add user model, API, and tests
# pick def5678 Other commit
```

When the rebase stops at that commit:

```bash
# Undo the commit but keep the changes staged
git reset HEAD~1

# Now selectively commit pieces
git add src/models/user.js
git commit -m "Add user model"

git add src/api/users.js
git commit -m "Add user API endpoint"

git add tests/user.test.js
git commit -m "Add user model and API tests"

# Continue the rebase
git rebase --continue
```

### Editing a Commit's Content

Use `edit` to modify the code in a previous commit:

```bash
git rebase -i HEAD~4
# Mark the target commit as "edit":
# edit abc1234 Add configuration loader
```

When the rebase stops:

```bash
# Make your changes
vim src/config.js

# Amend the commit with the new changes
git add src/config.js
git commit --amend --no-edit

# Continue the rebase
git rebase --continue
```

## Rebase Onto

`git rebase --onto` lets you move a range of commits to a completely different base:

```bash
# Move commits from one branch base to another
git rebase --onto new-base old-base feature

# Example: Move a feature branch from develop to main
#
# Before:
#   main ──── C1 ── C2
#                     │
#   develop ──────── D1 ── D2
#                           │
#   feature ────────────── F1 ── F2
#
# Command:
git rebase --onto main develop feature
#
# After:
#   main ──── C1 ── C2
#                     │
#                     └── F1' ── F2' (feature, now based on main)
#
#   develop ──────── D1 ── D2 (unchanged)
```

This is useful when you accidentally branched from the wrong base, or when the intermediate branch is no longer needed.

## Autosquash: The Clean Way

Git supports special commit message prefixes that work with `--autosquash`:

```bash
# Make your main commit
git commit -m "Add user authentication"

# Later, fix a bug in that commit
git commit -m "fixup! Add user authentication"
# Note: the message after "fixup!" must match the original EXACTLY

# Later, want to add to that commit
git commit -m "squash! Add user authentication"
# This will combine and let you edit the message

# Now rebase with --autosquash
git rebase -i --autosquash main
# Git automatically reorders and marks fixup!/squash! commits
```

```
Without --autosquash (you'd manually arrange):
  pick abc1234 Add user authentication
  pick def5678 Add profile page
  pick 9ab0123 fixup! Add user authentication
  pick cde4567 squash! Add user authentication

With --autosquash (Git arranges for you):
  pick abc1234 Add user authentication
  fixup 9ab0123 fixup! Add user authentication
  squash cde4567 squash! Add user authentication
  pick def5678 Add profile page
```

```bash
# Enable autosquash by default
git config --global rebase.autoSquash true

# Now every git rebase -i will automatically arrange fixup!/squash! commits
```

<Callout type="tip" title="The fixup! Workflow">
This is one of the most efficient development workflows:
1. Make your commits normally
2. When you find a mistake in a previous commit, use `git commit --fixup=<hash>`
3. Before creating your PR, run `git rebase -i --autosquash main`
4. The fixup commits are automatically absorbed into the original commits
</Callout>

## The Golden Rule of Rebase

<Callout type="warning" title="NEVER Rebase Shared/Public Branches">
**Never rebase commits that have been pushed to a shared branch that others are working on.**

Rebase creates NEW commits with different hashes. If you rebase commits that others have already based their work on, their commits will reference commit hashes that no longer exist in the history. This causes massive confusion, duplicate commits, and merge conflicts for everyone on the team.

```
Safe to rebase:
  ✓ Your local feature branch (before pushing)
  ✓ Your feature branch that only you use (with force-push)
  ✓ Commits that haven't been pushed yet

Never rebase:
  ✗ main or develop (shared branches)
  ✗ Any branch that other people have checked out
  ✗ Commits that are part of merged PRs
```

If you are unsure whether someone else is using a branch, do not rebase it. Use merge instead.
</Callout>

## Handling Rebase Conflicts

When replaying commits, Git may encounter conflicts at each step:

```bash
git rebase main
# CONFLICT (content): Merge conflict in src/app.js
# error: could not apply abc1234... Add feature
#
# Resolve all conflicts manually, mark them as resolved with
# "git add/rm <conflicted_files>", then run "git rebase --continue".
# You can instead skip this commit: "git rebase --skip".
# To abort the rebase and get back to where you started: "git rebase --abort".
```

### Resolving Conflicts During Rebase

```bash
# Step 1: See which files conflict
git status
# Unmerged paths:
#   both modified: src/app.js

# Step 2: Open the file and resolve conflicts
# Remove the conflict markers (<<<, ===, >>>) and keep the right code
vim src/app.js

# Step 3: Mark as resolved
git add src/app.js

# Step 4: Continue the rebase
git rebase --continue
# Git applies the resolved commit and moves to the next one
# If more conflicts arise, repeat steps 1-4
```

### Rebase Controls

```bash
# Continue after resolving conflicts
git rebase --continue

# Skip the current commit entirely
git rebase --skip
# WARNING: This discards the current commit's changes

# Abort the rebase and return to the original state
git rebase --abort
# This is your safety net — as if the rebase never happened
```

### Recovering from a Bad Rebase

If a rebase goes wrong and you have already continued past the point of `--abort`:

```bash
# The reflog records the state before the rebase
git reflog
# abc1234 HEAD@{0}: rebase (continue): Add feature
# def5678 HEAD@{1}: rebase (continue): Fix model
# 9ab0123 HEAD@{2}: rebase (start): checkout main
# cde4567 HEAD@{3}: commit: Your last commit before rebase ← THIS ONE

# Reset to the state before the rebase started
git reset --hard HEAD@{3}
# Your branch is back to exactly where it was before the rebase
```

<Callout type="tip" title="Create a Backup Branch Before Rebasing">
If you are nervous about a rebase, create a backup branch first:
```bash
git branch backup-before-rebase
git rebase -i main
# If everything goes wrong:
git reset --hard backup-before-rebase
git branch -D backup-before-rebase
```
This gives you a simple, visible recovery point.
</Callout>

## Real-World Rebase Patterns

### Pattern 1: Clean Up Before PR

```bash
# You've been working for a week — 15 commits, messy history
git log --oneline main..feature/auth
# f15 Add final test
# f14 Fix CI
# f13 Oops, forgot file
# f12 WIP
# f11 Actually fix the bug
# f10 Fix bug
# f09 Refactor tests
# f08 Add more tests
# f07 Debug
# f06 Fix lint
# f05 Add middleware
# f04 WIP auth
# f03 Fix typo
# f02 Add auth routes
# f01 Add auth model

# Clean up into logical commits
git rebase -i main

# Result: 4 clean commits
# f04' Add authentication model and routes
# f03' Add auth middleware with JWT validation
# f02' Add comprehensive auth test suite
# f01' Configure CI for auth module
```

### Pattern 2: Sync with Main via Rebase

```bash
# Your feature branch is behind main
git switch feature/payments
git fetch origin

# Rebase onto the latest main
git rebase origin/main

# Resolve any conflicts along the way
# Your commits now sit on top of the latest main

# Push (force-with-lease because history changed)
git push --force-with-lease origin feature/payments
```

### Pattern 3: Autosquash Workflow

```bash
# Regular development — commit normally
git commit -m "Add payment processing"
# ... work continues ...
git commit -m "Add payment validation"
# ... find a bug in the first commit ...
git commit --fixup=HEAD~1   # fixup! Add payment processing
# ... add forgotten test ...
git commit --fixup=HEAD~2   # fixup! Add payment validation

# Before creating PR, clean up with autosquash
git rebase -i --autosquash main
# Git automatically absorbs the fixup commits
```

set -euo pipefail

# Commit Cleanup Tool — analyzes branch commits and suggests cleanup opportunities

GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
CYAN='\033[0;36m'

# Check if base branch exists

    echo "Usage: $0 [base-branch]"
    exit 1

COMMITS=$(git log --oneline "$BASE_BRANCH..HEAD" 2>/dev/null)

if [ -z "$COMMITS" ]; then

    echo "Make sure you're on a feature branch."
    exit 0

echo -e "${BOLD}${BLUE}  Base: ${BASE_BRANCH}${NC}"
echo -e "${BOLD}${BLUE}  Commits: ${COMMIT_COUNT}${NC}"

fi

    echo "$COMMITS" | grep -i -n "^[a-f0-9]* update\b\|^[a-f0-9]* fix\b\|^[a-f0-9]* change\|^[a-f0-9]* stuff\|^[a-f0-9]* misc\|^[a-f0-9]* minor" | while read -r line; do

## Key Takeaways

- Rebase replays commits onto a new base, creating a linear history without merge commits
- Interactive rebase (`git rebase -i`) lets you pick, reword, edit, squash, fixup, and drop commits
- **pick** keeps a commit as-is; **reword** changes the message; **squash/fixup** combines with the previous commit; **drop** removes it
- Use interactive rebase to clean up messy development history before creating a PR
- The `fixup!` prefix and `--autosquash` flag automate the squash workflow
- `git rebase --onto` moves commits to a completely different base branch
- **The golden rule: never rebase commits that have been pushed to a shared branch** — rebase rewrites history (new hashes), which breaks other people's work
- When rebase conflicts occur, resolve them per-commit with `git rebase --continue`, or bail out with `git rebase --abort`
- Use `git reflog` to recover from a bad rebase — find the pre-rebase state and `git reset --hard` to it
- Create a backup branch before complex rebases for an easy escape hatch
- The typical professional workflow: develop with messy commits, clean up with interactive rebase, then create a polished PR
