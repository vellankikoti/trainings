<Callout type="story" title="The Story">
You just force-pushed to main and wiped out your colleague's work. Or you committed a 500MB video file. Or you made 10 commits that should have been 1. Or you deleted a branch with important work. Every one of these mistakes is recoverable in Git — if you know the right tool. Git almost never deletes data permanently. Commits you think are "gone" are still in the object database. Branches you deleted still have their commits in the reflog. The reflog is your time machine — it records every movement of HEAD for at least 90 days. Once you understand Git's undo tools, you'll experiment fearlessly.
</Callout>

## The Undo Decision Flowchart

Before diving into each tool, here's how to choose the right one:

```
What do you want to undo?
│
├── Changes in working directory (not staged, not committed)?
│   └── git restore <file>
│
├── Staged changes (added but not committed)?
│   └── git restore --staged <file>
│
├── A commit on a shared branch (others have pulled it)?
│   └── git revert <commit>
│
├── Recent commits on YOUR branch (not pushed)?
│   ├── Keep changes staged?     → git reset --soft HEAD~N
│   ├── Keep changes unstaged?   → git reset --mixed HEAD~N
│   └── Discard everything?      → git reset --hard HEAD~N
│
├── Untracked files (new files you don't want)?
│   └── git clean -fd
│
└── Something you thought was lost forever?
    └── git reflog + git cherry-pick or git reset
```

## Discarding Working Directory Changes

When you've edited a file and want to throw away your changes, returning it to the last committed state:

### git restore (Modern — Git 2.23+)

```bash
# Discard changes to a specific file
git restore src/app.js

# Discard changes to multiple files
git restore src/app.js src/config.js

# Discard ALL working directory changes
git restore .

# Restore a file to a specific commit's version
git restore --source=HEAD~3 src/app.js
# The file now matches what it looked like 3 commits ago
# but this is a working directory change — not committed yet
```

### git checkout -- (Legacy)

```bash
# Same as git restore, but older syntax
git checkout -- src/app.js

# Discard all changes
git checkout -- .
```

<Callout type="warning" title="These Changes Are Gone Forever">
Unlike almost everything else in Git, discarding working directory changes is permanent. There's no reflog, no object, no recovery. Git never tracked these changes in the first place. If you've spent hours editing a file and run `git restore` on it, that work is gone. When in doubt, stash instead of restore: `git stash` saves your changes and you can always get them back with `git stash pop`.
</Callout>

## Unstaging Files

When you've used `git add` but haven't committed yet, and you want to remove files from the staging area (while keeping the changes in your working directory):

### git restore --staged (Modern)

```bash
# Unstage a specific file
git restore --staged src/app.js

# Unstage everything
git restore --staged .

# The changes are still in your working directory — just not staged
git status
# Changes not staged for commit:
#   modified: src/app.js
```

### git reset HEAD (Legacy)

```bash
# Same effect, older syntax
git reset HEAD src/app.js

# Unstage everything
git reset HEAD
```

### Partial Unstaging

```bash
# Unstage specific hunks (interactive)
git restore --staged --patch src/app.js
# Git shows you each chunk and asks y/n

# Or with the legacy command
git reset --patch HEAD src/app.js
```

<Callout type="tip" title="Unstage vs Discard">
Unstaging (`git restore --staged`) is safe — your changes remain in the working directory. Discarding (`git restore`) is destructive — changes are lost. A common pattern: unstage first, review the changes, then decide whether to re-stage or discard.
</Callout>

## git reset — Moving the Branch Pointer

`git reset` moves the current branch pointer to a different commit. The three modes control what happens to the changes:

### --soft: Keep Changes Staged

```bash
# Move branch pointer back 1 commit, keep changes staged
git reset --soft HEAD~1
```

```
Before:
  main → C3 → C2 → C1
  Working: clean
  Staging: clean

After git reset --soft HEAD~1:
  main → C2 → C1
  Working: C3's changes present
  Staging: C3's changes staged (ready to re-commit)
```

Use case: You want to redo the last commit message, or combine the last few commits:

```bash
# Redo the last commit with a better message
git reset --soft HEAD~1
git commit -m "Better commit message"

# Squash the last 3 commits into 1
git reset --soft HEAD~3
git commit -m "Feature: user authentication (squashed)"
```

### --mixed: Keep Changes Unstaged (Default)

```bash
# Move branch pointer back 1 commit, unstage changes
git reset HEAD~1
# same as: git reset --mixed HEAD~1
```

```
Before:
  main → C3 → C2 → C1
  Working: clean
  Staging: clean

After git reset --mixed HEAD~1:
  main → C2 → C1
  Working: C3's changes present
  Staging: empty (changes are unstaged)
```

Use case: You committed too many files and want to selectively re-add:

```bash
# Undo the commit, then selectively re-stage
git reset HEAD~1
git add src/feature.js src/feature.test.js
git commit -m "Add feature with tests"
# The other files are still changed but not committed
```

### --hard: Discard Everything

```bash
# Move branch pointer back 1 commit, discard ALL changes
git reset --hard HEAD~1
```

```
Before:
  main → C3 → C2 → C1
  Working: clean
  Staging: clean

After git reset --hard HEAD~1:
  main → C2 → C1
  Working: clean (C3's changes GONE)
  Staging: clean (C3's changes GONE)
```

<Callout type="warning" title="--hard Is Dangerous (But Recoverable)">
`git reset --hard` discards committed changes and any uncommitted work. However, the "lost" commit still exists in Git's object database for at least 90 days. You can find it with `git reflog` and recover it. But any uncommitted working directory changes ARE permanently gone. Always check `git status` and `git stash` before using `--hard`.
</Callout>

### Reset to a Specific Commit

```bash
# Reset to a specific commit hash
git reset --soft abc1234

# Reset to where a branch points
git reset --mixed origin/main

# Reset to a tag
git reset --hard v2.0.0

# Reset using relative references
git reset --soft HEAD~5     # 5 commits back
git reset --mixed HEAD^     # 1 commit back (^ = ~1)
```

## git revert — Safe Undo for Shared History

`git revert` creates a NEW commit that undoes a previous commit. The original commit stays in history. This is the safe way to undo changes on shared branches:

```bash
# Revert the most recent commit
git revert HEAD
# Opens editor for the revert commit message
# Default: "Revert 'Original commit message'"

# Revert a specific commit
git revert abc1234

# Revert without opening editor
git revert --no-edit HEAD

# Revert multiple commits (one revert commit each)
git revert HEAD~2..HEAD
```

```
Before:
  main → C3 → C2 → C1

After git revert C3:
  main → R3 → C3 → C2 → C1
  (R3 undoes C3's changes, but C3 is still in history)
```

### Revert vs Reset

| Aspect | git reset | git revert |
|---|---|---|
| Modifies history | Yes (moves branch pointer) | No (adds new commit) |
| Safe for shared branches | No | Yes |
| Creates new commits | No | Yes |
| Changes are recoverable | Via reflog | Original commit preserved |
| Use when | Branch is local/unpushed | Branch is shared/pushed |

```bash
# On your local branch (not pushed) — reset is fine
git reset --soft HEAD~1

# On a shared branch (already pushed) — use revert
git revert HEAD
git push origin main
```

### Reverting a Merge Commit

Merge commits have two parents, so you must tell Git which parent to keep:

```bash
# Revert a merge commit, keeping the first parent (usually main)
git revert -m 1 <merge-commit-hash>

# -m 1 means "undo changes from the second parent (feature branch)"
# -m 2 would mean "undo changes from the first parent (main)"
```

<Callout type="warning" title="Reverting Merges Has Side Effects">
If you revert a merge commit, Git remembers that the branch was "un-merged." If you try to merge the same branch again later, Git thinks the changes are already applied (and reverted). You'll need to "revert the revert" before re-merging. This is a common source of confusion.
</Callout>

## git clean — Removing Untracked Files

`git clean` removes files that Git isn't tracking. Useful for cleaning up build artifacts, generated files, or experiments:

```bash
# Preview what would be deleted (dry run — ALWAYS do this first)
git clean -n
# Would remove build/output.js
# Would remove temp.log
# Would remove experiment.py

# Remove untracked files
git clean -f

# Remove untracked files AND directories
git clean -fd

# Remove untracked AND ignored files (nuclear option)
git clean -fdx
# WARNING: This removes node_modules/, .env, build/, etc.

# Interactive mode — choose file by file
git clean -i
```

<Callout type="warning" title="git clean Is Permanent">
Like `git restore` for working directory changes, `git clean` permanently deletes files that Git never tracked. There is no undo. Always run `git clean -n` (dry run) first to see what would be deleted.
</Callout>

## git reflog — Your Safety Net

The reflog records every time HEAD moves. Every commit, reset, checkout, merge, rebase, cherry-pick — it's all logged. This is how you recover from almost any mistake.

```bash
# View the reflog
git reflog
# abc1234 HEAD@{0}: commit: Add payment feature
# def5678 HEAD@{1}: reset: moving to HEAD~1
# 9ab0123 HEAD@{2}: commit: Accidentally committed huge file
# cde4567 HEAD@{3}: checkout: moving from feature to main
# fgh7890 HEAD@{4}: commit: Fix login bug
```

### Recovering from Hard Reset

```bash
# Oops! You just did this:
git reset --hard HEAD~3
# Three commits of work are "gone"

# But the reflog knows:
git reflog
# abc1234 HEAD@{0}: reset: moving to HEAD~3
# def5678 HEAD@{1}: commit: Third commit (the one you want back)
# 9ab0123 HEAD@{2}: commit: Second commit
# cde4567 HEAD@{3}: commit: First commit

# Recover by resetting to the reflog entry
git reset --hard HEAD@{1}
# All three commits are back!

# Or more precisely:
git reset --hard def5678
```

### Recovering a Deleted Branch

```bash
# Oops! You deleted a branch with important work
git branch -D feature/important-work
# Deleted branch feature/important-work (was abc1234).

# The commits are still in the reflog
git reflog
# ... look for the last commit on that branch ...

# Recreate the branch pointing to that commit
git branch feature/important-work abc1234
# Branch is back with all its commits!
```

### Recovering from a Bad Rebase

```bash
# You rebased and it went wrong
git rebase main
# ... conflicts, mistakes, confusion ...

# Find the state before the rebase
git reflog
# ... HEAD@{5}: rebase (start): checkout main
# abc1234 HEAD@{6}: commit: Your last good commit before rebase

# Reset to before the rebase
git reset --hard HEAD@{6}
# Back to normal!
```

<Callout type="tip" title="Reflog Retention">
By default, reflog entries are kept for 90 days (for reachable commits) and 30 days (for unreachable commits). You can configure this:
```bash
git config --global gc.reflogExpire 180.days
git config --global gc.reflogExpireUnreachable 90.days
```
After the retention period, `git gc` removes the entries and the orphaned commits may be truly deleted.
</Callout>

## git cherry-pick — Rescue Specific Commits

Cherry-pick copies a specific commit onto your current branch. Useful for rescuing work from deleted or botched branches:

```bash
# Copy a specific commit to your current branch
git cherry-pick abc1234

# Cherry-pick multiple commits
git cherry-pick abc1234 def5678

# Cherry-pick without committing (just stage the changes)
git cherry-pick --no-commit abc1234

# Cherry-pick a range of commits
git cherry-pick abc1234..def5678
```

### Rescue Workflow

```bash
# You deleted a branch but need two specific commits from it
git reflog | grep "important-work"
# Find the commit hashes

git switch main
git cherry-pick abc1234    # rescue first commit
git cherry-pick def5678    # rescue second commit
```

## Choosing the Right Undo Tool

Here's a practical summary for real-world scenarios:

```bash
# "I edited a file and want to throw away the changes"
git restore src/app.js

# "I staged a file by accident"
git restore --staged src/app.js

# "I committed but haven't pushed — want to redo the commit"
git reset --soft HEAD~1
# (edit, re-stage, recommit)

# "I committed but haven't pushed — want to completely undo"
git reset --hard HEAD~1

# "I pushed a bad commit — need to undo it publicly"
git revert HEAD
git push

# "I have random untracked files everywhere"
git clean -n    # preview first!
git clean -fd   # then remove

# "I did something terrible and need to go back in time"
git reflog
git reset --hard HEAD@{N}

# "I need one specific commit from another branch"
git cherry-pick <commit-hash>
```

<MiniProject title="Git Safety Net">

```bash
cat > ~/git-safety-net.sh << 'SCRIPT'
#!/bin/bash
set -euo pipefail

# Git Safety Net — friendly interface for common undo operations with safety checks

# Colors for output
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
NC='\033[0m'

# Check we're in a git repo
if ! git rev-parse --git-dir &>/dev/null; then
    echo -e "${RED}Error: Not in a git repository${NC}"
    exit 1
fi

CURRENT_BRANCH=$(git branch --show-current 2>/dev/null || echo "DETACHED")

show_menu() {
    echo ""
    echo -e "${BLUE}============================================${NC}"
    echo -e "${BLUE}  Git Safety Net${NC}"
    echo -e "${BLUE}  Branch: ${CURRENT_BRANCH}${NC}"
    echo -e "${BLUE}============================================${NC}"
    echo ""
    echo "  1) Discard changes to a file (git restore)"
    echo "  2) Unstage a file (git restore --staged)"
    echo "  3) Undo last commit, keep changes (git reset --soft)"
    echo "  4) Undo last commit, unstage changes (git reset --mixed)"
    echo "  5) Undo last commit, discard all (git reset --hard)"
    echo "  6) Revert a commit safely (git revert)"
    echo "  7) Clean untracked files (git clean)"
    echo "  8) View reflog (recovery history)"
    echo "  9) Recover from reflog entry"
    echo "  10) Show current status"
    echo "  q) Quit"
    echo ""
}

confirm() {
    local message=$1
    echo -e "${YELLOW}$message${NC}"
    read -r -p "Are you sure? (yes/no): " response
    [ "$response" = "yes" ]
}

action_discard_changes() {
    echo ""
    echo -e "${YELLOW}--- Modified files (working directory) ---${NC}"
    local modified
    modified=$(git diff --name-only 2>/dev/null)
    if [ -z "$modified" ]; then
        echo "No modified files in working directory."
        return
    fi
    echo "$modified"
    echo ""
    read -r -p "Enter filename to discard (or 'all'): " target
    if [ "$target" = "all" ]; then
        if confirm "This will PERMANENTLY discard ALL working directory changes."; then
            git restore .
            echo -e "${GREEN}All changes discarded.${NC}"
        fi
    elif [ -n "$target" ]; then
        if confirm "This will PERMANENTLY discard changes to '$target'."; then
            git restore "$target"
            echo -e "${GREEN}Changes to '$target' discarded.${NC}"
        fi
    fi
}

action_unstage() {
    echo ""
    echo -e "${YELLOW}--- Staged files ---${NC}"
    local staged
    staged=$(git diff --cached --name-only 2>/dev/null)
    if [ -z "$staged" ]; then
        echo "No staged files."
        return
    fi
    echo "$staged"
    echo ""
    read -r -p "Enter filename to unstage (or 'all'): " target
    if [ "$target" = "all" ]; then
        git restore --staged .
        echo -e "${GREEN}All files unstaged. Changes preserved in working directory.${NC}"
    elif [ -n "$target" ]; then
        git restore --staged "$target"
        echo -e "${GREEN}'$target' unstaged. Changes preserved in working directory.${NC}"
    fi
}

action_reset_soft() {
    echo ""
    echo -e "${YELLOW}--- Last 5 commits ---${NC}"
    git log --oneline -5
    echo ""
    read -r -p "How many commits to undo? (default: 1): " count
    count=${count:-1}
    echo ""
    echo "This will undo $count commit(s) but KEEP all changes staged."
    echo "You can immediately recommit with a new message."
    if confirm "Proceed with soft reset?"; then
        git reset --soft "HEAD~${count}"
        echo -e "${GREEN}Reset complete. Changes are staged and ready to recommit.${NC}"
        echo ""
        git status --short
    fi
}

action_reset_mixed() {
    echo ""
    echo -e "${YELLOW}--- Last 5 commits ---${NC}"
    git log --oneline -5
    echo ""
    read -r -p "How many commits to undo? (default: 1): " count
    count=${count:-1}
    echo ""
    echo "This will undo $count commit(s) and UNSTAGE the changes."
    echo "Changes will remain in your working directory."
    if confirm "Proceed with mixed reset?"; then
        git reset "HEAD~${count}"
        echo -e "${GREEN}Reset complete. Changes are in working directory (unstaged).${NC}"
        echo ""
        git status --short
    fi
}

action_reset_hard() {
    echo ""
    echo -e "${RED}!!! WARNING: DESTRUCTIVE OPERATION !!!${NC}"
    echo ""
    echo -e "${YELLOW}--- Last 5 commits ---${NC}"
    git log --oneline -5
    echo ""

    # Safety check: warn about uncommitted changes
    if [ -n "$(git status --porcelain)" ]; then
        echo -e "${RED}You have uncommitted changes that will be PERMANENTLY lost:${NC}"
        git status --short
        echo ""
    fi

    read -r -p "How many commits to undo? (default: 1): " count
    count=${count:-1}

    echo ""
    echo -e "${RED}This will undo $count commit(s) and DISCARD all changes.${NC}"
    echo -e "${RED}Working directory and staging area will be clean.${NC}"
    echo -e "${YELLOW}Recovery IS possible via 'git reflog' for committed changes.${NC}"
    echo -e "${RED}Uncommitted changes will be PERMANENTLY lost.${NC}"

    if confirm "Type 'yes' to proceed with HARD reset:"; then
        # Record reflog message for easy recovery
        BEFORE=$(git rev-parse HEAD)
        git reset --hard "HEAD~${count}"
        echo -e "${GREEN}Hard reset complete.${NC}"
        echo -e "${YELLOW}Recovery point: ${BEFORE}${NC}"
        echo -e "${YELLOW}To undo: git reset --hard ${BEFORE}${NC}"
    fi
}

action_revert() {
    echo ""
    echo -e "${YELLOW}--- Last 10 commits ---${NC}"
    git log --oneline -10
    echo ""
    read -r -p "Enter commit hash to revert: " hash
    if [ -n "$hash" ]; then
        echo ""
        echo "This will create a NEW commit that undoes the changes in $hash."
        echo "Original commit will remain in history. Safe for shared branches."
        if confirm "Proceed with revert?"; then
            # Check if it's a merge commit
            PARENT_COUNT=$(git cat-file -p "$hash" | grep -c "^parent" || true)
            if [ "$PARENT_COUNT" -gt 1 ]; then
                echo "This is a merge commit. Reverting with -m 1 (keeping first parent)."
                git revert -m 1 "$hash"
            else
                git revert "$hash"
            fi
            echo -e "${GREEN}Revert complete. New commit created.${NC}"
        fi
    fi
}

action_clean() {
    echo ""
    echo -e "${YELLOW}--- Untracked files (dry run) ---${NC}"
    local preview
    preview=$(git clean -n -d 2>/dev/null)
    if [ -z "$preview" ]; then
        echo "No untracked files to clean."
        return
    fi
    echo "$preview"
    echo ""
    echo -e "${RED}These files will be PERMANENTLY deleted (no recovery possible).${NC}"
    if confirm "Proceed with clean?"; then
        git clean -fd
        echo -e "${GREEN}Untracked files removed.${NC}"
    fi
}

action_reflog() {
    echo ""
    echo -e "${YELLOW}--- Reflog (last 20 entries) ---${NC}"
    git reflog --oneline -20
    echo ""
    echo "Each entry shows a state you can recover to."
    echo "Use option 9 to recover to a specific entry."
}

action_recover() {
    echo ""
    echo -e "${YELLOW}--- Reflog (last 20 entries) ---${NC}"
    git reflog --oneline -20
    echo ""
    read -r -p "Enter reflog ref to recover to (e.g., HEAD@{3} or hash): " ref
    if [ -n "$ref" ]; then
        echo ""
        echo "This will reset your branch to: $ref"
        echo -e "${YELLOW}Commit info:${NC}"
        git log --oneline -1 "$ref" 2>/dev/null || echo "Could not find ref: $ref"

        if confirm "Reset to this state?"; then
            git reset --hard "$ref"
            echo -e "${GREEN}Recovered to ${ref}.${NC}"
            echo ""
            git log --oneline -5
        fi
    fi
}

    echo -e "${YELLOW}--- Repository Status ---${NC}"
    echo ""
    echo "Branch: $CURRENT_BRANCH"
    echo ""
    echo "Working tree:"
    git status --short
    echo ""
    echo "Recent commits:"
```

## Key Takeaways

- `git restore <file>` discards working directory changes (permanent — no recovery)
- `git restore --staged <file>` unstages changes without losing them
- `git reset --soft` moves the branch pointer but keeps changes staged — perfect for rewording commits
- `git reset --mixed` (default) moves the pointer and unstages — for selective re-staging
- `git reset --hard` discards everything — use with caution, but recoverable via reflog
- `git revert` creates a new commit that undoes a previous one — safe for shared branches
- `git clean -fd` removes untracked files and directories (permanent — always dry-run first with `-n`)
- `git reflog` is your safety net — it records every HEAD movement for 90 days
- Deleted branches, hard resets, and bad rebases can all be recovered through the reflog
- `git cherry-pick` can rescue specific commits from any ref, including reflog entries
- When in doubt about shared branches: use `revert`. When working locally: `reset` is faster
