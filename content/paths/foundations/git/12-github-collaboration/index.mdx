<Callout type="story" title="The Story">
You join a team of 10 developers. Everyone's pushing to the same repo. Features are being built, bugs are being reported, releases are being tagged. Without a collaboration workflow, it's chaos. GitHub is not just a place to store code — it is where coordination happens. Issues track work, project boards visualize progress, and branch protection prevents disasters. The difference between a team that ships reliably and one that constantly breaks things is not talent — it is process. GitHub gives you the tools; you just need to know how to use them.
</Callout>

## GitHub vs Git

Git is the version control tool. GitHub is a platform built on top of Git that adds collaboration features.

```
Git (the tool):              GitHub (the platform):
  - Commits                    - Issues
  - Branches                   - Pull Requests
  - Merges                     - Project Boards
  - Tags                       - Actions (CI/CD)
  - Remotes                    - Branch Protection
  - History                    - Teams & Permissions
                               - Releases
                               - Discussions
                               - Code Search
```

You can use Git without GitHub (with GitLab, Bitbucket, or self-hosted servers). But GitHub is the most widely used platform, and understanding its features is essential for modern DevOps.

## GitHub Issues

Issues are the primary way to track work in a GitHub repository — bugs, features, tasks, and questions.

### Creating Issues

```bash
# Create an issue with GitHub CLI
gh issue create --title "Fix login timeout" --body "Users are getting logged out after 5 minutes. Should be 30 minutes."

# Create with labels and assignee
gh issue create \
  --title "Add dark mode support" \
  --body "Users have requested dark mode. See design spec in #42." \
  --label "enhancement" \
  --label "ui" \
  --assignee "@me"

# Create with a milestone
gh issue create \
  --title "Update dependencies" \
  --body "Several packages have security updates available." \
  --label "maintenance" \
  --milestone "v2.0"

# List open issues
gh issue list

# List issues with filters
gh issue list --label "bug" --assignee "@me"

# View a specific issue
gh issue view 42
```

Create `.github/ISSUE_TEMPLATE/` to standardize issue creation:

```yaml
  - type: textarea

      description: Steps to reproduce the behavior
      value: |
        1. Go to '...'
        2. Click on '...'
        3. See error

      required: true
  - type: textarea
    id: expected
    attributes:
      label: Expected Behavior
      description: What you expected to happen
    validations:
      required: true
  - type: dropdown
    id: severity
    attributes:
      label: Severity
      options:
        - Critical
        - High
        - Medium
        - Low
    validations:
      required: true
```

### Linking Commits to Issues

Reference issues in commit messages to create links. Use special keywords to automatically close issues:

```bash
# Reference an issue (creates a link but doesn't close it)
git commit -m "Increase session timeout — see #42"

# Close an issue automatically when the commit reaches the default branch
git commit -m "Fix login timeout — Fixes #42"
git commit -m "Resolve session expiry — Closes #42"
git commit -m "Handle token refresh — Resolves #42"

# Reference multiple issues
git commit -m "Update auth module — Fixes #42, Closes #43"
```

<Callout type="tip" title="Issue References Work Everywhere">
You can reference issues in commit messages, PR descriptions, comments, and even other issues. The format `#42` always creates a clickable link. The keywords `Fixes`, `Closes`, and `Resolves` (case-insensitive) only auto-close when the commit lands on the default branch (usually via a merged PR).
</Callout>

GitHub Projects provide Kanban-style boards for tracking work across issues and PRs.

```bash
# Create a project with GitHub CLI (uses Projects v2)
```

### Automation Rules

Projects can automatically move items based on events:

```
Auto-move rules:
  - When issue is assigned → move to "To Do"
  - When PR is opened → move to "In Progress"
  - When PR is merged → move to "Done"
  - When issue is closed → move to "Done"
```

## Repository Settings

### Branch Protection Rules

Branch protection prevents direct pushes and enforces code quality on important branches:

```bash
# View branch protection (GitHub CLI)
gh api repos/{owner}/{repo}/branches/main/protection

# Common protection rules for "main":
# 1. Require pull request reviews before merging
#    - Required approving reviews: 1 (or 2 for critical repos)
#    - Dismiss stale reviews when new commits are pushed
#    - Require review from code owners
#
# 2. Require status checks to pass before merging
#    - Required checks: CI/CD pipeline, linting, tests
#    - Require branches to be up to date before merging
#
# 3. Require conversation resolution before merging
#
# 4. Restrict who can push to the branch
#    - Only allow maintainers to push directly
#
# 5. Do not allow force pushes
#
# 6. Do not allow deletions
```

```
Without branch protection:
  Developer → git push origin main → Main branch updated
  (No review, no tests, no safety net)

With branch protection:
  Developer → Create PR → Reviews ✓ → CI passes ✓ → Merge
  (Every change is reviewed and tested)
```

<Callout type="warning" title="Set Up Protection Early">
Many teams add branch protection only after someone accidentally pushes broken code to main. Do not wait for a disaster. Set up protection rules when you create the repository. At minimum: require one PR review and require status checks to pass.
</Callout>

### Teams and Permissions

```
Permission Levels:
  Read    — View code, create issues, comment
  Triage  — Manage issues and PRs (labels, assignees)
  Write   — Push to non-protected branches, merge PRs
  Maintain — Manage repo settings (no destructive actions)
  Admin   — Full control (delete repo, manage access)

Team Structure Example:
  @org/developers    → Write access
  @org/leads         → Maintain access
  @org/devops        → Admin access
  @org/contractors   → Read access
```

## GitHub Actions Basics

GitHub Actions is GitHub's built-in CI/CD platform. It runs automated workflows triggered by events.

### Basic Workflow File

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

# When to run
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# What to run
jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      # Check out the code
      - uses: actions/checkout@v4

      # Set up the runtime
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      # Install dependencies
      - run: npm ci

      # Run linting
      - run: npm run lint

      # Run tests
      - run: npm test

      # Run build
      - run: npm run build

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --audit-level=high
```

### Common Workflow Triggers

```yaml
on:
  # Push to specific branches
  push:
    branches: [main, develop]

  # Pull requests targeting specific branches
  pull_request:
    branches: [main]

  # Scheduled runs (cron syntax)
  schedule:
    - cron: '0 0 * * 1'   # Every Monday at midnight

  # Manual trigger
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deploy environment'
        required: true
        default: 'staging'

  # When a release is published
  release:
    types: [published]
```

<Callout type="tip" title="Start Simple with CI">
Your first GitHub Actions workflow should just run your tests on every PR. That alone catches most bugs before they reach main. Add linting, security scanning, and deployment workflows as your project matures. A 10-line workflow that runs `npm test` is better than no automation at all.
</Callout>

```bash
    def __init__(self):

        if not title or not title.strip():
            raise ValueError("Task title cannot be empty")
        if priority not in self.PRIORITIES:
            raise ValueError(f"Priority must be one of: {list(self.PRIORITIES.keys())}")
        task = {
            "title": title.strip(),
            "done": False,
            "priority": priority

        self.tasks.append(task)
        return len(self.tasks)

    def complete_task(self, task_number):
        if not isinstance(task_number, int):
            raise TypeError("Task number must be an integer")
        if task_number < 1 or task_number > len(self.tasks):
            raise IndexError(f"Task {task_number} does not exist (1-{len(self.tasks)})")
        self.tasks[task_number - 1]["done"] = True
        return True

        if not self.tasks:
            print("  No tasks yet.")

        if sort_by_priority:
```

## Key Takeaways

- GitHub adds collaboration features on top of Git: issues, PRs, project boards, Actions, and access controls
- Every repository should have README.md, LICENSE, CONTRIBUTING.md, .gitignore, and issue/PR templates
- Issues are the primary tracking mechanism — link commits with `Fixes #N` to auto-close issues when merged
- Branch protection rules prevent direct pushes to main and require PR reviews and passing CI checks
- CODEOWNERS automatically assigns reviewers based on which files a PR modifies
- GitHub Actions runs automated workflows (tests, linting, builds) on push and PR events
- Use labels, milestones, and project boards to organize work and visualize progress
- The GitHub CLI (`gh`) lets you create repos, issues, PRs, and manage workflows from the terminal
- Releases combine Git tags with release notes and downloadable assets for distribution
- Start with simple protection rules and CI — even basic test automation prevents most production issues
