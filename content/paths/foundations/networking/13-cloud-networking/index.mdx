## What Is a VPC?

A Virtual Private Cloud (VPC) is your isolated, private network within a cloud provider. Think of it as your own data center network, but virtual вҖ” defined entirely in software and managed through an API.

```
Traditional Data Center:
  Physical routers вҶ’ Physical switches вҶ’ Physical servers
  Cables, racks, and hardware you manage

VPC (Virtual Private Cloud):
  Virtual routers вҶ’ Virtual subnets вҶ’ Virtual machines
  All defined in code, provisioned in seconds

Every VPC is:
  - Isolated from other customers' VPCs
  - Contained within a region (e.g., us-east-1)
  - Defined by a CIDR block (your private IP range)
  - Fully controlled by you: subnets, routes, firewalls
```

## VPC CIDR Blocks

When you create a VPC, you assign it a CIDR block вҖ” the range of private IP addresses available within the VPC.

```bash
# Common VPC CIDR choices:

# /16 вҖ” Large VPC (65,536 IPs)
# Best for: Production environments with many services
10.0.0.0/16     # 10.0.0.0 - 10.0.255.255

# /20 вҖ” Medium VPC (4,096 IPs)
# Best for: Smaller environments, development
10.1.0.0/20     # 10.1.0.0 - 10.1.15.255

# /24 вҖ” Small VPC (256 IPs)
# Best for: Testing, single-purpose VPCs
10.2.0.0/24     # 10.2.0.0 - 10.2.0.255
```

<Callout type="warning" title="Plan CIDR Blocks Carefully">
VPC CIDR blocks cannot overlap if you plan to connect VPCs via peering or transit gateway. Plan your IP addressing scheme before creating VPCs.

```
BAD: Two VPCs that might need to connect
  Production VPC:  10.0.0.0/16
  Staging VPC:     10.0.0.0/16   вҶҗ Same range! Cannot peer.

GOOD: Non-overlapping ranges
  Production VPC:  10.0.0.0/16
  Staging VPC:     10.1.0.0/16
  Development VPC: 10.2.0.0/16
```
</Callout>

## Subnets: Public vs Private

Subnets divide your VPC into smaller networks, each in a specific Availability Zone (AZ). The critical distinction is public vs private.

```
VPC: 10.0.0.0/16
в”ңв”Җв”Җ Public Subnet (10.0.1.0/24) вҖ” AZ us-east-1a
в”Ӯ   в”ңв”Җв”Җ Has a route to the Internet Gateway
в”Ӯ   в”ңв”Җв”Җ Instances CAN have public IPs
в”Ӯ   в””в”Җв”Җ Used for: Load balancers, bastion hosts, NAT gateways
в”Ӯ
в”ңв”Җв”Җ Public Subnet (10.0.2.0/24) вҖ” AZ us-east-1b
в”Ӯ   в””в”Җв”Җ Same config, different AZ (for redundancy)
в”Ӯ
в”ңв”Җв”Җ Private Subnet (10.0.10.0/24) вҖ” AZ us-east-1a
в”Ӯ   в”ңв”Җв”Җ NO route to Internet Gateway
в”Ӯ   в”ңв”Җв”Җ Instances CANNOT have public IPs
в”Ӯ   в”ңв”Җв”Җ Outbound internet via NAT Gateway
в”Ӯ   в””в”Җв”Җ Used for: Application servers, workers
в”Ӯ
в”ңв”Җв”Җ Private Subnet (10.0.11.0/24) вҖ” AZ us-east-1b
в”Ӯ   в””в”Җв”Җ Same config, different AZ
в”Ӯ
в”ңв”Җв”Җ Data Subnet (10.0.20.0/24) вҖ” AZ us-east-1a
в”Ӯ   в”ңв”Җв”Җ NO internet access at all (no NAT)
в”Ӯ   в””в”Җв”Җ Used for: Databases, caches (RDS, ElastiCache)
в”Ӯ
в””в”Җв”Җ Data Subnet (10.0.21.0/24) вҖ” AZ us-east-1b
    в””в”Җв”Җ Same config, different AZ
```

### Subnet Sizing

```bash
# AWS reserves 5 IPs in every subnet:
#   .0   вҖ” Network address
#   .1   вҖ” VPC router
#   .2   вҖ” DNS server
#   .3   вҖ” Reserved for future use
#   .255 вҖ” Broadcast (not used in VPC, but reserved)

# So a /24 subnet gives you 251 usable IPs (256 - 5)
# A /20 subnet gives you 4,091 usable IPs (4,096 - 5)

# Sizing guidelines:
#   /24 (251 IPs)   вҖ” Small tier, limited services
#   /22 (1,019 IPs) вҖ” Medium tier, room to grow
#   /20 (4,091 IPs) вҖ” Large tier, Kubernetes nodes
```

## NAT Gateway

A NAT Gateway allows instances in private subnets to initiate outbound connections to the internet (for updates, API calls, etc.) while preventing inbound connections from the internet.

```
Private Instance needs to download packages:

  Private Instance (10.0.10.50)
       в”Ӯ
       в–ј
  Private Subnet Route Table:
    0.0.0.0/0 вҶ’ NAT Gateway
       в”Ӯ
       в–ј
  NAT Gateway (in Public Subnet)
    Translates: 10.0.10.50 вҶ’ NAT's public IP (54.x.x.x)
       в”Ӯ
       в–ј
  Internet Gateway
       в”Ӯ
       в–ј
  Internet (apt-get update, pip install, etc.)

Response follows the reverse path back to the private instance.
The internet only sees the NAT Gateway's public IP.
```

<Callout type="tip" title="NAT Gateway Costs">
NAT Gateways are expensive вҖ” they charge per hour AND per GB of data processed. For non-production environments, consider a NAT instance (a small EC2 instance running NAT) or VPC endpoints to avoid NAT costs entirely for AWS service traffic.
</Callout>

## Route Tables

Every subnet has a route table that determines where traffic goes. This is the mechanism that makes a subnet public or private.

```bash
# Public Subnet Route Table:
# Destination        Target
# 10.0.0.0/16       local          вҶҗ VPC internal traffic stays in VPC
# 0.0.0.0/0         igw-abc123     вҶҗ Everything else goes to Internet Gateway

# Private Subnet Route Table:
# Destination        Target
# 10.0.0.0/16       local          вҶҗ VPC internal traffic stays in VPC
# 0.0.0.0/0         nat-xyz789     вҶҗ Everything else goes through NAT Gateway

# Data Subnet Route Table (fully isolated):
# Destination        Target
# 10.0.0.0/16       local          вҶҗ Only VPC internal traffic
# (No default route вҖ” no internet access at all)
```

```bash
# AWS CLI examples:

# Create a route table
aws ec2 create-route-table --vpc-id vpc-abc123

# Add a route to the internet gateway (makes it a public route table)
aws ec2 create-route \
    --route-table-id rtb-public123 \
    --destination-cidr-block 0.0.0.0/0 \
    --gateway-id igw-abc123

# Add a route to the NAT gateway (makes it a private route table)
aws ec2 create-route \
    --route-table-id rtb-private456 \
    --destination-cidr-block 0.0.0.0/0 \
    --nat-gateway-id nat-xyz789

# Associate a route table with a subnet
aws ec2 associate-route-table \
    --route-table-id rtb-public123 \
    --subnet-id subnet-pub1
```

## Security Groups

Security groups are stateful firewalls attached to individual instances (EC2, RDS, etc.). They control inbound and outbound traffic at the instance level.

```bash
# Security Group properties:
#   - STATEFUL: If you allow inbound, the response is automatically allowed out
#   - Default DENY inbound: Must explicitly allow inbound traffic
#   - Default ALLOW outbound: All outbound traffic is allowed by default
#   - Applied to instances, not subnets
#   - Can reference other security groups (powerful for tier-based access)

# Web Server Security Group
# Inbound Rules:
#   Type      Protocol  Port    Source
#   HTTP      TCP       80      0.0.0.0/0         вҶҗ Anyone can access HTTP
#   HTTPS     TCP       443     0.0.0.0/0         вҶҗ Anyone can access HTTPS
#   SSH       TCP       22      10.0.1.0/24       вҶҗ SSH only from public subnet

# Outbound Rules:
#   Type      Protocol  Port    Destination
#   All       All       All     0.0.0.0/0         вҶҗ Can reach anything (default)

# Application Server Security Group
# Inbound Rules:
#   Type      Protocol  Port    Source
#   Custom    TCP       8080    sg-webserver       вҶҗ Only from web server SG
#   SSH       TCP       22      sg-bastion         вҶҗ Only from bastion SG

# Database Security Group
# Inbound Rules:
#   Type      Protocol  Port    Source
#   PostgreSQL TCP      5432    sg-appserver       вҶҗ Only from app server SG
#   PostgreSQL TCP      5432    sg-bastion         вҶҗ Only from bastion for admin
```

<Callout type="tip" title="Security Group Best Practice: Reference Other SGs">
Instead of using IP addresses in security group rules, reference other security groups. This way, when you scale (add more app servers), they automatically get access to the database because they share the same security group.

```
BAD:  Allow 10.0.10.50 on port 5432 (breaks when you add more servers)
GOOD: Allow sg-appserver on port 5432 (works for any server in that SG)
```
</Callout>

## Network ACLs (NACLs)

NACLs are stateless firewalls attached to subnets. They provide an additional layer of security on top of security groups.

```bash
# NACL properties:
#   - STATELESS: Must explicitly allow both inbound AND outbound
#   - Rules are numbered and evaluated in order (lowest number first)
#   - First matching rule wins (unlike security groups where all rules apply)
#   - Applied to subnets (affects all instances in the subnet)
#   - Default NACL allows all traffic

# Example NACL for a public subnet:
# Inbound Rules:
#   Rule#  Type      Protocol  Port Range  Source        Action
#   100    HTTP      TCP       80          0.0.0.0/0    ALLOW
#   110    HTTPS     TCP       443         0.0.0.0/0    ALLOW
#   120    SSH       TCP       22          10.0.0.0/16  ALLOW
#   130    Ephemeral TCP       1024-65535  0.0.0.0/0    ALLOW  вҶҗ Return traffic!
#   *      All       All       All         0.0.0.0/0    DENY   вҶҗ Default deny

# Outbound Rules:
#   Rule#  Type      Protocol  Port Range  Destination   Action
#   100    HTTP      TCP       80          0.0.0.0/0    ALLOW
#   110    HTTPS     TCP       443         0.0.0.0/0    ALLOW
#   120    Ephemeral TCP       1024-65535  0.0.0.0/0    ALLOW  вҶҗ Return traffic!
#   *      All       All       All         0.0.0.0/0    DENY
```

### Security Groups vs NACLs

```
Feature              Security Group           NACL
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
Stateful?            Yes                      No
Applied to?          Instance                 Subnet
Default inbound      Deny                     Allow (default NACL)
Default outbound     Allow                    Allow (default NACL)
Rule evaluation      All rules evaluated      First match wins
Allow/Deny rules     Allow only               Allow and Deny
Use case             Primary firewall         Subnet-level guard

Recommendation: Use security groups as your primary firewall.
Use NACLs only for broad subnet-level blocking (e.g., block a known
malicious IP range from an entire subnet).
```

## VPC Peering

VPC peering connects two VPCs so instances in each can communicate using private IPs, as if they were in the same network.

```
VPC A (10.0.0.0/16) вҶҗв”Җв”Җ Peering Connection в”Җв”ҖвҶ’ VPC B (10.1.0.0/16)

Instance in VPC A (10.0.1.50) can reach Instance in VPC B (10.1.2.30)
directly via private IP вҖ” no internet involved.

Requirements:
  - CIDR blocks must not overlap
  - Both VPCs must accept the peering request
  - Route tables in both VPCs must be updated
  - Security groups must allow the traffic

Limitations:
  - Not transitive: If A peers with B and B peers with C,
    A CANNOT reach C through B
  - Maximum 125 peering connections per VPC
```

```bash
# Route table entry for VPC peering:
# In VPC A's route table:
#   Destination    Target
#   10.1.0.0/16   pcx-abc123    вҶҗ Traffic to VPC B goes through peering

# In VPC B's route table:
#   Destination    Target
#   10.0.0.0/16   pcx-abc123    вҶҗ Traffic to VPC A goes through peering
```

## VPC Endpoints

VPC endpoints let you access AWS services (S3, DynamoDB, etc.) without going through the internet or NAT Gateway. This saves money and improves security.

```
Without VPC Endpoint:
  Private Instance вҶ’ NAT Gateway вҶ’ Internet Gateway вҶ’ S3
  (Costs money for NAT, data goes over the internet)

With VPC Endpoint:
  Private Instance вҶ’ VPC Endpoint вҶ’ S3
  (Free for Gateway endpoints, stays on AWS network)

Two types:
  Gateway Endpoint:  Free, for S3 and DynamoDB only
                     Adds a route to your route table
  Interface Endpoint: Costs money, for 100+ other AWS services
                     Creates an ENI in your subnet
```

```bash
# Create a Gateway VPC endpoint for S3
aws ec2 create-vpc-endpoint \
    --vpc-id vpc-abc123 \
    --service-name com.amazonaws.us-east-1.s3 \
    --route-table-ids rtb-private456

# This adds a route like:
#   Destination             Target
#   pl-63a5400a (S3 prefix) vpce-s3endpoint
```

## Transit Gateway

When you have many VPCs that need to communicate, peering connections become unmanageable. A Transit Gateway acts as a central hub.

```
Without Transit Gateway (mesh of peering connections):
  VPC A в”Җв”Җ peer в”Җв”Җ VPC B
  VPC A в”Җв”Җ peer в”Җв”Җ VPC C
  VPC B в”Җв”Җ peer в”Җв”Җ VPC C
  VPC A в”Җв”Җ peer в”Җв”Җ VPC D
  ...
  N VPCs = N*(N-1)/2 peering connections = nightmare

With Transit Gateway (hub and spoke):
  VPC A в”Җв”Җв”җ
  VPC B в”Җв”Җв”Ө
  VPC C в”Җв”Җв”јв”Җв”Җ Transit Gateway в”Җв”Җ On-Premises (VPN)
  VPC D в”Җв”Җв”Ө
  VPC E в”Җв”Җв”ҳ

  Each VPC connects once to the Transit Gateway.
  Routing is centralized and manageable.
```

### Multi-AZ for High Availability

```
Why multi-AZ?
  - Each AZ is a physically separate data center
  - If AZ-1a goes down, AZ-1b keeps running
  - Load balancers distribute across AZs automatically
  - RDS multi-AZ failover is automatic

Minimum for production:
  - 2 AZs (3 for critical workloads)
  - Subnets in each AZ for each tier
  - Load balancer spanning all AZs
  - Database replication across AZs
```

## Key Takeaways

- A VPC is your isolated virtual network in the cloud вҖ” all physical networking concepts apply, just virtualized
- Public subnets have a route to the Internet Gateway; private subnets route through a NAT Gateway
- Security groups are stateful firewalls on instances; NACLs are stateless firewalls on subnets
- Reference security groups (not IPs) in rules so access scales automatically with your fleet
- NAT Gateways are expensive вҖ” use VPC endpoints for AWS service traffic to save money
- Plan CIDR blocks carefully вҖ” they cannot overlap if VPCs need to peer
- Three-tier architecture across multiple AZs is the standard production design
- Everything in cloud networking can be defined as code with Terraform or CloudFormation
