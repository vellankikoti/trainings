## Storage Hierarchy

```
Physical Disk (/dev/sda)
  └── Partition Table (GPT or MBR)
       ├── Partition 1 (/dev/sda1) → Filesystem (ext4) → Mounted at /boot
       ├── Partition 2 (/dev/sda2) → Filesystem (ext4) → Mounted at /
       └── Partition 3 (/dev/sda3) → Swap
```

## Viewing Disks and Partitions

```bash
# List all block devices
lsblk
# NAME   MAJ:MIN RM   SIZE RO TYPE MOUNTPOINT
# sda      8:0    0   100G  0 disk
# ├─sda1   8:1    0     1G  0 part /boot
# ├─sda2   8:2    0    97G  0 part /
# └─sda3   8:3    0     2G  0 part [SWAP]

# Detailed disk info
sudo fdisk -l /dev/sda

# Filesystem info
df -Th
# Shows filesystem type (ext4, xfs, etc.) with -T

# View UUIDs (used in fstab)
sudo blkid
```

## Filesystems

| Filesystem | Use Case | Notes |
|---|---|---|
| ext4 | Default for most Linux | Mature, reliable, supports up to 1EB |
| xfs | RHEL default, large files | Better for large files, can't shrink |
| btrfs | Snapshots, compression | Copy-on-write, good for backups |
| tmpfs | RAM-based temp storage | `/tmp`, `/run` — lost on reboot |
| swap | Virtual memory | Used when RAM is full |

### Creating a Filesystem

```bash
# Format a partition as ext4
sudo mkfs.ext4 /dev/sdb1

# Format as XFS
sudo mkfs.xfs /dev/sdb1

# Format with a label
sudo mkfs.ext4 -L "data-vol" /dev/sdb1

# Check filesystem integrity
sudo fsck /dev/sdb1        # ext4
sudo xfs_repair /dev/sdb1  # XFS (must be unmounted)
```

## Mounting Filesystems

### Manual Mount

```bash
# Create a mount point
sudo mkdir -p /mnt/data

# Mount a partition
sudo mount /dev/sdb1 /mnt/data

# Mount with specific options
sudo mount -o rw,noexec,nosuid /dev/sdb1 /mnt/data

# Mount a read-only filesystem
sudo mount -o ro /dev/sdb1 /mnt/data

# Verify the mount
mount | grep sdb1
df -h /mnt/data

# Unmount
sudo umount /mnt/data
# or by device:
sudo umount /dev/sdb1
```

### Common Mount Options

| Option | Meaning |
|---|---|
| `rw` / `ro` | Read-write / Read-only |
| `noexec` | Can't execute binaries (security) |
| `nosuid` | Ignore SUID/SGID bits (security) |
| `nodev` | Ignore device files (security) |
| `noatime` | Don't update access time (performance) |
| `defaults` | rw, suid, dev, exec, auto, nouser, async |

### Permanent Mounts with `/etc/fstab`

```bash
# View current fstab
cat /etc/fstab

# Format:
# <device>              <mount point>  <type>  <options>        <dump>  <pass>
# UUID=abc-123-def      /mnt/data      ext4    defaults,noatime 0       2
# /dev/sdb1             /mnt/backup    xfs     defaults         0       0
```

| Field | Meaning |
|---|---|
| device | UUID, device path, or LABEL |
| mount point | Where to mount |
| type | Filesystem type (ext4, xfs, etc.) |
| options | Mount options |
| dump | Backup flag (0 = no backup) |
| pass | fsck order (0 = skip, 1 = root, 2 = others) |

```bash
# Add a permanent mount
echo "UUID=$(sudo blkid -s UUID -o value /dev/sdb1) /mnt/data ext4 defaults,noatime 0 2" | sudo tee -a /etc/fstab

# Test fstab without rebooting
sudo mount -a

# ALWAYS test before rebooting — a bad fstab can prevent boot!
sudo findmnt --verify
```

<Callout type="warning" title="fstab Safety">
A mistake in `/etc/fstab` can prevent your system from booting. Always use `UUID=` instead of device paths (device names can change), and always test with `mount -a` before rebooting.
</Callout>

## LVM — Logical Volume Manager

LVM adds a flexible layer between physical disks and filesystems. You can resize volumes, span multiple disks, and create snapshots.

```
Physical Volumes (PVs): /dev/sdb, /dev/sdc
         ↓
Volume Group (VG): vg-data (combines PVs into a pool)
         ↓
Logical Volumes (LVs): lv-app (50GB), lv-db (100GB), lv-logs (20GB)
         ↓
Filesystems: ext4 on each LV, mounted at /opt/app, /var/lib/db, /var/log
```

### LVM Commands

```bash
# View physical volumes
sudo pvs
sudo pvdisplay

# View volume groups
sudo vgs
sudo vgdisplay

# View logical volumes
sudo lvs
sudo lvdisplay

# Create LVM setup (example with /dev/sdb)
# 1. Create physical volume
sudo pvcreate /dev/sdb

# 2. Create volume group
sudo vgcreate vg-data /dev/sdb

# 3. Create logical volumes
sudo lvcreate -L 50G -n lv-app vg-data
sudo lvcreate -L 100G -n lv-db vg-data
sudo lvcreate -l 100%FREE -n lv-logs vg-data    # use remaining space

# 4. Create filesystems
sudo mkfs.ext4 /dev/vg-data/lv-app
sudo mkfs.ext4 /dev/vg-data/lv-db

# 5. Mount
sudo mkdir -p /opt/app /var/lib/db
sudo mount /dev/vg-data/lv-app /opt/app
sudo mount /dev/vg-data/lv-db /var/lib/db
```

### Extending Volumes (The Killer Feature)

```bash
# Extend a logical volume by 20GB
sudo lvextend -L +20G /dev/vg-data/lv-db

# Resize the filesystem to use the new space
sudo resize2fs /dev/vg-data/lv-db     # ext4
# or
sudo xfs_growfs /dev/vg-data/lv-db    # XFS

# Do both in one command
sudo lvextend -L +20G --resizefs /dev/vg-data/lv-db

# Add a new disk to an existing volume group
sudo pvcreate /dev/sdc
sudo vgextend vg-data /dev/sdc
# Now you have more space to allocate!
```

## Key Takeaways

- `lsblk` shows disk layout; `df -Th` shows filesystem usage with types
- Always use `UUID=` in `/etc/fstab`, not device paths (they can change)
- Test fstab changes with `mount -a` before rebooting
- LVM allows live-resizing of volumes — essential for production servers
- Use `noexec,nosuid,nodev` mount options for security on data partitions
- Monitor disk usage — 90%+ needs immediate attention
- Check inodes too (`df -i`) — you can run out before running out of space
