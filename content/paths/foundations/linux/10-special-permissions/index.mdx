## SUID — Set User ID

When SUID is set on an executable, it runs with the permissions of the **file owner**, not the user who runs it.

```bash
# The classic example: passwd
ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root 68208 ... /usr/bin/passwd
#    ^
#    s instead of x = SUID is set
```

Why does `passwd` need SUID? Regular users need to change their own password, but the password hash is stored in `/etc/shadow` which only root can write. SUID lets `passwd` run as root temporarily.

### Setting and Removing SUID

```bash
# Set SUID (numeric: add 4000)
chmod u+s program
chmod 4755 program

# Remove SUID
chmod u-s program

# Verify
ls -l program
# -rwsr-xr-x  = SUID set, owner has execute
# -rwSr-xr-x  = SUID set, owner does NOT have execute (capital S = warning)
```

<Callout type="warning" title="Security Warning">
SUID on root-owned executables is a major attack vector. If an attacker can replace a SUID binary or find a vulnerability in one, they get root access. Never set SUID on scripts — only compiled binaries. Audit SUID files regularly.
</Callout>

### Finding All SUID Files

```bash
# Find all SUID files on the system
find / -perm -4000 -type f 2>/dev/null

# Common legitimate SUID binaries:
# /usr/bin/passwd     — change password
# /usr/bin/sudo       — run as root
# /usr/bin/su         — switch user
# /usr/bin/ping       — raw socket access (older systems)
# /usr/bin/mount      — mount filesystems
# /usr/bin/umount     — unmount filesystems
```

## SGID — Set Group ID

SGID works differently depending on whether it's on a file or directory.

### SGID on Files

The file runs with the permissions of the **group owner**:

```bash
ls -l /usr/bin/wall
# -rwxr-sr-x 1 root tty 15344 ... /usr/bin/wall
#         ^
#         s in group execute position = SGID
```

### SGID on Directories (More Common)

When SGID is set on a directory, new files and subdirectories inherit the **group of the directory**, not the group of the user who created them. This is essential for shared directories:

```bash
# Without SGID: files get the creator's default group
mkdir /opt/project
touch /opt/project/file1.txt
ls -l /opt/project/file1.txt
# -rw-r--r-- 1 ubuntu ubuntu ... file1.txt  (ubuntu's group)

# With SGID: files inherit the directory's group
sudo mkdir /opt/shared
sudo chown root:developers /opt/shared
sudo chmod 2775 /opt/shared
#           ^
#           2 = SGID bit

touch /opt/shared/file2.txt
ls -l /opt/shared/file2.txt
# -rw-rw-r-- 1 ubuntu developers ... file2.txt  (inherited group!)
```

### Setting SGID

```bash
# Set SGID (numeric: add 2000)
chmod g+s directory/
chmod 2775 directory/

# Remove SGID
chmod g-s directory/

# Verify
ls -ld directory/
# drwxrwsr-x  = SGID set on directory
```

## Sticky Bit

The sticky bit on a directory means only the file owner (or root) can delete or rename files inside it, even if others have write permission. The classic example is `/tmp`:

```bash
ls -ld /tmp
# drwxrwxrwt 15 root root 4096 ... /tmp
#          ^
#          t = sticky bit set
```

Without sticky bit: anyone with write permission to `/tmp` could delete anyone else's files. With sticky bit: you can only delete your own files.

### Setting Sticky Bit

```bash
# Set sticky bit (numeric: add 1000)
chmod +t directory/
chmod 1777 directory/

# Remove sticky bit
chmod -t directory/

# Verify
ls -ld directory/
# drwxrwxrwt  = sticky bit set (t in others execute position)
# drwxrwxrwT  = sticky bit set, but others don't have execute (T = warning)
```

### Practical Use: Shared Upload Directory

```bash
# Create a shared uploads directory
sudo mkdir /var/uploads
sudo chown root:www-data /var/uploads
sudo chmod 1773 /var/uploads
#           ^^^^
#           1 = sticky bit (users can only delete own files)
#           7 = owner rwx
#           7 = group rwx
#           3 = others wx (can create files but not list)
```

## Special Permissions Summary

| Permission | Numeric | On Files | On Directories |
|---|---|---|---|
| SUID | 4000 | Runs as file owner | (no effect) |
| SGID | 2000 | Runs as file group | New files inherit directory group |
| Sticky | 1000 | (no effect) | Only owner can delete their files |

The four-digit numeric permission: `4755` = SUID + rwxr-xr-x

## Access Control Lists (ACLs)

Standard permissions only allow one owner and one group. ACLs let you set permissions for specific users and groups beyond that:

```bash
# Check if filesystem supports ACLs
mount | grep acl

# View ACLs on a file
getfacl /var/log/syslog

# Set ACL: give alice read access to a file
setfacl -m u:alice:r /opt/app/config.yaml

# Set ACL: give developers group read+write
setfacl -m g:developers:rw /opt/app/config.yaml

# Set default ACL on directory (inherited by new files)
setfacl -d -m g:developers:rw /opt/app/

# Remove a specific ACL entry
setfacl -x u:alice /opt/app/config.yaml

# Remove all ACLs
setfacl -b /opt/app/config.yaml
```

### ACL Indicator

When a file has ACLs, `ls -l` shows a `+` at the end of the permission string:

```bash
ls -l /opt/app/config.yaml
# -rw-rw-r--+ 1 root root 1024 ... config.yaml
#           ^
#           + means ACLs are set
```

### Reading ACL Output

```bash
getfacl /opt/app/config.yaml
# # file: opt/app/config.yaml
# # owner: root
# # group: root
# user::rw-           ← owner permissions
# user:alice:r--       ← specific user ACL
# group::r--          ← owning group permissions
# group:developers:rw- ← specific group ACL
# mask::rw-           ← maximum effective permissions
# other::r--          ← everyone else
```

### ACL Use Cases

```bash
# Scenario: Multiple teams need different access to the same directory
sudo mkdir -p /opt/shared-app

# DevOps team: full access
sudo setfacl -m g:devops:rwx /opt/shared-app

# QA team: read + execute only
sudo setfacl -m g:qa:rx /opt/shared-app

# Specific developer: write access
sudo setfacl -m u:john:rwx /opt/shared-app

# Make these defaults for new files
sudo setfacl -d -m g:devops:rwx /opt/shared-app
sudo setfacl -d -m g:qa:rx /opt/shared-app

getfacl /opt/shared-app
```

## Key Takeaways

- SUID (`4000`): file runs as the file's owner — security-critical, audit regularly
- SGID (`2000`): on directories, new files inherit the directory's group — essential for shared workflows
- Sticky bit (`1000`): prevents users from deleting others' files in shared directories (like `/tmp`)
- ACLs provide fine-grained permissions beyond the user/group/other model
- Use `find -perm -4000` to find SUID files — review any unfamiliar ones
- World-writable files and directories without sticky bit are security risks
- `getfacl` and `setfacl` manage ACLs; a `+` in `ls -l` indicates ACLs are present
