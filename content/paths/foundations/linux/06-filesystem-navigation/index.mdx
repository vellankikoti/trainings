## Where Am I Right Now? -- `pwd`

Let me start with the simplest command in Linux, because it solves the most fundamental question: where am I?

`pwd` stands for **print working directory**. It tells you your exact location in the filesystem, every single time, with zero ambiguity.

```bash
pwd
# /home/ubuntu

cd /etc/nginx
pwd
# /etc/nginx
```

Here's the thing -- your shell prompt usually shows your current directory, but it often abbreviates it. You might see `~` instead of `/home/ubuntu`, or just the last folder name. When you're SSH'd into a server at 2 AM troubleshooting something, and you've been jumping between directories for twenty minutes, `pwd` is the command that grounds you. It's your "You Are Here" pin on the map.

<Callout type="tip" title="When pwd saves you from yourself">
I've seen engineers accidentally edit the wrong config file because they thought they were in `/etc/nginx` when they were actually in `/etc/apache2`. A quick `pwd` before editing anything on a production server is a habit that will save you someday. Make it automatic.
</Callout>

## Seeing What's Around You -- `ls`

If `pwd` tells you where you are, `ls` tells you what's here. You'll type this command more than almost any other in your career. Let's make sure you really know it.

### The Basics

```bash
# Plain ls — just the names
ls

# Long format — the one you'll actually use
ls -l

# Show hidden files (files starting with a dot)
ls -a

# The combo you'll type a thousand times: all files, long format, human-readable sizes
ls -lah
```

Let me be honest: you're going to live in `ls -lah` mode. The `-l` gives you details, the `-a` shows hidden files (and in Linux, a surprising amount of important stuff is hidden), and the `-h` makes file sizes readable -- `4.2K` instead of `4296`.

### Sorting -- Because Order Matters

```bash
# Sort by modification time — newest first
ls -lt

# Sort by modification time — oldest first (reverse)
ls -ltr

# Sort by size — largest first
ls -lS

# Sort by size — smallest first
ls -lSr
```

Here's a real scenario where sorting matters: you're investigating that disk space alert. You SSH into the server and need to find the biggest files in `/var/log`:

```bash
ls -lSh /var/log
```

The largest files appear at the top. In two seconds, you can see that `syslog` is 2.3 GB and that's your culprit. Without the sort flag, you'd be scrolling through dozens of files trying to eyeball which one is biggest.

### Reading `ls -l` Output Like a Pro

This is one of those things that looks intimidating the first time you see it, but once someone walks you through it, you'll read it as naturally as reading a sentence. Let me break down every piece:

```bash
ls -l /etc/passwd
# -rw-r--r-- 1 root root 2847 Jan 15 10:32 /etc/passwd
```

Here's what each part means, left to right:

```
-  rw-  r--  r--  1  root  root  2847  Jan 15 10:32  /etc/passwd
|  |    |    |    |  |     |     |     |              |
|  |    |    |    |  |     |     |     |              filename
|  |    |    |    |  |     |     |     last modified timestamp
|  |    |    |    |  |     |     size in bytes
|  |    |    |    |  |     group owner
|  |    |    |    |  file owner
|  |    |    |    number of hard links
|  |    |    permissions for everyone else (other)
|  |    permissions for the group
|  permissions for the owner
file type: - = file, d = directory, l = symlink
```

Let me translate that `/etc/passwd` line into plain English: "This is a regular file. The owner (root) can read and write it. The group (root) can only read it. Everyone else can only read it. It's 2,847 bytes. It was last touched on January 15th at 10:32 AM."

<Callout type="info" title="Why this matters for DevOps">
When a web server can't read a config file, or an application can't write to its log directory, the answer is almost always in the `ls -l` output. Wrong owner. Wrong group. Wrong permissions. You'll learn permissions deeply in a later lesson, but start reading `ls -l` output now so it becomes second nature.
</Callout>

Let's look at a directory listing so you can see how different file types look:

```bash
ls -la /etc/nginx/
# drwxr-xr-x 8 root root 4096 Mar  1 09:15 .
# drwxr-xr-x 92 root root 4096 Feb 28 14:22 ..
# -rw-r--r-- 1 root root 1077 Feb 28 14:22 fastcgi.conf
# -rw-r--r-- 1 root root 1482 Feb 28 14:22 nginx.conf
# lrwxrwxrwx 1 root root   22 Feb 28 14:22 modules -> /usr/lib/nginx/modules
# drwxr-xr-x 2 root root 4096 Mar  1 09:15 sites-enabled
```

See how the first character tells you the type? `d` for directory, `-` for regular file, `l` for symlink. That `modules -> /usr/lib/nginx/modules` is a symbolic link pointing somewhere else. Once you can read this at a glance, you have X-ray vision into any directory on any server.

## Moving Around -- `cd`

`cd` is how you move. It stands for **change directory**, and there's more to it than you might expect.

```bash
# Go to an absolute path
cd /var/log

# Go to your home directory — three ways, same result
cd ~
cd $HOME
cd          # cd with no arguments always takes you home

# Go to the previous directory you were in (like the "back" button)
cd -

# Go up one directory
cd ..

# Go up two directories
cd ../..

# Go into a subdirectory (relative to where you are)
cd nginx/
```

That `cd -` trick is a gem. Let me show you why. Say you're bouncing between a config directory and a log directory while debugging:

```bash
cd /etc/nginx           # look at the config
cd /var/log/nginx       # check the logs
cd -                    # back to /etc/nginx
cd -                    # back to /var/log/nginx
```

It's like a toggle switch between two locations. When you're troubleshooting and constantly flipping between config and logs, this saves a surprising amount of time and typing.

## Absolute vs. Relative Paths -- The GPS Analogy

This is one of those concepts that clicks instantly once you have the right mental model. Here's how I think about it.

**Absolute paths** are like a full street address: "742 Evergreen Terrace, Springfield, USA." No matter where you are in the world, that address takes you to the same place. In Linux, absolute paths start with `/` (the root of the filesystem):

```bash
/var/log/nginx/error.log
/etc/ssh/sshd_config
/home/ubuntu/.bashrc
```

**Relative paths** are like giving directions from where you're standing: "Go two blocks north, turn left." The destination depends on where you start. In Linux, relative paths don't start with `/`:

```bash
# If you're in /var/log:
nginx/error.log           # means /var/log/nginx/error.log
../spool/cron             # means /var/spool/cron (up one, then into spool/cron)
../../etc/hosts           # means /etc/hosts (up two, then into etc)
```

**Home-relative paths** use `~` as a shortcut for your home directory:

```bash
~/scripts/backup.sh       # means /home/ubuntu/scripts/backup.sh
~/.ssh/config             # means /home/ubuntu/.ssh/config
```

Here's a quick reference:

| Type | Starts with | Example | What it means |
|---|---|---|---|
| Absolute | `/` | `/var/log/nginx/error.log` | Full path from the root -- always the same |
| Relative | anything else | `../log/nginx/error.log` | From wherever you are right now |
| Home-relative | `~` | `~/scripts/deploy.sh` | From your home directory |

<Callout type="warning" title="When to use which">
In scripts, **always use absolute paths**. A script might run from any directory, and a relative path in a script is a bug waiting to happen. I've seen deployment scripts that worked fine when run from `/home/deploy` but broke catastrophically when a cron job ran them from `/`. At the command line, use whatever's fastest -- usually relative paths for nearby files, absolute for anything far away.
</Callout>

## Finding Files -- The `find` Command

Here's where things get powerful. `find` is the Swiss Army knife of file searching. It walks through directories in real time, checking every file against whatever criteria you give it.

The basic pattern is: `find WHERE WHAT-TO-LOOK-FOR`

### Find by Name

```bash
# Find a file by exact name
find /etc -name "nginx.conf"

# Case-insensitive search (when you're not sure about capitalization)
find /home -iname "readme*"

# Find all YAML config files
find /etc -name "*.yaml" -o -name "*.yml"
```

### Find by Type

```bash
# Only files (not directories)
find /var/log -type f -name "*.log"

# Only directories
find /etc -type d -name "nginx"

# Only symbolic links
find /etc -type l
```

### Find by Size -- The Disk Space Saver

Remember that disk space alert from our opening story? Here's how you actually solve it:

```bash
# Files larger than 100MB
find /var/log -type f -size +100M

# Files larger than 1GB (the real culprits)
find / -type f -size +1G 2>/dev/null

# Files smaller than 1KB (often empty or stub files)
find /tmp -type f -size -1k

# Files between 10MB and 100MB
find /var -type f -size +10M -size -100M
```

<Callout type="tip" title="That 2>/dev/null trick">
When you search from `/` (the root), you'll hit directories you don't have permission to read, and `find` will complain with "Permission denied" over and over. Adding `2>/dev/null` redirects those error messages to the void so you only see actual results. You'll use this pattern all the time.
</Callout>

### Find by Time -- The Change Detective

This is where `find` becomes invaluable for incident response. When someone says "something changed on the server," time-based search is how you figure out what.

```bash
# Files modified in the last 24 hours
find /etc -type f -mtime -1

# Files modified in the last 30 minutes
find /etc -type f -mmin -30

# Files modified more than 7 days ago
find /tmp -type f -mtime +7

# Files accessed in the last hour
find /var/log -type f -amin -60
```

Let me explain the time units because they trip people up:
- `-mtime` counts in **days**. `-mtime -1` means "less than 1 day ago" (last 24 hours)
- `-mmin` counts in **minutes**. `-mmin -30` means "less than 30 minutes ago"
- The `m` in `mtime` is for **modification** time. There's also `atime` (access) and `ctime` (status change)

### Find + Action -- Do Something With the Results

Finding files is great. Finding files AND doing something with them in one command? That's where real productivity lives.

```bash
# Find large log files and show their sizes
find /var/log -type f -size +50M -exec ls -lh {} \;

# Find old temp files and delete them
find /tmp -type f -mtime +30 -exec rm {} \;

# Find config files and search inside them for a specific string
find /etc -name "*.conf" -exec grep -l "listen 80" {} \;

# Find recently changed files and copy them to a backup
find /etc -type f -mmin -60 -exec cp {} /tmp/recent-changes/ \;
```

That `-exec` syntax looks weird, I know. Here's the pattern: `-exec COMMAND {} \;` The `{}` is a placeholder for each file that `find` discovers, and `\;` marks the end of the command. Think of it as saying "for each file you find, run this command on it."

<Callout type="info" title="The + optimization">
If you're running `-exec` on thousands of files, using `+` instead of `\;` at the end is much faster. `\;` runs the command once per file. `+` batches files together like `xargs`:

```bash
# Slow: runs ls once per file
find /var/log -name "*.log" -exec ls -lh {} \;

# Fast: runs ls once with all files as arguments
find /var/log -name "*.log" -exec ls -lh {} +
```
</Callout>

### Real DevOps `find` Patterns

These are commands I've used on real production servers. Bookmark these:

```bash
# DISK SPACE: Find the biggest files on the system
find / -type f -size +100M -exec ls -lh {} + 2>/dev/null | sort -k5 -rh

# SECURITY: Find config files changed in the last day (possible unauthorized changes)
find /etc -type f -mtime -1 -ls

# SECURITY: Find world-writable files (anyone can modify them)
find / -type f -perm -o+w 2>/dev/null | head -20

# SECURITY: Find SUID files (run with elevated permissions)
find / -type f -perm -4000 2>/dev/null

# CLEANUP: Find files owned by a user who no longer exists
find /home -nouser -type f 2>/dev/null

# CLEANUP: Find empty files and directories
find /var -empty -type f

# DEBUGGING: Find broken symbolic links
find /etc -xtype l

# DEBUGGING: Find all files owned by the nginx user
find / -user nginx -type f 2>/dev/null | head -20
```

## Quick Lookups -- `locate`

`find` is powerful, but it's also slow on large filesystems because it checks every single file in real time. When you just need to quickly find where something lives, `locate` is your speed demon.

`locate` works by searching a pre-built database instead of walking the filesystem:

```bash
# Find any file with "nginx" in the path
locate nginx.conf

# Case-insensitive search
locate -i readme

# Count matches instead of listing them
locate -c "*.service"

# Show only files that actually still exist (database might be stale)
locate -e nginx.conf
```

There's one important catch:

```bash
# Update the database (must be root)
sudo updatedb
```

The `locate` database is typically updated once a day by a cron job. So if you just created a file five minutes ago, `locate` won't know about it until you run `sudo updatedb` or wait for the daily update. For real-time searches, use `find`. For "I know this file exists somewhere, where is it?" searches, use `locate`.

<Callout type="info" title="Installing locate">
On some minimal server installations, `locate` isn't installed by default. If you get "command not found," install it:

```bash
# Debian/Ubuntu
sudo apt install mlocate

# RHEL/CentOS/Fedora
sudo dnf install mlocate

# Then build the initial database
sudo updatedb
```
</Callout>

## Finding Commands -- `which` and `type`

Sometimes the file you're looking for isn't a config or a log -- it's a command itself. "Which version of Python is running?" "Where does `node` actually live?" "Is `ls` an alias or a real binary?"

### `which` -- Where Does This Command Live?

```bash
which python3
# /usr/bin/python3

which nginx
# /usr/sbin/nginx

# Show ALL matches in your PATH (not just the first)
which -a python
# /usr/bin/python
# /usr/local/bin/python
```

That `-a` flag is important. If you have multiple versions of something installed, `which -a` shows all of them. The first one in the list is the one that runs when you type the command.

### `type` -- The Full Story

`type` goes deeper than `which`. It tells you not just where something lives, but what it actually is:

```bash
type cd
# cd is a shell builtin

type ls
# ls is aliased to 'ls --color=auto'

type python3
# python3 is /usr/bin/python3

type mkcd
# mkcd is a function
```

This matters more than you'd think. I've seen teams spend thirty minutes debugging why a command "wasn't working" only to discover that someone had created an alias that was overriding the real command. `type` reveals the truth.

<Callout type="tip" title="The DevOps detective combo">
When an application says it can't find a binary, or when the wrong version of something is running, this is your investigation sequence:

```bash
# Where is it?
which python3

# What exactly is it?
type python3

# What version?
python3 --version

# Is there a symlink involved?
ls -la $(which python3)

# Follow the symlink chain all the way
readlink -f $(which python3)
```

This sequence has solved "it works on my machine" problems more times than I can count.
</Callout>

## Seeing the Big Picture -- `tree`

Sometimes you need to see a directory structure at a glance. `ls` shows you one level. `tree` shows you everything.

```bash
# Install tree (it's not always there by default)
sudo apt install tree    # Debian/Ubuntu
sudo yum install tree    # RHEL/CentOS
```

```bash
# Show the structure of a directory
tree /etc/nginx
# /etc/nginx
# ├── conf.d
# │   └── default.conf
# ├── fastcgi.conf
# ├── mime.types
# ├── modules -> /usr/lib/nginx/modules
# ├── nginx.conf
# ├── sites-available
# │   └── default
# └── sites-enabled
#     └── default -> /etc/nginx/sites-available/default

# Limit the depth (essential for large directories)
tree -L 2 /etc

# Show only directories (great for understanding structure)
tree -d -L 3 /var

# Show file sizes
tree -sh /home/ubuntu/project

# Show hidden files too
tree -a -L 1 ~/
```

<Callout type="warning" title="Be careful with tree">
Running `tree` on a large directory without the `-L` depth limit can produce thousands of lines of output. Always use `-L 2` or `-L 3` when you're exploring unfamiliar territory. On production servers, `tree -L 2 /` gives you a beautiful high-level map without drowning you in detail.
</Callout>

When I onboard onto a new project or server, one of the first things I run is:

```bash
tree -L 2 -d /opt
tree -L 2 /etc/nginx
tree -L 1 /var/log
```

In thirty seconds, I have a mental map of how the application is laid out, how the web server is configured, and what log files exist. That mental map pays for itself every time I need to find something later.

<MiniProject title="Server Investigation Script">

Let's bring everything together. In the real world, when you first SSH into an unfamiliar server, you need to quickly understand what's on it. This script automates that reconnaissance:

```bash
cat > ~/server-recon.sh << 'SCRIPT'
#!/bin/bash
# Server Reconnaissance Script
# Run this when you SSH into an unfamiliar server for the first time

echo "============================================"
echo "  SERVER RECON REPORT"
echo "  Generated: $(date)"
echo "  Hostname:  $(hostname)"
echo "============================================"
echo ""

echo "--- Current Location ---"
pwd
echo ""

echo "--- Disk Usage Overview ---"
df -h / /var /tmp 2>/dev/null | column -t
echo ""

echo "--- Top 10 Largest Files on System ---"
find / -type f -size +50M -exec ls -lh {} + 2>/dev/null | sort -k5 -rh | head -10
echo ""

echo "--- Recently Modified Config Files (last 24h) ---"
find /etc -type f -mtime -1 2>/dev/null | head -15
echo "(showing max 15 results)"
echo ""

echo "--- Log Directory Sizes ---"
du -sh /var/log/*/ 2>/dev/null | sort -rh | head -10
echo ""

echo "--- Web Server Check ---"
if which nginx > /dev/null 2>&1; then
    echo "Nginx: $(nginx -v 2>&1)"
    echo "Config: $(find /etc/nginx -name '*.conf' | head -5)"
elif which apache2 > /dev/null 2>&1; then
    echo "Apache: $(apache2 -v 2>&1 | head -1)"
else
    echo "No web server detected"
fi
echo ""

echo "--- Key Directories ---"
for dir in /opt /srv /var/www /etc/nginx /etc/apache2; do
    if [ -d "$dir" ]; then
        count=$(find "$dir" -maxdepth 1 -not -path "$dir" | wc -l)
        echo "  $dir ($count items)"
    fi
done
echo ""

echo "--- Running Services Listening on Ports ---"
ss -tlnp 2>/dev/null | head -10 || netstat -tlnp 2>/dev/null | head -10
echo ""

echo "--- File Type Summary in /etc ---"
find /etc -type f 2>/dev/null | sed 's/.*\.//' | sort | uniq -c | sort -rn | head -10
echo ""

echo "============================================"
echo "  RECON COMPLETE"
echo "============================================"
SCRIPT

chmod +x ~/server-recon.sh

# Run it
~/server-recon.sh
```
