## Setting Up WSL 2 — Your Linux Inside Windows

This is my top recommendation for Windows users. WSL 2 runs a real Linux kernel inside Windows. Not a simulation. Not an emulator. A real Linux system, fully integrated with your Windows machine.

### Before We Start

You need:
- Windows 10 (version 2004 or later) or Windows 11
- Admin access to your computer
- About 10 minutes

### Let's Do This

**Step 1: Open PowerShell as Administrator**

Right-click the Start button → "Windows Terminal (Admin)" or "PowerShell (Admin)"

You should see a blue terminal window. If it asks "Do you want to allow this app to make changes?" — say Yes.

**Step 2: Install WSL**

Type this one command and press Enter:

```powershell
wsl --install
```

That's it. One command. It will:
- Enable the WSL feature
- Download the Linux kernel
- Install Ubuntu (the default distro)

You'll see progress bars. It takes 2-5 minutes depending on your internet.

**Step 3: Restart Your Computer**

When it says it's done, restart. I know, restarting is annoying. Do it anyway.

**Step 4: Set Up Your Linux User**

After restart, a terminal window will open automatically (or open "Ubuntu" from the Start menu). It'll ask you to create a username and password.

A few things:
- **Pick a simple username** — all lowercase, no spaces. Something like `ravi` or `sarah`.
- **The password won't show as you type** — this freaks everyone out the first time. It's a security feature. Just type your password and press Enter. You can't see it, but it IS being recorded.
- **Remember this password** — you'll need it whenever you use `sudo` (which is a lot).

```bash
# After setting username and password, you'll see something like:
ravi@DESKTOP-ABC123:~$

# That blinking cursor? That's your Linux terminal. You're in.
```

**Step 5: Update Everything**

Your first real Linux command as a system administrator:

```bash
sudo apt update && sudo apt upgrade -y
```

`sudo` means "run this as the administrator." `apt` is Ubuntu's package manager. You just told your system "check for updates and install them all." This is something you'll do on every server you ever manage.

**Step 6: Install the Tools We'll Need**

```bash
sudo apt install -y curl wget git vim htop tree net-tools
```

This installs a toolkit of essentials. Don't worry about what each one does yet — we'll cover every single one in upcoming lessons.

<Callout type="tip" title="Bonus: Accessing files between Windows and Linux">
Your Windows C: drive is available inside Linux at `/mnt/c/`. Your Linux files are available from Windows at `\\wsl$\Ubuntu\`. This means you can edit files in VS Code on Windows and run them in Linux. Best of both worlds.
</Callout>

## Setting Up a VirtualBox VM — Full Isolation

VirtualBox gives you a completely separate Linux machine running inside your current operating system. It's a bit more setup, but it gives you the clearest picture of "this is a separate computer running Linux."

### What You'll Need

- VirtualBox installed ([virtualbox.org](https://www.virtualbox.org/))
- Ubuntu Server 22.04 ISO ([ubuntu.com/download/server](https://ubuntu.com/download/server))
- About 20 minutes

### Step by Step

**1. Create the VM**

Open VirtualBox → Click "New" → Fill in:
- **Name:** `linux-lab`
- **Type:** Linux
- **Version:** Ubuntu (64-bit)
- **Memory:** 2048 MB (2 GB minimum, 4 GB is better if you can spare it)
- **Hard disk:** Create new, VDI format, dynamically allocated, 20 GB

Click "Create."

**2. Mount the Ubuntu ISO**

Select your VM → Settings → Storage → Click the empty disk icon → Choose the Ubuntu ISO you downloaded.

**3. Start and Install**

Click "Start." The Ubuntu installer will boot. Follow along:

- **Language:** English
- **Install:** Ubuntu Server
- **Network:** Accept the defaults (it'll auto-detect)
- **Storage:** "Use entire disk" (it's a virtual disk, nothing to worry about)
- **Your info:** Pick a name, a server name (`linux-lab`), a username, and a password
- **SSH server:** Yes, install it. We'll need this.
- **Featured snaps:** Skip all of them

Let it install. Takes 5-10 minutes. Reboot when it asks.

**4. Log In and Update**

After reboot, you'll see a login prompt. Enter your username and password.

```bash
# First things first:
sudo apt update && sudo apt upgrade -y

# Install essentials:
sudo apt install -y curl wget git vim htop tree net-tools
```

**5. Set Up SSH Access (optional but recommended)**

This lets you connect from your normal terminal instead of the VirtualBox window:

```bash
# Inside the VM, find your IP:
ip addr show | grep "inet " | grep -v 127.0.0.1

# From your host machine:
ssh your-username@that-ip-address
```

<Callout type="info" title="When to use VirtualBox vs. WSL">
VirtualBox gives you a completely isolated environment — it's great for practicing things like network configuration or disk management where you want to experiment without any risk. WSL is better for daily use because it's faster and more integrated. Both are real Linux.
</Callout>

## Spinning Up a Cloud Server — The Real Deal

This option gives you the experience closest to what you'll actually do on the job. You're going to rent a server in a data center somewhere and connect to it over the internet. This is literally how production infrastructure works.

### Option A: AWS Free Tier

```bash
# 1. Go to aws.amazon.com and create an account
#    (requires credit card, but t2.micro is FREE for 12 months)
#
# 2. Go to EC2 → "Launch Instance"
#    - Name: "linux-lab"
#    - AMI: Ubuntu Server 22.04 LTS
#    - Instance type: t2.micro (Free tier eligible)
#    - Key pair: Create new → download the .pem file → keep it safe
#    - Security group: Allow SSH (port 22) from your IP
#    - Click "Launch Instance"
#
# 3. Connect from your terminal:

# First, make the key file secure (Linux/Mac):
chmod 400 ~/Downloads/your-key.pem

# Then connect:
ssh -i ~/Downloads/your-key.pem ubuntu@your-public-ip-address
```

### Option B: DigitalOcean (Simpler)

```bash
# 1. Go to digitalocean.com (new accounts get $200 free credit)
#
# 2. Create → Droplets
#    - Ubuntu 22.04
#    - Basic plan → $6/month
#    - Choose a region close to you
#    - Authentication: SSH key (recommended) or Password
#
# 3. Connect:
ssh root@your-droplet-ip
```

### After You Connect

No matter which cloud provider you used:

```bash
# Update the system:
sudo apt update && sudo apt upgrade -y

# Install essentials:
sudo apt install -y curl wget git vim htop tree net-tools

# Check your server's specs:
echo "CPU cores: $(nproc)"
echo "Memory: $(free -h | grep Mem | awk '{print $2}')"
echo "Disk: $(df -h / | tail -1 | awk '{print $2}')"
```

<Callout type="warning" title="Seriously — set a billing alert">
I've mentored people who forgot about a running instance and got a surprise bill. Go to your cloud provider's billing page RIGHT NOW and set an alert for $5. It takes 30 seconds and saves you from a bad day. Also — stop or terminate your instance when you're done practicing. You can always create a new one.
</Callout>

## macOS — You're Already Closer Than You Think

Here's a nice surprise: macOS is built on Unix (BSD), which is Linux's close cousin. About 90% of the Linux commands we'll learn in this course work natively in your Mac's Terminal app. No installation needed.

### Getting Started

```bash
# Open Terminal: Applications → Utilities → Terminal
# Or the fast way: Cmd + Space, type "Terminal", press Enter

# Verify you're in business:
whoami
pwd
ls
uname -a
```

That's it. You're ready for most of this course.

### A Few Differences to Know About

| Linux | macOS | Notes |
|---|---|---|
| `apt install` | `brew install` | macOS uses Homebrew as its package manager |
| `free -h` | `vm_stat` or `top -l 1` | Different memory commands |
| `ss -tlnp` | `lsof -i -P` | Different network commands |
| GNU coreutils | BSD coreutils | Some flags differ (e.g., `sed -i` needs `''` on Mac) |

### Install Homebrew (macOS Package Manager)

If you don't have it yet:

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

Then install some tools that match what Linux has:

```bash
brew install coreutils gnu-sed gawk wget htop tree
```

### When You'll Need Real Linux

For Docker and Kubernetes lessons later in this course, you'll need actual Linux. When we get there, I'll show you how to set up:

```bash
# Lima — lightweight Linux VMs for Mac
brew install lima
limactl start default
limactl shell default
# Now you're in a real Ubuntu VM
```

But for now, your Mac Terminal is perfect. Let's keep moving.

## Verification — Let's Make Sure Everything Works

Regardless of which path you chose, run these commands. Every single one should work:

<Exercise number={1} title="The Verification Checklist">

Run each command. If any of them fails, check the troubleshooting section below.

```bash
# 1. Are you logged in?
whoami
# Should print your username

# 2. Where are you?
pwd
# Should print something like /home/yourusername

# 3. What system is this?
cat /etc/os-release | grep PRETTY_NAME
# Should print your distro name and version

# 4. Is the internet working?
ping -c 3 google.com
# Should show 3 successful pings

# 5. Does the package manager work?
sudo apt update 2>/dev/null && echo "Package manager: OK" || echo "Not apt-based (that's fine on macOS)"

# 6. Is disk space available?
df -h /
# Should show available space

# 7. Is memory available?
free -h 2>/dev/null || echo "Memory: $(sysctl hw.memsize 2>/dev/null | awk '{print $2/1073741824 " GB"}')"
```

Every command worked? You're set. If something failed, read on.

## When Things Go Wrong (Troubleshooting)

If you hit a wall during installation, don't panic. Here's a fix for every common problem:

<Callout type="danger" title="Common Issues and Fixes">

**"WSL --install does nothing" or "WSL won't start"**
1. Make sure virtualization is enabled in your BIOS. Restart → press F2/Del during boot → find "VT-x" or "AMD-V" → enable it
2. Run in PowerShell (Admin): `dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart`
3. Restart and try again

**"VirtualBox VM won't start — error about VT-x"**
- Same BIOS fix as above — enable virtualization
- On Windows: Disable Hyper-V first: `bcdedit /set hypervisorlaunchtype off` then restart

**"Permission denied" when running commands**
- Prefix the command with `sudo`: `sudo apt update`
- If sudo itself fails, you might not have admin privileges. On cloud: check you're using the right user (usually `ubuntu` for AWS, `root` for DigitalOcean)

**"Connection refused" when trying to SSH**
```bash
# On the server, check if SSH is running:
sudo systemctl status ssh

# If it's not installed:
sudo apt install openssh-server
sudo systemctl start ssh
sudo systemctl enable ssh

# Check firewall isn't blocking it:
sudo ufw allow ssh
```

**"apt update" gives errors**
```bash
# Check internet connectivity first:
ping -c 3 google.com

# If ping works but apt doesn't, DNS might be off:
echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf

# Then retry:
sudo apt update
```

</Callout>

Still stuck? That's OK. Debugging installation issues is actually great practice for the job. Try searching the exact error message you see — that's what experienced engineers do too.
