## The Five Stages: From Power Button to Login Prompt

When a Linux machine boots, five stages happen in sequence. Every single time. No exceptions. Understanding this chain is like understanding the order of operations in math — once you know it, everything makes sense.

```
Power Button
    │
    ▼
┌──────────────┐
│  1. BIOS/UEFI │  "Is the hardware even working?"
└──────┬───────┘
       ▼
┌──────────────┐
│  2. GRUB      │  "Where's the kernel? Let me load it."
└──────┬───────┘
       ▼
┌──────────────┐
│  3. Kernel    │  "I'm in charge now. Let me set up hardware."
└──────┬───────┘
       ▼
┌──────────────┐
│  4. systemd   │  "Time to start all the services."
└──────┬───────┘
       ▼
┌──────────────┐
│  5. Login     │  "Ready. Who are you?"
└──────┘
```

Let me walk you through each one. And I promise, by the end, this sequence will feel as natural as breathing.

### Stage 1: BIOS/UEFI — "Is Anyone Home?"

The very first thing that runs when you press the power button isn't Linux. It isn't even software you installed. It's **firmware** — code that lives on the motherboard itself.

There are two flavors:

- **BIOS** (Basic Input/Output System) — The legacy system. It's been around since the 1980s. Simple, limited, but it works.
- **UEFI** (Unified Extensible Firmware Interface) — The modern replacement. Faster boot times, supports Secure Boot, handles larger disks.

Here's what the firmware does, in plain English:

1. **POST** (Power-On Self-Test) — It checks: "Do I have a CPU? Is the RAM working? Are the disks connected?" If any of this fails, you'll hear beep codes or see nothing at all.
2. **Find a boot device** — It looks through its boot order (hard drive, USB, network) and finds the first one with a bootloader.

<Callout type="tip" title="Real-world context">
On cloud servers (AWS, GCP, Azure), you never see the BIOS/UEFI stage — the hypervisor handles it. But on physical servers in a data center? Understanding POST failures and boot device order is how you diagnose hardware problems. If you ever work with bare metal infrastructure, this knowledge becomes critical.
</Callout>

**DevOps relevance:** If a server fails at this stage, it's a hardware problem. No amount of SSH will help you. You're calling the data center team or opening a cloud provider support ticket.

### Stage 2: Bootloader (GRUB) — "Let Me Find the Kernel"

Once the firmware finds a boot device, it hands control to the **bootloader**. On Linux, that's almost always **GRUB** (GRand Unified Bootloader).

You've probably seen GRUB without realizing it — it's that menu that sometimes flashes by when a machine starts:

```
                    GNU GRUB  version 2.06

 ┌────────────────────────────────────────────────────┐
 │ Ubuntu                                             │
 │ Ubuntu, with Linux 5.15.0-89-generic               │
 │ Ubuntu, with Linux 5.15.0-89-generic (recovery)    │
 │ Advanced options for Ubuntu                        │
 └────────────────────────────────────────────────────┘
```

GRUB's job is simple but critical:

1. Show you a menu of available kernels (if there are multiple)
2. Load the selected kernel from disk into memory
3. Pass boot parameters to the kernel (like which filesystem to mount as root)

GRUB's configuration lives at `/boot/grub/grub.cfg`. You generally don't edit this file directly — instead, you modify files in `/etc/default/grub` and run `update-grub` to regenerate it.

<Callout type="warning" title="Don't mess with GRUB unless you mean it">
A bad GRUB configuration can make your server completely unbootable. On a remote server, that means you're locked out until someone physically accesses the machine or you use a cloud provider's recovery console. I've seen engineers take down production servers by casually running `update-grub` after a bad edit. Always double-check `/etc/default/grub` before regenerating.
</Callout>

**DevOps relevance:** After kernel updates, GRUB gets a new entry. If a new kernel has issues, you can select the previous kernel from the GRUB menu to recover. This is your "undo button" for bad kernel updates.

### Stage 3: Kernel — "I'm Running the Show Now"

This is where Linux truly takes over. GRUB loaded the kernel into memory, and now it springs to life. Here's what happens:

1. **Decompresses itself** — The kernel is stored compressed. It unpacks into memory.
2. **Initializes hardware** — CPU configuration, memory controller, device drivers. All the hardware the system needs.
3. **Mounts the root filesystem** — Using the root device specified by GRUB (usually something like `root=/dev/sda1` or `root=UUID=...`).
4. **Starts PID 1** — The very first user-space process. The ancestor of literally everything else.

You can see what the kernel was told to do and what it found during boot:

```bash
# See the kernel command line — what GRUB told the kernel
cat /proc/cmdline

# See kernel boot messages — everything the kernel discovered
dmesg | head -40

# Find the specific kernel version that booted
dmesg | grep -i "linux version"
```

Here's the thing about the kernel that I want you to really internalize: **the kernel is the bridge between hardware and everything else.** Every process, every file operation, every network packet — it all goes through the kernel. When it boots successfully, the foundation is solid. When it doesn't? That's a kernel panic, and it's one of the scariest messages you'll ever see on a server.

**DevOps relevance:** Kernel boot messages (`dmesg`) are incredibly useful for diagnosing hardware issues, driver problems, and storage failures. When a server is acting weird after a reboot, `dmesg` is one of the first places to look.

### Stage 4: Init System (systemd) — "Let's Start Everything"

Here's where it gets really practical for your daily work. The kernel starts one process — just one — and that process is responsible for starting *everything else*. On modern Linux, that process is **systemd**, and it runs as **PID 1**.

Let me show you why PID 1 is such a big deal:

```bash
# Verify: who is PID 1?
ps -p 1 -o pid,comm,args
```

systemd is the parent of all processes. Think of it as the conductor of an orchestra — it reads its score (configuration files), knows what instruments (services) need to play, knows what order they need to start in, and kicks everything off.

Here's what systemd does during boot:

1. **Reads the default target** — Which is like a blueprint for "what kind of system should this be?"
2. **Resolves dependencies** — Figures out the right order to start services
3. **Starts services in parallel** — This is why modern Linux boots so fast compared to the old days
4. **Monitors everything** — If a service crashes, systemd can restart it automatically

#### Boot Targets (The "What Kind of System" Question)

systemd uses **targets** to define what the system should look like when it's done booting. Think of targets as presets:

| Target | What It Means | When You'd Use It |
|---|---|---|
| `poweroff.target` | Shut down the system | System is turning off |
| `rescue.target` | Single-user mode, minimal services | Emergency repairs |
| `multi-user.target` | Full system, no GUI | **Servers (this is what you'll see most)** |
| `graphical.target` | Full system with desktop environment | Desktop machines |
| `reboot.target` | Reboot the system | System is restarting |

<Callout type="info" title="This is important for your career">
Servers almost always boot to `multi-user.target`. There's no desktop environment on a production server — why would there be? Nobody is sitting at a monitor clicking icons. Everything is managed through SSH and the command line. When someone says "the server booted to runlevel 3," they're using the old terminology for `multi-user.target`.
</Callout>

```bash
# What target does your system boot to?
systemctl get-default

# Want to change it? (careful on remote servers!)
# sudo systemctl set-default multi-user.target
```

### Stage 5: Login — "Who Goes There?"

Once systemd has started all the services defined by the target, the last piece falls into place: a login prompt.

On a server, this is usually a text-based prompt served by `getty` (the program that manages virtual terminals). On a desktop, it's a graphical login screen like GDM or LightDM.

For you as a DevOps engineer, by the time you see a login prompt (either on a console or via SSH), all five stages have completed successfully. The system is up. If you *don't* see a login prompt, something in one of the first four stages broke — and now you know exactly which stages to investigate.

## Working with systemd — Your Daily Driver

Let me show you the systemd commands you'll actually use in your career. I'm not going to dump a huge reference table on you. Instead, let me walk you through the commands in the order you'd naturally use them.

### "How did the boot go?"

```bash
# How long did the entire boot take?
systemd-analyze

# Output looks like:
# Startup finished in 3.456s (kernel) + 12.789s (userspace) = 16.245s

# What took the longest? (blame game, literally)
systemd-analyze blame | head -10

# Output shows you things like:
#  8.123s NetworkManager-wait-online.service
#  2.456s snapd.service
#  1.234s dev-sda1.device
```

That `systemd-analyze blame` command is gold. I've used it countless times to figure out why a server takes forever to boot. Nine times out of ten, it's a network service waiting for a timeout.

### "What's running right now?"

```bash
# List all active services
systemctl list-units --type=service --state=running

# Check a specific service
systemctl status sshd

# Is SSH enabled to start on boot?
systemctl is-enabled sshd

# What services are set to start on boot?
systemctl list-unit-files --state=enabled | head -20
```

### "What depends on what?"

```bash
# Show the dependency tree — what does multi-user.target need?
systemctl list-dependencies multi-user.target | head -25

# What does a specific service need before it can start?
systemctl list-dependencies sshd
```

## Troubleshooting Boot Failures — The Real Stuff

Alright, this is the section that will actually save you at 2 AM. Let me walk you through how boot failures really happen and how you diagnose them.

### The Five Usual Suspects

I've seen a lot of boot failures over the years. They almost always fall into one of these categories:

<Callout type="danger" title="The most common boot killers">

**1. Full disk — The silent assassin**
The disk fills up (usually `/var/log` growing out of control), and suddenly services can't write temp files during boot. The system hangs or starts and immediately becomes unstable.

**2. Bad `/etc/fstab` entry — The typo that locks you out**
Someone adds a mount point to `/etc/fstab` with a typo, or references a disk that no longer exists. At boot, systemd tries to mount it, fails, and drops you into emergency mode.

**3. Failed critical service — The dependency chain collapse**
A service that other services depend on fails to start. This cascades — everything that depends on it also fails. Suddenly half your system is down.

**4. Kernel panic — The worst-case scenario**
A bad kernel update, corrupt filesystem, or hardware failure causes the kernel itself to crash during initialization. You see "Kernel panic - not syncing" and the system is completely frozen.

**5. Network timeout — The slow death**
A service is configured to wait for a network resource (DHCP, NFS mount, DNS) that never responds. The boot process just... hangs. For minutes. Sometimes forever.

</Callout>

### Your Diagnostic Toolkit

Here's the exact workflow I use when a server won't boot or boots in a degraded state:

```bash
# STEP 1: Are any services in a failed state?
systemctl --failed

# STEP 2: What happened during the last boot?
# Show only errors and above
journalctl -b -p err --no-pager

# STEP 3: Did the disk fill up?
df -h

# STEP 4: Is /etc/fstab sane?
cat /etc/fstab

# STEP 5: If the current boot is fine, but the LAST boot failed:
# Look at the previous boot's logs
journalctl -b -1 -p err --no-pager

# STEP 6: Check kernel messages for hardware issues
dmesg | grep -iE "error|fail|panic" | tail -20

# STEP 7: How long is boot taking? Is something hanging?
systemd-analyze blame | head -15
```

<Callout type="tip" title="The journalctl trick that saves time">
The `-b` flag means "this boot." The `-b -1` flag means "the previous boot." This is incredibly powerful — if a server rebooted unexpectedly and came back up fine, you can still investigate what went wrong during the failed boot. The `-p err` flag filters to only show errors, which cuts through the noise fast.
</Callout>

### What To Do When You Can't Even Log In

Sometimes the boot failure is so bad you can't get a normal login. Here's what you do:

<TabGroup>
<Tab label="Cloud Server (AWS, GCP, etc.)">

**Use the provider's serial console:**
1. In AWS: EC2 Console -> Instance -> Actions -> Monitor & Troubleshoot -> EC2 Serial Console
2. In GCP: Compute Engine -> VM Instances -> Serial Console
3. In Azure: Virtual Machine -> Serial Console

The serial console shows you exactly where the boot process is stuck. You can also:
- Detach the disk and mount it on another instance
- Use the recovery/rescue boot option
- Check boot logs through the provider's console logging

</Tab>
<Tab label="Physical Server / VM">

**Boot into rescue mode:**
1. Restart the server and hold `Shift` (BIOS) or press `Escape` (UEFI) to get to the GRUB menu
2. Select "Advanced options" -> "(recovery mode)"
3. This boots with minimal services — just enough to fix things

**From rescue mode, common fixes:**
```bash
# Fix a bad fstab entry
nano /etc/fstab
# Comment out the bad line with #

# Fix a full disk
du -sh /var/log/*
rm /var/log/old-massive-logfile.log

# Rebuild GRUB if needed
update-grub

# Check and repair filesystem
fsck /dev/sda1
```

</Tab>
</TabGroup>

## Real DevOps Scenarios — Where This Knowledge Saves You

Let me share some real scenarios where boot knowledge made the difference. These are the kinds of situations you'll face in your career.

<Callout type="story" title="Scenario 1: The midnight fstab disaster">
A junior engineer added a new NFS mount to `/etc/fstab` on a production server. The entry was correct... except the NFS server was behind a firewall rule that hadn't been applied yet. Next time the server rebooted (during a maintenance window), it hung for 5 minutes trying to mount the NFS share, then dropped into emergency mode. No SSH. No services. The entire application was down.

**The fix:** Booted into rescue mode through the cloud console, commented out the NFS line in `/etc/fstab`, rebooted, and the server came back in 20 seconds. Then fixed the firewall rule and re-enabled the mount properly.

**The lesson:** Always use the `nofail` option in fstab for non-critical mounts. It tells systemd "if this mount fails, keep booting anyway."
</Callout>

<Callout type="story" title="Scenario 2: The kernel update that went wrong">
After a routine `apt upgrade` that included a kernel update, a server rebooted into the new kernel and immediately kernel-panicked. Turns out the new kernel didn't have the right driver for the server's RAID controller.

**The fix:** Used GRUB to boot the previous kernel (it was still in the GRUB menu), then pinned that kernel version to prevent it from being upgraded until a patched version was available.

**The lesson:** This is why GRUB keeps old kernel entries. They're your safety net. Never clean up all old kernels — keep at least one previous version.
</Callout>

<Callout type="story" title="Scenario 3: The slow boot mystery">
A server that normally booted in 15 seconds started taking over 3 minutes. Nothing appeared to be broken — just slow. The team lived with it for weeks until it started causing issues with their auto-scaling group (new instances took too long to join the load balancer).

**The fix:** Ran `systemd-analyze blame` and found that `NetworkManager-wait-online.service` was taking 120 seconds. A DNS server in the network config had gone offline, and this service was waiting for the full timeout before giving up. Updated the DNS config, boot time dropped back to 15 seconds.

**The lesson:** `systemd-analyze blame` is the first thing you run when boot is slow. It tells you exactly where the time is being spent.
</Callout>

<MiniProject title="Boot Diagnostics Script">

Let's build a proper diagnostic script that you can keep in your toolkit. This is the kind of thing you'd save on your laptop and use whenever you need to quickly assess a server's boot health.

```bash
#!/bin/bash
# boot-diagnostics.sh — Quick boot health assessment
# Usage: bash boot-diagnostics.sh

RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

echo "========================================"
echo "  BOOT DIAGNOSTICS REPORT"
echo "  Generated: $(date)"
echo "  Hostname:  $(hostname)"
echo "========================================"
echo ""

# --- System Identity ---
echo "--- System Info ---"
echo "  Kernel:     $(uname -r)"
echo "  Init:       $(ps -p 1 -o comm=)"
echo "  Target:     $(systemctl get-default 2>/dev/null || echo 'N/A')"
echo "  Uptime:     $(uptime -p 2>/dev/null || uptime)"
echo ""

# --- Boot Timing ---
echo "--- Boot Timing ---"
BOOT_TIME=$(systemd-analyze 2>/dev/null | head -1)
if [ -n "$BOOT_TIME" ]; then
    echo "  $BOOT_TIME"
    echo ""
    echo "  Slowest services:"
    systemd-analyze blame 2>/dev/null | head -5 | while read line; do
        echo "    $line"
    done
else
    echo "  Could not analyze boot time"
fi
echo ""

# --- Failed Services ---
echo "--- Service Health ---"
FAILED_COUNT=$(systemctl --failed --no-legend 2>/dev/null | wc -l)
if [ "$FAILED_COUNT" -eq 0 ]; then
    echo -e "  ${GREEN}All services healthy${NC}"
else
    echo -e "  ${RED}$FAILED_COUNT failed service(s):${NC}"
    systemctl --failed --no-legend 2>/dev/null | while read line; do
        echo "    $line"
    done
fi
echo ""

# --- Disk Space ---
echo "--- Disk Space (boot killers) ---"
df -h / /boot /var 2>/dev/null | awk 'NR==1 || NR>1 {printf "  %-20s %5s used (%s)\n", $6, $5, $3}'
echo ""

# --- Recent Boot Errors ---
echo "--- Errors from Current Boot ---"
ERROR_COUNT=$(journalctl -b -p err --no-pager 2>/dev/null | grep -v "^--" | wc -l)
if [ "$ERROR_COUNT" -le 1 ]; then
    echo -e "  ${GREEN}No errors in current boot${NC}"
else
    echo -e "  ${YELLOW}$ERROR_COUNT error entries found. Top ones:${NC}"
    journalctl -b -p err --no-pager 2>/dev/null | grep -v "^--" | tail -5 | while read line; do
        echo "    $line"
    done
fi
echo ""

# --- fstab Check ---
echo "--- Filesystem Table (/etc/fstab) ---"
FSTAB_ENTRIES=$(grep -v "^#\|^$" /etc/fstab 2>/dev/null | wc -l)
echo "  $FSTAB_ENTRIES active mount entries"
# Check for nofail
NOFAIL_MISSING=$(grep -v "^#\|^$\|swap\|/boot\| / " /etc/fstab 2>/dev/null | grep -v "nofail" | wc -l)
if [ "$NOFAIL_MISSING" -gt 0 ]; then
    echo -e "  ${YELLOW}$NOFAIL_MISSING non-root entries without 'nofail' option${NC}"
    echo "  (Consider adding 'nofail' to prevent boot hangs)"
fi
echo ""

echo "========================================"
echo "  Diagnostics complete."
echo "========================================"
```

Save this as `boot-diagnostics.sh` and run it:

```bash
chmod +x boot-diagnostics.sh
bash boot-diagnostics.sh
```

This script checks everything we've talked about — boot timing, failed services, disk space, boot errors, and fstab safety. Keep it in your dotfiles or on a USB drive. You'll use it more often than you think.
